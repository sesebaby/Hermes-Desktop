# 微软 BitNet 与 bitnet.cpp 资料核查总结

日期：2026-04-09  
用途：把本轮关于 `BitNet / bitnet.cpp` 的公开资料核查结果落成一份能直接看的中文总结。  
范围：只总结微软官方仓库、微软研究论文、arXiv 论文里能查到的内容；不把二手营销文案当事实。

## 1. 结论先说

微软的 `bitnet.cpp` 这件事，核心方向是真的，但网上很多说法确实吹大了。

可以确认的点：

- 微软确实做了 `bitnet.cpp`
- 它确实是给 `BitNet b1.58` 这类三值权重模型做高效推理
- 官方确实宣称在 CPU 上比常规全精度基线快很多，能耗也低很多
- 官方也确实写过：在研究配置里，`100B` 级别的 `BitNet b1.58` 推理可以在单 CPU 上跑到“接近人类阅读速度”

但不能直接照着营销文案说的点：

- 不是“2026 年 3 月突然首发的重磅炸弹”
- 不是“随便一台普通旧电脑都能轻松 5-7 token/s 跑 100B”
- 不是“官方已经公开放出一个人人都能直接下载的 100B 微软模型”
- 更不是“GPU 被颠覆，以后推理都不用 GPU 了”

一句大白话：

`bitnet.cpp` 真正厉害的地方，是它把“CPU 跑 1.58-bit / 三值模型”这件事从论文概念推进到了可实测、可开源、可对比的阶段；但它还没有夸张到“几年前随便一台电脑都能稳稳本地飞起 100B”。

## 2. 官方时间线

按微软官方仓库 `README` 里能直接看到的时间线：

- `2023-10-17`：`BitNet: Scaling 1-bit Transformers for Large Language Models`
- `2024-02-27`：`The Era of 1-bit LLMs: All Large Language Models are in 1.58 Bits`
- `2024-10-17`：`bitnet.cpp 1.0 released`
- `2024-10-21`：微软 CPU 推理论文 `1-bit AI Infra: Part 1.1, Fast and Lossless BitNet b1.58 Inference on CPUs`
- `2025-02-18`：论文 `Bitnet.cpp: Efficient Edge Inference for Ternary LLMs`
- `2025-04-14`：微软官方 `2B` 参数模型上 Hugging Face
- `2025-05-20`：官方 GPU inference kernel
- `2026-01-15`：BitNet CPU Inference Optimization

所以，“`2026 年 3 月微软扔出 BitNet.cpp`”这个说法不准确。  
更准确的说法是：

- `bitnet.cpp` 在 `2024 年 10 月` 就已经正式发布
- `2025 年` 又有后续论文和官方模型
- `2026 年 1 月` 有一轮 CPU 优化更新

## 3. BitNet b1.58 到底是什么

按论文说法，`BitNet b1.58` 的核心特征是：

- Transformer 层权重只允许取 `-1 / 0 / +1`
- 这 3 个值对应的信息量大约是 `1.58 bits per weight`

这件事的直接好处有两个：

- 模型权重占用能大幅下降
- 计算路径可以围绕低比特、三值权重专门优化

这里还要补一句，不然后面很容易看混：

- `1.58 bits per weight` 更接近“理论信息量 / 背景概念”
- `bitnet.cpp` 实际 kernel 的存储格式并不一定都正好是 `1.58 bpw`
- 论文里不同 kernel 还有 `1.67 bpw`、`2 bpw` 这类实现

但这里要注意一个关键边界：

`bitnet.cpp` 的“lossless inference”说的是：  
**在论文定义的 BitNet b1.58 推理语境下，对原生按 `BitNet b1.58` 训练方案来的模型，如果推理实现严格对齐训练约束，就可以做到无损推理。**

它不是说：

- 任意 FP16 / FP32 大模型随手压成 `1.58 bit` 都能无损
- 任意现有模型都能白嫖成 `BitNet`
- 任意 ternary LLM 都能自动套用同样的“lossless”结论

## 4. 官方到底宣称了什么性能

### 4.1 2024 年微软 CPU 推理论文

微软在 `2024-10-21` 的论文里给出的结论是：

- x86 CPU：速度提升 `2.37x` 到 `6.17x`
- ARM CPU：速度提升 `1.37x` 到 `5.07x`
- x86 平台能耗下降 `71.9%` 到 `82.2%`
- ARM 平台能耗下降 `55.4%` 到 `70.0%`

这部分是有官方论文依据的，不是纯营销文案。

### 4.2 100B 单 CPU 跑 5-7 token/s 这句话有没有出处

有出处。

微软在仓库 README 和论文里都写过类似表述：

- `bitnet.cpp can run a 100B BitNet b1.58 model on a single CPU`
- `achieving speeds comparable to human reading (5-7 tokens per second)`

但这里最容易被二手文案偷换概念。

因为论文里真正列出来的测试机器，不是“随便一台普通旧电脑”，而是：

- `Apple M2 Ultra + 64GB`
- `Intel i7-13700H + 64GB`

在 `2025` 论文的表格里，`100B` 大致是：

- `Apple M2 Ultra`：最高大约 `7.45 tok/s`
- `Intel i7-13700H`：最高大约 `1.65-1.69 tok/s`

还要注意，论文附录里也写了：这些被测的大模型尺寸很多属于 **research dummy setups / 研究配置**，是为了展示推理性能和 kernel 表现，不等于微软已经公开发布了一个成熟可下载的官方 `100B BitNet` 模型。

所以正确理解应该是：

- **高带宽、高规格 CPU 平台上，100B ternary 模型确实可能接近 5-7 tok/s**
- **普通消费级 x86 电脑并不能默认拿到这个数字**

## 5. 为什么很多营销文案会说得太夸张

### 5.1 把“单 CPU 能跑”偷换成“普通旧电脑轻松跑”

微软论文的测试对象本身就不普通。  
尤其 `Apple M2 Ultra` 的统一内存带宽非常高，论文里还专门说了：  
`bitnet.cpp` 的性能很受带宽影响。

论文甚至明确提到：

- `Apple M2 Ultra` 带宽 `800 GB/s+`
- `Intel i7-13700H` 带宽不到 `100 GB/s`

这也是为什么同样是单 CPU，`M2 Ultra` 和 `i7-13700H` 的 100B 表现差距很大。

### 5.2 把“支持 100B dummy layout / 研究配置”偷换成“官方开放 100B 成熟模型”

微软仓库目前能直接看到的 **微软官方发布模型**，主线是：

- `BitNet-b1.58-2B-4T`

README 里另外列出的 **支持演示 / 支持推理的第三方或社区模型** 还有：

- `0.7B`
- `3.3B`
- `8B`
- `Falcon3 1B-10B`

也就是说：

- 官方确实证明过 `100B` 这一级别的研究配置推理能力和 kernel 表现
- 但这不等于“微软已经公开放出一个成熟的官方 100B BitNet 模型给大家直接下载用”

### 5.3 把“CPU 推理有突破”偷换成“GPU 被淘汰”

这也不对。

因为微软官方仓库自己就在继续推进：

- CPU inference
- GPU inference kernel
- 后续还提到 NPU

所以更准确的说法是：

- 在端侧、本地、边缘部署里，CPU 对 `1.58-bit / ternary` 模型的实用性大幅提高了
- 但训练、大规模推理、通用高吞吐部署里，GPU 仍然重要

## 6. 对“模型体积降到几 GB”这类说法要怎么理解

网上常见说法是：

- “100B 从几百 GB 直接掉到几 GB”

这个说法太粗暴，容易误导。

如果只按 `1.58 bits/weight` 粗算，`100B` 权重本体当然会比 FP16 小很多，但：

- 实际打包格式不只看理论 bit 数
- 还会涉及 kernel 对齐、索引、scale、embedding、上下文缓存等额外开销
- 真正能不能跑，还要看内存容量和带宽，不是只看静态权重文件大小

所以：

- “比传统全精度模型小得多”是真的
- “100B 只剩几 GB、谁都能装”这种说法不靠谱

## 7. 按微软公开数据，对我当前电脑的保守判断

本轮实际查到的本机配置：

- CPU：`AMD Ryzen 7 3800X`
- 核心 / 线程：`8C / 16T`
- 内存：`32GB DDR4-3200`
- 显卡：`NVIDIA GTX 1050 Ti 4GB`

这里真正关键的是：

- 主要看 CPU 和内存
- `GTX 1050 Ti 4GB` 对 `bitnet.cpp` 这条 CPU 推理路子帮助不大

### 7.1 为什么不能直接照搬微软的最好成绩

微软论文里的 x86 基准机是：

- `Intel i7-13700H`
- `64GB RAM`

而且论文明确指出带宽很重要。  
我的机器是：

- 更老一代桌面 CPU
- 只有 `32GB`
- 内存带宽条件不如 `M2 Ultra` 那类统一内存平台

所以不能直接把论文里 `13B` 的 `10.99 tok/s` 当成我这台机器的实测结果。

### 7.2 如果目标是最好超过 10 token/s

按微软公开表格，再结合我机器规格做保守判断：

下面这段是 **基于微软 `i7-13700H 64GB` 数据，对 `Ryzen 7 3800X + 32GB` 做的经验外推**，不是微软官方承诺，也不是我这台机器的实测 benchmark。

- 最稳妥的上限：`7B`
- 可以碰一碰，但不保证稳过 10：`8B`
- 大概率掉到 10 以下：`10B` 到 `13B`
- `30B`：不建议按这台 `32GB` 机器当现实目标
- `70B / 100B`：更不建议按这台机子做实用目标

更直白点说：

- 如果目标是“本地长期能用，而且速度别太难受”，优先盯 `7B`
- 如果目标是“就想冲最大”，那是另一个问题，不等于“能实用”

## 8. 这件事对本地 AI 有什么实际意义

`bitnet.cpp` 真正有价值的，不是给营销号一句“CPU 颠覆 GPU”。

它真正说明的是：

1. 原生为 `1-bit / 1.58-bit / ternary` 训练的模型，确实能把推理成本打下来  
2. 端侧、本地、边缘部署以后不一定只能靠大显卡  
3. 模型设计路线开始更重视“训练时就为了推理效率做架构约束”，而不只是事后量化  
4. 对想做本地 AI 的个人和小团队来说，未来有可能出现一批“能力还行、硬件要求明显更低”的模型路线

但这条路线仍然有现实前提：

- 模型本身得按这条路训练
- 推理框架得专门适配
- 硬件带宽仍然重要
- 公开模型生态目前还远没到“100B 人人现成可用”的成熟度
