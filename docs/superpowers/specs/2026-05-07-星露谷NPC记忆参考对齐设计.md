# 星露谷 NPC 记忆参考对齐设计

## 背景

当前手测里，海莉和潘妮的 `MEMORY.md` 被自主行动日志污染。里面出现了很多不该成为长期记忆的内容，比如 `Autonomy tick trace_xxx`、移动失败、等待、工具预算、调试格式、命令结果和 JSON 协议片段。

这个问题的核心不是“缺少更聪明的记忆压缩器”，而是当前项目有一条偏离参考项目的自动写入路径：`NpcAutonomyLoop` 每轮自主行动后，把本轮决策摘要直接写进 `MEMORY.md`。参考项目没有这种“每轮行动都硬写长期记忆”的设计。

本设计只做参考项目对齐。开发阶段旧记忆没有兼容价值，旧的污染 `MEMORY.md` 不作为迁移对象。

## 参考项目事实

参考项目的长期记忆是小容量、人工整理味道很重的文件型记忆。

- `MEMORY.md`：agent 自己的长期笔记，放稳定事实、长期观察、项目环境事实。
- `USER.md`：agent 对用户的长期了解，比如偏好、沟通方式、长期背景。
- `memory` 工具是写长期记忆的入口，动作只有 `add`、`replace`、`remove`。
- `replace` 和 `remove` 靠 `old_text` 找到旧条目，不靠 trace id、命令 id 或游戏 tick id。
- `add` 超过容量上限会拒绝，并提示先 `replace` 或 `remove`。
- 安全扫描只挡提示注入、密钥外泄这类危险内容，不做游戏字段黑名单。
- 会话开始时读取 `MEMORY.md` 和 `USER.md`，形成当轮系统提示词用的快照。
- 会话中途写入磁盘可以发生，但不会偷偷改掉当前系统提示词里的旧快照。

参考项目里的 `todo` 不是长期记忆。

- `todo` 是当前 agent / 当前会话的任务列表。
- 状态只有 `pending`、`in_progress`、`completed`、`cancelled`。
- 上下文压缩后，只把 `pending` 和 `in_progress` 重新注入上下文。
- `completed` 和 `cancelled` 不重新注入，避免模型反复做完事。
- `todo` 不接管 `MEMORY.md` 的内容，也不保存历史行动日志。

参考项目里的上下文压缩也不是 `MEMORY.md` 压缩。

- 上下文压缩处理的是长会话上下文。
- `MEMORY.md` 的整理方式是模型通过 `replace` 或 `remove` 改条目。
- 容量不够时，`add` 失败，模型必须先删改旧条目。

## 参考链重建

按参考项目看，记忆链路是下面这样。

1. 触发：模型认为某个事实值得长期保存时，主动调用 `memory` 工具。
2. 快照：会话开始时读取 `MEMORY.md` 和 `USER.md`，作为当前上下文的固定快照。
3. 提示词组装：固定快照进入系统提示词；`todo` 只在需要时提供当前任务列表。
4. 解析和校验：`memory` 工具校验动作、目标、内容、容量和安全风险。
5. 执行：`add` 追加条目，`replace` 替换旧条目，`remove` 删除旧条目。
6. 权威写回：工具把结果写回 `MEMORY.md` 或 `USER.md`。
7. 可见面：写入结果作为工具结果返回给模型；玩家不需要确认每条 NPC 记忆。

参考项目没有把行动 trace、命令结果、移动失败、等待状态自动投进长期记忆的层。

## 实施前项目偏离点

当前项目已经有参考项目式的基础设施：

- `src/memory/MemoryManager.cs`：固定 `MEMORY.md` / `USER.md` 文件存储，支持 `AddAsync`、`ReplaceAsync`、`RemoveAsync`，并有容量上限。
- `src/Tools/MemoryTool.cs`：`memory` 工具入口。
- `src/Tools/TodoWriteTool.cs`：当前会话任务列表工具。

真正偏离的是自主循环曾经直接写记忆：

- `src/runtime/NpcAutonomyLoop.cs` 曾经在每轮行动后调用 `WriteMemoryAsync(...)`。
- `WriteMemoryAsync` 曾经写入 `Autonomy tick {traceId}: ...`。

这条路径绕过了参考项目的“模型主动调用 `memory` 工具”边界，也把 trace id 和行动摘要伪装成长期记忆。

## 设计决策

本次对齐只做一件事：删除或关闭 `NpcAutonomyLoop` 的自动写 `MEMORY.md` 路径。

对齐后的规则：

- NPC 自主循环普通 tick 不再自动写 `MEMORY.md`。
- NPC 要保存长期事实，只能通过现有 `memory` 工具写。
- `MemoryManager`、`MemoryTool`、`BuiltinMemoryPlugin` 继续保留，因为它们已经接近参考项目方案。
- `todo` 继续只做当前会话任务列表，不接长期记忆，也不接 trace。
- `runtime.jsonl` 继续放执行证据、trace、命令、失败和调试信息。
- `session_search` 继续用于查历史会话，不替代长期记忆。
- Dreamer 不参与这次 `MEMORY.md` 整理，因为参考项目的 memory / todo 边界不靠 Dreamer。

明确不做：

- 不新增 trace id 黑名单。
- 不新增 command id 黑名单。
- 不新增 Stardew 字段过滤器。
- 不把内容自动分流到 `todo`。
- 不新增记忆数据库。
- 不做向量记忆。
- 不做评分、淘汰、每日整理、阈值整理。
- 不做旧 `MEMORY.md` 兼容迁移。

## 数据边界

| 数据 | 只负责什么 | 不负责什么 |
| --- | --- | --- |
| `MEMORY.md` | NPC 长期事实、稳定观察、长期偏好 | tick 日志、trace id、移动失败、工具预算 |
| `USER.md` | 对玩家或用户的长期了解 | NPC 自己每轮行动记录 |
| `todo` | 当前会话未完成任务和正在做的事 | 长期记忆、历史日志、trace 存档 |
| `runtime.jsonl` | 自主循环执行证据、trace、命令、失败原因 | NPC 长期记忆 |
| transcript / `session_search` | 历史对话和历史检索 | 结构化长期记忆整理 |

## 当前问题怎么落地解决

海莉和潘妮的污染记忆，根因不是模型“太爱记小事”，而是宿主每轮自动写入。

落地动作应该是：

1. 删掉或关闭 `NpcAutonomyLoop` 里每轮自动 `WriteMemoryAsync` 的调用。
2. 删掉或改掉相关测试里“自主循环会写 NPC 本地记忆”的旧断言。
3. 新增测试证明普通自主 tick 不会改 `MEMORY.md`。
4. 保留测试证明 `memory` 工具仍能写入、替换、删除长期事实。
5. 开发环境里的旧污染 `MEMORY.md` 可以直接清空或重建测试 profile，不设计迁移流程。

## 验收标准

- NPC 自主循环跑一轮后，`MEMORY.md` 不新增 `Autonomy tick` 条目。
- `MEMORY.md` 不再出现由宿主自动写入的 trace id。
- 通过 `memory` 工具写入长期事实仍然生效。
- `add` 超容量时仍然拒绝，并要求先 `replace` 或 `remove`。
- `todo` 没有被改成长期记忆存储。
- `runtime.jsonl` 仍保留 trace、命令、失败和调试证据。
- 代码里没有新增本地黑名单、游戏字段过滤器、自动分流器或新记忆存储。

## 测试建议

- 删除或重写 `RunOneTickAsync_WithDecisionResponse_WritesNpcLocalMemory` 这类旧测试。
- 新增测试：普通自主 tick 不写 `MEMORY.md`。
- 新增或保留测试：NPC 通过 `memory` 工具写长期事实仍然生效。
- 保留 `MemoryTool` / `MemoryManager` 的 `add`、`replace`、`remove`、容量上限测试。
- 保留 `todo` 的当前会话任务行为，不把它和长期记忆混测。

## 参考映射矩阵

| 参考项目行为 | 当前项目对应 | 本次处理 |
| --- | --- | --- |
| `MEMORY.md` / `USER.md` 是固定文件型长期记忆 | `MemoryManager` 已是固定文件型存储 | 保留 |
| `memory` 工具负责 `add` / `replace` / `remove` | `MemoryTool` 已提供入口 | 保留 |
| 容量超限时拒绝 `add` | `MemoryManager.AddAsync` 已有容量判断 | 保留 |
| `todo` 是当前会话任务列表 | `TodoWriteTool` 已有会话任务能力 | 保留原边界 |
| 上下文压缩不等于压缩 `MEMORY.md` | 当前不应新增 `MEMORY.md` 压缩器 | 不新增 |
| 没有每轮行动自动写长期记忆 | `NpcAutonomyLoop` 当前会自动写 | 删除或关闭这条路径 |
| 没有游戏字段黑名单 | 当前不需要新增 | 不新增 |
| 没有自动分流到 `todo` | 当前不需要新增 | 不新增 |

## 偏离记录

当前设计没有申请偏离参考项目。

如果后续实现时有人想加本地过滤器、自动分流、评分整理、向量记忆或旧文件迁移，必须先单独记录为偏离，并说明参考项目没有这么做。
