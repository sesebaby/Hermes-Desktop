# Hermes Desktop 接入 Stardew Valley 多 NPC 村庄模式设计方案

## 1. 背景与目标

本方案目标是在 Hermes Desktop 现有能力基础上，接入 Stardew Valley（星露谷物语），实现多 NPC 村庄模式。

项目目标来自现有深度访谈规格：
- 改造 Hermes Desktop 项目，接入星露谷物语
- 实现多 NPC 村庄模式
- 每个 NPC 拥有独立人格、独立记忆、独立自主决策能力
- MVP 先实现 1-3 个 NPC 并发运行

本方案明确参考 `hermescraft` 的核心架构思想，但不盲目复制 Minecraft 特有实现，而是优先复刻其“Agent 自己通过工具理解世界、桥接层只暴露能力接口”的结构。

### MVP 目标
- 先支持 Haley（海莉）与 Penny（潘妮）两名 NPC
- 近景时 NPC 以独立 Agent 运行
- 每个 NPC 自己维护自己的长期记忆与人格作用域
- 玩家在场时支持 NPC 群聊
- 使用桥接层能力接口，不通过主控转述世界事实

### 长期目标
- 扩展到 3 个以上 NPC
- 玩家不在场时 NPC 之间自然群聊
- 更复杂的社交图、行为规划、采集、生产和经济系统
- 必要时演进到多进程 / 分布式架构

---

## 2. 核心设计结论

1. 架构采用：**平行 NPC Agent（并行 NPC 智能体） + 非智能调度器服务**。
2. 不保留“主 Agent 语义导演”这一角色。
3. NPC 身份、记忆区域、人格区域必须由系统绑定，不能依赖提示词或主控传参。
4. 桥接层只提供能力接口，不提供语义摘要或事实转述。
5. NPC Agent 自己通过工具理解当前场景。
6. 每个 NPC 自己维护并写入自己的 canonical memory（规范长期记忆）。
7. 完整设计支持“玩家在场 / 不在场”两类群聊；MVP 先只实现玩家在场群聊。
8. 运行模式完整保留模式 A / 模式 B 两种；MVP 先实现模式 A。
9. 最终设计文档必须同时包含：Hermes Desktop 当前仓库证据 + hermescraft 参考 mod 证据。

## 5. 设计原则

1. **NPC 身份由系统绑定，不由提示词决定。**
2. **调度器只调度，不做语义导演。**
3. **桥接层只提供能力接口，不提供语义摘要。**
4. **NPC Agent 自己使用工具感知和理解场景。**
5. **每个 NPC 自己维护并写入自己的长期记忆。**
6. **玩家在场时优先保证近景真实感。**
7. **完整设计写全，MVP 只截断实现范围，不截断愿景。**
8. **每个关键结论必须能追溯到当前仓库证据和 / 或参考 mod 证据。**

## 6. 总体架构设计

```text
SMAPI / Stardew
    ^
    |
桥接层（HTTP / 进程通信）
    ^
    |
Hermes 工具接口（Stardew 工具）
    ^
    |
平行 NPC Agent（Haley / Penny / ...）
    ^
    |
非智能调度器服务
```

### 架构分工
- **非智能调度器服务**：检测视野变化、查注册表、创建 / 回收 NPC Agent、维护运行表、处理节日 / 剧情让位。
- **平行 NPC Agent**：各自独立使用工具理解场景、执行近景行为、维护自己的会话与长期记忆。
- **Hermes 工具接口**：给 Agent 暴露 Stardew 专用能力。
- **桥接层**：接收工具调用，转成 SMAPI 侧操作。
- **SMAPI / Stardew**：游戏实际执行层。

## 7. 核心组件设计

### 7.1 NpcRegistry（NPC 注册表）
职责：
- 保存 `npcId -> 绑定信息`
- 作为 NPC 身份、记忆路径、人格作用域的系统真相表

### 7.2 NpcRuntimeBinding（NPC 运行时绑定）
职责：
- 表达单个 NPC 在运行时必须被绑定的依赖集合
- 包括 `memoryDir`、`soulScope`、`sessionNamespace` 等

### 7.3 NpcRuntimeFactory（NPC 运行时工厂）
职责：
- 根据 `NpcRuntimeBinding` 创建已经绑定好的 NPC Agent
- 直接构造并注入专属 `MemoryManager`、`TranscriptStore`、`ContextManager`、`SoulService`

### 7.4 NpcSceneScheduler（NPC 场景调度器）
职责：
- 维护视野集合
- 决定进入视野 / 持续停留 / 离开视野时的创建与回收
- 维护每个 NPC 同时只有一个近景运行体

### 7.5 NpcRuntimeHandle（NPC 运行句柄）
职责：
- 保存运行中 NPC 实例的元数据：
  - `npcId`
  - `agentId`
  - `sessionId`
  - `backgroundTaskId`
  - `cancelTokenSource`
  - `status`
  - `lastSeenAt`

### 7.6 StardewBridgeServer（桥接服务）
职责：
- 在 SMAPI mod 中暴露 HTTP / 进程通信接口
- 只负责能力接口，不负责语义摘要

### 7.7 Stardew 工具集
职责：
- 让 NPC Agent 以 hermescraft 模式调用游戏能力
- 工具层只暴露高层能力，不暴露底层 SMAPI 细节

## 8. 生命周期设计

### 三态状态机
1. **未激活**：不在玩家视野内，没有近景 Agent 在运行。
2. **近景活跃**：在玩家视野内，近景 Agent 正在运行。
3. **回收中**：已离开玩家视野，正在停止 Agent、等待收尾。

### 三个事件
- **进入视野**：未激活 -> 近景活跃
- **持续停留**：近景活跃 -> 近景活跃
- **离开视野**：近景活跃 -> 回收中 -> 未激活

### 规则
- 同一 NPC 同一时刻只允许一个近景运行体
- 持续停留时只保活，不重建
- 回收由调度器发出结束信号，不靠 Agent 自己猜测

---

## 9. 绑定与创建机制

### 目标
确保 NPC 身份、记忆目录、人格目录完全准确，不依赖主控传参，不依赖提示词，不依赖模型自我识别。

### 设计
- 调度器只负责“是否拉起 / 是否回收”
- 不负责“这个 Agent 是谁”
- `NpcRegistry` + `NpcRuntimeFactory` 共同完成系统级绑定

### 关键原则
- 被唤醒的 Agent 不是“被告诉自己是谁”
- 而是“在被创建时就已经是那个 NPC”

## 11. 通信方式设计

### 方案对比
- **HTTP**：hermescraft 已验证，隔离性好，易调试
- **命名管道 / 进程通信**：更快，但实现和调试复杂度更高
- **进程内共享库**：延迟最低，但耦合最强

### 选型结论
**MVP 选 HTTP**。

### 端口冲突处理
- 支持配置默认端口
- 支持端口范围扫描
- 支持自动向后寻找可用端口
- SMAPI mod 把当前绑定端口写入发现文件
- Hermes Desktop 启动时读取发现文件连接

## 13. 第一批 NPC 方案

### MVP 首批 NPC
- Haley（海莉）
- Penny（潘妮）

### 选择原因
- **Haley**：人格外放，近景对话价值高，更容易验证“像不像活人”。
- **Penny**：性格温和、稳定，适合测试持续对话、情绪和关系连续性。

### 测试价值
- 两人差异大，更容易验证人格隔离
- 两人都属于村庄核心 NPC，利于在公共场景测试群聊

## 14. 运行模式设计

### 模式 A：后台独立运行（MVP 先做）
- 不修改现有 Hermes Desktop 主 UI
- NPC 系统作为后台模块运行
- 用户正常使用现有 UI，不受打断

### 模式 B：UI 集成运行（后续阶段）
- Hermes Desktop 增加 NPC 管理面板
- 显示在线 NPC、生命周期状态、日志、最近活动
- 允许手动暂停 / 重启 / 查看某个 NPC 状态

### 文档要求
模式 A 和模式 B 都必须写入最终方案；MVP 只实现模式 A。

## 17. 风险与缓解

### 风险 1：端口冲突
缓解：端口范围 + 自动回退 + 发现文件

### 风险 2：身份绑定错误
缓解：注册表 + 工厂注入，禁止靠提示词绑定

### 风险 3：多 NPC 记忆 / 人格串写
缓解：每 NPC 独立 `memoryDir` / `soulScope` / `sessionNamespace`

### 风险 4：桥接层做太多语义转述，长出双轨
缓解：桥接层只暴露能力接口，不提供语义摘要

### 风险 5：群聊复杂度过早膨胀
缓解：完整设计保留，MVP 只实现玩家在场群聊

### 风险 6：UI 集成过早导致范围失控
缓解：模式 A 先行，模式 B 后置

### 布局原则
- 静态配置与运行时状态分离
- 桥接状态与调度状态分离
- 每个 NPC 拥有自己的 `memory / marks / social / runtime` 文件边界
- 设计采用文件化 / 目录化持久化，不引入数据库优先方案
