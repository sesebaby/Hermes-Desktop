# Hermes Desktop 接入 Stardew Valley 多 NPC 村庄模式设计方案

## 1. 背景与目标

本方案目标是在 Hermes Desktop 现有能力基础上，接入 Stardew Valley（星露谷物语），实现多 NPC 村庄模式。

项目目标来自现有深度访谈规格：
- 改造 Hermes Desktop 项目，接入星露谷物语
- 实现多 NPC 村庄模式
- 每个 NPC 拥有独立人格、独立记忆、独立自主决策能力
- MVP 先实现 1-3 个 NPC 并发运行

本方案明确参考 `hermescraft` 的核心架构思想，但不盲目复制 Minecraft 特有实现，而是优先复刻其“Agent 自己通过工具理解世界、桥接层只暴露能力接口”的结构。

### MVP 目标
- 先支持 Haley（海莉）与 Penny（潘妮）两名 NPC
- 近景时 NPC 以独立 Agent 运行
- 每个 NPC 自己维护自己的长期记忆与人格作用域
- 玩家在场时支持 NPC 群聊
- 使用桥接层能力接口，不通过主控转述世界事实

### 长期目标
- 扩展到 3 个以上 NPC
- 玩家不在场时 NPC 之间自然群聊
- 更复杂的社交图、行为规划、采集、生产和经济系统
- 必要时演进到多进程 / 分布式架构

## 2. 核心设计结论

1. 架构采用：**平行 NPC Agent（并行 NPC 智能体） + 非智能调度器服务**。
2. 不保留“主 Agent 语义导演”这一角色。
3. NPC 身份、记忆区域、人格区域必须由系统绑定，不能依赖提示词或主控传参。
4. 桥接层只提供能力接口，不提供语义摘要或事实转述。
5. NPC Agent 自己通过工具理解当前场景。
6. 每个 NPC 自己维护并写入自己的 canonical memory（规范长期记忆）。
7. 完整设计支持“玩家在场 / 不在场”两类群聊；MVP 先只实现玩家在场群聊。
8. 运行模式完整保留模式 A / 模式 B 两种；MVP 先实现模式 A。
9. 最终设计文档必须同时包含：Hermes Desktop 当前仓库证据 + hermescraft 参考 mod 证据。
10. Hermes Desktop 采用**一个应用内托管多个 NPC runtime**，不为 MVP 复制 hermescraft 的“一 NPC 一进程”启动形态。
11. CLI 可以存在，但 CLI、Hermes typed tools、未来 UI 面板都只能是 `StardewCommandService` 的适配器，禁止各自直接调用 SMAPI HTTP 形成双轨。
12. Prompt 组装顺序必须绑定 Hermes Desktop 现有 `ContextManager` / `PromptBuilder` 链路，禁止新增独立 `StardewPromptAssembler`。
13. SocialRouter 只做消息投递，不做情绪判断、意图判断、关系推理或剧情导演。
14. NPC 数据隔离键必须至少包含 `gameId + saveId + npcId + profileId`，不能只用 `npcId`。
15. SMAPI HTTP 线程只能读缓存、入队命令、返回命令编号；真正读写 `Game1` / NPC / 地图对象必须在 SMAPI 游戏循环事件中执行。
16. 参考实现必须落成“三方矩阵”：`hermescraft` 参考链条、Hermes Desktop 当前代码接入点、Stardew 映射规则必须逐层对应。
17. Bridge API / command 契约必须固定 request / response / error / status / 幂等规则；CLI、typed tools、UI 不得各自定义 JSON 或错误码。
18. 跨天保留 NPC 的记忆、承诺、意图；不默认保留未验证的物理动作执行现场。第二天继续任务必须重新观察并重新下发命令。
19. MVP 必须留下结构化可观测证据：每 NPC transcript、activity、bridge command trace、prompt preview 必须能按 `traceId` / `commandId` 串起来查。
20. Stardew 配置必须进入 Hermes Desktop 现有 `HERMES_HOME/config.yaml` 配置体系，禁止新增 `.claude/stardew.json` 或其他第二配置源。
21. Bridge HTTP 是本机轻量接口：MVP 默认只绑定 `127.0.0.1`，除 `/health` 外必须带启动时生成的 bridge token，禁止默认 LAN 暴露。
22. Haley / Penny 必须有固定 persona seed pack；核心人格只能由 seed pack 进入各自 runtime 的 `SOUL.md`，禁止实现阶段临场生成或塞进初始化 prompt。
23. NPC 身份必须经 `StardewNpcCatalog` 统一规范化：路径只用 `npcId`，SMAPI 只用 `smapiName`，玩家点名只经 `aliases` 归一化，禁止各层各自匹配名字。
24. MVP 工具必须固定 request `payload` 与 response `data` DTO；禁止只固定 envelope，而让工具内部 `data` 自由发挥。
25. 多 NPC 并发时，观察类工具可并发，物理动作命令必须经过 `ResourceClaimRegistry` 互斥；同一 NPC、同一目标 tile、同一交互 tile、同一目标对象不能被多个命令同时占用。
26. NPC 自主性采用受控常驻 `NpcAutonomyLoop`，贴近 hermescraft 的长会话 agent 模式；事件只做唤醒 / 优先级输入，不能退化成“有事件才临时问一次 AI”。

## 5. 设计原则

1. **NPC 身份由系统绑定，不由提示词决定。**
2. **调度器只调度，不做语义导演。**
3. **桥接层只提供能力接口，不提供语义摘要。**
4. **NPC Agent 自己使用工具感知和理解场景。**
5. **每个 NPC 自己维护并写入自己的长期记忆。**
6. **玩家在场时优先保证近景真实感。**
7. **完整设计写全，MVP 只截断实现范围，不截断愿景。**
8. **每个关键结论必须能追溯到当前仓库证据和 / 或参考 mod 证据。**
9. **一个应用可以托管多个 NPC runtime，但每个 runtime 的 home、SOUL、memory、transcript、social、marks 必须硬隔离。**
10. **同一条 Stardew 命令只能有一个服务源；CLI、typed tools、UI 只能调用同一个命令服务。**
11. **Prompt 组装顺序固定，不允许实现阶段自行发明新顺序。**
12. **SocialRouter 只是“邮递员”，不是导演。**
13. **Stardew 行为工具必须按 Stardew 任务语义定义，不能把 Minecraft 的 `collect` 等动作照搬进 MVP。**
14. **参考实现对齐必须同时考虑当前 Hermes 代码接入点，不能只抄参考 mod 的表面流程。**
15. **Bridge 契约、命令状态、错误码、幂等规则必须是共享契约，不允许 CLI / typed tools / UI 各写一份。**
16. **跨天续做必须重新观察和重新下发命令；旧的物理动作 command 不能默认跨天继续跑。**
17. **多 NPC 运行必须可追踪、可回放、可调试；没有结构化 trace 的实现不能算通过 MVP。**
18. **Haley / Penny 人格必须来自固定 seed pack，不能由实现 AI 临场编写、临场改写或重复注入。**
19. **NPC 名字不能散落匹配；所有显示名、中文名、别名、SMAPI 名都必须先映射成稳定 `npcId`。**
20. **MVP 每个工具的 payload / data 都必须有命名 DTO 与 golden JSON，不能用自由 JSON 字典补实现。**
21. **多 NPC 可以并发思考，但物理动作不能并发抢同一资源；资源占用必须由 bridge / command 层统一裁决。**
22. **活跃 NPC 必须有自己的常驻自主循环；调度器负责启动、暂停、恢复、停止，不替 NPC 做一次性决策。**

## 6. 总体架构设计

```text
SMAPI / Stardew
    ^
    |
桥接层（HTTP / 进程通信）
    ^
    |
Hermes 工具接口（Stardew 工具）
    ^
    |
平行 NPC Agent（Haley / Penny / ...）
    ^
    |
非智能调度器服务
```

### 运行形态
- MVP 采用**一个 Hermes Desktop 应用内托管多个 NPC runtime**。
- 这不是把 Haley、Penny 都塞进同一个 Agent；而是在同一进程内创建多个边界清楚的 `Agent + SoulService + ContextManager + MemoryManager + TranscriptStore` 组合。
- 参考 mod 的 landfolk 是每个角色一个 `HERMES_HOME` 启动进程；Hermes Desktop 的等价映射是每个 NPC 一个 runtime home / profile，而不是每个 NPC 一个外部 Hermes 进程。
- 长期可以演进到多进程 / 分布式，但 MVP 不以多进程为默认形态。

### 架构分工
- **非智能调度器服务**：检测视野变化、查注册表、创建 / 回收 NPC Agent、维护运行表、处理节日 / 剧情让位。
- **平行 NPC Agent**：各自独立使用工具理解场景、执行近景行为、维护自己的会话与长期记忆。
- **Hermes 工具接口**：给 Agent 暴露 Stardew 专用能力。
- **桥接层**：接收工具调用，转成 SMAPI 侧操作。
- **SMAPI / Stardew**：游戏实际执行层。

### 参考实现三方落地矩阵
本方案不能只说“参考 hermescraft”，必须把参考链条逐层落到 Hermes Desktop 当前代码和 Stardew 目标实现上。

| 链条层 | hermescraft 参考实现 | Hermes Desktop 当前接入点 | Stardew 映射规则 |
| --- | --- | --- | --- |

## 7. 核心组件设计

### 7.1 NpcRegistry（NPC 注册表）
职责：
- 保存 `npcId -> 绑定信息`
- 作为 NPC 身份、记忆路径、人格作用域的系统真相表
- 只接受已经由 `StardewNpcCatalog` 规范化后的 `npcId`

### 7.1.1 StardewNpcCatalog（Stardew NPC 身份目录）
职责：
- 作为 `npcId`、SMAPI 内部名、玩家显示名、中文名、别名、persona seed pack 的唯一映射源。
- 为 SocialRouter、StardewCommandService、NpcRuntimeFactory、SMAPI bridge client 提供统一身份解析。

MVP 最小字段：

```json
{
  "npcId": "haley",
  "smapiName": "Haley",
  "displayNames": ["Haley", "海莉"],
  "aliases": ["haley", "Haley", "海莉"],
  "profileIds": ["default"],
  "defaultProfileId": "default",
  "personaSeedPack": "src/game/stardew/personas/haley/default"
}
```

硬规则：
- 路径、runtime state、trace、memory、transcript 只使用稳定小写 `npcId`，例如 `haley`、`penny`。
- SMAPI 查询、NPC 查找、友谊 / 行为条件只使用 `smapiName`，例如 `Haley`、`Penny`。
- 玩家输入、点名、群聊 target 必须先经 `aliases` 归一化为 `npcId`。
- `displayNames` 只用于 UI / prompt 可读文本，不能作为路径 key 或 SMAPI 查询 key。
- SocialRouter 禁止自己做字符串匹配；它只能调用 `StardewNpcCatalog.ResolveAlias(...)`。
- typed tools、`sdv` CLI、UI、Bridge client 禁止各自维护名字映射表。
- 未知或歧义别名必须返回 `invalid_target`，不能让模型猜最像的 NPC。
- `npc_default_cast` 只能写 `npcId` 列表，不能写 `Haley`、`海莉` 或 `smapiName`。

### 7.2 NpcRuntimeBinding（NPC 运行时绑定）
职责：
- 表达单个 NPC 在运行时必须被绑定的依赖集合
- 包括 `gameId`、`saveId`、`npcId`、`profileId`
- 包括 `runtimeHome`、`memoryDir`、`transcriptDir`、`soulScope`、`sessionNamespace`
- 包括 `enabledSkillNames`

硬规则：
- `npcId` 不能单独作为数据隔离 key。
- MVP 即使只有一个人格配置，也必须使用 `profileId = "default"`。
- 任一 runtime 的 `SoulService`、`ContextManager`、`PromptBuilder`、`MemoryManager`、`TranscriptStore` 必须由 `NpcRuntimeBinding` 创建 / 注入，不能复用全局单例依赖。

### 7.3 NpcRuntimeFactory（NPC 运行时工厂）
职责：
- 根据 `NpcRuntimeBinding` 创建已经绑定好的 NPC Agent
- 直接构造并注入专属 `MemoryManager`、`TranscriptStore`、`ContextManager`、`SoulService`
- 把 NPC 专属 `SOUL.md` 放在该 runtime 的 `SoulContext` 来源中
- 把 Stardew 必选规则、命令契约、观察-思考-行动循环绑定进稳定系统上下文

### 7.4 NpcSceneScheduler（NPC 场景调度器）
职责：
- 维护视野集合
- 决定进入视野 / 持续停留 / 离开视野时的创建与回收
- 维护每个 NPC 同时只有一个近景运行体
- 启动、暂停、恢复、停止对应 NPC 的 `NpcAutonomyLoop`

硬规则：
- `NpcSceneScheduler` 不调用 LLM，不替 NPC 组装“下一步应该做什么”。
- 进入近景时创建 runtime 并启动常驻自主循环。
- 持续停留时只保活 / 投递事件，不重复启动第二个循环。
- 离开近景、切存档、返回标题时发出停止信号，由 `NpcAutonomyLoop` 收尾。

### 7.5 NpcRuntimeHandle（NPC 运行句柄）
职责：
- 保存运行中 NPC 实例的元数据：
  - `npcId`
  - `agentId`
  - `sessionId`
  - `backgroundTaskId`
  - `cancelTokenSource`
  - `status`
  - `lastSeenAt`
  - `autonomyLoopTask`
  - `autonomySessionId`
  - `lastAutonomyHeartbeatAt`
  - `lastAutonomyExitReason`

### 7.5.1 NpcAutonomyLoop（NPC 常驻自主循环）
职责：
- 作为活跃 NPC 的“脑子”常驻运行。
- 持有该 NPC 的 `Agent`、`Session`、`NpcRuntimeBinding`、enabled skills、工具集合和 cancellation token。
- 进入循环后让 NPC 自己反复观察、思考、行动、读聊天、读命令、轮询任务。
- 接收 SocialRouter / bridge / scheduler 投递的事件，但事件只影响优先级，不替 NPC 决策。

参考映射：
- hermescraft 是每个角色一个长期 `hermes chat --yolo -q "$PROMPT"` 进程。
- Stardew Desktop 不再启动多个外部 Hermes 进程，而是在同一应用内为每个活跃 NPC 启动一个受控 `NpcAutonomyLoop`。
- 这是 `equivalent mapping`：保持“每个角色一个长会话脑子”的 authority owner，只把进程边界映射为应用内 runtime 边界。

自主循环启动 prompt 只允许描述循环任务，不允许塞人格：

```text
You are now active in Stardew Valley.
Repeat your Stardew loop while active:
1. observe game status, self state, nearby, chat, commands
2. decide what matters
3. speak, remember, mark, complete command, or request a task when needed
4. while a task runs, poll task_status plus read_chat / commands
Stop cleanly if runtime state becomes paused, stopping, or bridge unavailable.
```

硬规则：
- 一个 `npcId + saveId + profileId` 同时只能有一个 `NpcAutonomyLoop`。
- `NpcAutonomyLoop` 使用同一个 `sessionId` 持续写 transcript，不为每次事件创建新 session。
- `NpcAutonomyLoop` 可以在内部调用一次长 `Agent.ChatAsync(...)`，并把 `MaxToolIterations` 显式设置为 Stardew autonomy 预算；如果 `ChatAsync(...)` 因达到预算或模型自然停止而返回，但 runtime 仍为 `active`，loop supervisor 必须记录原因、短暂 cooldown，然后用同一 session 续跑。
- `NpcAutonomyLoop` 不能在 SMAPI `UpdateTicked` 线程内执行；它只能在 Hermes Desktop 后台任务中运行。
- `paused` / `stopping` / `ReturnedToTitle` / `DayEnding` / `Saving` 必须能取消或收尾 loop，不能让后台 LLM 继续向旧存档发命令。
- 长任务期间禁止让模型空等；必须像 hermescraft 一样轮询 `task_status`、`read_chat`、`commands`。

### 7.5.2 NpcAutonomyBudget（NPC 自主预算）
职责：
- 控制多个常驻 NPC loop 对 LLM 的并发和总量。
- 在 ChatClient 或 Agent 调用层做全局门闸，而不是锁住整个 `NpcAutonomyLoop`。

MVP 默认：
- `npc_autonomy_max_tool_iterations = 100`
- `npc_autonomy_max_concurrent_llm_requests = 1`
- `npc_autonomy_restart_cooldown_seconds = 5`
- `npc_autonomy_max_restarts_per_scene = 3`
- `npc_autonomy_start_stagger_seconds = 4`

硬规则：
- 多个 NPC loop 可以同时存活、等待工具、轮询 bridge；但 LLM 请求必须经过 `NpcAutonomyBudget`。
- MVP 默认同一时间只允许一个 LLM request，防止多 NPC 同时烧模型和触发 provider 限流。
- 预算限制不能由 prompt 表达，必须由代码 enforce。
- 超预算时 NPC runtime 进入 `paused` 或 `blocked`，并写入 trace；不能静默停止。

### 7.6 StardewBridgeServer（桥接服务）
职责：
- 在 SMAPI mod 中暴露 HTTP / 进程通信接口
- 只负责能力接口，不负责语义摘要
- HTTP 请求线程只允许读取已缓存状态、写入命令队列、返回 `commandId`
- 实际读写 Stardew `Game1` / NPC / 地图对象必须在 SMAPI 游戏循环事件内执行
- 统一记录 bridge request / response / command 状态，供 Hermes 侧按 `traceId` 查询

### 7.7 Stardew 工具集
职责：
- 让 NPC Agent 以 hermescraft 模式调用游戏能力
- 工具层只暴露高层能力，不暴露底层 SMAPI 细节

### 7.8 StardewCommandService（Stardew 命令服务）
职责：
- 作为 Stardew 命令的唯一业务入口
- 负责命令注册、参数规范化、调用 bridge client、结果渲染
- 负责生成或传递 `traceId` / `requestId` / `commandId`
- 负责把 bridge error code 映射为 typed tool / CLI / UI 一致的结果模型

硬规则：
- Hermes typed tools 必须调用 `StardewCommandService`
- `sdv` CLI 必须调用 `StardewCommandService`
- 未来 UI 管理面板必须调用 `StardewCommandService`
- 禁止 typed tool、CLI、UI 各自直接拼 HTTP 请求
- 禁止 tool-only 逻辑、CLI-only 逻辑、UI-only 逻辑
- 禁止 typed tool、CLI、UI 各自维护一套 request / response DTO、错误码或命令状态枚举

### 7.9 StardewSkillBinding（Stardew 游戏专有 skill 绑定）
职责：
- 映射参考 mod 的 `skills/*.md` 思路，为 Stardew 建立游戏专有 skill prompt
- 在 `NpcRuntimeBinding.EnabledSkillNames` 中显式声明本 NPC 启用哪些 skill
- 在 runtime 创建时把启用 skill 的 prompt 注入稳定系统上下文

硬规则：
- `skill_invoke` 只能作为可选查询工具，不能作为核心规则唯一来源。
- 当前 Hermes Desktop 不解析 `triggers`，所以不能照搬 hermescraft 的 `triggers` 期待自动触发。
- Haley / Penny 的人格写在各自 `SOUL.md`；Stardew 通用规则、工具契约、游戏循环写在稳定系统上下文或共享 Stardew skill prompt。

### 7.10 StardewPersonaSeedPack（NPC 人格种子包）
职责：
- 为 Haley / Penny 提供固定、可审查、可测试的人格源文件。
- 作为 `NpcRuntimeFactory` 创建 runtime profile 时复制 / 初始化 `SOUL.md` 的唯一来源。
- 记录每个 NPC 的事实摘要、说话风格边界、禁止事项、可用 skill 列表。

MVP seed pack 最小结构：

```text
src/game/stardew/personas/
├─ haley/
│  └─ default/
│     ├─ SOUL.md
│     ├─ facts.md
│     ├─ voice.md
│     ├─ boundaries.md
│     └─ skills.json
└─ penny/
   └─ default/
      ├─ SOUL.md
      ├─ facts.md
      ├─ voice.md
      ├─ boundaries.md
      └─ skills.json
```

硬规则：
- MVP 必须至少提交 `haley/default` 与 `penny/default` 两套 seed pack。
- 核心人格正文只能进入对应 runtime 的 `SOUL.md`，禁止再写进初始化 prompt、工具 prompt 或命令 prompt。
- 禁止实现 AI 在运行时临场生成 Haley / Penny 核心人格。
- 禁止把两个 NPC 的人格合写到一个共享 prompt 或共享 `SOUL.md`。
- 禁止直接复制大段原版台词；seed pack 使用手写事实摘要、风格边界和行为约束，不把 prompt 做成台词库。
- `skills.json` 只能声明该 NPC 启用的 Stardew skill 名称；skill prompt 本体仍归 `StardewSkillBinding` 管。
- `NpcRuntimeFactory` 首次创建 profile 时从 seed pack 初始化 runtime `SOUL.md`；后续 NPC 长期记忆写入 memory，不回写 seed pack。

### 7.11 SocialRouter（社交路由器）
职责：
- 按“谁能听见 / 看见 / 被点名”投递消息
- 维护每个 NPC 的 `inbox`、`commandQueue`、`overhear`、`socialSummary`
- 生成可审计的消息 envelope

硬规则：
- SocialRouter 不判断情绪，不推断意图，不替 NPC 做关系决策。
- 对 `Haley: ...` 这类点名消息，只有 Haley 收到 command；附近 NPC 只能进入 overhear。
- 对 `Haley,Penny: ...` 这类群组消息，只有被点名 NPC 收到 direct / group command。
- 广播消息进入可听见 NPC 的 inbox，但是否回应由各 NPC Agent 自己决定。

## 8. 生命周期设计

### 8.1 Hermes Desktop NPC runtime 三态状态机
1. **未激活**：不在玩家视野内，没有近景 Agent 在运行。
2. **近景活跃**：在玩家视野内，近景 Agent 正在运行。
3. **回收中**：已离开玩家视野，正在停止 Agent、等待收尾。

### 8.2 Hermes Desktop 三个事件
- **进入视野**：未激活 -> 近景活跃
- **持续停留**：近景活跃 -> 近景活跃
- **离开视野**：近景活跃 -> 回收中 -> 未激活

### 8.3 Hermes Desktop 规则
- 同一 NPC 同一时刻只允许一个近景运行体
- 持续停留时只保活，不重建
- 回收由调度器发出结束信号，不靠 Agent 自己猜测

### 8.4 Hermes Desktop runtime 扩展状态
三态用于解释近景唤醒主流程，但实现时必须支持更细的运行状态：

- `inactive`：不在近景，没有运行体。
- `starting`：正在创建 runtime、绑定依赖、加载 `SOUL.md` 与 skill。
- `active`：近景活跃，可正常响应。
- `pausing`：游戏暂时不可执行动作，正在进入暂停。
- `paused`：保留 runtime，但不下发新物理动作。
- `resuming`：游戏恢复可控，正在刷新观察并恢复对话 / 命令处理。
- `stopping`：离开视野或返回标题，正在停止运行体。
- `stopped`：运行体已停止，可重新创建。
- `recovering`：Agent、bridge 或 command queue 出错后正在恢复。
- `failed`：恢复失败，需要人工或监督器处理。

硬规则：
- `paused` 不等于失败；剧情、节日、菜单、玩家不可控、游戏窗口不活跃都应优先进入暂停 / blocked，而不是反复重启 Agent。
- 从 `paused` / `recovering` 回到 `active` 前，必须重新调用观察工具刷新事实。
- `active` 表示 `NpcAutonomyLoop` 存活或正在按预算续跑；不是“只等事件触发一次回答”。
- `failed` 状态必须写入结构化 trace，不能只写普通文本日志。

### 8.5 SMAPI bridge 生命周期
SMAPI 侧生命周期必须按游戏事件驱动：
- `GameLaunched`：启动 bridge HTTP 服务，绑定端口，写入 `bridge-port.json`。
- `SaveLoaded` / `DayStarted`：建立当前 `saveId`，刷新 NPC catalog、地图缓存、可见性缓存。
- `UpdateTicked`：处理命令队列，执行需要触碰 `Game1` / NPC / map 的操作，刷新近景感知。
- `Warped` / NPC 位置变化：更新玩家附近 NPC 可见性。
- `DayEnding` / `Saving`：暂停接收新长任务，flush 状态。
- `ReturnedToTitle`：清空 save 级 runtime 状态，bridge 进入 unavailable。

硬规则：
- SMAPI `UpdateTicked` 只处理游戏状态、缓存、命令队列和资源 claim；禁止直接调用 LLM 或触发新的 `Agent.ChatAsync(...)`。
- HTTP handler 不直接修改 `Game1`、NPC、地图、物品、事件状态。
- HTTP handler 对动作请求只入队并返回 `commandId`。
- `UpdateTicked` 执行命令前必须检查 `Context.IsWorldReady`、玩家是否自由、剧情 / 节日 / 菜单状态。
- 每条命令必须有可查询状态：`queued`、`running`、`succeeded`、`failed`、`cancelled`、`blocked`、`expired`、`stuck`。
- 命令执行必须幂等可恢复；重复轮询不能重复执行同一动作。

### 8.5.1 多 NPC 命令并发与资源占用
问题边界：
- 多个 NPC runtime 可以同时思考、同时调用观察工具、同时读聊天和社交状态。
- 但 Stardew 的物理世界是单个游戏循环，NPC / tile / 物品 / 地图对象不能让多个命令同时写。

证据对应：
- 当前 Hermes 单 Agent 内部只对白名单工具并行；这不覆盖多个 NPC runtime 同时调用 bridge 的情况。
- hermescraft 用单个 `currentTask` 限制后台任务并发。
- Stardew Squad 用 `_claimedTaskTargets` 与 `_claimedInteractionSpots` 避免多个 NPC 抢同一任务目标或交互位置。

新增组件：
- `ResourceClaimRegistry`
  - 运行在 SMAPI bridge / command executor 层。
  - 保存当前 save 内的物理资源占用。
  - 只由 `UpdateTicked` 命令执行循环修改。

最小 claim key：
- `npc:<npcId>`
- `targetNpc:<npcId>`
- `tile:<locationName>:<x>:<y>`
- `interactionTile:<locationName>:<x>:<y>`
- `object:<locationName>:<objectId>:<x>:<y>`
- `terrain:<locationName>:<x>:<y>`
- `command:<commandId>`

并发规则：
- observation 工具：`game_status`、`self_state`、`nearby`、`look`、`read_chat`、`overhear`、`social`、`marks` 可以并发读取缓存。
- social 写工具：`chat`、`complete_command` 必须按 `traceId` / `commandId` 有序写入日志，但不占用物理 tile。
- physical / long task：后续 `goto`、`interact`、`collect` 等必须先申请 claim，再进入 `running`。
- MVP 中即使 physical tool 不开放，`task_status`、`cancel_task`、命令队列也必须按这套状态机预留字段，避免后续扩展重写。

硬规则：
- 同一 `npcId` 同时只能有一个 `queued | running` 的物理动作命令。
- 同一目标 tile / interaction tile / object / terrain 同时只能被一个物理动作命令占用。
- claim 申请失败必须返回 `command_conflict`，并在 error 中给出冲突资源与占用 `commandId`；不能静默等待、不能新开第二条命令抢资源。
- claim 必须在 `succeeded`、`failed`、`cancelled`、`blocked`、`expired`、`stuck`、`DayEnding`、`Saving`、`ReturnedToTitle` 时释放。
- claim 不由 Hermes typed tool 或 CLI 本地判断；只能由 `StardewCommandService` 请求 bridge 后，以 bridge / command executor 的结果为准。
- Stardew 动作工具不得加入 Hermes `ParallelSafeTools`，除非该工具是纯观察、只读缓存，并有测试证明不会写游戏状态。

### 8.6 跨天与存档边界
跨天不是“Agent 失忆”，但也不能默认让旧物理命令继续跑。

规则：
- 跨天保留 NPC 的长期记忆、承诺、意图和社交上下文。
- 跨天不默认保留未验证的物理动作执行现场，例如昨天的目标 tile、路径、目标 NPC、目标物品、作物状态。
- `DayEnding` / `Saving` 时 bridge 进入 `draining`：停止接收新的长动作，允许短状态查询、flush 和必要取消。
- 未完成物理命令必须按类型转为 `blocked` / `expired` / `cancelled`，并写明原因，例如 `day_transition`。
- `DayStarted` 后如果 Agent 想继续昨天的意图，必须先重新观察 `game_status`、`self_state`、`nearby`、`look`，再下发新的 command。
- 旧 `commandId` 不能跨天继续执行；第二天继续做必须创建新 `commandId`，并用 trace 关联到旧意图。

这条规则只约束物理动作执行现场，不禁止 NPC 在记忆里记得“昨天答应今天继续做什么”。

## 9. 绑定与创建机制

### 目标
确保 NPC 身份、记忆目录、人格目录完全准确，不依赖主控传参，不依赖提示词，不依赖模型自我识别。

### 设计
- 调度器只负责“是否拉起 / 是否回收”
- 不负责“这个 Agent 是谁”
- `NpcRegistry` + `NpcRuntimeFactory` 共同完成系统级绑定

### 关键原则
- 被唤醒的 Agent 不是“被告诉自己是谁”
- 而是“在被创建时就已经是那个 NPC”

### Prompt 组装固定顺序
Stardew NPC runtime 必须沿用 Hermes Desktop 现有顺序：
1. `SoulContext`
2. stable `SystemPrompt`
3. `[Session State]`
4. `[Retrieved Context]`
5. recent turns
6. current user message

其中 `SoulContext` 内部顺序固定为：
1. `SOUL.md`
2. `USER.md`
3. `AGENTS.md`
4. recent mistakes
5. recent habits

硬规则：
- 禁止新增 `StardewPromptAssembler` 自己决定 prompt 顺序。
- 禁止把 Haley / Penny 人格同时写进 `SOUL.md` 和初始化 prompt，避免双份人格漂移。
- NPC 人格归各自 runtime 的 `SOUL.md`。
- Stardew 通用硬规则、工具契约、观察-思考-行动循环归稳定 `SystemPrompt` 或启用 skill prompt。
- 当前 Hermes Desktop 不自动解析 skill `triggers`，所以 Stardew 必选 skill 必须由 `NpcRuntimeBinding.EnabledSkillNames` 显式注入。

## 10. 工具接口设计

### 10.1 原则
- 桥接层只暴露能力接口
- 不提供 `scene_summary` 之类的语义摘要接口
- NPC Agent 自己调用工具理解场景
- 优先复刻 hermescraft 的工具面结构
- MVP 工具优先覆盖观察、社交、命令队列、生命周期，不优先覆盖采集 / 制作 / 存取箱子

桥接层可以返回事实压缩，例如 `status`、`look`、`nearby`、`map` 中的可见对象列表、位置、时间、天气、NPC 状态；但禁止返回导演式内容，例如：
- “Haley 现在应该安慰 Penny”
- “Penny 对玩家有点失望”
- “此时应该触发三人剧情”
- “推荐 NPC 下一步去做什么”

### 10.2 第一批工具（MVP）
- `game_status`
- `self_state`
- `nearby`
- `look`
- `read_chat`
- `chat`
- `commands`
- `complete_command`
- `overhear`
- `social`
- `task_status`
- `cancel_task`
- `mark`
- `marks`

### 10.2.1 MVP 工具 DTO 固定契约
证据对应：
- 当前 Hermes typed tool 只从 `ITool.ParametersType` 生成工具入参 schema，`ToolResult` 不会自动约束返回 DTO。
- hermescraft 的 `/status`、`/nearby`、`/look`、`/chat`、`/commands`、`/task` 等端点都有稳定返回形状。

因此 Stardew MVP 不能只固定 Bridge envelope。每个工具必须同时固定：
- request `payload` DTO
- response `data` DTO
- tool result 渲染规则
- fake bridge golden JSON

契约源：
- `StardewBridgeDtos.cs`：Bridge envelope、error、status、auth、trace、discovery DTO。
- `StardewCommandContracts.cs`：每个 command / tool 的 `payload` 与 `data` DTO。
- `Tools/Stardew/*Tool.cs`、`sdv` CLI、未来 UI 只能引用这些 DTO，不得复制 JSON shape。

| 工具 | request payload DTO | response data DTO |
| --- | --- | --- |
| `game_status` | `{}` | `{ gameId, saveId, day, season, year, timeOfDay, weather, locationName, isWorldReady, isPlayerFree, isFestival, isEventUp }` |
| `self_state` | `{ npcId }` | `{ npcId, smapiName, locationName, tile, facingDirection, isVisibleToPlayer, currentActivity, blockedReason }` |
| `nearby` | `{ npcId, radius }` | `{ originNpcId, radius, npcs, objects, terrain }` |
| `look` | `{ npcId, range }` | `{ npcId, locationName, visibleNpcs, visibleObjects, exits, hazards, rawFacts }` |
| `read_chat` | `{ npcId, sinceMessageId, limit }` | `{ messages }` |
| `chat` | `{ npcId, channel, text, targetNpcIds }` | `{ messageId, deliveredTo, overheardBy }` |
| `commands` | `{ npcId, status, limit }` | `{ commands }` |
| `complete_command` | `{ npcId, commandId, resultSummary }` | `{ commandId, status }` |
| `overhear` | `{ npcId, sinceMessageId, limit }` | `{ messages }` |
| `social` | `{ npcId }` | `{ npcId, recentEvents, relationshipFacts }` |
| `task_status` | `{ npcId, commandId }` | `{ commandId, status, progress, blockedReason, errorCode }` |
| `cancel_task` | `{ npcId, commandId, reason }` | `{ commandId, status }` |
| `mark` | `{ npcId, name, locationName, tile, note }` | `{ markId, npcId, name, locationName, tile }` |
| `marks` | `{ npcId }` | `{ marks }` |

嵌套 DTO 必须固定：
- `tile`：`{ x, y }`
- `npcs[]`：`{ npcId, smapiName, displayName, locationName, tile, distance, relationshipHint }`
- `objects[]`：`{ objectId, displayName, locationName, tile, category, state }`
- `terrain[]`：`{ kind, locationName, tile, state }`
- `messages[]`：`{ messageId, from, channel, text, createdAt, commandId, targets, hearers }`
- `commands[]`：`{ commandId, text, from, targets, status, createdAt }`
- `recentEvents[]`：`{ eventId, kind, participants, summary, createdAt }`
- `relationshipFacts[]`：`{ subjectNpcId, objectId, relationType, value, source, updatedAt }`
- `marks[]`：`{ markId, npcId, name, locationName, tile, note, createdAt }`

硬规则：
- MVP 生产代码禁止用 `Dictionary<string, object>`、匿名对象或自由 JSON 拼出工具 `payload` / `data`。
- 工具可以把 DTO 渲染成模型可读文本，但文本必须由 DTO 派生，不能绕过 DTO 另写一套结果。
- 任何出现 NPC 身份的 DTO 字段必须包含稳定 `npcId`；涉及 SMAPI 对象时同时保留 `smapiName`，并遵守 `StardewNpcCatalog`。
- `data` 禁止包含 `recommendedNextAction`、情绪判断、关系推理、剧情导演结论。
- 新增 MVP 工具必须先补 `StardewCommandContracts.cs` DTO、golden JSON、CLI/tool 同源测试，再写工具实现。

### 10.3 第二批工具（扩展）
- `goto`
- `interact`
- `collect`
- `deposit`
- `withdraw`
- `craft`
- `eat`
- 其他 Stardew 特化工具

### 10.4 Stardew 行为工具门槛
`goto` / `interact` / `collect` 不进入 MVP，除非先定义完整 Stardew-native 任务语义：
- 目标类型：NPC、物品、作物、树、矿石、怪物、家具、地块等。
- 目标 tile 与交互 tile。
- 路径可达性与占位冲突。
- 节日、剧情、菜单、玩家不可控状态下的让位。
- 动作动画、结果确认、失败原因。
- 任务状态轮询与取消。

参考 Stardew Squad 的任务类型说明：Stardew 行为不是一个泛化 `collect` 能覆盖的动作集合，而是 `Watering`、`Harvesting`、`Foraging`、`Fishing`、`Petting` 等具体任务。

## 11. 通信方式设计

### 方案对比
- **HTTP**：hermescraft 已验证，隔离性好，易调试
- **命名管道 / 进程通信**：更快，但实现和调试复杂度更高
- **进程内共享库**：延迟最低，但耦合最强

### 选型结论
**MVP 选 HTTP**。

### CLI / Tool / UI 单一命令源
可以提供 `sdv` CLI，但必须满足：
- `sdv` CLI 只做参数解析、展示结果、返回退出码。
- Hermes typed tools 只做 tool 参数 schema 与模型可读结果适配。
- 未来 UI 面板只做界面交互。
- 三者都调用同一个 `StardewCommandService`。

禁止：
- CLI 直接调用 SMAPI HTTP。
- typed tool 直接调用 SMAPI HTTP。
- UI 直接调用 SMAPI HTTP。
- CLI 和 typed tool 各自维护一套命令列表、结果格式、缓存、错误码。

### Bridge API 固定契约
所有 Bridge API 必须使用统一 envelope，禁止各层自由发明字段。

请求 envelope：

```json
{
  "traceId": "string",
  "requestId": "string",
  "idempotencyKey": "string",
  "gameId": "string",
  "saveId": "string",
  "npcId": "string",
  "profileId": "string",
  "commandType": "string",
  "payload": {}
}
```

HTTP 认证 header：

```http
Authorization: Bearer <bridgeToken>
```

响应 envelope：

```json
{
  "ok": true,
  "traceId": "string",
  "requestId": "string",
  "commandId": "string",
  "status": "queued | running | succeeded | failed | cancelled | blocked | expired | stuck",
  "data": {},
  "error": null,
  "state": {}
}
```

错误 envelope：

```json
{
  "ok": false,
  "traceId": "string",
  "requestId": "string",
  "commandId": "string",
  "status": "failed | blocked | expired | stuck",
  "data": null,
  "error": {
    "code": "string",
    "message": "string",
    "retryable": false
  },
  "state": {}
}
```

固定错误码：
- `bridge_unavailable`
- `world_not_ready`
- `player_not_free`
- `festival_blocked`
- `cutscene_blocked`
- `menu_blocked`
- `day_transition`
- `invalid_target`
- `invalid_state`
- `command_conflict`
- `command_not_found`
- `command_expired`
- `command_stuck`
- `idempotency_conflict`
- `bridge_unauthorized`
- `bridge_stale_discovery`

硬规则：
- `StardewBridgeDtos.cs` 与 `StardewCommandContracts.cs` 是契约源。
- typed tools、`sdv` CLI、未来 UI 只能引用契约源，不能复制 DTO。
- `commandId` 由 `StardewCommandService` / bridge contract 统一生成或登记，不能由各 adapter 自己生成。
- 同一个 `idempotencyKey` 重放时不得创建第二条物理动作命令。
- observation 类 GET 可以无 `commandId`，但仍必须有 `traceId` / `requestId`。
- 除 `/health` 外，Bridge API 必须校验 `Authorization: Bearer <bridgeToken>`。
- `/health` 允许无 token，但只能返回 alive / version / minimal readiness，不返回 saveId、NPC 状态、命令状态或其他敏感运行时细节。

### 端口冲突处理
- 支持配置默认端口
- 支持端口范围扫描
- 支持自动向后寻找可用端口
- SMAPI mod 把当前绑定端口写入发现文件
- Hermes Desktop 启动时读取发现文件连接

### Bridge 本机安全边界
这是轻量本机防误调用，不是账号 / 权限系统。

硬规则：
- MVP 默认绑定 `127.0.0.1`，禁止默认绑定 `0.0.0.0`。
- LAN 暴露不进入 MVP；后续如需 LAN，必须单独配置开关、单独验收。
- SMAPI bridge 每次启动生成随机 `bridgeToken`，写入 `bridge-port.json`。
- `bridge-port.json` 至少包含 `host`、`port`、`bridgeToken`、`pid`、`saveId`、`startedAt`、`modVersion`；`GameLaunched` 阶段 `saveId` 可以为空，`SaveLoaded` / `DayStarted` 后必须更新为当前存档身份。
- Hermes Desktop 读取发现文件后必须校验 host 是 loopback、token 存在、端口可连、发现文件不是陈旧状态。
- 发现文件缺 token、host 非 loopback、进入存档后 saveId 与当前目标存档不匹配、端口 health 不匹配时，必须拒绝连接并返回 `bridge_stale_discovery` 或 `bridge_unauthorized`。
- token 只能防误调用、旧进程串线和普通网页 / 脚本误打端口；不把它描述成本机恶意程序防护。

### 结构化可观测性与调试
MVP 即使不做完整 UI，也必须留下可查询的结构化证据。

必须记录：
- 每个 NPC runtime 独立 transcript：模型对话、system messages 来源、当前 user message。
- 每个 NPC runtime 独立 activity：tool call、参数摘要、结果摘要、耗时、状态。
- 每个 bridge request：`traceId`、`requestId`、HTTP method、endpoint、status code、error code。
- 每个 command：`commandId`、`npcId`、`saveId`、`profileId`、状态变更、执行 tick、失败原因。
- 每次 prompt 组装：最终 messages 顺序、启用 skill 列表、`SOUL.md` 来源路径。

MVP 调试入口：
- `sdv debug runtime <npcId>`：查看 runtime 状态、路径、最近 activity。
- `sdv debug command <commandId>`：查看 command 状态、bridge 结果、SMAPI 执行记录。
- `sdv debug prompt <npcId>`：导出该 NPC 最近一次最终 messages，用于验证 prompt 顺序和 skill 注入。
- `sdv debug trace <traceId>`：串起 typed tool / command service / bridge / SMAPI 执行记录。

硬规则：
- 不允许只依赖普通文本日志判断多 NPC 正确性。
- trace 文件必须是 JSONL 或等价结构化格式，便于测试断言。
- `traceId` 必须能跨越 Hermes tool call、`StardewCommandService`、bridge HTTP、SMAPI command queue、`UpdateTicked` 执行结果。
- 模式 B UI 可以后置，但 MVP 的 CLI / 文件化调试能力不能后置。

## 12. 社交与群聊设计

### 12.1 参考 mod 已实现的场景功能
必须参考并映射以下六大能力：
1. 场景感知
2. 社交感知
3. 群聊路由
4. 命令队列
5. 后台任务
6. 地标记忆

### 12.2 Stardew 对应设计
- 完整设计目标：玩家在场 / 不在场都允许 NPC 群聊
- MVP：先只实现**玩家在场群聊**
- 群聊建立在“感知范围内 NPC 可互相看见 / 听见 / 交互”的基础上
- 不通过调度器转述场景，由 NPC 自己感知和决定是否参与群聊

### 12.3 SocialRouter 消息 envelope
所有社交消息进入 Hermes 前必须被包装为 envelope：

```json
{
  "messageId": "string",
  "channel": "public | direct | group | overheard | system_event",
  "from": "player | npcId | system",
  "targets": ["npcId"],
  "hearers": ["npcId"],
  "locationId": "string",
  "saveId": "string",
  "text": "string",
  "createdAt": "iso-8601"
}
```

### 12.4 路由规则
- `public`：投递给可听见 NPC 的 inbox。
- `direct`：只投递给目标 NPC 的 commandQueue；附近 NPC 可进入 overhear。
- `group`：只投递给被点名 NPC 的 commandQueue；附近非目标 NPC 可进入 overhear。
- `overheard`：只作为旁听事实，不可被当作对自己的命令。
- `system_event`：用于节日、剧情、日夜切换、bridge 状态等事实通知。

硬规则：
- SocialRouter 不负责“让谁应该说话”。
- SocialRouter 不负责“判断谁喜欢谁”。
- SocialRouter 不负责“决定 NPC 应不应该吃醋 / 生气 / 解释”。
- 每个 commandQueue 项必须有 `commandId`。
- NPC 完成点名命令后必须调用 `complete_command(commandId)`，避免下轮重复处理同一命令。

- 两人差异大，更容易验证人格隔离
- 两人都属于村庄核心 NPC，利于在公共场景测试群聊

### 16.2 生命周期测试

- 持续停留不会启动第二个 `NpcAutonomyLoop`
- 离开视野回收

- `GameLaunched` 写入 bridge port 发现文件
- `DayStarted` / `SaveLoaded` 建立 save 级缓存
- `UpdateTicked` 处理命令队列
- `ReturnedToTitle` 清空 save runtime 状态

补充硬验收：

- CLI 与 typed tools 对同一 fake bridge 输入应产生一致请求与一致结果模型。
- 每个 MVP 工具都必须有 fake bridge golden JSON，覆盖 request `payload`、response `data`、error envelope。

- Bridge request / response / error envelope 必须符合 `StardewBridgeDtos.cs` / `StardewCommandContracts.cs`。
- 同一 `idempotencyKey` 重放不会产生第二条物理动作命令。
- 错误码只能使用固定枚举，不能返回自由文本当作错误类型。
- Bridge 默认绑定 `127.0.0.1`，测试必须证明不会绑定 `0.0.0.0`。
- 除 `/health` 外，无 `Authorization: Bearer <bridgeToken>` 的请求必须返回 `bridge_unauthorized`。
- `bridge-port.json` 缺 token、host 非 loopback、目标存档已知后 saveId 不匹配或超过发现文件最大年龄时，Hermes Desktop 必须拒绝连接。
- `/health` 无 token 时只能返回最小健康信息，不能泄露 command / NPC / save runtime 状态。

### 16.8 Prompt 与 skill 注入测试
- 单 NPC runtime 的最终 messages 顺序符合 `SoulContext -> SystemPrompt -> Session State -> Retrieved Context -> recent turns -> current user`。
- `SOUL.md` 中的人格不会被初始化 prompt 重复覆盖。
- `EnabledSkillNames` 中的 Stardew skill prompt 会进入稳定系统上下文。
- 未启用 skill 不会进入该 NPC runtime。
- 不依赖 `triggers` 自动触发核心规则。
- Haley runtime 只加载 `haley/default/SOUL.md` 初始化出的 profile `SOUL.md`，不会加载 Penny 的人格内容。
- Penny runtime 只加载 `penny/default/SOUL.md` 初始化出的 profile `SOUL.md`，不会加载 Haley 的人格内容。
- seed pack 缺 `SOUL.md`、`facts.md`、`voice.md`、`boundaries.md` 或 `skills.json` 时，runtime factory 必须失败并给出明确错误，不能临场补写。
- prompt preview 必须证明人格只出现在 `SoulContext` 层，不出现在初始化 prompt / tool prompt / command prompt 的重复副本中。

## 17. 风险与缓解

### 风险 1：端口冲突
缓解：端口范围 + 自动回退 + 发现文件

### 风险 2：身份绑定错误
缓解：注册表 + 工厂注入，禁止靠提示词绑定

### 风险 3：多 NPC 记忆 / 人格串写
缓解：每 NPC 独立 `memoryDir` / `soulScope` / `sessionNamespace`

### 风险 4：桥接层做太多语义转述，长出双轨
缓解：桥接层只暴露能力接口，不提供语义摘要

### 风险 5：群聊复杂度过早膨胀
缓解：完整设计保留，MVP 只实现玩家在场群聊

### 风险 6：UI 集成过早导致范围失控
缓解：模式 A 先行，模式 B 后置

### 风险 7：CLI / tool / UI 各写一套命令逻辑
缓解：写死 `StardewCommandService` 为唯一命令服务源；CLI、typed tools、UI 只做 adapter；测试必须证明同源。

### 风险 8：Prompt 顺序被实现阶段重写
缓解：沿用 `ContextManager` / `PromptBuilder` 现有顺序；禁止新增独立 Stardew prompt assembler；用 golden test 固定最终 messages 顺序。

### 风险 9：SocialRouter 演变成剧情导演
缓解：SocialRouter 只投递 envelope；关系、情绪、是否回应由各 NPC Agent 自己决定。

### 风险 10：跨存档 / 跨人格污染
缓解：runtime 路径必须包含 `gameId + saveId + npcId + profileId`；MVP 也保留 `profileId = default`。

### 风险 11：HTTP 线程直接改 Stardew 游戏对象
缓解：HTTP handler 只入队；`UpdateTicked` 在游戏循环内检查状态后执行。

### 风险 12：照搬 Minecraft 动作工具
缓解：MVP 不引入 `collect` 等 Minecraft 风格动作；行为扩展阶段必须先定义 Stardew-native 任务语义。

### 风险 13：引用证据不可复现
缓解：证据索引只能引用当前仓库存在的文件和行号；zip 内证据必须先恢复到可引用路径再写入行号。

### 风险 14：只抄参考 mod 表面流程，忽略当前 Hermes 代码边界
缓解：新增三方落地矩阵；每层必须同时写参考实现、Hermes 当前接入点、Stardew 映射规则。

### 风险 15：Bridge API 契约自由发挥，API 层长出双轨
缓解：`StardewBridgeDtos.cs` 与 `StardewCommandContracts.cs` 是唯一契约源；CLI、typed tools、UI 只能引用，不得复制。

### 风险 16：跨天任务盲目续跑，导致旧地图 / 旧目标 / 旧路径污染新一天
缓解：跨天只保留记忆和意图；物理命令必须 `blocked` / `expired` / `cancelled`，第二天重新观察后新建 command。

### 风险 17：多 NPC 出错后无法定位
缓解：MVP 必须提供 per-NPC transcript / activity / command trace / prompt preview；没有结构化 trace 不能算通过。

### 风险 18：新增 `.claude/stardew.json` 导致配置双轨
缓解：Stardew 配置只能落入现有 `HERMES_HOME/config.yaml` 的 `stardew:` section；实现如需 `StardewOptions` / `StardewConfig` 类型，也只能解析这个 section，不能新增项目内 JSON 配置文件。

### 风险 19：本机 HTTP bridge 被旧进程、网页或脚本误调用
缓解：MVP 默认只绑定 `127.0.0.1`；除 `/health` 外所有请求必须带启动生成的 `bridgeToken`；发现文件缺 token、过期、host 非 loopback 或 saveId 不匹配时拒绝连接。

### 风险 20：实现 AI 临场生成 Haley / Penny 人格，导致人格漂移
缓解：MVP 必须提交 Haley / Penny 固定 persona seed pack；runtime 只能从 seed pack 初始化对应 profile `SOUL.md`；prompt preview / golden test 证明人格未被重复注入或串 NPC。

### 风险 21：Haley / 海莉 / haley / SMAPI `npc.Name` 混用导致路由和存档错乱
缓解：`StardewNpcCatalog` 是唯一身份映射源；路径只用 `npcId`，SMAPI 只用 `smapiName`，玩家输入只经 `aliases` 归一化；未知或歧义名字返回 `invalid_target`。

### 风险 22：MVP 工具 data 自由发挥，CLI / typed tools / UI 形成隐形双轨
缓解：每个 MVP 工具必须固定 request `payload` DTO 与 response `data` DTO；`StardewBridgeDtos.cs` / `StardewCommandContracts.cs` 是唯一契约源，fake bridge golden test 必须证明 CLI、typed tools、未来 UI 使用同一 DTO。

### 风险 23：多个 NPC 同时抢同一 tile / 目标对象，导致路径、动画、任务状态互相覆盖
缓解：物理动作命令必须经 `ResourceClaimRegistry` 申请 `npcId`、target tile、interaction tile、object / terrain claim；冲突返回 `command_conflict`，命令终止和生命周期切换时释放 claim。

### 风险 24：把 NPC 自主性做成事件触发的一次性问答，偏离 hermescraft 常驻 agent 模式
缓解：活跃 NPC 必须运行受控 `NpcAutonomyLoop`；事件只进入 inbox / priority queue，不能替代常驻观察-思考-行动循环。

### 风险 25：照搬 hermescraft 常驻循环但没有预算，导致模型调用失控
缓解：`NpcAutonomyBudget` 必须在代码层限制 max tool iterations、并发 LLM request、重启次数和 start stagger；超预算进入 `paused` / `blocked` 并写 trace。

## 22. 配置与数据文件布局设计

### Hermes Desktop 侧配置
配置必须写入现有 Hermes Desktop 配置文件：

```text
<HERMES_HOME>/config.yaml
```

当前代码依据：
- `Desktop/HermesDesktop/Services/HermesEnvironment.cs:23-28` 固定 `HermesConfigPath = Path.Combine(HermesHomePath, "config.yaml")`。
- `Desktop/HermesDesktop/Services/HermesEnvironment.cs:769-809` 当前通用读写接口按顶层 section 读取 / 写入。
- `Desktop/HermesDesktop/Services/HermesEnvironment.cs:826-875` 当前 section 写入是扁平 key-value，因此 Stardew 配置也必须先按扁平 section 设计。

示例 section：

```yaml
stardew:
  bridge_default_host: 127.0.0.1
  bridge_default_port: 8745
  bridge_port_range: 8745-8765
  bridge_allow_lan: false
  bridge_token_source: generated_runtime
  bridge_discovery_max_age_seconds: 30
  bridge_discovery_file: mods/StardewHermesBridge/runtime/bridge-port.json
  npc_catalog_source: bundled
  npc_default_cast: haley,penny
  npc_max_active_scene_agents: 3
  npc_default_profile_id: default
  npc_autonomy_max_tool_iterations: 100
  npc_autonomy_max_concurrent_llm_requests: 1
  npc_autonomy_restart_cooldown_seconds: 5
  npc_autonomy_max_restarts_per_scene: 3
  npc_autonomy_start_stagger_seconds: 4
  skills_required: stardew-core,stardew-social
  skills_optional: stardew-farming,stardew-navigation
  mvp_enable_player_present_group_chat_only: true
  mvp_enable_offscreen_npc_chat: false
```

硬规则：
- 禁止新增 `.claude/stardew.json`。
- 禁止新增项目目录内 Stardew 专用配置文件作为主配置源。
- `StardewOptions` / `StardewConfig` 如需新增，只能从 `HermesEnvironment.HermesConfigPath` 的 `stardew:` section 读取。
- `bridge-port.json` 是 SMAPI bridge 的运行时发现文件，不是 Hermes 配置源。
- `bridge_allow_lan` MVP 必须保持 `false`；实现不能因为配置存在就开放 LAN 行为。
- `bridge_token_source` MVP 只能是 `generated_runtime`，不读取用户手写固定 token。
- `npc_default_cast` 只能写 `npcId`，不能写显示名、中文名或 SMAPI 内部名。
- 配置放在 `config.yaml`；运行时状态、trace、prompt preview、NPC profile 数据放在 `HermesHomePath/runtime/stardew/`。
