## RALPLAN-DR

### 原则

1. 持久化的 `todo` 工具结果 JSON 是任务事实的唯一权威来源。
2. 任何任务归档 / 历史表面都必须是派生的、只读的、可重建的，并且绝不能回流到提示注入，也不能修改活跃任务。
3. 保留参考语义：todo 顺序就是优先级；只有 active / in-progress todo 会被重新注入；已完成 / 已取消的任务不会作为活跃工作重新出现；不完整归档仍然必须可见，并标明不完整。
4. 纯 UI 形态 / 样式差异是允许的，作为标题本地化适配：WinUI 布局、颜色、图标和控件不需要与 TUI 参考实现完全一致。
5. 范围保持为 desktop-only，避免过早耦合 `TaskManager` / Coordinator。

### 主要决策驱动

1. 当前桌面实现已经在工具执行、会话作用域、transcript 投影、历史装载和 active-task 提示注入方面基本对齐。
2. 剩余的语义差距包括模型可见的 schema 指引、缺失 id / content 的测试覆盖、实时面板的优先级顺序，以及用户可检查的按轮次任务历史等价物。
3. 由于存在重复持久化路径，任何归档 / 历史读模型都必须是幂等的，并且不能变成第二个事实来源。

### 可行方案

#### 方案 A：消除可避免的语义偏差

实现缺失的语义部分：更完整的 schema 指引、按源顺序渲染面板，以及派生的任务历史 / 归档读模型。

优点：
- 与参考效果最接近。
- 避免在核心语义上留下待批准的受控偏差。
- 保持参考权威边界。

缺点：
- 需要新增归档 / 历史派生测试和一个较小的桌面表面。

#### 方案 B：保持当前 UI 行为并延后归档

保留 active-first 的面板排序，仅依赖持久化的工具 JSON 和实时面板。

优点：
- 补丁更小。

缺点：
- 无法诚实地声称达成同等效果，因为缺少参考中的顺序即优先级和按轮次可检查性。
- 需要明确的受控偏差批准。

#### 方案 C：把 todos 桥接到 `TaskManager`

将会话 todos 映射到持久化的桌面任务基础设施。

优点：
- 复用现有的持久化任务概念。

缺点：
- 会偏离参考的会话级 todo 语义。
- 过早地把普通聊天规划和 Coordinator 时代的任务生命周期耦合在一起。

### 选定方案

方案 A。

## 参考链

| 层 | 参考行为 | 桌面映射 |
| --- | --- | --- |
| 触发 | 模型调用单一 `todo` 工具；省略 `todos` 即为读取。 | `TodoTool` 面向模型；`todo_write` 是隐藏的兼容别名。 |
| 快照 | 每个 agent/session 只有一个有序的 todo 存储。 | 以桌面 session id 为键的 `SessionTodoStore`。 |
| 提示 / 摘要组装 | 仅在压缩后注入 pending / in_progress todo。 | `ContextManager` -> `PromptBuilder.ActiveTaskContext`。 |
| 解析 / 规范化 | 缺失 id -> `?`；缺失 content -> `(no description)`；无效 status -> `pending`；重复 id 保留最后一次出现。 | `SessionTodoStore.Validate` 和 `DeduplicateById`。 |
| 执行器 | `todo_tool` 修改存储并返回完整列表加摘要。 | `TodoTool.ExecuteAsync`。 |
| 权威写回 | 完整的 `tool.complete` 结果是事实来源；tool-start 参数不是。 | 通过 `SessionTaskProjectionService` 观察持久化 transcript 工具消息。 |
| 玩家可见层 | 实时 todo 面板加按轮次的 transcript 轨迹。 | WinUI 实时任务面板加派生的任务历史 / 归档表面。 |

## 实施计划

### 提交 1：用测试锁定参考语义

- 为缺失 id / content 的规范化补充测试。
- 为参考行为指引补充 schema 指引一致性测试。
- 增加面板源顺序测试。
- 在实现前增加归档派生测试：
  - 重复持久化只生成一行归档
  - 从持久化的工具消息装载归档
  - 忽略格式错误的 todo 消息
  - 仅完成的归档折叠 / 从活跃状态清除
  - 不完整归档被标记为不完整

### 提交 2：收紧 `todo` 工具契约

- 扩展 `TodoTool.Description`，纳入与参考等价的行为指引：
  - 用于复杂的 3 步以上或多任务工作
  - 省略参数即可读取
  - 替换 / 合并语义
  - 列表顺序就是优先级
  - 只能有一个 `in_progress` 项
  - 完成后立即标记
  - 失败项要取消并添加修订项
  - 每次调用都返回完整的当前列表
- 继续把 `todo_write` 仅作为隐藏的旧别名保留为可执行状态。

### 提交 3：在实时面板中保留优先级顺序

- 修改 `SessionTaskPanelModel`，使其渲染权威快照顺序。
- 除非之后批准为受控语义偏差，否则不要把 active 项排在 done 项之前。
- 纯标签 / 颜色 / 布局仍作为桌面本地 UI 选择。

### 提交 4：添加派生的任务历史读模型

- 增加一个扫描 transcript 消息的归档 / 历史派生服务。
- 规范性事实仍然是已持久化的 `todo` 工具 JSON。
- 归档是只读且可重建的。
- 不要把 `SessionTaskProjectionService.HydrateSessionAsync` 用于归档构建，因为它会修改 `SessionTodoStore`。
- 归档输出要与 `FormatActiveTasksForInjection` 分开。

### 提交 5：幂等性与桌面表面

- 在每个 assistant completion 边界恰好归档一次。
- 若可用，首选键：`sessionId + assistantMessageId`。
- 当前回退键：`sessionId + latest todo tool-call/content hash + assistant completion boundary hash`。
- 去重重复的 todo 工具结果和重复的 assistant 边界。
- 将派生归档暴露为 transcript/history 行或 task-history 区段：
  - 仅完成 / 已取消的行要折叠或标记为不活跃
  - pending / in_progress 行要明显标记为未完成
  - 保留源顺序

### 提交 6：验证与清理

- 运行 todo 工具、投影、面板模型、归档派生、聊天服务生命周期和上下文注入的针对性测试。
- 运行完整的桌面测试项目。
- 使用 Lore trailers 为每个小而已验证的补丁提交。

## 验收标准

1. 面向模型的工具定义暴露 `todo`，而不是 `todo_write`。
2. `todo` 的读 / 写 / 合并都会返回完整的当前列表加摘要。
3. 缺失 id / content 和无效 status 的规范化方式与参考一致。
4. 实时任务面板保留 todo 源顺序作为优先级顺序。
5. completed / cancelled-only 的 todo 不会被重新注入到提示上下文中。
6. 工具完成会仅根据持久化的工具结果 JSON 更新实时状态。
7. 会话重新加载时，装载的是最新有效的 `todo` 结果，而不是参数或摘要。
8. 任务历史 / 归档可以从 transcript 数据中删除并重建。
9. 重复持久化只产生一行归档。
10. 格式错误的 todo 消息不会导致 replay 崩溃，也不会修改活跃状态。
11. 不完整的归档快照会被标记为不完整。
12. 纯 WinUI 视觉差异是可接受的，不会阻止对齐。

## ADR 草案

### 标题

采用派生的、忠实于参考实现的桌面 todo 历史，以实现 session 任务对齐

### 状态

已提出；已获得共识批准用于实施。

### 背景

参考项目使用的是 session 作用域的 `todo` 存储、完整列表的工具结果、仅活跃项的提示重注入、实时 todo 面板更新，以及按轮次的 transcript todo 归档。桌面项目现在已经对齐了大部分工具和投影语义，但仍需要源顺序渲染、更完整的工具指引，以及一个桌面版的按轮次可检查性等价物。

### 决策

将持久化的 `todo` 工具结果 JSON 作为唯一的任务规范来源。增加一个派生的、可重建的桌面 task-history 读模型，用于玩家可见的归档语义。保留 live 和 archived 视图中的源顺序。接受 WinUI 的视觉差异作为标题本地化适配。

### 备选方案

- 直接映射到 `TaskManager`：被拒绝，因为这会改变 session todo 语义。
- 延后归档：除非明确批准，否则被拒绝，因为这会阻止达成同等效果。
- 将归档持久化为规范状态：被拒绝，因为这会制造第二个事实来源。

### 结果

正面：
- 无需复制 TUI 运行时即可达到与参考实现效果一致的对齐。
- 权威边界清晰。
- 更利于回放 / 重载时对任务进度做诊断。

负面：
- 需要新增一个派生读模型和测试。
- 需要围绕重复消息持久化处理幂等性。

### 后续事项

- 在完成对齐加固后，再决定是否应将持久化的 `TaskManager` 任务作为独立分组视图与 session todos 一起展示。
- 如果 `Message` 之后获得稳定 id，则把归档键迁移为 `sessionId + assistantMessageId`。
