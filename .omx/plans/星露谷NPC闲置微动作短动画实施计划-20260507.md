# 星露谷 NPC 闲置微动作短动画实施计划

## RALPLAN-DR Summary

### Principles

1. 单一路径原则：闲置微动作只走一条固定接线，不保留 bridge 随机兜底或 selector 双轨。
2. 宿主决策边界清晰：parent 只决定是否尝试 `idle_micro_action` 与其轻量语义，本地小模型只在固定枚举内做选择，bridge 只执行与回报，不代决策、不随机补洞。
3. 可见性优先：只在玩家可见且 NPC 处于 idle 时触发，避免对不可见世界做“假活跃”。
4. 轻微且有意义：微动作必须轻微表达 NPC 当前心理/任务，不是纯随机装饰。
5. 证据闭环：`displayed/skipped/blocked/interrupted` 必须统一沉到 bridge 返回、runtime log、bridge log。

### Drivers

1. 用户明确要求微动作表达当前心理/任务，而不是无语义随机抖动。
2. 现有项目已具备 parent/local executor/bridge/overlay/log/test 基础设施，应该复用而不是另起系统。
3. 当前 `private_chat_reply_closed` 与 overlay 事件存在交叉，必须借本次合约升级彻底收束 channel 边界。

### Options

1. 纯 bridge 随机 idle 表现。
   - 结论：否决。违反“本地小模型只从固定动作枚举中选择”和“bridge 不代决策、不随机兜底”。
2. parent 直接输出原始动画帧或 overlay 指令。
   - 结论：否决。违反 schema 白名单与 host/bridge 分层。
3. parent 决定尝试 idle，local executor 在固定枚举中选动作，host/bridge 做安全门与执行。
   - 结论：采纳。满足边界、复用现有基础设施、风险最低。

## 需求摘要

为 Stardew NPC 增加“闲置微动作”能力，在不改变日程、不生成文本、不进入私聊、不移动 NPC 的前提下，让玩家可见范围内处于 idle 的 NPC 偶发执行轻微、有语义的表情、转头、小幅身体动作或短 idle animation。

新增能力必须满足以下硬约束：

1. 微动作要轻微表达 NPC 当前心理/任务，不是纯随机装饰。
2. 只在玩家可见且 NPC idle 时触发。
3. 本地小模型只从固定动作枚举中选择。
4. 充分利用当前项目已有基础设施。
5. 不允许双轨；不要 bridge 规则随机 / local selector 两套路径。
6. 单一路径写死：
   `NpcLocalActionIntent.IdleMicroAction -> NpcLocalExecutorRunner -> stardew_idle_micro_action -> GameActionType.IdleMicroAction -> StardewCommandService -> StardewBridgeRoutes.ActionIdleMicroAction -> BridgeHttpHost -> BridgeCommandQueue -> overlay/body animation -> displayed/skipped/blocked/interrupted`
7. tool 不得直接触碰 overlay；host 不得旁路直调 bridge 内部对象；bridge 不代决策、不随机兜底。
8. `NpcLocalActionIntent.IdleMicroAction` 使用白名单 schema；`speech/destination/target/frame/rawAnimationId` 等字段不属于 schema。
9. overlay 合约升级为显式 `BubbleChannel/BubbleKind`；`private_chat_reply_closed` 只能由 `private_chat` channel 产生，idle 永不触发。
10. `displayed/skipped/blocked/interrupted` 统一沉到 bridge 返回、runtime log、bridge log。
11. `idle_animation_once` allowlist 使用结构化配置：`alias`、最大时长、是否 loop、恢复帧 owner；禁止 `raw frame/frameIndex/rawAnimationId`。
12. interrupt cleanup 必须覆盖：NPC 开始移动、warp/remove、menu/cutscene/event、玩家离开可见范围、新 idle 命令覆盖旧命令、runtime drain/clear。
13. 测试必须补 route/DTO 对齐和 `BridgeCommandQueue` 生命周期回归。

## 非目标

1. 不发 `stardew_speak`，不生成文字气泡，不做碎碎念。
2. 不进入 `private_chat`、手机线程、`open_private_chat` 流程。
3. 不改变 tile、不做 wander、不做跨地图导航。
4. 不让 parent 或 local model 直接控制 overlay 实现细节。
5. 不做第二条 bridge-only idle fallback 路径。
6. 不做复杂情绪系统、长期 persona 权重系统或通用动画编辑器。
7. 不开放任意原始动画 key、帧号、frame index、raw animation id。

## 唯一接线矩阵

| 层 | 唯一入口/出口 | 责任 | 禁止 |
|---|---|---|---|
| Parent autonomy | `NpcAutonomyLoop` 产出 `action=idle_micro_action` | 判断当前是否值得尝试一个轻微 idle 微动作，并提供与当前心理/任务相关的简短 reason | 不输出 overlay 字段、不输出动画帧、不直接调 tool |
| Intent contract | `NpcLocalActionIntent.IdleMicroAction` | 白名单解析：只接受固定字段与固定动作枚举 | 不接受 `speech/destination/target/frame/rawAnimationId` |
| Local executor | `NpcLocalExecutorRunner` | 仅向本地小模型暴露 `stardew_idle_micro_action` 单工具，要求只从固定枚举选择 | 不暴露 speak/private chat/move/overlay 相关工具 |
| Executor tool | `stardew_idle_micro_action` | 结构化提交 idle 微动作请求 | 不直接访问 overlay 或 bridge 内部对象 |
| Host action | `GameActionType.IdleMicroAction`，实际位置 `src/game/core/GameAction.cs` | 将工具调用变成标准游戏动作 | 不绕过 `StardewCommandService` |
| Command service | `StardewCommandService.SubmitIdleMicroActionAsync` | 构造 DTO/envelope 并发往 bridge route | 不做 bridge 内部执行逻辑 |
| Route constant | `StardewBridgeRoutes.ActionIdleMicroAction`，实际位置 `src/games/stardew/StardewCommandContracts.cs` | 唯一路由名 | 不复用 `ActionSpeak`/`ActionOpenPrivateChat` |
| HTTP host | `BridgeHttpHost.HandleIdleMicroActionAsync` | 只做 DTO 反序列化与入队调用 | 不直接操作 overlay/body animation |
| Bridge execution | `BridgeCommandQueue` | 安全门、冷却、生命周期、interrupt cleanup、结果归一 | 不随机选动作、不兜底换动作 |
| UI execution | `NpcOverheadBubbleOverlay` / body animation hook | 按 `BubbleChannel/BubbleKind` 显示或执行 | 不再以私聊关闭事件混用 idle channel |
| Result sink | bridge response + runtime log + bridge log | 统一记录 `displayed/skipped/blocked/interrupted` | 不出现只在其中一层可见的状态 |

## 原子落盘要求

实现时必须把以下改动作为同一批原子链路落盘，不允许只改其中一段后宣称完成：

1. `NpcLocalActionIntent`
2. `NpcLocalExecutorRunner`
3. local executor tool surface
4. `GameActionType`
5. `StardewBridgeRoutes` 常量
6. `StardewCommandService`
7. Desktop DTO
8. bridge DTO
9. `BridgeHttpHost`
10. `BridgeCommandQueue`
11. overlay channel/kind
12. 对应回归测试

原因：当前代码结构下，如果只补局部，容易出现“编译通过，但链路中段断掉”的假完成。

## 动作与 Schema 设计

### 固定动作枚举

1. `emote_happy`
2. `emote_question`
3. `emote_sleepy`
4. `emote_music`
5. `look_left`
6. `look_right`
7. `look_up`
8. `look_down`
9. `look_around`
10. `tiny_hop`
11. `tiny_shake`
12. `idle_pose`
13. `idle_animation_once`

### 语义约束

1. parent 的 `reason` 必须体现当前心理/任务，例如“等待玩家回应”“思考接下来去哪”“短暂放松一下”，而不是“随便做个动作”。
2. local model 只能在固定枚举中做选择，不能生成新动作名。
3. `look_around/tiny_hop/tiny_shake/idle_pose/idle_animation_once` 都必须保持轻微，不得演变成走位或强戏剧化表演。

### `NpcLocalActionIntent.IdleMicroAction` 白名单字段

1. `action`
2. `reason`
3. `idleMicroAction.kind`
4. `idleMicroAction.animationAlias`
5. `idleMicroAction.intensity`
6. `idleMicroAction.ttlSeconds`

### 明确禁止字段

1. `speech`
2. `destinationId`
3. `target`
4. `text`
5. `message`
6. `dialogue`
7. `frame`
8. `frameIndex`
9. `rawAnimationId`
10. `locationName`
11. `x`
12. `y`

### `idle_animation_once` allowlist 配置

每个条目必须是结构化配置，而不是原始帧或原始动画 id：

1. `alias`
2. `maxDurationMs`
3. `allowLoop`
4. `restoreFrameOwner`

首版建议 alias 仅允许少量人工审核项，例如：

1. `idle_tinker`
2. `idle_clean`
3. `idle_stretch`

## 结果编码

不要新增 `status=skipped`。结果编码必须兼容现有 `StardewCommandStatuses` 终态体系，避免连带改动 `IsTerminalStatus(...)`、状态映射、日志解析和既有测试。

统一编码方式：

1. `displayed` -> `status=completed`, `result=displayed`
2. `skipped` -> `status=completed`, `result=skipped`, `reasonCode=cooldown|not_visible|menu_blocked|...`
3. `blocked` -> `status=blocked`, `result=blocked`, `reasonCode=unsupported_action|animation_not_allowed|...`
4. `interrupted` -> `status=interrupted`, `result=interrupted`, `reasonCode=move_started|warp|event|...`

bridge response、runtime log、bridge log 必须使用同一套 `result/reasonCode` vocabulary。

## Overlay / Bubble 合约升级

### 新合约

将现有 bubble/phone/private-chat 混用语义升级为显式二元组：

1. `BubbleChannel`
   - `player_dialogue`
   - `private_chat`
   - `move_thought`
   - `idle_micro`
2. `BubbleKind`
   - `text`
   - `emote`
   - `look`
   - `fidget`
   - `idle_animation`

### 强制边界

1. `private_chat_reply_closed` 只能由 `BubbleChannel=private_chat` 产生。
2. `idle_micro` 永不触发 `private_chat_reply_closed`。
3. idle 微动作的 overlay/log 不能再复用现有 `private_chat` 或 `move_thought` 语义字段偷渡。

## 验收标准

1. parent 只在 NPC idle 且玩家可见时，才可能产出 `action=idle_micro_action`。
2. 微动作 reason 能反映当前心理/任务，不是随机 filler。
3. local executor 对 `IdleMicroAction` 只暴露 `stardew_idle_micro_action` 一把工具。
4. `stardew_idle_micro_action` 只走 `GameActionType.IdleMicroAction -> StardewCommandService -> StardewBridgeRoutes.ActionIdleMicroAction -> BridgeHttpHost -> BridgeCommandQueue`。
5. bridge 不会替模型随机选择动作，也不会在动作不支持时偷换动作。
6. `NpcLocalActionIntent.IdleMicroAction` 是白名单 schema，禁止字段有明确拒绝测试。
7. `idle_animation_once` 只接受 allowlist alias，且具备最大时长、loop 控制、恢复帧 owner。
8. `displayed/skipped/blocked/interrupted` 同时出现在 bridge 返回、runtime log、bridge log 中，语义一致。
9. `private_chat_reply_closed` 只可能由 `private_chat` channel 产出，idle 路径永不产出。
10. interrupt cleanup 覆盖移动、warp/remove、menu/cutscene/event、玩家离开可见范围、新 idle 覆盖旧 idle、runtime drain/clear。
11. route/DTO 对齐测试和 `BridgeCommandQueue` 生命周期回归通过。
12. 不引入双轨执行路径。

## 风险

1. 可见性判定过松，导致玩家看不到时仍执行 idle。
   - 缓解：以 bridge 侧 viewport/scene gate 为准，parent 仅表达尝试意图。
2. idle animation 恢复不完整，NPC 卡在异常姿势。
   - 缓解：allowlist 配置中显式指定恢复帧 owner，并做生命周期回归。
3. overlay channel 改造影响现有 private chat 流程。
   - 缓解：保留 private_chat 原语义不变，只新增显式 channel/kind 并加回归测试。
4. local model 仍可能尝试输出非法字段。
   - 缓解：intent parser 白名单拒绝，tool schema 也只接受最小字段。
5. `BridgeCommandQueue` 新增 idle lifecycle 后与 move lifecycle 互相干扰。
   - 缓解：明确 active idle 与 active move 的互斥关系，并增加 interrupt 回归。

## ADR

### Decision

采用“parent 决定可尝试 idle micro action，本地小模型只在固定枚举里选择，host/bridge 做安全门与执行”的单一路径方案。

### Drivers

1. 需要有语义但低风险的 idle 表现。
2. 需要严格遵守 host/local/bridge 边界。
3. 需要复用现有 Hermes Desktop 与 Stardew bridge 基础设施。

### Alternatives Considered

1. bridge 侧随机动作池。
   - 未选：bridge 不应代决策，且会形成双轨。
2. parent 直接输出 raw animation/frame。
   - 未选：破坏白名单 schema 与安全边界。
3. host 直接调 bridge overlay。
   - 未选：破坏服务边界且难测试。

### Why Chosen

该方案同时满足用户最新边界、现有架构事实与最小风险实施路径，且便于通过 focused tests 与日志证据验证。

### Consequences

1. 需要扩展 contract、tool、GameAction、DTO、route、bridge queue、overlay channel。
2. 需要补一批 route/DTO/queue 生命周期回归测试。
3. 后续若扩展更强 idle 行为，仍必须沿这条单一路径扩展，不能回退到 bridge 随机。

### Follow-ups

1. 首版稳定后再评估 NPC 个性化权重。
2. 第二阶段再考虑更强 idle 行为如 sit/swim/wander，但需单独评审。
