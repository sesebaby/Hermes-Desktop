# 星露谷 NPC 常驻自治循环手动验证清单

## 这份清单是干什么的

这份清单专门用来验证一件事：

**现在的 NPC 是否真的像桌面里的 Hermes agent 一样，能在自己的 home 里长期运行、长期记忆、长期恢复，而不是只在点按钮时短暂跑一下。**

它不是代码测试规格，也不是实现计划。它就是给人拿着一条条过的手测清单。

---

## 这次手测测什么

这次只测下面 6 类东西：

1. NPC 是否真的常驻跑着
2. NPC 是否真的用自己的 home、自己的会话、自己的记忆
3. 私聊和自治是否会互相让路，不会打架
4. 命令执行中断、重启、重连后，NPC 是否还能接着恢复
5. 多个 NPC 一起跑时，是否会出现卡死、堆积、失忆
6. 桌面面板看到的状态，是否和游戏里实际表现对得上

## 这次不测什么

这次不测下面这些：

1. 剧情质量好不好
2. 对话文风像不像原作
3. 复杂经济系统、交易系统、送礼系统
4. 自定义素材、动画、演出效果
5. 模型供应商本身的稳定性

---

## 测试前准备

### 环境准备

1. 桌面程序使用当前最新代码启动
2. `StardewHermesBridge` 已正确加载
3. 游戏存档能正常进入
4. 配置里已经启用要测试的 NPC，例如海莉、潘妮
5. 模型和工具都能正常调用

### 建议先确认的观察面

进入桌面 Dashboard，确认能看到 NPC runtime 面板，重点看这几项：

1. `Loop` / `Wait`
2. `Session` / `Action`
3. `Ingress` / `Pending` / `Cursor`
4. `LastTraceId`
5. `LastError`
6. `Bridge` 是否已连接

如果这些都看不到，先不要开始后面的手测。

---

## 第一组：常驻是否真的活着

### 用例 1：进入游戏后，不点任何按钮，NPC 也会自己跑

**步骤**

1. 启动桌面程序
2. 启动游戏并进入目标存档
3. 不点 Dashboard 上的手动 tick
4. 等待 30 到 60 秒
5. 观察海莉、潘妮的 runtime 面板

**预期**

1. NPC 不需要手动点 tick 也会更新
2. `LastTraceId` 会变化
3. `Loop` 应该出现 `Running`，或者在合理情况下短暂停到 `Paused`
4. `LastAutomaticTickAtUtc` 对应的时间会刷新

**失败信号**

1. 只有点手动 tick 才变化
2. 一直停在 `NotStarted`
3. `LastTraceId` 长时间不变
4. 没有报错，但就是完全不动

---

### 用例 2：切出游戏、切回游戏，常驻循环不会直接死掉

**步骤**

1. 保持游戏运行
2. Alt+Tab 切出 10 到 20 秒
3. 切回游戏
4. 观察 Dashboard

**预期**

1. runtime 不会直接消失
2. 如果 bridge 短暂抖动，恢复后能继续跑
3. 不会因为一次切出切回就永久卡死

**失败信号**

1. bridge 恢复后 NPC 仍一直不再运行
2. 状态卡死在某个旧值
3. 需要重启整个桌面程序才恢复

---

## 第二组：home、会话、记忆是否真隔离

### 用例 3：海莉记住玩家名字，之后还能答对

**步骤**

1. 在游戏里只和海莉私聊
2. 明确告诉她：“我叫远古牛哥，请记住”
3. 正常结束这次私聊
4. 等待一段时间，让自治循环继续跑
5. 再次和海莉私聊，问“我叫什么”

**预期**

1. 海莉能答出“远古牛哥”或者等价正确表达
2. 中间经过自治循环、命令执行、bridge 轮询后，记忆仍在

**失败信号**

1. 海莉回答不知道
2. 海莉像第一次见到玩家一样
3. 记忆只有当场有效，过一会就没了

---

### 用例 4：潘妮不会串用海莉的记忆

**步骤**

1. 先完成“海莉记住玩家名字”的用例
2. 不给潘妮同样信息
3. 直接问潘妮“我叫什么”

**预期**

1. 潘妮不应该凭空拿到海莉那边的私人记忆
2. 她可以不知道，也可以按她自己的上下文回答，但不能像共享脑子一样直接复述海莉那条私聊记忆

**失败信号**

1. 潘妮直接读出了只告诉海莉的内容
2. 多个 NPC 明显共用一套私聊记忆

---

### 用例 5：重启桌面程序后，NPC 记忆还能继续

**步骤**

1. 先完成“海莉记住玩家名字”的用例
2. 关闭桌面程序
3. 重新启动桌面程序
4. 重新连上游戏
5. 再问海莉“我叫什么”

**预期**

1. 海莉还能答对
2. 不需要重新建立人格或重新灌入 SOUL 才能恢复

**失败信号**

1. 重启后失忆
2. runtime 恢复了，但 transcript 没恢复
3. 只有桌面 agent 记得，NPC 不记得

---

## 第三组：私聊和自治是否会互相让路

### 用例 6：私聊期间，NPC 自治会暂停，不会抢着说别的话

**步骤**

1. 让海莉进入私聊会话
2. 私聊窗口保持打开，不要立刻结束
3. 同时观察 Dashboard 上海莉的 runtime 状态

**预期**

1. 海莉的自治 loop 会暂停
2. `PauseReason` 应该体现为私聊相关原因
3. 私聊结束后，自治会恢复

**失败信号**

1. 私聊时海莉还在继续跑自治动作
2. 私聊和自治同时抢同一个 NPC 的输出
3. 私聊结束后自治不恢复

---

### 用例 7：私聊被取消后，lease 会释放

**步骤**

1. 打开海莉私聊
2. 立即取消或关闭
3. 观察 Dashboard 上海莉状态
4. 等待下一轮自治

**预期**

1. `Session` 相关占用会消失
2. 海莉下一轮能恢复自治

**失败信号**

1. 私聊虽然关了，但 `Session` 一直挂着
2. 之后海莉永久不再自治

---

## 第四组：命令执行、恢复、重连是否闭环

### 用例 8：NPC 发起移动或说话后，命令状态会正确推进

**步骤**

1. 让 NPC 自治运行一段时间
2. 观察是否触发 `move` 或 `speak`
3. 同时看 Dashboard 的 `Action` / `Pending`

**预期**

1. 命令提交后能看到 pending / action 状态
2. 命令完成后，pending / action 会清掉
3. 不会出现明明早就执行完，但控制器里一直挂着旧命令

**失败信号**

1. action 已结束，但状态不清
2. claim 不释放，后续动作被堵住
3. commandId 丢了以后彻底恢复不了

---

### 用例 9：bridge 短暂断开后，恢复连接还能接着跑

**步骤**

1. 让游戏和桌面先处于正常联通状态
2. 人为制造一次短暂 bridge 中断，再恢复
3. 观察 NPC runtime 状态变化

**预期**

1. 断开时会暂停
2. 恢复后会重新 attach
3. attach 后仍能继续自治和私聊

**失败信号**

1. 恢复后永远不再运行
2. 同一个 NPC 重连后像换了一个新实例
3. 记忆、cursor、pending 状态全部丢空

---

### 用例 10：切换存档后，不会把旧存档状态串进新存档

**步骤**

1. 在存档 A 中让海莉和潘妮各自运行一段时间
2. 退出到标题界面
3. 切到存档 B
4. 观察 Dashboard

**预期**

1. save 变了以后，runtime scope 会切换
2. 不会把 A 的 pending、cursor、lease 直接带进 B
3. 但同一 save 内自己的 home 和记忆仍按当前设计保留

**失败信号**

1. 新存档继承旧存档的命令状态
2. 旧存档的会话 lease 还压在新存档上

---

## 第五组：多 NPC 并发和长跑稳定性

### 用例 11：海莉和潘妮一起跑，彼此不会拖死

**步骤**

1. 同时启用海莉和潘妮
2. 保持游戏正常运行至少 10 分钟
3. 期间分别与她们私聊、走动、等待
4. 观察 Dashboard

**预期**

1. 两个 NPC 都会持续更新
2. 不会出现一个活着、另一个永久卡死
3. 私聊其中一个时，另一个还能继续自治

**失败信号**

1. 某个 NPC 长时间不再变化
2. 一个 NPC 的私聊把另一个 NPC 也一起暂停了
3. `LastError` 不断重复同一错误

---

### 用例 12：慢模型下连续运行 15 到 30 分钟，不会越跑越糟

**步骤**

1. 使用较慢模型或人为制造较慢响应
2. 保持至少 15 分钟，最好 30 分钟
3. 中间穿插几次私聊、几次地图切换、几次等待
4. 每隔几分钟观察一次 Dashboard

**重点看什么**

1. `Loop` 有没有长期卡在异常状态
2. `Pending` 有没有越来越多但不消
3. `Cursor` 有没有明显不再前进
4. `LastError` 有没有持续刷同一种错误
5. `Ingress` 有没有异常升高后再也下不来

**预期**

1. 即使偶尔暂停，也能恢复
2. 不会越跑越卡
3. 不会出现“表面没报错，实际上已经不工作”

**失败信号**

1. 长跑后某个 NPC 彻底不再更新
2. 一段时间后开始频繁失忆
3. 状态面板显示正常，但游戏里明显已经不响应

---

## 第六组：桌面观察面板是否说真话

### 用例 13：Dashboard 显示要和实际行为对得上

**步骤**

1. 在不同场景下观察面板：
   - 正常自治
   - 私聊进行中
   - 命令执行中
   - bridge 短断
   - 恢复后重新运行
2. 对照游戏里真实表现

**预期**

1. `Loop / Wait` 能反映当前运行状态
2. `Session / Action` 能反映当前占用和命令
3. `Pending / Cursor` 能反映事件与控制器进度
4. `LastError` 真有错时才变化

**失败信号**

1. 面板说在 Running，但游戏里已经死了
2. 面板说没有 Session，但私聊实际上还锁着
3. 面板信息明显滞后或乱跳

---

## 建议记录方式

每次手测建议至少记下面这些：

1. 测试时间
2. 存档名
3. 测试 NPC
4. 当前模型
5. 做了哪些操作
6. 预期是否成立
7. 如果失败，失败前最后一个 `LastTraceId`
8. 如果失败，Dashboard 上最后看到的 `Loop / Session / Pending / Cursor / LastError`

---

## 通过标准

要算“这一轮手测过关”，至少要满足：

1. 海莉和潘妮都完成一轮完整验证
2. 长期记忆验证通过
3. 私聊与自治互斥验证通过
4. bridge 恢复验证通过
5. 多 NPC 并发 10 分钟以上不出现永久卡死
6. 慢模型长跑 15 分钟以上不出现明显退化

如果上面有任何一条不过，就不能算最终闭环。

---

## 当前最值得优先测的顺序

如果时间有限，建议按这个顺序测：

1. 海莉记住玩家名字，之后还能答对
2. 私聊期间自治暂停，结束后恢复
3. 桌面重启后记忆仍在
4. bridge 短断后恢复
5. 海莉和潘妮并发 10 分钟
6. 慢模型长跑 15 到 30 分钟
