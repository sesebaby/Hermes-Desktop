## RALPLAN-DR

### Principles

1. Desktop/core 拥有私聊编排、会话状态、Agent 调用与记忆边界；mod 仅保留桥接和宿主 UI。
2. 复用抽象只上移状态机与契约，不上移 Stardew/SMAPI 专属 UI、菜单、game loop 约束。
3. 以类型化配置替代硬编码 Stardew 常量，同时保留 Stardew 现有行为作为兼容基线。
4. 先用测试锁住现状与目标行为，再做抽象；每一步都必须可回退、可验证。

### Decision Drivers

1. 当前 `src/games/stardew/StardewPrivateChatOrchestrator.cs` 的状态迁移逻辑大体通用，但事件名、`gameId`、默认 NPC、prompt 与 runner 文案绑死 Stardew/Haley。
2. 旧共识计划已要求 Desktop/core 拥有私聊编排；把 UI 或 LLM 下沉到 mod 会直接违反既定边界。
3. `E-2026-0013` 和 `E-2026-0014` 明确要求 Stardew UI 继续留在 SMAPI/game-loop 可见菜单层，因此“跨 mod 复用”不能通过抽走 UI 达成。

### Options

#### Option A: 在 `src/game/core/` 新建通用私聊状态机内核，Stardew 包装为配置 + runner 适配层

优点：
- 真正把可复用部分提升到 game/core，后续其他 mod 只需提供事件契约、命令元数据和 agent runner。
- 能保留 Stardew 的 `BridgeCommandQueue` / `PrivateChatInputMenu` 不动，只替换 Desktop/core 编排入口。
- 测试可直接覆盖“通用状态机 + Stardew 兼容包装”两层，不依赖具体 mod UI。

缺点：
- 需要拆分现有 `StardewPrivateChatOrchestrator`，短期会改到 core 与 Stardew 两层测试。
- 若抽象过度，容易把一次稳定的修复再度泛化失控。

#### Option B: 保持 orchestrator 位于 `src/games/stardew/`，仅把事件名、prompt、gameId 参数化

优点：
- 改动最小，最快。
- 对现有 `StardewPrivateChatBackgroundService`、`App.xaml.cs` 和测试冲击最小。

缺点：
- 复用性是名义上的，后续其他 mod 仍要依赖 Stardew 命名空间或复制文件。
- “每个游戏 mod 可复用”的目标无法被清晰证明，Architect 很可能会否决这是伪抽象。

#### Option C: 为每个游戏各自保留一份 orchestrator，只共享少量 helper

优点：
- 最低抽象风险，不动现有 Stardew 稳定路径。

缺点：
- 直接放弃本任务目标，重复逻辑会继续分叉。
- 后续修复会在多个 mod 间重复传播，不符合 Desktop/core 拥有编排的长期方向。

### Recommended

推荐 `Option A`，但采用“薄核心 + 薄 Stardew 包装”的克制抽象：
- 在 `src/game/core/` 放通用状态机、配置契约、会话状态与协作接口。
- 保留 `src/games/stardew/` 中的 Stardew wrapper/background service/runner，实现 Stardew 事件名、prompt、`gameId` 和默认 NPC 的兼容注入。
- 不把 `Mods/StardewHermesBridge/**` 上移，也不要求新增依赖。

推荐原因：
- 它满足“跨 mod 可复用”的实际目标，同时保留已修好的 Stardew UI 行为与边界。
- 它把高风险变化限制在 Desktop/core，不触碰 `E-2026-0013` / `E-2026-0014` 已锁定的 mod 约束。
- 现有 `GameActionType.OpenPrivateChat`、`GameEventRecord`、`IGameAdapter` 已提供足够的通用地基，抽象成本可控。

6. `E-2026-0013` 与 `E-2026-0014` 的约束保持成立：Stardew UI 仍只由 mod/game loop/可见 `IClickableMenu` 路径负责。

## ADR

### Decision

将当前 Stardew 私聊状态机拆成“core 通用私聊状态机 + Stardew 兼容包装层”。通用状态机进入 `src/game/core/`；Stardew 继续保留 wrapper、background service、agent runner 与 mod UI。

### Drivers

1. 任务目标是“每个游戏 mod 可复用”，而不是“让 Stardew 更可配置一点”。
2. 既有共识已规定 Desktop/core 拥有编排；mod 只做桥接/UI。
3. 已修复的 Stardew UI 行为依赖 SMAPI/game loop/真实 `IClickableMenu`，不能被抽入 core。

### Alternatives Considered

- `Option A`：core 通用状态机 + Stardew wrapper。
- `Option B`：仅参数化现有 `StardewPrivateChatOrchestrator`。
- `Option C`：每个游戏各自维持独立 orchestrator，只共享 helper。

### Why Chosen

`Option A` 是唯一同时满足这三点的方案：
- 真正实现跨 mod 复用。
- 不破坏 Stardew 已修好的行为。
- 不把 mod UI 或 LLM 责任移到错误边界。

它也最符合当前代码结构：`IGameAdapter`、`GameActionType.OpenPrivateChat`、`GameEventRecord` 已经是通用抽象，只缺一个通用私聊状态机内核。

### Consequences

正向结果：
- 后续新游戏 mod 可以只实现事件/命令契约和 UI bridge，而复用 Desktop/core 私聊编排。
- Stardew 仍拥有稳定的兼容包装与现有人工验证路径。

代价与风险：
- 短期要维护一段 wrapper 映射代码，命名会出现 core/stardew 双层并存。
- 如果抽象范围失控，容易把一次稳定修复再变成泛化重构；因此必须用 TDD 和小步迁移约束。

### Follow-ups

1. Architect 重点审查通用契约是否过宽，尤其是 prompt/idempotency/action-factory 是否有过度抽象迹象。
2. Critic 重点审查 TDD 顺序是否足以锁住 Stardew 回归，以及验收是否真的证明“非 Stardew 配置可复用”。
3. 若后续第二个游戏接入，再决定是否需要把 background service 也上提为通用轮询宿主；本次不提前设计。
