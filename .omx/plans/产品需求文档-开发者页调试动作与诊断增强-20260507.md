# 产品需求文档：开发者页调试动作与诊断增强

## 1. 背景

开发者页第一阶段已经提供只读 NPC 观察台，用于查看 NPC 的 `SOUL.md`、`MEMORY.md`、`USER.md`、模型回复、工具调用、委托、任务与待办、runtime 日志。当前手测跨地图能力仍缺一个稳定前置条件：把海莉或潘妮放到 `Town`，再要求她使用地图 skill 选择 `Beach` 明确坐标并通过本地执行层执行。

现有 `Mods/StardewHermesBridge/Commands/TestTeleportCommand.cs` 注册了 `teleport <海莉|haley|潘妮|penny>`，但它的行为是 `Game1.warpFarmer(...)`，即把玩家传送到 NPC 附近，不是把 NPC 放到镇上。因此需要新增一个短手测命令，并把后续调试动作收敛到开发者页。

## 2. RALPLAN-DR 摘要

### 原则

- 单一路径：SMAPI 控制台命令、bridge HTTP 调试端点、桌面开发者页按钮必须复用同一套 bridge 调试动作核心。
- 明确副作用：`放到镇上` 是调试 teleport，不能被记录或解释为自然跨地图移动成功。
- 观察优先：开发者页继续服务于定位断点，受控动作必须配确认、执行反馈和审计。
- 中文一致：普通 UI 文案、文档标题、页签和按钮用中文；日志、JSON、路径、工具名、模型原文保持原样。
- 脱敏导出：诊断包默认本地生成，必须移除 bridge token、API key、Authorization 和凭据类字段。

### 决策驱动

- 手测效率：没有稳定把 NPC 放回 `Town` 的短命令，就无法快速复现 `Town -> Beach` 跨地图目标。
- 架构一致性：现有 NPC runtime、transcript、todo、runtime.jsonl、bridge HTTP 面已经接线，新增能力应挂在这些链路上。
- 可诊断性：用户需要同时看到模型回复、工具调用、委托、任务、待办、执行结果和 route probe。

### 可选方案

#### 方案 A：bridge 调试动作核心 + SMAPI 短命令 + 开发者页复用

做法：在 bridge 侧抽出 `BridgeNpcDebugRepositioner`，SMAPI 控制台命令和 `/debug/npc/reposition` HTTP 端点都调用它；桌面 `StardewNpcDebugActionService` 再调用该 HTTP 端点。

优点：一套放置逻辑、一套日志、一套失败码；同时满足短命令和开发者页受控动作。

缺点：需要新增 bridge debug endpoint 和桌面 service 方法。

#### 方案 B：只加 SMAPI 短命令

优点：最快，风险最低。

缺点：第二阶段开发者页仍无法闭环，用户还要在 SMAPI 控制台和桌面之间切换。

#### 方案 C：直接实现完整跨地图分段执行

优点：如果成功，用户不需要手动放置 NPC。

缺点：与当前“小步 route probe”节奏冲突，失败时更难定位层级。

### 推荐

采用方案 A。短命令解决即时手测，开发者页通过同一 bridge 调试动作复用该能力，不创建第二套执行路径。

## 3. 阶段范围

### 第一阶段：短手测命令与共用 bridge 调试动作

新增短命令：

```text
npc_town haley
npc_town penny
npc_town 海莉
npc_town 潘妮
```

行为：

- 将目标 NPC 放到 `Town` 的固定可站立点，首选 `BridgeDestinationRegistry` 的 `town.square`，必要时在附近寻找开放 tile。
- 保留现有 `teleport` 命令语义，不覆盖、不改名。
- 明确记录 `debug_npc_reposition_*` 日志，包含 NPC、原地点、目标地点、目标 tile、结果或失败码。
- 命令失败时输出结构化原因：`world_not_ready`、`invalid_target`、`location_not_found`、`target_blocked`。

限制：

- 该命令是显式调试 teleport，不进入自然移动验收。
- 首版只允许已知 NPC 和固定 `Town` 目标，不开放任意地图或任意坐标。

### 第二阶段：受控调试动作迁移到开发者页

在 `开发者 / NPC 观察台` 增加中文分区 `受控调试`。

动作范围：

- `放到镇上`：调用 bridge debug reposition endpoint。
- `单步唤醒`：复用 `StardewAutonomyTickDebugService.RunOneTickAsync(...)`。
- `发送调试台词`：复用 `StardewNpcDebugActionService.SpeakAsync(...)`。
- `刷新观察台`、`打开运行目录`：沿用现有开发者页能力。

UI 规则：

- 有副作用动作必须显示确认说明或二次点击保护。
- 执行中禁用相关按钮，执行后显示中文结果和原始错误码。
- bridge 未连接、未选 NPC、世界未加载、动作执行中时，按钮禁用并给出中文原因。
- `Agent > Runtime` 保留摘要和“打开开发者 NPC 观察台”跳转，不再重复固定调试按钮。

### 第三阶段：追踪筛选、诊断包导出、上下文注入差异视图

追踪筛选：

- 支持按 NPC、追踪 id、事件类型、commandId、工具名、错误码、文本关键词筛选。
- 筛选基于现有 `runtime.jsonl` 和 transcript 投影，不新增 trace 数据库。

诊断包导出：

- 默认导出到 `%LOCALAPPDATA%\hermes\hermes-cs\diagnostics\`。
- 文件名建议：`npc-diagnostics-<npcId>-<yyyyMMdd-HHmmss>.zip`。
- 包含所选 NPC 的 runtime 目录摘要、选中 trace、相关 transcript 摘要、Hermes 日志尾部、SMAPI 日志尾部、脱敏 bridge discovery、诊断说明。
- 强制脱敏字段：`bridgeToken`、`Authorization`、`apiKey`、`apikey`、`secret`、`password`、`token`、`connectionString`。
- 默认不包含完整 `SOUL.md`、`MEMORY.md`、`USER.md` 原文；如后续需要包含原文，必须作为显式选项另评审。

上下文注入差异：

- 展示 NPC 调用中人格、记忆、技能、世界事实、待办、检索内容、最近消息等 block 的注入前后状态。
- 默认展示 block 名称、字符数、估算 token、hash、是否裁剪、短预览。
- 优先把 block 级元数据写入现有 `runtime.jsonl` 的诊断事件，不建新存储。

## 5. 验收标准

- SMAPI 控制台输入 `npc_town haley` 或 `npc_town penny` 后，对应 NPC 位于 `Town` 的可站立 tile。
- SMAPI 日志明确写出 `debug_npc_reposition`，不与自然移动的 `task_move_enqueued/task_completed` 混淆。
- 从 `Town` 发起“去海边”的自然语言目标时，跨地图证据仍只以 mechanical target、`stardew_navigate_to_tile`、`routeProbe`、bridge task status 为准。
- 开发者页出现中文 `受控调试` 分区，选中 NPC 后可执行 `放到镇上`、`单步唤醒`、`发送调试台词`。
- 所有普通 UI 字符串中文一致；不出现 `task/TODO/diff/replay` 这类混排控件标签。
- `Agent > Runtime` 不再重复固定调试按钮，保留跳转到开发者页。
- 追踪列表能按事件类型、追踪 id、commandId、错误码和关键词筛选。
- 诊断包导出为本地 zip，默认位置为 `%LOCALAPPDATA%\hermes\hermes-cs\diagnostics\`，并脱敏敏感字段。
- 上下文差异页签展示注入块、裁剪状态、长度变化和短预览。
- 不新增第二套 NPC runtime、todo、memory、transcript、trace 数据库或工具 lane。

## 6. ADR

### 决策

采用“bridge 调试动作核心 + SMAPI 短命令 + 开发者页复用”的路线。短命令用于即时手测，开发者页动作通过同一个 bridge endpoint 复用该能力，诊断增强继续读取现有 runtime/transcript/log 链路。

### 驱动

- 用户需要稳定把 NPC 放回 `Town` 以触发 `Town -> Beach` 跨地图测试。
- 现有 `teleport` 命令只移动玩家，不满足需求。
- 开发者页一期已经是 NPC 观察中心，后续调试动作应收敛到这里。
- 不能创建第二套 runtime、记忆、任务、trace 或工具通道。

### 备选

- 只加 SMAPI 命令：最快，但桌面开发者页仍无法完成受控调试闭环。
- 只加桌面按钮：不满足短命令和 SMAPI 手测便利。
- 直接做完整跨地图分段执行：风险过大，不利于先定位 route probe。

### 后果

- bridge 会新增一个明确受限的 debug action surface。
- 开发者页从只读观察台进入“受控动作 + 观察”阶段。
- 诊断包导出和上下文差异需要严格脱敏和体积限制。

### 后续

- 继续校准 `skills/gaming/stardew-navigation/SKILL.md` 中关于跨地点移动的文案。
- 如果 route probe 通过，再规划真正的跨地图第一段移动执行。
