# 测试规格：桌面端开发者页与 NPC 观察台

## 1. 测试目标

验证第一阶段 `开发者 / NPC 观察台` 是一个只读、中文、一致复用现有后端基础设施的 WinUI 页面。测试重点不是游戏动作是否成功，而是开发者能否从 UI 看到 NPC 状态、资料文件、模型回复、工具调用、委托链路、任务待办和日志入口，并且实现不引入双轨存储或 UI 阻塞风险。

## 2. 范围

### 必测

- 新导航项与页面加载。
- 中文资源和无障碍名称。
- NPC runtime 只读快照投影。
- NPC 文件路径解析与只读读取。
- `runtime.jsonl` trace 解析。
- transcript 消息读取、模型回复和工具调用投影。
- 明确不要求 provider HTTP 原始请求/响应包；只验证 Hermes 已持久化消息、reasoning、tool call 与诊断摘要。
- `agent` 工具结果的委托链路投影。
- `SessionTodoStore` / `NpcRuntimeTaskView` 的任务与待办展示。
- 日志/目录打开入口。
- 构建与相关单元测试。

### 不测

- 真实 Stardew 游戏移动、说话、私聊等副作用动作。
- 完整 MSIX 发布流程。
- 全量历史 trace 查询性能。
- 编辑 `SOUL.md`、`MEMORY.md`、`USER.md`。

## 3. 测试数据源与依据

| 能力 | 依据 | 测试要求 |
| --- | --- | --- |
| runtime snapshot | `NpcRuntimeWorkspaceService.GetSnapshot()` 位于 `Desktop/HermesDesktop/Services/NpcRuntimeWorkspaceService.cs:35` | 用伪造 supervisor/实例或临时 runtime 数据验证投影 |
| NPC namespace | `NpcNamespace` 定义 `SOUL.md`、memory、transcripts、trace、activity 路径，见 `src/runtime/NpcNamespace.cs:17`、`:32`、`:38`、`:40`、`:44`、`:46` | 临时目录创建 NPC home，验证路径与读取 |
| activity log | `NpcRuntimeLogRecord` 字段见 `src/runtime/NpcRuntimeLogWriter.cs:45` | JSONL 解析要容忍坏行和未知 action type |
| transcript | `TranscriptStore.LoadSessionAsync` 见 `src/Transcript/TranscriptStore.cs:76` | 保存 assistant/tool 消息后能读取并投影 |
| tool calls | `Message.ToolCalls` 与 `ToolCall.Arguments` 见 `src/Core/Models.cs:15`、`:84` | 工具名、参数、结果和 call id 可显示 |
| messages 表 | `tool_calls`、`reasoning_content` 字段见 `src/Search/SessionSearchIndex.cs:96`、`:103` | 有 reasoning 时展示，无 reasoning 时不报错 |
| todo | `SessionTodoStore.Read/Write` 见 `src/tasks/SessionTodoStore.cs:21`、`:33` | 展示 pending/in_progress/blocked/failed/cancelled |
| 子 agent | `AgentTool.ExecuteAsync` 输出子 agent 文本，见 `src/Tools/AgentTool.cs:35` | transcript 中 `agent` tool result 能归入委托链路 |

### runtime log parser

- `ParseRuntimeLog_WithObservationDiagnosticExecutorAndHostAction_MapsChineseEventKinds`

### transcript 投影

测试：

- assistant final content 展示到 `模型回复`。
- assistant `ToolCalls` 展示到 `工具调用`。
- tool role 的 `ToolName`、`ToolCallId` 和 result content 与 tool call 匹配。
- `agent` 工具输出进入 `委托链路`。
- `todo` / `todo_write` 工具结果进入任务与待办投影时不新建 `TaskManager` 数据。
- 没有 provider raw payload 时显示中文说明，不把缺失视为失败。

## 5. 资源与中文 UI 测试

### 资源要求

- 所有新增普通 UI 字符串使用 `x:Uid` 或 `ResourceLoader`。
- 新增资源键同时写入 `Desktop/HermesDesktop/Strings/en-us/Resources.resw` 和 `Desktop/HermesDesktop/Strings/zh-cn/Resources.resw`。
- 中文环境下新增页面普通标签为中文。
- 原始内容区域可显示日志、JSON、路径、工具名和模型原文。

### 可脚本检查

建议在实现后运行：

```powershell
rg -n "Content=\"|Text=\"" .\Desktop\HermesDesktop\Views\DeveloperPage.xaml
rg -n "task|TODO|diff|replay|Runtime|Trace|Tool" .\Desktop\HermesDesktop\Strings\zh-cn\Resources.resw
```

检查结果需要人工判定：原始技术词在资源中不应作为控件标签出现；如果用于说明“原始日志保留 tool 名称”，可以接受。

## 6. 可访问性测试

每个交互控件必须满足：

- `AutomationProperties.Name` 或 `x:Uid` 资源提供的 AutomationProperties 名称。
- 关键控件有稳定 `AutomationProperties.AutomationId`。
- NPC 列表、trace 列表、详情页签、刷新按钮、打开日志按钮可键盘访问。
- 不只依赖颜色表达状态，状态文本要可读。

手测清单：

- 用 Tab 键遍历 `开发者 / NPC 观察台`。
- 使用键盘选择 NPC、切换详情页签、刷新。
- Windows 高对比度模式下检查文本可读。
- Narrator 能读出刷新、跟随最新、打开目录等按钮含义。

## 7. 性能测试

要求：

- 长列表使用 `ListView` 或虚拟化控件。
- 文件与日志读取使用 async I/O。
- 单个文件预览限制大小，例如首版可只读前/后固定字节数并提供路径。
- `runtime.jsonl` 读取默认尾部读取或最大行数限制。
- 不在 UI 线程执行大文件解析。

建议测试：

- 构造 5,000 行 `runtime.jsonl`，确认加载不会冻结 UI。
- 构造 1 MB `MEMORY.md`，确认页面显示截断说明或预览限制。
- 快速切换 NPC 时不会交叉显示旧 NPC 的异步结果。

## 9. 手动验收流程

1. 启动桌面应用。
2. 左侧导航选择 `开发者`。
3. 确认页面标题为 `NPC 观察台`，普通 UI 文案为中文。
4. 选择一个 NPC。
5. 查看当前概览：状态、session、trace、错误、动作槽、私聊锁、下一次唤醒、任务摘要。
6. 打开资料区：查看 `SOUL.md`、`MEMORY.md`、`USER.md`。
7. 选择最新 trace。
8. 查看时间线：观察事实、模型请求、模型回复、意图解析、本地执行、工具调用、游戏桥接、结果。
9. 切到 `模型回复`，确认显示原始模型内容或中文缺失说明。
10. 切到 `工具调用`，确认显示工具名、参数、结果、call id。
11. 切到 `委托链路`，确认 `agent` 工具结果可见；没有委托时显示中文空状态。
12. 切到 `任务与待办`，确认与 NPC session 的 todo snapshot 一致。
13. 打开日志入口，确认能打开 runtime 目录和对应日志路径。

## 10. 回归风险检查

- `Dashboard` NPC 卡片仍能加载。
- `Agent > Runtime` 不因新页面而崩溃。
- `Dashboard` NPC 卡片的入口跳到 `开发者 / NPC 观察台` 后仍保留原有 runtime 摘要可见性。
- `Agent > Runtime` 首版仍能执行原有固定调试按钮，或明确显示跳转到开发者页，不得静默删除功能。
- `ChatPage` 会话任务面板不受 NPC todo 投影影响。
- `StardewNpcAutonomyBackgroundService` 不因只读观察台加载而启动额外 runtime。
- `NpcRuntimeSupervisor.Snapshot()` 不被 UI 高频轮询压垮。

## 11. 执行前复核

- 复核 `Agent` 写入 transcript 的实际 helper 和保存行为，确认 assistant final content、assistant `ToolCalls`、tool result、reasoning 字段能通过 `TranscriptStore.LoadSessionAsync` 读回。
- 复核 `runtime.jsonl` 现有 action/stage 枚举，先用真实或测试日志建立中文事件映射，不凭空新增 UI 分类。
- 复核 `AgentTool` 的子 agent 输出是否作为 tool result 存入 transcript；如果没有，只显示中文缺失说明，不另建委托存储。
- 复核 `TaskManager` 与 `SessionTodoStore` 的边界，确保页面只把会话待办展示为 `任务与待办`，不误写长期任务系统。

## 12. 完成标准

- 所有新增服务公共方法有单元测试。
- 构建通过。
- 相关测试通过或明确未测原因。
- 新页面普通 UI 文案中文一致。
- 没有新增依赖。
- 没有新增第二套后端存储。
- 文件和日志读取不阻塞 UI。
- 剩余风险写入最终报告。
