# 桌面与 NPC 智能体 HermesCraft 核心同源化审慎计划

## 0. 任务定义

目标：基于当前仓库证据，模仿 `external/hermescraft-main` 的核心方案，修复 Desktop agent / NPC agent 当前所有已发现的同源装配、runtime ownership、身份 provenance、prompt/tool 语义分叉问题，使 NPC 成为“标准 Hermes agent + 独立身份隔离 + 游戏专属能力通过 bridge/tool/skill 接入”的常驻运行时，而不是散落在 private-chat / debug / host 等路径里的临时缩水 runtime。

本计划明确采用 deliberate mode：这是跨 `src/runtime`、`src/game/core`、`src/games/stardew`、Desktop 启动接线与测试基线的高风险架构调整。

本计划同时保留以下边界：

- 继续引用 HermesCraft 方案，但只吸收其“standard Hermes agent + per-agent identity/runtime root + 同源能力面”的核心，不回退成“一 NPC 一进程”。
- 保持“单 Desktop 进程内托管多个 NPC runtime”。
- 不新增独立 `StardewPromptAssembler`。
- 不把 all-in-one host 重构设为唯一主线；是否由 `NpcRuntimeHost` 成为 live path 主入口，必须放到 contract 与 provenance 收敛之后再做证据判定。

## 2. RALPLAN-决策记录摘要

### 2.1 原则

1. NPC 必须是 normal Hermes agent；“不同 NPC”与“Desktop vs NPC”的合法差异只能来自显式 contract 输入，而不是各路径私造装配。
2. 必须先冻结 canonical runtime 句柄 / runtime ownership：任何 live NPC path 都只能通过同一句柄取得或创建 agent/runtime instance，不允许绕过句柄直接按场景临时 `new` agent。
3. runtime identity key 必须显式对齐仓库硬约束：至少是 `gameId + saveId + npcId + profileId`；`sessionId` / `source` 只能作为审计、会话或 trace 维度，不能替代隔离键。
4. 必须形式化两层 contract：
   - 基础 Hermes 组合契约：定义所有 Hermes agent 共享的 prompt builder、built-in tools、discovered tools、skills guidance、context/runtime collaborators。
   - Stardew 增强契约：只定义 Stardew 在基础契约之上的 persona/session/root、game bridge、game tools、scene supplement 与 NPC-specific binding/provenance。
5. 身份事实必须来自可审计 provenance，而不是默认值或局部 patch；`saveId` / `sessionId` / `source` 必须贯穿 descriptor、runtime handle、日志与测试，但只有 `gameId + saveId + npcId + profileId` 能定义 runtime 隔离。
6. discovered tools 生命周期本轮明确选择 `runtime-refresh with explicit rebind rules`：MCP connect/reconnect/config reload 后允许刷新工具面，但只能由 canonical runtime owner 在显式安全边界执行 rebind，禁止各入口各自取快照。
7. 保持本仓库 topology 约束：单 Desktop 进程内托管多个 NPC runtime，不回退成“一 NPC 一进程”，也不新增独立 `StardewPromptAssembler`。

### 2.2 决策驱动

1. 当前系统缺少被写死的 persistent runtime loop owner，private-chat、autonomy/debug、可能的 host path 都能各自临时造 context/agent/adapter/loop。
2. runtime identity key 必须与现有 `NpcRuntimeSupervisor.BuildKey(gameId, saveId, npcId, profileId)` 保持一致，防止 `sessionId` / `source` 被误用为隔离键。
3. Desktop MCP 是异步初始化，而 NPC live path 直接读 `McpManager.Tools.Values`；因此 discovered tools 必须明确选择 `runtime-refresh with explicit rebind rules`，否则长寿命 runtime 行为不可复现。
4. 当前“只差 persona/session/namespace/memory directory”的说法没有被形式化，导致 base Hermes 能力与 Stardew 增量能力持续缠绕。
5. 是否由 `NpcRuntimeHost` 成为 live path 主入口目前缺少证据；计划必须先收敛 key、owner、rebind contract，再验证 host/live path 优先级。

### 2.3 可行方案

#### 方案 A：先冻结 identity key + canonical runtime owner + runtime-refresh/rebind 规则，再让所有 live NPC path 通过 canonical handle 取 runtime，host 角色最后按证据决定

优点：
- 直接对准 architect 反馈的根问题，不假设 `NpcRuntimeHost` 已是产品主入口。
- 能同时写死 runtime ownership、identity key、discovered-tools refresh/rebind contract，而不是只修表层 builder。
- 允许 host 在证据不足时保持为实现支撑或工具入口，而不阻塞 live path 收敛。

缺点：
- 需要先定义清晰 contract、owner 与 rebind 边界，再回头收口各入口，第一阶段抽象与验证工作量更高。
- 需要补一批 provenance / lifecycle / parity 测试，短期红测面较大。

#### 方案 B：直接把 `NpcRuntimeHost` 提升为唯一 live path 主入口，再让其他路径全部挂到 host，由 host 隐式承担 owner/rebind

优点：
- 叙事简单，看起来像“一次收口所有路径”。
- 如果 host 本来就是主入口，实现上可能较直观。

缺点：
- 目前没有证据证明 host 是真实 live path，强推会把计划绑在半成品假设上。
- 即便共享了 host，如果 runtime ownership、identity key、rebind rules、two-layer contract 没冻结，仍可能只是把临时组装搬进 host。

#### 方案 C：继续保留 private-chat / debug 的场景内组装，只要求它们共享 `AgentCapabilityAssembler` 并按需读取 `McpManager.Tools.Values`

优点：
- 改动面最小。
- 短期内最容易让个别用例恢复。

缺点：
- 不能解决 canonical handle / ownership 缺失的问题。
- “共享 builder 但继续临时造 agent”仍然违背 HermesCraft 核心原则。
- `gameId + saveId + npcId + profileId` 隔离键、`sessionId/source` 审计边界与 discovered tools refresh/rebind 规则仍会继续漂移。

#### discovered tools 生命周期子决策

##### 方案 A：session 稳定快照

优点：
- 实现最简单，启动后工具面固定。
- 单次会话内行为更易复现。

缺点：
- 无法覆盖 `InitializeMcpAsync(...)` 完成后才出现的工具。
- reconnect / config reload 后 runtime 会继续持有过期工具面。
- Desktop 与 NPC 长寿命 runtime 的能力面仍可能分叉。

##### 方案 B：runtime-refresh 并显式 rebind 规则

优点：
- 能覆盖首次 MCP connect、后续 reconnect、config reload / tool catalog 变化。
- 由 canonical owner 控制 rebind，行为边界可审计、可测试。
- 保持单进程多 NPC runtime 下的共享发现机制，同时避免 mid-turn 热替换。

缺点：
- 需要补 generation / pending-rebind / safe boundary 状态与测试。
- 对现有调用方的模糊行为容忍度更低。

##### 方案 C：原始 live provider / 隐式变更

优点：
- 不需要显式缓存或 rebind 状态。

缺点：
- runtime 行为会直接暴露给异步 MCP 变更。
- 无法证明 private-chat / autonomy / debug 在何时完成 rebind。
- 问题出现时难以通过日志或测试定位。

### 2.4 推荐方案

推荐 **Option A**，并在其中显式采用 `runtime-refresh with explicit rebind rules`。

原因：

- 这是唯一同时满足 HermesCraft 原则、architect 反馈、以及本仓库单进程多 NPC runtime 约束的方案。
- 它不预设 `NpcRuntimeHost` 必须是 live path 主入口，但要求所有 live NPC path 最终都服从同一个 canonical runtime owner / canonical handle。
- 它把“共享 builder”提升为“共享基础契约 + Stardew 增强契约 + 精确 identity key + runtime-refresh/rebind 规则 + 可审计 provenance”的完整闭环，而不是表面收口。
- 相比 `session-stable snapshot`，它不会把 NPC 永久锁死在启动时的工具面；相比 `raw live provider / implicit mutation`，它把刷新时机收敛为显式 rebind 规则，因此更适合 Desktop 的异步 MCP 初始化。

### 2.5 事前验尸

#### 场景 1：共享了 builder，但没有被写死的 canonical runtime owner / handle

- Trigger：`private-chat` / `debug` / 未来新路径仍能直接 `new NpcRuntimeContextFactory()` 或 `new NpcAgentFactory()` 按场景临时造 agent。
- Containment：所有 live path 只能经由 `NpcRuntimeSupervisor` 获取 handle；外层 consumer 不得直接持有 runtime factory。
- Proof：`NpcRuntimeSupervisorTests`、`NpcRuntimeContextFactoryTests`、`NpcAgentFactoryTests` 断言 live path 不存在绕过入口；runtime 日志字段固定包含 `runtimeOwner=NpcRuntimeSupervisor`。

#### 场景 2：两层契约没有拆清，Stardew 增量继续污染 base Hermes 契约

- Trigger：Prompt supplement、game tools、persona/session/root 再次混进 base contract。
- 约束范围：基础 Hermes 组合契约只保留共享 Hermes 能力；Stardew 增量只能落在增强契约。
- Proof：`PrivateChatOrchestratorTests`、`StardewPrivateChatWiringTests`、`NpcNamespaceTests` 断言 base contract 不再携带场景补丁或 NPC 业务身份。

#### 场景 3：去掉 `haley/manual-debug`，但把 `sessionId` / `source` 错当运行时键

- Trigger：不同入口开始拿 `sessionId` 或 `source` 替代 `gameId + saveId + npcId + profileId`。
- Containment：`NpcRuntimeSupervisor.BuildKey(...)` 仍是唯一隔离键来源；descriptor、namespace、日志只把 `sessionId` / `source` 当审计字段。
- Proof：`StardewNpcBindingResolverTests`、`NpcNamespaceTests`、`NpcRuntimeSupervisorTests` 和 runtime trace 字段同时断言 `runtimeHandleKey`、`sessionId`、`source` 不混用。

#### 场景 4：discovered tools 选择了 refresh，但没有显式 rebind 规则

- Trigger：`InitializeMcpAsync(...)` 完成、reconnect 或 config reload 后，旧 runtime 仍绑定旧工具集，或者在执行中途被热替换。
- Containment：只允许 `NpcRuntimeSupervisor` 在下一 reply turn / 下一 tick / 下一次 debug request 前执行 rebind，并维护 `rebindGeneration` / `pendingRebind`。
- Proof：`NpcRuntimeLogWriterTests`、`StardewAutonomyTickDebugServiceTests`、`StardewPrivateChatOrchestratorTests` 断言日志字段 `toolCatalogVersion`、`rebindGeneration`、`pendingRebind`。

### 2.6 扩展测试计划

#### 契约 / 归属单测

- canonical runtime 句柄测试：
  - 任意 live NPC path 必须通过同一 runtime handle 解析/获取 runtime instance。
  - runtime owner 必须负责 agent/context/adapter/loop/cancellation/restart budget。
  - 任何直接 `new NpcRuntimeContextFactory()` / `new NpcAgentFactory()` 的 live path 都会被测试或代码结构约束捕获。
- base composition contract tests：
  - Desktop agent 与 NPC agent 共享同一 base prompt builder、skills guidance、built-in tools 与 discovered tools 语义。
- Stardew 增强契约测试：
  - persona、save/session provenance、namespace/root、game tools、scene supplement 仅通过 augmentation 输入生效，不污染 base contract。

#### 来源证明单测

- `StardewNpcRuntimeBindingResolver` / descriptor factory：
  - binding 必须来自 catalog/manifest/pack。
  - runtime identity key 必须至少由 `gameId + saveId + npcId + profileId` 组成。
  - `sessionId` / `source` 只能作为审计/会话维度，且必须在 descriptor、runtime handle、日志对象间一致但不参与隔离键替代。
- `NpcNamespace`：
  - `SOUL.md`、memory、sessions、transcripts、activity、trace 根目录必须与 provenance 对齐。

#### Prompt / 工具单测

- prompt parity：
  - NPC prompt 包含与 Desktop 同源的 base system prompt 与 `skillsMandatoryPrompt`。
  - private-chat / autonomy 只能通过 scene supplement 叠加差异。
- discovered tools lifecycle：
  - 本轮固定选择 `runtime-refresh with explicit rebind rules`。
  - refresh 触发源至少覆盖 `InitializeMcpAsync(...)` 首次 connect、后续 reconnect、config reload / tool catalog 变化。
  - rebind 只能发生在 owner 控制的安全边界：private-chat 的下一次 reply turn 前、autonomy loop 的下一 tick 前、debug 请求进入执行前；禁止 mid-turn / mid-tool-call 热替换。
  - rebind 后 runtime identity key、namespace、memory、transcript、restart budget 不变，只更新 tool registry / agent binding / loop adapter 依赖。

#### 集成测试

- `StardewPrivateChatBackgroundService` + orchestrator + runner：
  - 真实 background poll 路径下，Penny 事件命中 Penny descriptor / provenance / runtime handle / namespace。
  - MCP tools 刷新后，下一次 reply turn 前由 owner 执行 rebind，不中断正在进行的 turn。
- `StardewAutonomyTickDebugService`：
  - debug/autonomy 通过 canonical runtime 句柄获取 runtime，而不是本地 `new` agent。
  - refresh 发生时，下一 tick / 下一次 debug request 进入前 rebind。
- `NpcRuntimeHost`：
  - 若证据表明 host 是 live path，验证它确实消费同一 handle/contract。
  - 若证据表明 host 不是 live path，验证它保持为实现支撑或工具入口而不分叉 contract。

#### E2E / host 冒烟测试

- 不把真实 Desktop 宿主级 E2E 硬塞进 `Desktop/HermesDesktop.Tests`；该工程当前是 `net10.0` 核心测试工程，现有模式主要是 source-level wiring 断言与纯服务集成，不承担 WinUI 宿主进程启动。
- 自动化验证分两层：
  - `Desktop/HermesDesktop.Tests` 继续负责 canonical handle、provenance、private-chat/autonomy wiring、runtime reuse / rebind 逻辑断言；
  - `tools/Hermes.SmokeProbe` 负责真实 Desktop 宿主启动 smoke，验证 `HermesDesktop.exe` 能启动，且引入本轮收敛后不会让 `InitializeMcpAsync(...)` / startup path 写入 fatal startup markers。
- 如果后续必须覆盖“启动后首次背景私聊命中 runtime reuse / rebind”的行为，则优先把它落在现有可跑的服务级集成测试里，而不是虚构一个当前工程结构不支持的 `DesktopNpcRuntimeE2eTests`。

## 3. 架构决策 / ADR

### 3.1 决策

采用“canonical runtime owner/handle + 精确 identity key + runtime-refresh with explicit rebind rules + 两层 composition contract + provenance-first 收敛”的方案。

具体决策：

1. 先定义 canonical runtime 句柄，作为所有 live NPC path 取得或创建 runtime instance 的唯一合法入口。
2. `NpcRuntimeSupervisor` 直接扩展为 canonical runtime owner service。
3. `NpcRuntimeInstance` 扩展为持有 runtime handle state：agent/context/adapter/loop/cancellation/restart budget/rebind generation/pending-rebind。
4. canonical runtime 句柄的 runtime identity key 固定为 `gameId + saveId + npcId + profileId`；`sessionId` / `source` 只能挂在 session、transcript、activity、trace 审计维度。
5. `NpcRuntimeHost` 若不是 live path，只能是 consumer/orchestrator，不是 owner。
6. discovered tools 生命周期固定采用 `runtime-refresh with explicit rebind rules`。
7. 再冻结两层 contract：
   - 基础 Hermes 组合契约
   - Stardew 增强契约
8. 之后统一 save/session/source provenance，再统一 prompt/tools/discovered tools 语义。
9. 最后才根据代码证据决定 `NpcRuntimeHost` 是 live path 主入口、实现支撑层，还是工具/编排入口。
10. 保持单 Desktop 进程内托管多个 NPC runtime；不引入“一 NPC 一进程”。
11. 不新增独立 `StardewPromptAssembler`；游戏专属能力继续通过 `StardewGameAdapter`、`StardewNpcToolFactory`、`skills/gaming/stardew-*.md` 接入。

### 3.2 基础 Hermes 组合契约

基础 Hermes 组合契约必须定义并固定以下共享语义：

- base system prompt / prompt builder 主链
- `skillsMandatoryPrompt`
- built-in tools 注册语义
- discovered tools provider 与生命周期语义
- discovered tools refresh generation / pending-rebind contract
- context/runtime collaborators
- agent build 所需的标准 Hermes capability surface

它不应包含：

- NPC persona 文本事实
- Stardew save / NPC binding / scene-specific prompt
- Stardew bridge/tools/skill augmentation
- 任何 `private-chat` / `autonomy` / `debug` 专用 prompt patch

### 3.3 Stardew 增强契约

Stardew 增强契约只负责在基础契约之上提供：

- npc descriptor / binding 解析结果
- `saveId` / `sessionId` / `source` provenance
- runtime identity key 中固定使用的 `gameId + saveId + npcId + profileId`
- `NpcNamespace` root、memory、transcript、activity、trace、SOUL seed
- Stardew game bridge、game tools、game skills
- scene supplement，例如 private-chat 或 autonomy 所需的薄层场景上下文

它不应重新定义：

- base prompt builder
- 一套新的 Hermes tool stack
- discovered tools 生命周期的独立规则
- runtime key 中额外混入 `sessionId` / `source` 当隔离维度

### 3.4 运行时句柄 / 归属

canonical runtime 句柄必须回答四个问题：

1. 谁拥有 runtime instance：
   - `NpcRuntimeSupervisor` 是 canonical runtime owner service，负责创建、缓存、复用、释放，以及 agent/context/adapter/loop/cancellation/restart budget。
   - `NpcRuntimeInstance` 持有 runtime handle state：agent/context/adapter/loop/cancellation/restart budget/rebind generation/pending-rebind。
2. 谁能取到 runtime instance：
   - private-chat、autonomy/debug、可能的 host/live path 都只能通过 handle 进入。
3. runtime identity key 是什么：
   - 固定至少为 `gameId + saveId + npcId + profileId`，并与 `NpcRuntimeSupervisor.BuildKey(...)`、`NpcNamespace` 路径结构对齐。
   - `sessionId` / `source` 只能用于 session、activity、transcript、trace 审计，不得模糊替代隔离键。
4. discovered tools refresh 后如何 rebind：
   - `NpcRuntimeSupervisor` 必须维护 discovered tools generation / pending-rebind 状态。
   - refresh 后只允许在显式安全边界 rebind：private-chat 下一 reply turn 前、autonomy 下一 tick 前、debug 下一请求执行前。
   - 禁止在进行中的 LLM turn、tool call、adapter action 中途热替换。

约束：

- 任何 live NPC path 不得直接 `new NpcRuntimeContextFactory()` / `new NpcAgentFactory()` 组装身份后绕过 canonical handle。
- `NpcRuntimeHost` 若不是 live path，只是 handle/contract 的 consumer 或 orchestration layer，不自动成为设计中心；除非后续证据证明其本身就是 live path 主入口。

### 3.5 Live Path / Host Path 优先级与验证顺序

验证顺序必须明确：

1. 先确认 identity key、canonical runtime owner、runtime-refresh/rebind 规则。
2. 再确认 live path 是否全部消费 canonical handle，而不是各自直接 new。
3. 再确认两层 contract。
4. 再确认 provenance 是否贯穿 descriptor -> runtime handle -> namespace -> logs/tests。
5. 再确认 prompt / tools / discovered tools 语义是否统一。
6. 最后才判断 live path 与 host path 的关系：
   - 若证据证明 `NpcRuntimeHost` 已是 live path，则将其收敛为 canonical handle 的主 consumer。
   - 若证据证明 `NpcRuntimeHost` 不是 live path，则保持其为实现支撑或工具入口，不强制提升为产品主入口。

### 3.6 驱动因素

1. architect 已明确指出：不能默认 host 是真实产品入口。
2. 没有被写死的 canonical runtime owner / ownership，即使共享 builder 也仍会继续临时造 agent。
3. runtime identity key 不显式写死为 `gameId + saveId + npcId + profileId`，后续一定会继续把 `sessionId` / `source` 混进隔离语义。
4. 不把基础 Hermes 契约与 Stardew 增强契约分层，后续还会继续在 prompt/tools/provenance 上长分叉。
5. discovered tools 生命周期与 rebind 规则必须成为显式设计选择，而不是异步 MCP 初始化的隐式副作用。

### 3.7 备选方案

- 直接把 `NpcRuntimeHost` 定为唯一主线：
  - 拒绝。当前证据不足，且会把计划绑在未验证假设上。
- 只统一 builder，不统一 ownership：
  - 拒绝。不能解决临时造 agent 的根因。
- 用 `sessionId` 或 `source` 扩张 runtime key：
  - 拒绝。违背 `NpcRuntimeSupervisor.BuildKey(...)` 与 `NpcNamespace` 目录隔离事实，也会让同一 NPC runtime 被会话维度打碎。
- 新造 `StardewPromptAssembler`：
  - 拒绝。违背现有 PromptBuilder / ContextManager 约束，并制造第二套 prompt 系统。
- 回退为一 NPC 一进程：
  - 拒绝。违背本仓库既定 runtime topology 约束。

#### discovered tools 生命周期备选方案

- session-stable snapshot：
  - 拒绝。它适合一次性会话，但不适合 Desktop `InitializeMcpAsync(...)` 之后才完成工具接线、且还存在 reconnect/reload 的事实。
- runtime-refresh with explicit rebind rules：
  - 采纳。它能覆盖首次 connect 与后续 refresh，同时把行为锁进 `NpcRuntimeSupervisor` 的安全边界。
- raw live provider / implicit mutation：
  - 拒绝。它不需要显式状态，但会把异步工具变化直接暴露给进行中的 runtime，导致行为不可证明。

### 3.8 选择原因

因为它是唯一能同时做到：

- 保留 HermesCraft 的“标准 Hermes agent + 独立 identity/runtime roots + 同源能力面”；
- 把 runtime key 明确锁定到 `gameId + saveId + npcId + profileId`，同时把 `sessionId/source` 降回审计维度；
- 把 discovered tools 生命周期明确锁定到 `runtime-refresh with explicit rebind rules`；
- 保留本仓库单进程多 NPC runtime 约束；
- 不把 host 半成品误当成架构中心；
- 让 runtime ownership、provenance、prompt/tools/discovered-tools 生命周期都落入可验证合同。

### 3.9 影响

- `App.xaml.cs`、runtime 工厂、private-chat、debug、可能的 host path 都要改为 canonical owner/handle consumer。
- 验证重点从“能不能跑通某个场景”升级为“是否存在任何绕过 canonical handle 的 live path、是否有人把 `sessionId/source` 当 runtime key、是否有人在错误边界 rebind tools”。
- host 相关改动会延后到 contract/provenance/prompt-tools 收敛之后，避免主线误判。

### 3.10 后续事项

- 本轮不设计多 NPC 并发私聊会话模型；仍维持 `single active private-chat session per bridge`。
- 本轮不引入多进程 host。
- 本轮不引入独立 `StardewPromptAssembler`。
- 若未来需要 autonomy capability subset，必须在现有 base contract 之上另立 ADR，而不是本轮顺手做特例。

## 4. 具体实施步骤

### 步骤 1：先冻结 identity key / runtime owner / runtime-refresh rebind 规则

目标：
- 明确 `NpcRuntimeSupervisor` 是 live NPC path 获取 runtime 的唯一合法入口与 canonical runtime owner。
- 明确 runtime identity key 固定为 `gameId + saveId + npcId + profileId`。
- 明确 `NpcRuntimeInstance` 持有 runtime handle state，并由 `NpcRuntimeSupervisor` 负责 agent/context/adapter/loop/cancellation/restart budget。
- 明确 discovered tools 采用 `runtime-refresh with explicit rebind rules`，以及 refresh 触发源和 rebind 安全边界。

工作：
- 审查现有 `NpcRuntimeHost`、`NpcRuntimeSupervisor`、`NpcRuntimeInstance`、`NpcRuntimeContextFactory`、`NpcAgentFactory`、private-chat/debug 的职责边界。
- 定义 canonical runtime 句柄的输入、输出、ownership 规则。
- 将 `NpcRuntimeInstance` 的状态面扩展到 agent/context/adapter/loop/cancellation/restart budget/rebind generation/pending-rebind。
- 定义 refresh generation / pending-rebind 状态以及 private-chat、autonomy、debug 的显式 rebind 边界。
- 建立代码结构或测试约束，禁止 live path 在外层直接 new runtime context/agent。

完成标志：
- 计划与代码层都能指出“live NPC path 只能经由哪个 handle 进来”。
- 文本和测试都明确 `gameId + saveId + npcId + profileId` 是唯一 runtime 隔离键，`sessionId/source` 不是。
- `NpcRuntimeHost` 在此阶段只作为待验证 consumer/host-path 候选，不被预设为 owner。

### 步骤 2：让所有 live paths 消费 canonical handle

目标：
- private-chat、autonomy/debug、以及任何当前 live NPC path 全部改为通过 canonical handle 取 runtime。

工作：
- private-chat runner / orchestrator 通过 canonical runtime 句柄获取 runtime。
- autonomy/debug 通过 canonical runtime 句柄获取 runtime。
- `NpcRuntimeHost` 若参与 live path，只能作为 consumer/orchestrator 调用 `NpcRuntimeSupervisor`；若不是 live path，则保持后置。
- 保持 `single active private-chat session per bridge` 语义不变。

完成标志：
- 已知 live path 不再自行 `new` context/agent。
- runtime owner 成为唯一持有 agent/context/adapter/loop/cancellation/restart budget 的地方。

### 步骤 3：统一 prompt / tools / augmentation 契约

目标：
- 形式化基础 Hermes 组合契约与 Stardew 增强契约。
- 统一 prompt / built-in tools / discovered tools refresh 语义，同时继续禁止独立 `StardewPromptAssembler`。

工作：
- 基于 `AgentCapabilityAssembler` 收敛 base contract。
- 明确 Stardew augmentation 只提供 persona/session/root、game bridge/tools/skills、scene supplement 与 provenance。
- 将 private-chat/autonomy 差异下沉为 augmentation supplement。
- 将 discovered tools refresh/rebind 规则挂到 owner/handle，而不是散在入口里读 `McpManager.Tools.Values`。

完成标志：
- 任何“共享能力面”都在 base contract。
- 任何“Stardew/NPC 差异”都在 augmentation contract。
- 没有设计要求依赖独立 `StardewPromptAssembler`。

### 步骤 4：统一身份来源（`saveId` / `sessionId` / `source`）

目标：
- 让所有 NPC 入口只能从 catalog/manifest/pack/descriptor provenance 得到业务身份。
- 让 key / session / source 的边界在 descriptor、namespace、日志、trace 中一致。

工作：
- 收敛 `StardewNpcRuntimeBindingResolver`、descriptor factory、runtime handle key 与 namespace root 的 provenance 字段。
- 去掉 `haley`、`manual-debug` 一类默认业务身份兜底。
- 让 logs / trace / transcripts / activity / tests 能回溯 `saveId`、`sessionId`、`source`，并明确它们不替代 runtime key。

完成标志：
- live NPC path 的身份来源可审计，且跨 descriptor、runtime、日志一致。
- `sessionId/source` 只留在会话/审计层。

### 步骤 5：补齐回归测试与可观测断言

目标：
- 把“ownership/contract/provenance/lifecycle 已统一”变成自动可验证事实。

工作：
- 增加 canonical runtime 句柄 / owner tests。
- 增加 base/augmentation contract parity tests。
- 增加 provenance 审计 tests。
- 增加 discovered tools refresh/rebind lifecycle tests。
- 增加 Penny 私聊与 autonomy/debug 路径 tests。
- 若需要，扩展 runtime/activity/transcript trace 字段方便断言。

完成标志：
- 删除任一关键修复后，相关测试必须失败。

### 步骤 6：最后判定 host 是否需要提升为 live path 主入口

目标：
- 在 contract、ownership、provenance、prompt/tools 统一后，再对 host/live path 做证据驱动决策。

工作：
- 验证 `NpcRuntimeHost` 当前是否真实承载产品 live path。
- 若是，则让它消费 canonical handle 并承担协调入口职责。
- 若不是，则保持为实现支撑或工具入口，避免强行架构中心化。

完成标志：
- host 的定位由代码证据决定，而不是由计划预设决定。

15. `single active private-chat session per bridge` 语义明确成立：
   - bridge 空闲时，任意 NPC 可开启会话；
   - 活跃会话期间，其他 NPC 不抢占；
   - 会话结束后，新的 NPC 可重新开启。
16. 每角色继续保持独立 `SOUL.md`、memory、sessions、transcripts、activity、trace 根目录。
17. 游戏专属能力仍通过 game bridge / game tools / game skills 接入，没有新增独立 `StardewPromptAssembler`，也没有回退为多进程。

## 8. 风险

1. canonical runtime owner / handle 若定义不清，会出现“形式上统一 contract，实质上仍然场景内临时造 agent”的伪收敛。
2. `gameId + saveId + npcId + profileId` 若没有被严格写死，后续仍会有人把 `sessionId` / `source` 混入 runtime 隔离逻辑。
3. 两层契约若拆分不严谨，Stardew 增强仍可能反向污染基础 Hermes 组合契约。
4. discovered tools 生命周期已选 `runtime-refresh with explicit rebind rules`；这可能暴露现有 Desktop 或 NPC 某些依赖于模糊行为的隐式假设，需要测试与调用方同步调整。
5. 去掉 `manual-debug` 默认身份后，现有调试入口可能需要显式提供 provenance 参数，调试基建与测试基线要同步补齐。
6. 这次不强制 `NpcRuntimeHost` 必须成为产品入口；如果证据证明它不是 live path，应保持为实现支撑或工具入口，而不是被计划强推为唯一主轴。
7. 这次不引入多进程，因此所有 ownership / lifetime / refresh / rebind 语义都必须在单进程多 runtime 拓扑内自洽解决。
8. 这次不引入独立 `StardewPromptAssembler`，因此所有 Stardew 差异都必须在 augmentation contract 内表达，不能靠新 prompt 子系统逃避边界设计。

## 9. 非目标

1. 不强制 `NpcRuntimeHost` 必须成为产品 live path 主入口；若证据不足或相反，保持其为实现支撑或工具入口即可。
2. 不实现“一 NPC 一进程”，也不引入多进程 runtime 模型。
3. 不新增独立 `StardewPromptAssembler`。
4. 不设计多 NPC 并发 private-chat 模型。
5. 不在本轮做 autonomy capability subset/policy gating。
6. 不修改与本任务无关的 transcript/openspec 脏改文件。

## 10. 推荐执行模式

### 推荐方案

推荐后续用 **`team`**，不是 `ralph`。

原因：

- 这是跨 runtime ownership、composition contract、Stardew flow/private-chat、Desktop DI、tests 的高风险收敛任务。
- 至少存在三条可以并行但需要 leader 集成的工作面：canonical handle + contracts、provenance + Stardew flows、tests + verification。
- host/live path 判定适合在前几条 lane 完成后由 leader 做证据整合，而不是一开始就把所有人绑到 host 改造线上。

### 建议分工

1. `team-executor` / `executor` 分工：canonical runtime 句柄 + 基础/增强契约
   - 负责 `AgentCapabilityAssembler`、runtime handle、`NpcRuntimeContextFactory`、`NpcAgentFactory`、`App.xaml.cs` 与 contract 边界收敛。
   - reasoning: `high`

2. `executor` / `debugger` lane：provenance + Stardew live paths
   - 负责 `StardewNpcRuntimeBindingResolver`、descriptor provenance、`StardewPrivateChatOrchestrator`、`StardewAutonomyTickDebugService`、`PrivateChatOrchestrator` 的消费者收敛。
   - reasoning: `high`

3. `test-engineer` / `verifier` lane：ownership/provenance/lifecycle/parity verification
   - 负责新增/修订 canonical-handle、provenance、discovered-tools lifecycle、Penny 私聊与 autonomy/debug 回归测试。
   - reasoning: `medium`

4. 可选 `critic` / `code-reviewer` lane：host/live-path 判定与回归审查
   - 专门检查是否错误预设 host 为主入口，或又偷偷引入新的 prompt patch、identity fallback、tool lifecycle drift。
   - reasoning: `high`

### 启动提示

建议后续 handoff 采用：

```text
$team
第 1 lane：canonical runtime 句柄 + 基础/增强契约
lane 2: provenance + stardew private-chat/autonomy live paths
lane 3: tests + verification
lane 4: host/live-path review gate
```

### 团队验证路径

1. lane 1 先冻结 canonical handle 与 two-layer contracts。
2. lane 2 在此基础上统一 provenance 与 live path consumers。
3. lane 3 跑 ownership/provenance/lifecycle 定向测试，再跑 Runtime/Stardew 全集。
4. lane 4 最后审查 host 是否应成为 live path 主入口；若证据不足，不强推提升。
