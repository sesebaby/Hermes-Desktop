# RALPLAN: Stardew Haley Direct Private Chat Agent Loop

## Status

Verdict: APPROVE for execution handoff after consensus review.

Context snapshot:
- `.omx/context/stardew-direct-private-chat-agent-loop-20260430T075824Z.md`

Reviewer notes:
- Planner selected the Desktop/core private-chat orchestrator approach.
- Architect approved the approach, with hard requirements for a single state machine, explicit `vanilla_dialogue_completed -> open_private_chat` transition, correlation/session dedupe, and Desktop/core ownership of agent execution.
- Critic first returned `ITERATE`; this plan incorporates the required concrete event contract, failure paths, concurrency behavior, reopen policy, interface boundaries, and test specification.

## User Flow

Target flow:

```text
Click Haley, finish vanilla dialogue
  -> HUD: 海莉知道你想和她聊天
  -> Desktop/core immediately opens private-chat input
  -> Player types a message
  -> Input closes; HUD: 海莉正在思考怎么回答你
  -> Desktop/core Haley agent generates reply
  -> Game displays Haley reply
  -> Optional: reopen input for another round
```

The key fix is that the Desktop/core runtime must actively consume the bridge event and send `/action/open_private_chat`. The current bridge-only HUD slice records `vanilla_dialogue_completed_fact`, but no Desktop consumer calls `OpenPrivateChat`, so the input never appears.

## RALPLAN-DR Summary

### Principles

1. Keep SMAPI as bridge/UI only: no LLM calls, no persona, no memory, no agent session state.
2. Keep Desktop/core as the owner of private-chat orchestration, Haley agent execution, tool calls, memory, and session state.
3. Preserve ordinary bridge events as facts; only `player_private_message_submitted` is a narrow explicit-user-input exception.
4. Make the private-chat loop a single state machine with explicit dedupe, timeout, failure, and concurrency rules.
5. Prefer typed event/command contracts over parsing human summaries.

### Decision Drivers

1. Manual testing already proved the missing component: `vanilla_dialogue_completed_fact` is recorded, but no Desktop/core consumer sends `/action/open_private_chat`.
2. The player must see immediate UI feedback and must not wait 10s for the Agent before the input appears.
3. The implementation must not turn the game event stream into a hidden semantic driver for general NPC autonomy.

### Options

#### Option A: Desktop/core dedicated private-chat orchestrator

Chosen.

Pros:
- Directly fixes the missing automatic `open_private_chat` command.
- Keeps LLM/session/persona/memory in Desktop/core.
- Separates player-driven chat from autonomous NPC ticking.
- Gives a small, testable state machine.

Cons:
- Adds a dedicated orchestration path that must not become an uncontrolled per-NPC special-case pile.

#### Option B: Put the special case into `NpcAutonomyLoop`

Rejected.

Reason: `NpcAutonomyLoop` owns autonomous observe/decide/action ticks. Player-submitted private chat is an explicit conversational input, not an ambient world fact. Mixing them would blur the fact-only event boundary and make autonomy behavior harder to reason about.

#### Option C: Let SMAPI directly open the input after vanilla dialogue

Rejected.

Reason: it is fastest locally, but it moves chat UX ownership into the mod and weakens the Desktop/core boundary. The bridge should expose UI primitives and record events; Desktop/core should decide when to invoke them.

## ADR

Decision: implement a Desktop/core `StardewPrivateChatOrchestrator` for Haley first. It polls bridge events, maps Haley `vanilla_dialogue_completed` facts to `OpenPrivateChat`, maps explicit `player_private_message_submitted` events to Haley agent input, displays the reply through `Speak`, and optionally reopens input under a bounded policy.

Drivers:
- The observed failure is an absent Desktop consumer, not a broken SMAPI text-entry path.
- Agent latency must not delay opening the player's input box.
- The game bridge must remain a thin boundary, not an Agent host.

Alternatives considered:
- Extend `NpcAutonomyLoop`.
- Open the private chat locally in SMAPI.
- Keep using only manual Desktop debug ticks.

Why chosen:
- It is the smallest architecture that fixes the UX defect while preserving ownership boundaries.
- It makes the exception explicit: player private-chat text is direct user input, not an inferred world event.
- It is testable without live Stardew Valley.

Consequences:
- Adds one Desktop/core orchestrator and a small options object.
- Requires typed event payloads instead of summary parsing.
- Reopen behavior must be policy-driven and capped to avoid loops.

Follow-ups:
- Generalize beyond Haley only after this single-NPC loop is stable.
- Consider promoting the orchestrator interface to multi-NPC private chat after more than one NPC needs it.
- Keep background autonomy scheduling separate from this player-driven loop.

## Event And Command Contract

Extend the bridge event contract so Desktop/core never parses `Summary` to recover machine data.

### DTO changes

Update both bridge-side and Desktop-side DTOs:

- `Mods/StardewHermesBridge/Bridge/BridgeCommandModels.cs`
- `src/games/stardew/StardewBridgeDtos.cs`
- `src/game/core/GameObservation.cs` if core event records need payload access

Add optional fields to event data:

```csharp
string? CorrelationId
JsonObject? Payload
```

Keep existing fields:

```csharp
string EventId
string EventType
string? NpcId
DateTime TimestampUtc
string Summary
```

If adding `JsonObject` to `GameEventRecord` is too broad, introduce a Stardew-specific event wrapper inside the orchestrator and keep the core game event record unchanged. Do not parse private message text out of `Summary`.

### Event types

`vanilla_dialogue_completed`
- Source: `Mods/StardewHermesBridge/ModEntry.RecordVanillaDialogueCompleted`.
- `NpcId`: SMAPI name, for this slice only `Haley` should trigger the private-chat orchestrator.
- `CorrelationId`: default to the event id when SMAPI records the event, or `dialogue:{eventId}` if a prefixed value is clearer.
- `Payload`: `{ "source": "formal_npc_click" }`.
- Effect: Desktop/core orchestrator may auto-open private chat.

`private_chat_opened`
- Source: `BridgeCommandQueue.OpenPrivateChat`.
- `NpcId`: opened NPC name.
- `CorrelationId`: the `conversationId` passed by Desktop/core, falling back to the command id/request id.
- `Payload`: `{ "conversationId": "...", "prompt": "..." }`.
- Effect: observation only; useful for logs/tests.
- This event is not the authoritative success signal for the Desktop state transition. The authoritative success signal is the `/action/open_private_chat` command response with `Ok == true` and `Data.Opened == true`.

`player_private_message_submitted`
- Source: `BridgeCommandQueue.OpenPrivateChat` text-entry submit handler.
- `NpcId`: opened NPC name.
- `CorrelationId`: same `conversationId` as the private-chat input.
- `Summary`: generic text such as `Player submitted a private chat message.` Do not duplicate raw player text in summary.
- `Payload`: `{ "conversationId": "...", "text": "...", "submittedAtUtc": "..." }`.
- Effect: narrow exception; Desktop/core routes `payload.text` directly to Haley private-chat agent execution.

`player_private_chat_cancelled` or timeout fallback
- Source: bridge if close/cancel can be detected; otherwise Desktop/core timeout.
- `CorrelationId`: current `conversationId`.
- Effect: end current private-chat session without Agent call.

### Command changes

Extend `OpenPrivateChatPayload` / `StardewOpenPrivateChatRequest` with:

```csharp
string? ConversationId
```

`StardewCommandService.SubmitOpenPrivateChatAsync` should read `action.Payload["conversationId"]` and send it to `/action/open_private_chat`.

The orchestrator should create:

```text
traceId = trace_private_chat_{guid}
idempotencyKey = private_chat:{saveId}:{npcId}:{dialogueEventId}
conversationId = pc_{dialogueEventId}
```

## Orchestrator Boundary

Add `StardewPrivateChatOrchestrator` under `src/games/stardew/`.

Responsibilities:
- Own the private-chat state machine.
- Poll `StardewEventSource` with a retained `GameEventCursor`.
- Filter to Haley for this slice.
- Call `StardewCommandService.SubmitAsync` with `GameActionType.OpenPrivateChat`.
- Route explicit player text to a Haley agent runner built from existing NPC runtime primitives.
- Call `GameActionType.Speak` through `StardewCommandService` for the final reply.
- Write trace/log entries useful for manual validation.

Non-responsibilities:
- It must not replace `NpcAutonomyLoop`.
- It must not infer semantic actions from ordinary events.
- It must not call SMAPI HTTP directly; use `StardewCommandService`.
- It must not create a second prompt assembler; reuse existing NPC runtime/context/agent factory surfaces.

Suggested narrow collaborator:

```csharp
public interface INpcPrivateChatAgentRunner
{
    Task<NpcPrivateChatReply> ReplyAsync(
        NpcPackManifest npc,
        string playerText,
        string conversationId,
        CancellationToken ct);
}
```

The runner should use existing `NpcRuntimeContextFactory`, `NpcAgentFactory`, NPC namespace/memory/transcript, and `StardewNpcToolFactory` where feasible. If this becomes too large for one slice, implement a minimal testable runner adapter first and keep the orchestrator independent from the concrete Agent construction.

Register the orchestrator in `Desktop/HermesDesktop/App.xaml.cs` near existing Stardew services. Start it as a cancellable background Desktop service after DI is built, not from SMAPI. It should reconnect/poll when bridge discovery becomes available.

## State Machine

States:

```text
Idle
PendingOpen
AwaitingPlayerInput
WaitingAgentReply
ShowingReply
OptionalReopen
FailedRecoverable
```

Transitions:

- `Idle + vanilla_dialogue_completed(Haley)`:
  - create `conversationId` from `eventId`,
  - mark `dialogueEventId` processed,
  - call `OpenPrivateChat`,
  - move to `PendingOpen`.

- `PendingOpen + open accepted`:
  - accepted means the `OpenPrivateChat` command response returned `Ok == true` and `Data.Opened == true`,
  - `private_chat_opened` remains a bridge observation/log event, not the state transition source,
  - move to `AwaitingPlayerInput`.

- `PendingOpen + retryable open failure` (`menu_blocked`, `world_not_ready`):
  - retry with short bounded backoff, then move to `FailedRecoverable -> Idle`.

- `PendingOpen + non-retryable open failure`:
  - log and move to `Idle`.

- `AwaitingPlayerInput + player_private_message_submitted` with matching `conversationId`:
  - move to `WaitingAgentReply`,
  - run Haley agent asynchronously,
  - bridge HUD shows `海莉正在思考怎么回答你`.

- `AwaitingPlayerInput + cancel/timeout`:
  - move to `Idle`,
  - no Agent call.

- `WaitingAgentReply + agent reply success`:
  - call `Speak`,
  - move to `ShowingReply`.

- `WaitingAgentReply + agent timeout/error/cancel`:
  - clear or expire thinking HUD,
  - log failure,
  - move to `Idle`.
  - Do not fabricate a Haley persona reply in SMAPI.

- `WaitingAgentReply + new vanilla_dialogue_completed`:
  - ignore or record `busy` for this session; do not open another input.

- `ShowingReply + speak accepted`:
  - move to `OptionalReopen`.

- `OptionalReopen`:
  - apply reopen policy; either call `OpenPrivateChat` again with a new turn id or return to `Idle`.

## Dedupe And Reconnect Rules

- Keep an in-memory set of processed `dialogueEventId` values for the active Desktop process.
- The idempotency key for `OpenPrivateChat` must include `saveId`, `npcId`, and `dialogueEventId`.
- Duplicate `vanilla_dialogue_completed` events with the same `eventId` or correlation id must not open a second input.
- While Desktop is running, bridge reconnect should resume from the retained `GameEventCursor` so buffered events after the cursor are processed once.
- Across Desktop restart, UI-trigger events are ephemeral: bootstrap by polling and advancing to the latest event without opening stale inputs. This avoids surprise chat windows from old dialogue completions after restart.
- If persistent cursor storage is added later, it must still not replay old UI-open commands after process restart.

## Reopen Policy

Add a small options object:

```csharp
public sealed record StardewPrivateChatOptions(
    string NpcId = "haley",
    PrivateChatReopenPolicy ReopenPolicy = PrivateChatReopenPolicy.OnceAfterReply,
    int MaxTurnsPerSession = 3,
    TimeSpan PlayerInputTimeout = default,
    TimeSpan AgentReplyTimeout = default);
```

Policy values:

```text
Never
OnceAfterReply
UntilCancelled
```

Default for this Haley slice:
- `OnceAfterReply`
- `MaxTurnsPerSession = 3`
- user cancel/empty submission ends the session immediately
- any agent error/timeout ends the session

This gives continuous conversation without allowing an infinite reopen loop.

## Bridge HUD Rules

Existing pending HUD remains:

```text
海莉知道你想和她聊天
她正在考虑是否回答你
```

Required adjustment:
- Pending HUD appears when vanilla Haley dialogue completion is recorded.
- Pending HUD clears when `OpenPrivateChat` succeeds.
- On `player_private_message_submitted`, bridge switches to thinking HUD:

```text
海莉正在思考怎么回答你
```

- Thinking HUD clears when `Speak` displays a reply, when the orchestrator sends a clear command if one exists, or by TTL fallback.
- These HUD changes are UI status only; they must not call LLMs or drive Agent behavior in SMAPI.

## Acceptance Criteria

1. A live Haley `vanilla_dialogue_completed` event causes Desktop/core to call `/action/open_private_chat` without manual debug tick.
2. The input opens before any Haley Agent reply is requested.
3. `player_private_message_submitted` carries raw player text in a typed payload and routes that text to the Haley private-chat Agent runner.
4. Ordinary bridge events remain facts and do not call the Agent.
5. Duplicate dialogue completion events do not open duplicate inputs.
6. While waiting for Haley's reply, new completion events do not open another input.
7. Agent success displays the reply through the existing `Speak` command path.
8. Agent timeout/error returns to a stable idle state and does not leave the HUD permanently stuck.
9. Other NPCs' vanilla dialogue completions do not trigger Haley's private-chat orchestrator.
10. SMAPI contains no LLM/persona/memory/session logic.
