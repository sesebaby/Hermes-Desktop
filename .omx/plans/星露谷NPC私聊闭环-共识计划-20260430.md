# RALPLAN: 星露谷 Haley 直接私聊 Agent 闭环

## 状态

审阅者备注：

- 架构师批准了该方案，但提出了硬性要求：必须使用单一状态机、明确的 `vanilla_dialogue_completed -> open_private_chat` 迁移、关联/会话去重，以及由 Desktop/core 负责 Agent 执行。
- 批评者最初返回 `ITERATE`；本计划已纳入所需的具体事件契约、失败路径、并发行为、重开策略、接口边界和测试规格。

## 用户流程

目标流程：

```text
点击 Haley，完成原版对话
  -> HUD: 海莉知道你想和她聊天
  -> Desktop/core 立即打开私聊输入框
  -> 玩家输入一条消息
  -> 输入框关闭；HUD: 海莉正在思考怎么回答你
  -> Desktop/core Haley Agent 生成回复
  -> 游戏显示 Haley 的回复
  -> 可选：重新打开输入框进行下一轮
```

关键修复点是 Desktop/core 运行时必须主动消费桥接事件并发送 `/action/open_private_chat`。当前仅有桥接层的 HUD 片段会记录 `vanilla_dialogue_completed_fact`，但没有 Desktop 消费者调用 `OpenPrivateChat`，因此输入框不会出现。

## RALPLAN-DR 摘要

### 原则

1. 保持 SMAPI 只做桥接/UI：不做 LLM 调用、不承担角色人格、不持久化记忆，也不保存 Agent 会话状态。
2. 保持 Desktop/core 作为私聊编排、Haley Agent 执行、工具调用、记忆和会话状态的拥有者。
3. 将普通桥接事件保留为事实；只有 `player_private_message_submitted` 是一个狭义的显式用户输入例外。
4. 将私聊闭环实现为单一状态机，并明确去重、超时、失败和并发规则。
5. 优先使用类型化事件/命令契约，不解析人类可读摘要来还原机器数据。

### 决策驱动因素

1. 手动测试已经证明缺失的组件是：`vanilla_dialogue_completed_fact` 已被记录，但没有 Desktop/core 消费者发送 `/action/open_private_chat`。
2. 玩家必须立即看到 UI 反馈，不能因为等待 Agent 而延迟 10 秒才出现输入框。
3. 实现不能把游戏事件流变成通用 NPC 自主行为的隐藏语义驱动器。

### 方案

#### 方案 A：Desktop/core 专用私聊编排器

已选。

优点：
- 直接修复缺失的自动 `open_private_chat` 命令。
- 将 LLM/会话/人格/记忆保留在 Desktop/core。
- 将玩家驱动的聊天与自动化 NPC tick 分离。
- 提供一个小而可测试的状态机。

缺点：
- 增加了一条专门的编排路径，且不能演变成失控的按 NPC 特例堆积。

#### 方案 B：把特殊情况放进 `NpcAutonomyLoop`

已拒绝。

原因：`NpcAutonomyLoop` 负责自主的观察/决策/行动 tick。玩家提交的私聊是显式会话输入，不是环境中的普通世界事实。把二者混在一起会模糊事实型事件边界，并让自主行为更难推理。

#### 方案 C：让 SMAPI 在原版对话结束后直接打开输入框

已拒绝。

原因：这在本地最快，但会把聊天 UI 的所有权移入 mod，并削弱 Desktop/core 边界。桥接层应暴露 UI 原语并记录事件；Desktop/core 应决定何时调用它们。

## ADR

决策：为 Haley 先实现一个 Desktop/core 的 `StardewPrivateChatOrchestrator`。它轮询桥接事件，将 Haley 的 `vanilla_dialogue_completed` 事实映射为 `OpenPrivateChat`，将显式的 `player_private_message_submitted` 事件映射为 Haley Agent 输入，通过 `Speak` 展示回复，并在受限策略下可选地重新打开输入框。

驱动因素：
- 观察到的故障是 Desktop 消费者缺失，而不是 SMAPI 文本输入路径损坏。
- Agent 延迟不能影响玩家输入框的打开。
- 游戏桥接必须保持为薄边界，而不是 Agent 主机。

备选方案：
- 扩展 `NpcAutonomyLoop`。
- 在 SMAPI 本地打开私聊。
- 继续只使用手动 Desktop 调试 tick。

选择原因：
- 这是在保留所有权边界的前提下，修复该 UX 缺陷所需的最小架构。
- 它把例外显式化：玩家私聊文本是直接用户输入，而不是推断出的世界事件。
- 它无需运行中的星露谷即可测试。

后果：
- 增加一个 Desktop/core 编排器和一个小型选项对象。
- 需要使用类型化事件负载，而不是摘要解析。
- 重新打开行为必须由策略驱动并设置上限，以避免循环。

后续事项：
- 只有在这个单 NPC 闭环稳定后，才推广到 Haley 之外。
- 在不止一个 NPC 需要时，再考虑把编排器接口推广为多 NPC 私聊。
- 保持后台自主调度与这个玩家驱动闭环分离。

## 事件与命令契约

扩展桥接事件契约，避免 Desktop/core 再通过解析 `Summary` 来还原机器数据。

### DTO 变更

同时更新桥接侧和 Desktop 侧 DTO：

- `Mods/StardewHermesBridge/Bridge/BridgeCommandModels.cs`
- `src/games/stardew/StardewBridgeDtos.cs`
- 如果核心事件记录需要负载访问，则更新 `src/game/core/GameObservation.cs`

为事件数据增加可选字段：

```csharp
string? CorrelationId
JsonObject? Payload
```

保留现有字段：

```csharp
string EventId
string EventType
string? NpcId
DateTime TimestampUtc
string Summary
```

如果给 `GameEventRecord` 添加 `JsonObject` 的影响范围太大，就在 orchestrator 内部引入一个 Stardew 专用事件包装器，并保持核心游戏事件记录不变。不要从 `Summary` 中解析私聊文本。

### 事件类型

`vanilla_dialogue_completed`
- 来源：`Mods/StardewHermesBridge/ModEntry.RecordVanillaDialogueCompleted`。
- `NpcId`：SMAPI 名称；在这个切片里只有 `Haley` 应触发私聊 orchestrator。
- `CorrelationId`：在 SMAPI 记录事件时默认使用事件 id；如果带前缀的值更清晰，则使用 `dialogue:{eventId}`。
- `Payload`：`{ "source": "formal_npc_click" }`。
- 效果：Desktop/core orchestrator 可以自动打开私聊。

`private_chat_opened`
- 来源：`BridgeCommandQueue.OpenPrivateChat`。
- `NpcId`：被打开的 NPC 名称。
- `CorrelationId`：Desktop/core 传入的 `conversationId`，若无则回退到 command id/request id。
- `Payload`：`{ "conversationId": "...", "prompt": "..." }`。
- 效果：仅用于观察；对日志/测试很有帮助。
- 这个事件不是 Desktop 状态迁移的权威成功信号。权威成功信号是 `/action/open_private_chat` 命令响应中的 `Ok == true` 且 `Data.Opened == true`。

`player_private_message_submitted`
- 来源：`BridgeCommandQueue.OpenPrivateChat` 的文本输入提交处理器。
- `NpcId`：被打开的 NPC 名称。
- `CorrelationId`：与私聊输入相同的 `conversationId`。
- `Summary`：使用通用文本，例如 `Player submitted a private chat message.`。不要在 summary 中重复原始玩家文本。
- `Payload`：`{ "conversationId": "...", "text": "...", "submittedAtUtc": "..." }`。
- 效果：这是一个狭义例外；Desktop/core 直接把 `payload.text` 路由给 Haley 私聊 Agent 执行。

`player_private_chat_cancelled` 或超时回退
- 来源：如果能检测到关闭/取消，则来自桥接层；否则来自 Desktop/core 的超时处理。
- `CorrelationId`：当前 `conversationId`。
- 效果：结束当前私聊会话，不触发 Agent 调用。

### 命令变更

为 `OpenPrivateChatPayload` / `StardewOpenPrivateChatRequest` 扩展以下字段：

```csharp
string? ConversationId
```

`StardewCommandService.SubmitOpenPrivateChatAsync` 应读取 `action.Payload["conversationId"]`，并将其发送到 `/action/open_private_chat`。

orchestrator 应创建：

```text
traceId = trace_private_chat_{guid}
idempotencyKey = private_chat:{saveId}:{npcId}:{dialogueEventId}
conversationId = pc_{dialogueEventId}
```

## Orchestrator 边界

在 `src/games/stardew/` 下添加 `StardewPrivateChatOrchestrator`。

职责：
- 拥有私聊状态机。
- 使用保留的 `GameEventCursor` 轮询 `StardewEventSource`。
- 针对这个切片仅过滤 Haley。
- 调用 `StardewCommandService.SubmitAsync` 并传入 `GameActionType.OpenPrivateChat`。
- 将显式玩家文本路由到一个 Haley Agent runner，该 runner 基于已有的 NPC 运行时原语构建。
- 通过 `StardewCommandService` 调用 `GameActionType.Speak` 输出最终回复。
- 写入对人工验证有帮助的 trace/log 记录。

非职责：
- 不能替代 `NpcAutonomyLoop`。
- 不能从普通事件中推断语义动作。
- 不能直接调用 SMAPI HTTP；必须使用 `StardewCommandService`。
- 不能创建第二套 prompt 组装器；应复用现有的 NPC runtime/context/agent factory 面。

建议的窄协作者：

```csharp
public interface INpcPrivateChatAgentRunner
{
    Task<NpcPrivateChatReply> ReplyAsync(
        NpcPackManifest npc,
        string playerText,
        string conversationId,
        CancellationToken ct);
}
```

runner 应尽可能使用现有的 `NpcRuntimeContextFactory`、`NpcAgentFactory`、NPC 命名空间/记忆/转录，以及 `StardewNpcToolFactory`。如果这在一个切片内变得过大，就先实现一个最小可测试的 runner 适配器，并保持 orchestrator 与具体 Agent 构造解耦。

在 `Desktop/HermesDesktop/App.xaml.cs` 中、靠近现有 Stardew 服务的位置注册 orchestrator。它应在 DI 构建完成后作为可取消的后台 Desktop 服务启动，而不是从 SMAPI 启动。桥接发现可用后，它应该重新连接/轮询。

## 状态机

状态：

```text
Idle
PendingOpen
AwaitingPlayerInput
WaitingAgentReply
ShowingReply
OptionalReopen
FailedRecoverable
```

迁移：

- `Idle + vanilla_dialogue_completed(Haley)`：
  - 由 `eventId` 创建 `conversationId`，
  - 标记 `dialogueEventId` 已处理，
  - 调用 `OpenPrivateChat`，
  - 进入 `PendingOpen`。

- `PendingOpen + 接受成功`：
  - 接受成功指 `OpenPrivateChat` 命令响应返回 `Ok == true` 且 `Data.Opened == true`，
  - `private_chat_opened` 仍然只是桥接观察/日志事件，不是状态迁移来源，
  - 进入 `AwaitingPlayerInput`。

- `PendingOpen + 可重试的打开失败`（`menu_blocked`、`world_not_ready`）：
  - 使用短的受限退避重试，然后进入 `FailedRecoverable -> Idle`。

- `PendingOpen + 不可重试的打开失败`：
  - 记录日志并进入 `Idle`。

- `AwaitingPlayerInput + player_private_message_submitted` 且 `conversationId` 匹配：
  - 进入 `WaitingAgentReply`，
  - 异步运行 Haley Agent，
  - bridge HUD 显示 `海莉正在思考怎么回答你`。

- `AwaitingPlayerInput + 取消/超时`：
  - 进入 `Idle`，
  - 不调用 Agent。

- `WaitingAgentReply + Agent 回复成功`：
  - 调用 `Speak`，
  - 进入 `ShowingReply`。

- `WaitingAgentReply + Agent 超时/错误/取消`：
  - 清除或让思考 HUD 过期，
  - 记录失败，
  - 进入 `Idle`。
  - 不要在 SMAPI 中伪造 Haley 的人格回复。

- `WaitingAgentReply + 新的 vanilla_dialogue_completed`：
  - 对这个会话忽略，或记录为 `busy`；不要再打开另一个输入框。

- `ShowingReply + speak accepted`：
  - 进入 `OptionalReopen`。

- `OptionalReopen`：
  - 应用重新打开策略；要么用新的 turn id 再次调用 `OpenPrivateChat`，要么返回 `Idle`。

## 去重与重连规则

- 为活动中的 Desktop 进程保留一个已处理 `dialogueEventId` 的内存集合。
- `OpenPrivateChat` 的幂等键必须包含 `saveId`、`npcId` 和 `dialogueEventId`。
- 带有相同 `eventId` 或 correlation id 的重复 `vanilla_dialogue_completed` 事件不能再次打开输入框。
- Desktop 运行期间，桥接重连应从保留的 `GameEventCursor` 继续，这样游标之后的缓冲事件只会处理一次。
- 跨 Desktop 重启时，UI 触发事件是短暂的：启动时应通过轮询并推进到最新事件，但不要打开旧输入框。这样可以避免进程重启后，旧的对话完成事件突然弹出聊天窗口。
- 如果以后添加持久化游标存储，它也必须在进程重启后不重放旧的 UI 打开命令。

## 重新打开策略

添加一个小型选项对象：

```csharp
public sealed record StardewPrivateChatOptions(
    string NpcId = "haley",
    PrivateChatReopenPolicy ReopenPolicy = PrivateChatReopenPolicy.OnceAfterReply,
    int MaxTurnsPerSession = 3,
    TimeSpan PlayerInputTimeout = default,
    TimeSpan AgentReplyTimeout = default);
```

策略值：

```text
Never
OnceAfterReply
UntilCancelled
```

这个 Haley 切片的默认值：
- `OnceAfterReply`
- `MaxTurnsPerSession = 3`
- 用户取消/空提交会立即结束会话
- 任何 Agent 错误/超时都会结束会话

这样既能连续对话，又不会形成无限重开循环。

## Bridge HUD 规则

现有的 pending HUD 保持不变：

```text
海莉知道你想和她聊天
她正在考虑是否回答你
```

必须调整为：
- 当记录到 Haley 的原版对话完成时，显示 pending HUD。
- 当 `OpenPrivateChat` 成功时，清除 pending HUD。
- 在 `player_private_message_submitted` 时，bridge 切换为 thinking HUD：

```text
海莉正在思考怎么回答你
```

- 当 `Speak` 显示回复、当 orchestrator 发送清除命令（如果存在）或通过 TTL 回退时，清除 thinking HUD。
- 这些 HUD 变化只是 UI 状态；它们不能在 SMAPI 中调用 LLM，也不能驱动 Agent 行为。

## 验收标准

1. 活体 Haley 的 `vanilla_dialogue_completed` 事件会驱动 Desktop/core 调用 `/action/open_private_chat`，无需手动调试 tick。
2. 输入框会在请求任何 Haley Agent 回复之前打开。
3. `player_private_message_submitted` 会在类型化 payload 中携带原始玩家文本，并把该文本路由给 Haley 私聊 Agent runner。
4. 普通桥接事件仍然只是事实，不会触发 Agent。
5. 重复的对话完成事件不会打开重复输入框。
6. 在等待 Haley 回复时，新的完成事件不会再次打开输入框。
7. Agent 成功时，通过现有的 `Speak` 命令路径显示回复。
8. Agent 超时/错误会回到稳定的 idle 状态，不会让 HUD 永久卡住。
9. 其他 NPC 的原版对话完成不会触发 Haley 的私聊 orchestrator。
10. SMAPI 中不包含 LLM/persona/记忆/会话逻辑。
