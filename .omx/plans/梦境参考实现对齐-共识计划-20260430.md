# RALPLAN-DR：梦境 / 长期记忆参考对齐

## 范围

规划 Hermes-Desktop 应如何将当前的 dream / long-term memory 能力更紧密地对齐到 `external/hermes-agent-main`，本阶段不修改源代码，也不在后续明确立项前改变当前 Dreamer 行为。

### 规划推断

- 最低风险的对齐路径，是扩展 Hermes 现有的 provider / orchestrator 路径，让精选记忆像 transcript recall 一样参与同一生命周期，而不是再创建第二条权威长期记忆管线。
- Dreamer 应继续保持为独立的创意 / 后台系统，后续如有需要，可以成为记忆信号的生产者，但不应成为权威的回忆 / 记忆机制。
- 与其把 `AutoDreamService` 直接恢复为主架构，不如把它视为想法和可复用 prompt 逻辑的来源，因为按原样唤醒它会在 `MemoryManager`、`MemoryReviewService` 和 `TurnMemoryCoordinator` 之外再引入一条并行的 consolidation 权威。

## 原则

1. 在后续执行任务明确要求修改前，保留当前 Dreamer 的 walk、scoring、digest 和 build-sprint 行为不变。
2. 通过生命周期与权责流对齐，而不是通过名称与参考项目一一对应。
3. 对写入、读取和压缩交接只保留一条权威的精选记忆路径，避免出现并行的记忆真相来源。
4. 在引入新服务或依赖之前，优先复用现有的 memory、search 和 context 接口。
5. 倾向于采用可增量、可开关的发布方式，并在扩展行为前补齐回归覆盖。

## 决策驱动因素

1. 尽量降低对当前运行中的桌面应用的行为风险，尤其是 Dreamer 启动和 agent turn 流程。
2. 尽快补齐参考实现中价值最高的差距：精选记忆生命周期参与、独立的 session recall，以及压缩前持久化 / 交接。
3. 将工作落在 Hermes 已经具备生产级接口的地方，使实现保持小而可测试。

## 可行方案

### 方案 A：扩展现有的 provider / orchestrator 路径

摘要：在当前 `IMemoryProvider` / `HermesMemoryOrchestrator` / `TurnMemoryCoordinator` 路径上增加精选记忆和压缩感知的生命周期行为，同时保持 Dreamer 独立。

优点：

- 复用现有、已经标注为 Python-compatible 的生命周期抽象。
- 让 transcript recall 和精选记忆保持为彼此独立但可协调的 provider，更接近参考能力划分。
- 能自然接入当前的 turn-start、sync、queue-prefetch 和 context-compaction 接口。
- 对 Dreamer 的风险最低，因为 Dreamer 不需要成为记忆权威。

缺点：

- 需要谨慎设计，避免精选记忆快照同时通过 plugin 注入和 turn-memory 注入而重复。
- 可能暴露出一些过时的仓库文档或旧的记忆假设，需要在执行过程中清理。

### 方案 B：恢复 `AutoDreamService`，让其成为权威 consolidation 器

摘要：注册并现代化 `AutoDreamService`，把它作为主要的长期记忆写入者，同时保留 Dreamer 独立。

优点：

- 复用现有的 transcript-consolidation worker，其中已经带有 soul extraction 概念。
- 概念上接近定期“学习循环”模型。

缺点：

- 会与 `MemoryManager`、`MemoryReviewService` 以及现有的 memory tools 产生权责冲突。
- 目前没有接入启动流程，也没有接入 turn-memory 生命周期。
- 与参考实现的 turn / compression 生命周期相比，更容易偏离。
- 回滚风险更高，因为它会重新激活休眠行为。

### 方案 C：把 Dreamer 折叠进权威记忆管线

摘要：让 Dreamer 自己负责长期记忆的整合和回忆决策。

优点：

- “做梦”和“记忆”可以归入一个叙事系统。

缺点：

- 违反了保留 Dreamer 角色不变的约束。
- 将创意 / 后台 worker 与核心 turn-time 记忆可靠性耦合在一起。
- 这是最弱证据基础下最大的架构变化。

## 推荐方案

推荐 **方案 A**。

原因：

- 它最能用 Hermes 现有代码里已经存在的 primitives 去对齐参考实现的能力划分。
- 它保留了当前 Dreamer 行为。
- 它为对齐压缩前和 turn 后的记忆生命周期语义提供了清晰路径，而无需恢复一个休眠的并行子系统。

## 权责矩阵

| 关注点 | 权威归属者 | 计划中的对齐规则 |
|---|---|---|
| 精选记忆写入 | `MemoryManager`，通过 `MemoryTool` 和 `MemoryReviewService` | 保留这条写入权责。默认不要让 Dreamer 或 `AutoDreamService` 直接写入最终的精选记忆。 |
| 精选记忆读取 / 注入 | 现有的 `BuiltinMemoryPlugin` 冻结系统提示快照 | 保留按会话冻结的语义。第一阶段对齐不要把精选记忆改成按 turn 动态召回。 |
| 动态 transcript recall | 通过 `HermesMemoryOrchestrator` 和 `TurnMemoryCoordinator` 的 `TranscriptMemoryProvider` | 保持 transcript recall 位于动态的 `<memory-context>` turn 注入通道中。 |
| 压缩前交接 | 目前只有 plugin 通道 | 在摘要之前增加明确的 memory handoff 合约。除非执行证明扩展 `IMemoryProvider` 更简洁，否则优先使用并列的 compression-participant 接口，而不是强迫所有 `IMemoryProvider` 实现都变成 compression-aware。 |
| Dreamer 输出 | `DreamerService` 写入 dreamer room artifacts | 默认保持为非权威的本地 artifact。未来如果 Dreamer 要产生记忆候选，必须通过精选记忆的权威路径进入。 |

## 精选记忆语义决策

在第一轮实现中，精选记忆保持为 **每会话冻结一次的系统提示快照**。这样既保留了当前行为，也避免了一种隐蔽的语义变化：会话中途的记忆编辑悄悄改变 turn-time 的 grounding。

动态召回仍然只保留给 transcript / session recall provider。若后续阶段希望精选记忆按 turn 刷新，就必须另起一份 ADR，因为这会改变 prompt 稳定性、缓存行为，以及重复注入风险。

## 压缩交接合约

方案 A 需要一个明确的合约变更。当前 `IMemoryProvider` 生命周期覆盖了 turn start、prefetch、sync 和 queued prefetch，但没有覆盖 pre-compression 或 session-rotation 交接。

首选的规划目标：

1. 增加一个窄一些的 compression handoff 接口，例如 `IMemoryCompressionParticipant`，其方法在压缩摘要或丢弃上下文之前执行。
2. 让 `HermesMemoryOrchestrator` 发现这些 participant，或者直接接收它们，并向 `ContextManager` 暴露一个统一的 handoff 方法。
3. 在 `ContextManager` 现有的摘要步骤之前调用这个 handoff。
4. 保持方法的副作用安全和尽力而为：失败只记录日志，不阻塞 compaction，除非后续执行证明必须阻塞。

第一轮被否决的目标：

- 把精选记忆当作普通的 `PrefetchAsync` 来源。这样会把冻结的精选记忆与动态 transcript recall 混在一起，并增加重复注入的风险。

## 注入与去重规则

只有一层负责精选记忆注入：现有的 plugin / system-prompt 层。provider / orchestrator 这一层可以协调生命周期和压缩交接，但绝不能再次把精选记忆追加到用户消息里。

transcript recall 继续通过 `TurnMemoryCoordinator` 负责动态的 `<memory-context>` 注入。去重测试必须断言：某条已知的精选记忆条目在组装后的 prompt / message 路径里只出现一次，不会在动态 recall 输出中再次重复。

## 需求摘要

1. 精选记忆必须像 transcript recall 一样参与同一套生命周期模型，而不能只靠 plugin / system-prompt 注入。
2. session search 和精选记忆必须保持为两个独立能力，并拥有各自独立的工具 / 面向用户语义。
3. 上下文压缩必须在摘要上下文被旋转掉之前，拥有一个明确的 memory-commit / memory-snapshot 交接路径。
4. Dreamer 必须保持可用，并且默认行为不变。
5. 任何 rollout 都必须带有 gate，并围绕当前的 turn preparation、response flow 和 Dreamer startup 做回归测试。
6. 在第一阶段中，精选记忆必须保持为每会话冻结；按 turn 动态读取精选记忆不在范围内。
7. `AutoDreamService` 必须默认保持未注册状态；Dreamer 仍然是唯一默认的 dream 相关后台循环。

## 验收标准

1. 存在一份单一、明确的长期记忆写入、读取和压缩交接权责模型，并清楚说明每一步由哪个 service 负责。
2. 实现设计不会在所选路径之外再引入第二个权威的记忆写入者。
3. 首次 turn 请求会让 transcript recall 继续留在动态 provider-prefetch 车道，而精选记忆在第一阶段只通过 lifecycle / handoff 合约参与；`PrefetchAsync` 不能注入精选记忆内容。
4. compaction / context summarization 包含一个定义明确的 pre-compression memory hook 或等价交接点，并且有测试覆盖该路径。
5. 除非另有单独范围明确要求，Dreamer startup 和 cycle 行为保持不变。
6. 工具合约保持区分：`memory` 负责精选记忆，`session_search` 负责搜索 / 浏览会话历史。
7. 测试覆盖 turn preparation、记忆注入顺序 / 去重、压缩钩子行为，以及 Dreamer startup 无回归。
8. 默认 rollout 不会注册 `AutoDreamService`，也不会让 Dreamer 成为权威记忆写入者。
9. 一条压缩交接测试能够证明新的 memory handoff 在 context summarization / replacement 之前运行。
10. 一条重复注入测试能够证明精选记忆不会同时通过 plugin prompt 注入和动态 turn-memory 增强被输出两次。

## 风险与缓解

### 风险：记忆上下文重复

缓解：
- 为每一层定义唯一的注入权责，并在 plugin block 与 turn-memory block 之间添加明确的去重规则。

### 风险：从冻结的精选记忆隐式转向动态 recall 的语义变化

缓解：
- 在第一阶段保留按会话冻结的精选记忆，并把这一行为写入测试。

### 风险：出现两个相互竞争的长期记忆写入者

缓解：
- 在后续明确选择之前，不要把 `AutoDreamService` 放进默认权威路径；让一个 service 负责写入权威。

### 风险：Dreamer 因意外耦合而回归

缓解：
- 把 Dreamer 集成视为可选的后续工作；不要让 Dreamer 依赖 turn-time 记忆成功与否。

### 风险：过时的内部文档掩盖当前行为

缓解：
- 在执行过程中，务必对齐 `wiki/systems/memory-system.md` 与当前代码，因为它仍然描述的是旧的按文件 YAML memory 行为，已经不符合 `MemoryManager` 的现状。

## ADR 草案

### 标题

通过扩展现有的 provider / orchestrator 路径，把 Hermes 长期记忆对齐到参考生命周期

### 状态

草案

### 决策

Hermes 应该通过围绕 `HermesMemoryOrchestrator` 和 `TurnMemoryCoordinator` 扩展当前 memory 生命周期，加入明确的 compression handoff 合约，来对齐参考项目的 dream / long-term-memory 行为，同时保留精选记忆作为每会话冻结的 system-prompt 快照，并让 Dreamer 继续作为独立且被保留的后台系统。

### 驱动因素

- 保留当前 Dreamer 行为。
- 复用生产代码中已经存在的生命周期接口。
- 以最少的新架构，补齐参考实现中关于精选记忆、session recall 分离和 compression handoff 的差距。

### 备选方案

- 恢复 `AutoDreamService` 作为主 consolidation 器。
- 让 Dreamer 成为记忆权威。

### 选择原因

- 这是风险最低、且现有代码支撑最强的路径。
- 它最适合参考实现的能力划分，而不会强迫重做 Dreamer。
- 它把长期记忆权威保持在 provider / orchestrator 路径中。

### 结果

- Hermes 很可能会新增一个 compression handoff participant 或精选记忆生命周期 adapter，以及更清晰的注入规则。
- 现有的 plugin / context 重叠必须被显式解决。
- 除非未来决策把它们提升为生产者角色并纳入同一权威路径，否则 `AutoDreamService` 和 Dreamer 会继续保持默认非权威状态。
- 在第一轮实现中，精选记忆不会变成按 turn 动态 recall。

## 共识评审说明

### 独立架构评审

- 结论：需要迭代，已在本版本中处理。
- 最强的反命题：方案 A 有把冻结的精选记忆与动态 transcript recall 压缩到同一条通道的风险。
- 真正的权衡张力：生命周期统一 vs. 语义分离。
- 采用的综合方案：用方案 A 负责协调和 compression handoff，但保留精选记忆的冻结快照语义，并维持单一的注入权责。
- 已应用的必需修改：
  - 权责矩阵
  - 明确的合约变更段落
  - 精选记忆语义决策
  - 强制性的去重 / 注入规则
  - 启动不重激活条件
  - 压缩顺序要求

### Critic 结论

- 结论：批准。
- 阻塞性问题：无。
- 已应用的非阻塞改进：
  - 验收和验证措辞现在区分了精选生命周期参与与动态 `PrefetchAsync` 注入
  - 实施步骤 4 现在明确把 `ContextManager` 作为主要的实时 compaction 接口
  - 启动接线现在明确保留当前默认值、plugin 顺序和 Dreamer 行为
- 执行时需要关注的剩余风险：
  - `TurnMemoryCoordinator` 会把 provider prefetch 追加到 `<memory-context>`，所以精选 prefetch 必须保持不生效或被排除
  - `BuiltinMemoryPlugin.OnPreCompressAsync` 已经会刷新 session snapshot，因此新的 handoff 顺序必须保留“当前 turn 冻结”与“下一边界刷新”这两个语义
  - 测试必须证明只存在一条精选记忆注入路径，并且 `AutoDreamService` 没有被激活
