# RALPLAN-DR: Dream / Long-Term Memory Reference Alignment

## Scope

Plan how Hermes-Desktop should align its current dream / long-term memory capability more closely with `external/hermes-agent-main`, without editing source in this phase and without changing current Dreamer behavior unless later explicitly scoped.

## Evidence vs Inference

### Direct repo-local evidence

- The provided snapshot identifies the reference behaviors to mirror: curated persistent memory, session search recall, and pre-compression memory lifecycle handling.
- Hermes already has a Python-style memory-provider lifecycle surface: [`src/memory/IMemoryProvider.cs`](D:/Projects/Hermes-Desktop/src/memory/IMemoryProvider.cs), [`src/memory/HermesMemoryOrchestrator.cs`](D:/Projects/Hermes-Desktop/src/memory/HermesMemoryOrchestrator.cs).
- The app currently wires only transcript recall into that lifecycle via [`src/search/TranscriptMemoryProvider.cs`](D:/Projects/Hermes-Desktop/src/search/TranscriptMemoryProvider.cs) and [`Desktop/HermesDesktop/App.xaml.cs`](D:/Projects/Hermes-Desktop/Desktop/HermesDesktop/App.xaml.cs).
- Curated memory already exists as fixed-file `MEMORY.md` / `USER.md` management through [`src/memory/MemoryManager.cs`](D:/Projects/Hermes-Desktop/src/memory/MemoryManager.cs) and the `memory` tool in [`src/Tools/MemoryTool.cs`](D:/Projects/Hermes-Desktop/src/Tools/MemoryTool.cs).
- Session search already exists as a separate tool and recall service via [`src/Tools/SessionSearchTool.cs`](D:/Projects/Hermes-Desktop/src/Tools/SessionSearchTool.cs) and [`src/search/TranscriptRecallService.cs`](D:/Projects/Hermes-Desktop/src/search/TranscriptRecallService.cs).
- Context summarization already has a pre-compression hook surface through `PluginManager.OnPreCompressAsync(...)` in [`src/Context/ContextManager.cs`](D:/Projects/Hermes-Desktop/src/Context/ContextManager.cs).
- Dreamer is currently a distinct background loop started from [`Desktop/HermesDesktop/App.xaml.cs`](D:/Projects/Hermes-Desktop/Desktop/HermesDesktop/App.xaml.cs) and implemented in [`src/dreamer/DreamerService.cs`](D:/Projects/Hermes-Desktop/src/dreamer/DreamerService.cs).
- `AutoDreamService` exists as a separate transcript-consolidation worker in [`src/dream/AutoDreamService.cs`](D:/Projects/Hermes-Desktop/src/dream/AutoDreamService.cs), but the snapshot and startup inspection show it is not currently registered in runtime startup.

### Planning inferences

- The closest low-risk alignment path is to extend Hermes' existing provider/orchestrator path so curated memory participates in the same lifecycle as transcript recall, instead of creating a second authoritative long-term-memory pipeline.
- Dreamer should remain a separate creative/background system and optionally become a memory signal producer later, rather than becoming the authoritative recall/memory mechanism.
- `AutoDreamService` is better treated as a source of ideas and reusable prompt logic than as the primary target architecture, because reviving it as-is would introduce a second consolidation authority beside `MemoryManager`, `MemoryReviewService`, and `TurnMemoryCoordinator`.

## Principles

1. Preserve current Dreamer walk, scoring, digest, and build-sprint behavior unless a later execution task explicitly scopes behavior changes.
2. Align by lifecycle and authority flow, not by name parity with the reference project.
3. Keep one authoritative curated-memory path for write/read/compression handoff; avoid parallel memory truth sources.
4. Reuse existing memory, search, and context seams before introducing new services or dependencies.
5. Prefer additive, feature-flagged rollout with regression coverage before behavior expansion.

## Decision Drivers

1. Minimize behavior risk to the currently running desktop app, especially Dreamer startup and agent turn flow.
2. Close the highest-value reference gaps: curated memory lifecycle participation, distinct session recall, and pre-compression persistence/handoff.
3. Land the work where Hermes already has production seams, so the implementation stays small and testable.

## Viable Options

### Option A: Extend the existing provider/orchestrator path

Summary: add curated-memory and compression-aware lifecycle behavior to the current `IMemoryProvider` / `HermesMemoryOrchestrator` / `TurnMemoryCoordinator` path, while keeping Dreamer separate.

Pros:

- Reuses the existing lifecycle abstraction already labeled as Python-compatible.
- Keeps transcript recall and curated memory as separate but coordinated providers, matching the reference capability split more closely.
- Integrates naturally with current turn-start, sync, queue-prefetch, and context-compaction seams.
- Lowest risk to Dreamer because Dreamer does not need to become the memory authority.

Cons:

- Requires careful design so curated memory snapshots do not get duplicated through both plugin injection and turn-memory injection.
- May expose some stale repo docs or older memory assumptions that need cleanup during execution.

### Option B: Revive `AutoDreamService` as the authoritative consolidator

Summary: register and modernize `AutoDreamService`, using it as the main long-term-memory writer while Dreamer remains separate.

Pros:

- Reuses an existing transcript-consolidation worker with soul extraction concepts already present.
- Conceptually close to a periodic "learning loop" model.

Cons:

- Creates authority tension with `MemoryManager`, `MemoryReviewService`, and current memory tools.
- Not currently wired into startup or the turn-memory lifecycle.
- More likely to diverge from the reference's turn/compression lifecycle than Option A.
- Higher rollout risk because it reactivates dormant behavior.

### Option C: Fold Dreamer into the authoritative memory pipeline

Summary: make Dreamer itself responsible for long-term-memory consolidation and recall decisions.

Pros:

- One narrative subsystem for "dreaming" and "memory."

Cons:

- Violates the preservation constraint by redefining Dreamer's role.
- Couples a creative/background worker to core turn-time memory reliability.
- Largest architectural change for the weakest evidence base.

## Recommended Option

Recommend **Option A**.

Reasoning:

- It best matches the reference capability split using current Hermes primitives already present in code.
- It preserves current Dreamer behavior.
- It gives a clear path to align pre-compression and post-turn memory lifecycle semantics without reviving a dormant parallel subsystem.

## Authority Matrix

| Concern | Authoritative owner | Planned alignment rule |
|---|---|---|
| Curated memory writes | `MemoryManager`, via `MemoryTool` and `MemoryReviewService` | Preserve this write authority. Do not let Dreamer or `AutoDreamService` write final curated memory directly by default. |
| Curated memory read/injection | Existing `BuiltinMemoryPlugin` frozen system-prompt snapshot | Preserve frozen-per-session semantics. Do not convert curated memory into per-turn dynamic recall in the first alignment pass. |
| Dynamic transcript recall | `TranscriptMemoryProvider` through `HermesMemoryOrchestrator` and `TurnMemoryCoordinator` | Keep transcript recall in the dynamic `<memory-context>` turn-injection lane. |
| Pre-compression handoff | Currently plugin lane only | Add a concrete memory handoff contract before summarization. Prefer a sibling compression-participant interface over forcing all `IMemoryProvider` implementations to become compression-aware unless execution proves extending `IMemoryProvider` is cleaner. |
| Dreamer output | `DreamerService` writes dreamer room artifacts | Keep as non-authoritative local artifacts by default. Future Dreamer memory candidates must enter through the curated-memory authority path. |
| AutoDream output | Dormant `AutoDreamService` / `ConsolidationAgent` | Do not register or start by default. Mine prompt/consolidation ideas only after explicit scoped approval. |

## Curated Memory Semantics Decision

Curated memory remains a **frozen-per-session system-prompt snapshot** in the first implementation pass. This preserves current behavior and avoids a hidden semantic shift where mid-session memory edits silently change turn-time grounding.

Dynamic recall remains limited to transcript/session recall providers. If a later phase wants curated memory refreshed per turn, that must be a separate ADR because it changes prompt stability, cache behavior, and duplicate-injection risk.

## Compression Handoff Contract

Option A requires an explicit contract change. The current `IMemoryProvider` lifecycle covers turn start, prefetch, sync, and queued prefetch, but not pre-compression or session-rotation handoff.

Preferred planning target:

1. Add a narrow compression handoff interface, for example `IMemoryCompressionParticipant`, with a method that runs before compaction summarizes or drops context.
2. Let `HermesMemoryOrchestrator` discover or receive those participants and expose a single handoff method to `ContextManager`.
3. Call the handoff before the existing summarization step in `ContextManager`.
4. Keep the method side-effect-safe and best-effort: failures are logged and do not block compaction unless execution later proves blocking is necessary.

Rejected first-pass target:

- Make curated memory an ordinary `PrefetchAsync` source. That would mix frozen curated memory with dynamic transcript recall and risks duplicate injection.

## Injection And Dedupe Rule

One layer owns curated-memory injection: the existing plugin/system-prompt layer. The provider/orchestrator lane may coordinate lifecycle and compression handoff, but it must not append curated memory again to the user message.

Transcript recall remains the owner of dynamic `<memory-context>` injection through `TurnMemoryCoordinator`. Dedupe tests must assert that a known curated-memory entry appears once in the assembled prompt/message path and is not repeated in dynamic recall output.

## Requirements Summary

1. Curated memory must participate in the same lifecycle model as transcript recall, not only via plugin/system-prompt injection.
2. Session search and curated memory must remain distinct capabilities with distinct tool/user-facing semantics.
3. Context compression must have an explicit memory-commit / memory-snapshot handoff path before summarized context is rotated away.
4. Dreamer must remain operational and behaviorally unchanged by default.
5. Any rollout must be gated and regression-tested around current turn preparation, response flow, and Dreamer startup.
6. Curated memory must remain frozen-per-session in the first pass; dynamic per-turn curated recall is out of scope.
7. `AutoDreamService` must remain unregistered by default; Dreamer remains the only default dream-related background loop.

## Out of Scope

- Replacing Dreamer with a new memory system.
- Enabling `AutoDreamService` in production by default as part of this alignment.
- Changing the `memory` or `session_search` user-facing responsibilities.
- Adding new third-party dependencies.

## Acceptance Criteria

1. There is a single documented authority model for long-term memory writes, reads, and compression handoff, and it names which service owns each step.
2. The implementation design introduces no second authoritative memory writer beside the chosen path.
3. A first-turn request keeps transcript recall in the dynamic provider-prefetch lane while curated memory participates only through the lifecycle/handoff contract in phase 1; `PrefetchAsync` must not inject curated memory content.
4. Compaction/context summarization includes a defined pre-compression memory hook or equivalent handoff point with test cases covering the path.
5. Dreamer startup and cycle behavior remain unchanged unless a separate scoped task explicitly opts into Dreamer-memory integration.
6. Tool contracts remain distinct: `memory` manages curated memory, `session_search` searches/browses session history.
7. Tests cover turn preparation, memory injection ordering/deduplication, compression hook behavior, and no-regression Dreamer startup.
8. The default rollout does not register `AutoDreamService` or make Dreamer an authoritative memory writer.
9. A compression handoff test proves the new memory handoff runs before context summarization/replacement.
10. A duplicate-injection test proves curated memory is not emitted both through plugin prompt injection and dynamic turn-memory augmentation.

## Risks and Mitigations

### Risk: duplicated memory context

Mitigation:
- Define one injection authority per layer and add explicit dedupe rules between plugin blocks and turn-memory blocks.

### Risk: hidden semantic shift from frozen curated memory to dynamic recall

Mitigation:
- Preserve frozen-per-session curated memory in the first pass and encode that behavior in tests.

### Risk: two competing long-term-memory writers

Mitigation:
- Keep `AutoDreamService` out of the default authority path unless later explicitly chosen; make one service responsible for write authority.

### Risk: Dreamer regression by accidental coupling

Mitigation:
- Treat Dreamer integration as optional follow-on work; do not make Dreamer depend on turn-time memory success.

### Risk: stale internal docs masking current behavior

Mitigation:
- During execution, reconcile `wiki/systems/memory-system.md` with current code because it still describes older per-file YAML memory behavior that no longer matches `MemoryManager`.

## ADR Draft

### Title

Align Hermes long-term memory to the reference lifecycle by extending the existing provider/orchestrator path

### Status

Draft

### Decision

Hermes should align to the reference project's dream/long-term-memory behavior by extending the current memory lifecycle around `HermesMemoryOrchestrator` and `TurnMemoryCoordinator` with an explicit compression handoff contract, while preserving curated memory as a frozen-per-session system-prompt snapshot and keeping Dreamer as a separate preserved background system.

### Drivers

- Preserve current Dreamer behavior.
- Reuse existing lifecycle seams already present in production code.
- Close the reference gaps around curated memory, session recall separation, and compression handoff with minimal new architecture.

### Alternatives Considered

- Revive `AutoDreamService` as the main consolidator.
- Make Dreamer the memory authority.

### Why Chosen

- Lowest-risk path with the strongest existing code support.
- Best fit for the reference capability split without forcing Dreamer redesign.
- Keeps long-term memory authority centralized in the provider/orchestrator path.

### Consequences

- Hermes will likely gain a new compression handoff participant or curated-memory lifecycle adapter and clearer injection rules.
- Some current plugin/context overlap must be resolved explicitly.
- `AutoDreamService` and Dreamer stay non-authoritative by default unless a future decision promotes them to producer roles into the same authority path.
- Curated memory does not become per-turn dynamic recall in the first implementation pass.
