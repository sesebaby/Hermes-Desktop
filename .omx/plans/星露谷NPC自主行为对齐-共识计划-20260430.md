## RALPLAN-DR Summary

### Principles

1. Prove the smallest real NPC life loop before expanding the village.
2. Keep NPC identity, memory, transcript, trace, activity, and prompt context isolated by `gameId + saveId + npcId + profileId`.
3. Let game actions go only through typed Stardew bridge contracts and `StardewCommandService`.
4. Preserve hard autonomy: the Agent owns cadence, observation, decision, and action.
5. Treat game/bridge events as facts only.
6. Keep prompt assembly on runtime-local `ContextManager` / `PromptBuilder`.
7. Cap cost and runaway behavior before richer behavior.

### Decision Drivers

1. The first meaningful milestone is a real loop: `self-driven tick -> observe -> context -> decide -> tool/action -> poll -> persist -> next tick`.
2. Implementation must reuse existing Hermes Desktop primitives: `NpcRuntime*`, `NpcNamespace`, `Agent`, `MemoryManager`, `ContextManager`, `PromptBuilder`, and Stardew bridge contracts.
3. Tests must run without live Stardew Valley or SMAPI.

### Viable Options

#### Option A: Put the loop directly in `NpcRuntimeInstance`

Pros:
- Fewest new classes.
- Lifecycle state is already there.

Cons:
- Mixes lifecycle state, background orchestration, LLM calls, bridge calls, trace, and memory in one class.
- Harder to test a single tick independently.

#### Option B: Add dedicated `NpcAutonomyLoop` owned by runtime lifecycle

Pros:
- Keeps `NpcRuntimeInstance` focused on lifecycle and snapshot state.
- Gives execution a direct, testable `RunOneTickAsync`.
- Provides a single allowed entry point into LLM/tool decisions.
- Makes the fact-only event boundary enforceable.

Cons:
- Adds a small orchestration layer and DI wiring.
- Requires care not to build a second prompt system.

#### Option C: Reuse generic `Agent` as the whole runtime immediately

Pros:
- Closest to hermescraft's long-session tool-using Agent model.
- Reuses existing Agent tool loop.

Cons:
- Too broad for Phase 1 unless NPC tool and context isolation are already solved.
- Risks exposing desktop tools to NPCs.

Chosen: Option B, with an NPC-scoped `Agent` inside the loop only after the local context and tool allowlist are constructed.

## Non-Negotiable Autonomy Boundary

The implementation must not be event-driven.

SMAPI events, bridge events, proximity changes, dialogue clicks, chat/inbox messages, scheduler ticks, and world-state changes may only be persisted as observation facts. They must not:

- call LLM completion,
- call `Agent.ChatAsync`,
- call `StardewCommandService.SubmitAsync`,
- call `move` or `speak`,
- enqueue semantic instructions such as "Haley should respond now."

Only `NpcAutonomyLoop.RunOneTickAsync` or a future explicit loop scheduler may enter the observe/decide/tool/action path. A scheduler may choose *when* to run a tick, but not *what* the NPC decides.

## Requirements Summary

- Add a bounded single-NPC autonomy loop for Haley first.
- The tick observes current facts before any decision.
- The tick builds NPC-local context through `ContextManager` / `PromptBuilder`.
- The Agent sees only NPC-safe tools.
- Move/speak/status actions go through `StardewCommandService`.
- Long commands poll status until terminal state or configured limit.
- Trace/activity/memory are written under the NPC namespace.
- Event ingestion is fact-only and has negative tests.

## Acceptance Criteria

1. Starting Haley creates the runtime namespace and a loop-capable handle while preserving supervisor snapshots.
2. `RunOneTickAsync` observes before decision or action.
3. Event/fact ingestion produces no LLM/tool/command side effects until the next explicit tick.
4. Haley context flows through runtime-local `ContextManager` / `PromptBuilder`.
5. The NPC Agent has a strict allowlist of Stardew/NPC-local tools only.
6. Move decisions submit `GameActionType.Move` through `StardewCommandService`.
7. Speak decisions submit `GameActionType.Speak` through `StardewCommandService`.
8. Long commands poll `task/status` until terminal or limited.
9. Bridge unavailable/world-blocked facts produce a no-op/paused trace, not a forced action.
10. A completed meaningful tick writes trace/activity and attempts concise NPC memory persistence.

## Implementation Steps

1. Add fact-only observation boundary.
   - Add `IStardewWorldObserver` and `StardewWorldObserver`.
   - Wrap `StardewBridgeRoutes.QueryStatus` for `StardewNpcStatusData`.
   - Add a passive observation fact type/store if event facts are needed.

2. Add NPC-scoped Stardew tools.
   - Add wrappers for status, move, speak, and task status.
   - Bind `npcId`, `saveId`, `traceId`, and idempotency internally from runtime context.
   - Reuse `StardewCommandService`; do not call SMAPI HTTP directly.

3. Wire runtime-local context using existing infrastructure.
   - Do not add `StardewPromptAssembler` or `NpcPromptContextBuilder`.
   - Copy/seed persona files into `NpcNamespace` as needed.
   - Construct local `SoulService`, `MemoryManager`, `TranscriptStore`, `ContextManager`, and `PromptBuilder`.
   - Add narrow skill binding that resolves names in `skills.json` to `skills/gaming/stardew-*.md` and feeds text through existing context inputs.

4. Add `NpcAutonomyLoop`.
   - Add `RunOneTickAsync(NpcRuntimeInstance instance, CancellationToken ct)`.
   - Flow: budget -> observe -> prepare context -> NPC-scoped Agent/tool allowlist -> decision turn -> command polling -> trace/activity/memory -> update `LastTraceId`.
   - Keep continuous background scheduling separate and explicit.

5. Wire lifecycle safely.
   - Extend runtime/supervisor/host only enough to create and start a loop-capable Haley runtime.
   - Do not auto-start all discovered NPCs at Desktop launch.
   - Preserve Dashboard discovery behavior.

6. Add tests first or alongside each slice.
   - Prioritize autonomy-boundary negative tests before action tests.

## Risks And Mitigations

- Hidden event control: enforce fact-only event ingestion with negative tests.
- Tool leakage: build NPC agents with local tool registry and assert allowlist.
- Prompt drift: require runtime-local `ContextManager` / `PromptBuilder` path in tests.
- Runaway cost: start with single tick, budget limits, cancellation, and no automatic app-start loop.
- Live game dependency: use fake observer, fake bridge client, and fake command service in tests.

## ADR

Decision: Implement Phase 1 as a dedicated, single-NPC, agent-driven `NpcAutonomyLoop` with `RunOneTickAsync`, using runtime-local Hermes context/memory/soul infrastructure and NPC-scoped Stardew tools.

Drivers:
- The highest-priority gap is autonomous NPC runtime behavior.
- The multi-NPC design requires NPC self-observation/self-decision and fact-only bridge semantics.
- Existing Hermes context and tool infrastructure should be reused, not bypassed.

Alternatives considered:
- Put loop directly in `NpcRuntimeInstance`.
- Reuse generic `Agent` as the whole runtime immediately.
- Event-driven NPC responses.
- Custom Stardew prompt assembler.

Why chosen:
- A dedicated loop gives a single enforceable autonomy boundary and a testable tick surface.
- Runtime-local `ContextManager` / `PromptBuilder` satisfies the design doc and avoids prompt-path drift.
- Fact-only events prevent the game from becoming a hidden semantic director.

Consequences:
- Adds a small orchestration class and local service factory/wiring.
- Requires explicit tests around tool allowlist and event non-driving behavior.
- Continuous background mode remains a later, safer step after single tick is proven.

Follow-ups:
- Add multi-NPC scheduler only after Haley tick passes tests.
- Add SocialRouter only as message delivery, not semantics.
- Add richer tools only after status/move/speak/task-status loop is stable.
