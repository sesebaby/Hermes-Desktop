# 桌面任务闭环 RALPLAN

## RALPLAN-DR

### 原则
1. 保持仅限桌面端的范围：不做 gateway、TUI、messenger 或 server 的实现工作。
2. 先以参考实现的任务语义为准：普通聊天应先使用类似 `todo` 的全量列表任务循环，再考虑任何 `CoordinatorService` 扩展。
3. 将持久化的 `todo` 工具结果消息视为任务状态的唯一权威来源；工具开始时的状态文本和活动摘要仅作参考。
4. 在普通聊天循环关闭并验证前，保持会话 `todo` 与持久化的项目/协调器任务彼此独立。
5. 任务状态应能从桌面会话历史和由投影层维护的最新快照中恢复，而不能只依赖运行时内存中的界面状态。

### 三个最重要的决策驱动因素
1. 当前不一致：普通聊天写入 `todo_write`，而右侧面板读取 `TaskManager`，因此用户可见的任务循环现在是断裂的。
2. 参考行为：`external/hermes-agent-main` 使用每个会话一个 `todo` 存储、全量 JSON 结果、完成时投影以及历史回填。
3. 范围纪律：`src/coordinator/CoordinatorService.cs` 和持久化的 `TaskManager` 任务，并不是关闭普通聊天任务循环的最短路径。

### 可行方案

#### 方案 A：把普通聊天 `todo` 直接统一进 `TaskManager`
- 概要：重写 `TodoWriteTool` 和 `TaskPanel`，让聊天创建的 `todo` 立即变成持久化的 `HermesTask` 记录。
- 优点：
  - 复用现有的持久化和面板基础设施。
  - 避免引入第二套桌面任务模型。
- 缺点：
  - 语义会偏离参考实现，后者是按会话范围且以全量列表为中心。
  - 会把每一步代理规划都强制映射到持久化项目任务生命周期状态（`Failed`、`Blocked`、依赖关系），而普通聊天当前并不需要这些。
  - 在基础循环还没跑通之前，就把耦合度提高到 `CoordinatorService` 时代的模型。

#### 方案 B：在现有 `TaskManager` 旁边增加一个符合参考实现的会话 `todo` 循环
- 概要：为普通聊天引入按会话划分的 `todo` 存储以及转写/回填路径，然后让 `TaskPanel` 把会话 `todo` 作为主要的实时展示面。
- 优点：
  - 与参考实现的事件/状态流非常接近。
  - 不依赖 `CoordinatorService`，即可修复聊天 -> 工具 -> 转写 -> 面板 这条断裂链路。
  - 仍然保留后续把会话 `todo` 与持久化项目任务整合的空间。
- 缺点：
  - 桌面应用中会引入第二套任务表示。
  - 需要明确面板在空态、实时态和归档态下的行为。

### 选定方案
方案 B。

原因：这是最小的、仅限桌面端的方案，既能关闭真实存在的功能缺口，又能保持与 `external/hermes-agent-main` 的任务语义一致，并避免过早耦合 `CoordinatorService`。

## 需求摘要

1. 普通聊天必须提供与参考实现兼容的 `todo` 工具契约，包含读取模式、写入模式、替换/合并行为，以及 `pending`、`in_progress`、`completed`、`cancelled` 状态。
2. 桌面应用必须维护按会话划分的 `todo` 状态，该状态在当前桌面会话期间保持可用，并且在重载或切换会话后可从转写历史重新恢复。
3. `TaskPanel` 必须从真实的聊天/工具结果中渲染实时任务状态，而不能只依赖无关的 `TaskManager` 数据。
4. 工具完成必须是投影到面板状态的权威时刻，因为工具开始时的参数可能只是部分合并载荷。
5. 投影边界必须仅限桌面端：`SessionTaskProjectionService` 是每会话 `todo` 快照的唯一权威拥有者，它通过转写/会话层消费持久化的 `Message { Role = "tool", ToolName, ToolCallId, Content }` 结果，而不是 `ActivityEntryAdded` 摘要或 gateway/TUI 事件封包。
6. `todo` 工具必须通过桌面端工具/会话参数路径显式限定为会话范围，这样单个全局工具注册不会把任务泄漏到其他聊天会话。
7. `todo` 工具必须提供一个与参考实现等价的自定义嵌套 schema/guidance 契约；仅靠通用工具 schema 构建器不够，因为它无法表达列表项、合并行为、状态和读写语义。
8. 轮次/会话历史必须保留足够新的任务状态，以便恢复活跃 `todo`，并在会话历史中展示已归档的完成/未完成状态。
9. 上下文准备/压缩必须能够通过会话任务投影快照，经由桌面实时提示路径（`ContextManager`/`PromptBuilder`）重新注入活跃任务上下文，而不是通过未使用的 gateway 或独立压缩入口。
10. 该计划不得新增 gateway、TUI、Telegram、Discord、Slack 或其他已退役的界面。

## 验收标准

1. 当模型在普通聊天中调用 `todo` 时，工具返回包含完整当前 `todo` 列表以及汇总计数的 JSON。
2. 只读的 `todo` 调用如果没有 `todos` 载荷，应返回当前会话列表且不修改它。
3. 合并模式会按 id 更新已有项，并在保持与参考工具一致的列表顺序语义下追加新项。
4. `cancelled` 状态被接受、持久化到会话状态中，并在面板投影和汇总计数里可见。
5. 成功完成 `todo` 工具后，右侧任务面板会自动更新，不需要手动刷新。
6. 通过 `HermesChatService.LoadSessionAsync` 重新加载已有会话后，`SessionTaskProjectionService.HydrateSessionAsync(sessionId)` 会从转写历史重建最新的 `todo` 快照，并将其显示在任务面板中。
7. 发生对话重置或新会话启动后，当前会话的实时 `todo` 会从面板中清除，但不会删除无关的持久化项目/协调器任务。
8. 活跃的 pending/in-progress `todo` 可以由 `ContextManager.PrepareContextAsync` 格式化后重新注入上下文，而只包含 completed/cancelled 的列表不会把已完成工作重新带回去。
9. `ActivityEntryAdded` 和 `ChatStreamEvent` 不是正确性所必需的；如果使用它们，也只能在投影服务解析出权威状态之后，用于通知界面进度。
10. 测试覆盖工具语义、转写回填、面板投影逻辑，以及从工具完成到可见任务状态的桌面事件循环。

## 风险与缓解措施

### 风险：工具名迁移会破坏提示行为或权限记忆
- 影响：
  - 现有的桌面提示文本、权限规则和测试目前都引用 `todo_write`.
- 缓解：
  - 规划一个临时别名期，让 `todo` 成为主入口，但 `todo_write` 仍能被解析。
  - 在同一个改动集里更新提示/工具允许列表和测试夹具。

### 风险：会话 `todo` 状态与持久化 `TaskManager` 状态会让面板混淆
- 影响：
  - 用户可能会看到不一致或重复的任务列表。
- 缓解：
  - 明确面板归属：普通聊天标签页内容以会话 `todo` 为先。
  - 在循环关闭并验证之前，延后任何混合视图。

### 风险：基于转写的工具结果解析很脆弱
- 影响：
  - 如果最后一条工具消息损坏或不完整，会话重载可能丢失当前任务。
- 缓解：
  - 只从有效的 `todo` JSON 工具结果载荷中回填。
  - 在工具成功完成时，保持最新的内存/会话快照同步。
  - 增加与参考回填场景对应的损坏历史回归测试。

### 风险：把界面遥测误当成领域状态
- 影响：
  - `ActivityEntryAdded` 和流事件可能让面板看起来是实时的，但实际上会丢失或破坏完整的 `todo` 快照。
- 缓解：
  - 将持久化的 `todo` 工具结果消息视为唯一权威输入。
  - 让回放/活动事件仅在投影服务更新快照之后，作为可选的界面进度信号。
  - 增加测试，证明被截断的活动摘要不能用于回填或更新任务。

### 风险：压缩注入落在错误层级
- 影响：
  - 模型可能会重复已完成任务，或者在压缩后丢失活跃上下文。
- 缓解：
  - 按参考实现规则：只重新注入 pending/in-progress 项。
  - 将活跃任务格式化放在会话投影/存储边界上，而不是分散到提示代码里。

## 验证步骤

1. 工具语义：
   - 为读取模式、替换模式、合并模式、按 id 去重、无效状态归一化，以及 `cancelled` 支持编写单元测试。
2. 转写恢复：
   - 测试证明最新的 `todo` 工具结果会在 `LoadSessionAsync` 之后回填会话 `todo`。
   - 测试证明非 `todo` 的工具历史和无效 JSON 不会污染任务状态。
3. 桌面事件循环：
   - 测试证明已持久化且完成的 `todo` 工具结果会在不需要手动刷新的情况下更新会话任务投影和面板侧状态。
   - 测试证明单靠 `ActivityEntryAdded` 摘要不能改变权威任务状态。
4. 生命周期：
   - 测试证明重置/新会话会清空实时会话 `todo`。
   - 测试证明会话重载会恢复它们。
5. 压缩/重建：
   - 测试证明 `ContextManager.PrepareContextAsync` 只会通过 `PromptBuilder` 注入活跃 `todo`。
   - 测试证明仅包含 completed/cancelled 的快照不会添加任务上下文。
6. 实现后的人工验证：
   - 启动桌面会话，发起一个多步骤任务，确认工具调用会让面板可见更新；然后重载会话，确认任务列表被恢复。

## 共识结果

- 规划者：方案 B 是正确的路径，在持久化 `TaskManager` 旁边增加一个符合参考实现的会话 `todo` 循环。
- 架构师：只有当 `SessionTaskProjectionService` 负责状态，并且由持久化的工具结果消息喂入，而不是由 `ActivityEntryAdded` 或流事件喂入时，方案 B 才是可接受的。
- 批评者：必须持续迭代，直到所有权、DI、schema/会话作用域、面板模型，以及实时压缩挂钩点都清晰明确。
- 按照这些修订后的最终计划状态：该方案已经可以进入实施阶段，前提是 `SessionTaskProjectionService` 作为强制性的权威边界。

## ADR 草案

### 标题
为普通桌面聊天采用一个按会话划分、符合参考实现的 `todo` 循环

### 状态
已提议

### 背景
- 桌面应用当前通过 `src/Tools/TodoWriteTool.cs` 暴露 `todo_write`，但 `Desktop/HermesDesktop/Views/Panels/TaskPanel.xaml.cs` 中可见的任务界面却读取 `src/tasks/TaskManager.cs`。
- 这意味着普通聊天的任务规划不会出现在任务面板里，因此用户无法完成一个真正的桌面任务闭环。
- 参考项目 `external/hermes-agent-main` 将任务实现为按会话划分的 `todo` 存储，支持全量写/读、完成时投影、转写回填以及压缩后的活跃任务重新注入。
- 桌面项目有意移除了 gateway/messenger 界面，因此当前范围只涵盖这些语义和事件流模式。

### 决策
用一个专用的会话 `todo` 存储和符合参考实现的 `todo` 契约来实现桌面普通聊天任务循环。增加一个 `SessionTaskProjectionService`，它消费权威的持久化 `todo` 工具结果消息，拥有每个会话的最新快照，并把这些实时会话状态作为 `TaskPanel` 的主数据源。

### 驱动因素
1. 以最小且符合参考实现的改动，关闭当前普通聊天 -> 工具 -> 面板的断层。
2. 保持桌面端范围，并避免重新引入已经退役的 gateway/TUI 界面。
3. 避免把 `TaskManager`/`CoordinatorService` 的持久化项目任务语义过早地压到普通会话规划上。

### 备选方案

#### 备选 1
把聊天 `todo` 直接映射到 `TaskManager`。

为什么不选：
- 这样会把普通聊天规划过早地绑定到持久化项目任务语义上，并且偏离参考循环。

#### 备选 2
通过 `CoordinatorService` 处理普通聊天的任务创建。

为什么不选：
- 当前证据表明聊天入口路径并不会调用 `CoordinatorService`，先扩展它也无法修复断裂的普通任务循环。

### 后果

#### 正面
- 普通聊天任务会变得可见、可恢复，并且与参考实现对齐。
- 该计划仍然保持在桌面范围内。
- 未来仍可把会话 `todo` 与持久化任务整合。
- 任务界面不再依赖被截断的回放/活动摘要来保证正确性。

#### 负面
- 桌面应用会暂时拥有两套职责不同的任务模型。
- 迁移期间需要做提示词/工具名兼容工作。
- 必须把一个小型投影服务接入转写/会话加载/重置/压缩路径。

### 后续事项
1. 在普通聊天循环稳定后，决定是否在同一面板中通过明确的过滤/分组来展示持久化的 `TaskManager` 任务。
2. 评估是否应把协调器创建的长期任务可选地同步到已归档的会话 `todo` 摘要中。
