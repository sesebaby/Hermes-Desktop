# Desktop Task Closure RALPLAN

## RALPLAN-DR

### Principles
1. Preserve desktop-only scope: no gateway, TUI, messenger, or server implementation work.
2. Be reference-faithful on task semantics first: ordinary chat should use a `todo`-style full-list task loop before any CoordinatorService expansion.
3. Treat persisted `todo` tool result messages as the authoritative task state source; tool start/status text and activity summaries are advisory only.
4. Keep session todos and persisted project/coordinator tasks distinct until the ordinary chat loop is closed and verified.
5. Make task state recoverable from desktop session history and a projection-owned latest snapshot, not just from live in-memory UI state.

### Top 3 Decision Drivers
1. Current mismatch: ordinary chat writes `todo_write`, while the right panel reads `TaskManager`, so the user-visible task loop is broken now.
2. Reference behavior: `external/hermes-agent-main` uses one per-session todo store, full-list JSON results, completion-time projection, and history hydration.
3. Scope discipline: `src/coordinator/CoordinatorService.cs` and persisted `TaskManager` tasks are not the shortest path to closing the ordinary chat task loop.

### Viable Options

#### Option A: Unify ordinary chat todos directly into `TaskManager`
- Summary: rewrite `TodoWriteTool` and `TaskPanel` so chat-created todos become durable `HermesTask` records immediately.
- Pros:
  - Reuses existing persistence and panel infrastructure.
  - Avoids introducing a second desktop task model.
- Cons:
  - Semantics diverge from the reference, which is session-scoped and full-list-oriented.
  - Forces every agent planning step into durable project-task lifecycle states (`Failed`, `Blocked`, dependencies) that ordinary chat does not currently need.
  - Raises coupling to CoordinatorService-era models before the basic loop is working.

#### Option B: Add a reference-faithful session todo loop alongside existing `TaskManager`
- Summary: introduce a per-session todo store and transcript/hydration path for ordinary chat, then make `TaskPanel` render session todos as its primary live surface.
- Pros:
  - Matches the reference event/state flow closely.
  - Closes the broken chat -> tool -> transcript -> panel loop without depending on CoordinatorService.
  - Keeps room for later integration between session todos and durable project tasks.
- Cons:
  - Introduces a second task representation in the desktop app.
  - Requires explicit panel behavior decisions for empty/live/archive states.

### Chosen Option
Option B.

Reason: it is the smallest desktop-only plan that closes the actual feature gap while staying source-faithful to `external/hermes-agent-main` task semantics and avoiding premature CoordinatorService coupling.

## Requirements Summary

1. Ordinary chat must expose a reference-compatible `todo` tool contract with read mode, write mode, replace/merge behavior, and statuses `pending`, `in_progress`, `completed`, `cancelled`.
2. The desktop app must maintain per-session todo state that survives across the active desktop session and can be rehydrated from transcript history after reload or session switch.
3. `TaskPanel` must render live task state from actual chat/tool results, not from unrelated `TaskManager` data only.
4. Tool completion must be the authoritative projection point into panel state because tool start arguments may be partial merge payloads.
5. The projection boundary must be desktop-only: `SessionTaskProjectionService` is the single authoritative owner of per-session todo snapshots and consumes persisted `Message { Role = "tool", ToolName, ToolCallId, Content }` results through the transcript/session layer, not `ActivityEntryAdded` summaries or gateway/TUI event envelopes.
6. The `todo` tool must be explicitly session-scoped through the desktop tool/session parameter path so one app-wide tool registration cannot leak tasks across chat sessions.
7. The `todo` tool must provide a custom nested schema/guidance contract equivalent to the reference; the generic tool schema builder is not enough for list items, merge behavior, statuses, and read/write semantics.
8. Turn/session history must preserve the latest task state sufficiently to recover active todos and to show archived completed/incomplete state in session history.
9. Context preparation/compression must be able to re-inject active task context from the session task projection snapshot through the live desktop prompt path (`ContextManager`/`PromptBuilder`), not through unused gateway or standalone compaction surfaces.
10. The plan must not add gateway, TUI, Telegram, Discord, Slack, or other retired surfaces.

## Acceptance Criteria

1. When the model calls `todo` during ordinary chat, the tool returns JSON containing the full current todo list plus summary counts.
2. A read-only `todo` call with no `todos` payload returns the current session list without mutating it.
3. Merge mode updates existing items by id and appends new items while preserving list order semantics consistent with the reference tool.
4. `cancelled` status is accepted, persisted in session state, and surfaced in panel projection and summary counts.
5. After a successful `todo` tool completion, the right-side Tasks panel updates without requiring a manual refresh click.
6. After reloading an existing session via `HermesChatService.LoadSessionAsync`, `SessionTaskProjectionService.HydrateSessionAsync(sessionId)` rebuilds the latest todo snapshot from transcript history and displays it in the Tasks panel.
7. After conversation reset or new session start, live session todos clear from the panel for the new session without deleting unrelated persisted project/coordinator tasks.
8. Active pending/in-progress todos can be formatted for context re-injection by `ContextManager.PrepareContextAsync`, while completed/cancelled-only lists do not reintroduce finished work.
9. `ActivityEntryAdded` and `ChatStreamEvent` are not required for correctness; if they are used, they only notify UI progress after the projection service has parsed authoritative state.
10. Tests cover tool semantics, transcript hydration, panel projection logic, and the desktop event loop from tool completion to visible task state.

## Risks And Mitigations

### Risk: Tool-name migration breaks prompt behavior or permission memory
- Impact:
  - Existing desktop prompt text, permission rules, and tests currently reference `todo_write`.
- Mitigation:
  - Plan for a temporary alias period where `todo` is primary but `todo_write` still resolves.
  - Update prompt/tool allowlists and test fixtures in the same change set.

### Risk: Session todo state and persisted `TaskManager` state confuse the panel
- Impact:
  - Users could see mismatched or duplicated task lists.
- Mitigation:
  - Make panel ownership explicit: ordinary chat tab content is session todos first.
  - Defer any blended view until after the loop is closed and validated.

### Risk: Tool result parsing from transcripts is brittle
- Impact:
  - Session reload could lose current tasks if the last tool message is malformed or partial.
- Mitigation:
  - Hydrate only from valid `todo` JSON tool result payloads.
  - Keep the latest in-memory/session snapshot synchronized on successful tool completion.
  - Add malformed-history regression tests mirroring the reference hydration cases.

### Risk: UI telemetry is mistaken for domain state
- Impact:
  - `ActivityEntryAdded` and stream events can make the panel appear live while losing or corrupting the full todo snapshot.
- Mitigation:
  - Treat persisted `todo` tool result messages as the only authoritative input.
  - Keep replay/activity events as optional UI progress signals after the projection service updates its snapshot.
  - Add tests proving truncated activity summaries cannot be used to hydrate or update tasks.

### Risk: Compaction injection lands in the wrong layer
- Impact:
  - The model may re-do completed tasks or lose active context after compression.
- Mitigation:
  - Mirror the reference rule: only pending/in-progress items are re-injected.
  - Keep active-task formatting on the session projection/store boundary rather than scattering it across prompt code.

## Verification Steps

1. Tool semantics:
   - Unit tests for read mode, replace mode, merge mode, dedupe-by-id, invalid status normalization, and `cancelled` support.
2. Transcript recovery:
   - Tests proving latest `todo` tool result hydrates session todos after `LoadSessionAsync`.
   - Tests proving non-`todo` tool history and invalid JSON do not pollute task state.
3. Desktop event loop:
   - Tests proving a completed persisted `todo` tool result updates session task projection and panel-facing state without manual refresh.
   - Tests proving `ActivityEntryAdded` summaries alone do not mutate authoritative task state.
4. Lifecycle:
   - Tests proving reset/new session clears live session todos.
   - Tests proving session reload restores them.
5. Compaction/rebuild:
   - Tests proving `ContextManager.PrepareContextAsync` injects only active todos through `PromptBuilder`.
   - Tests proving completed/cancelled-only snapshots do not add task context.
6. Manual verification after implementation:
   - Start a desktop session, prompt a multi-step task, verify tool call produces visible panel updates, reload the session, and confirm the task list is restored.

## Consensus Result

- Planner: Option B is the right lane, adding a reference-faithful session todo loop alongside durable `TaskManager`.
- Architect: Option B is acceptable only if `SessionTaskProjectionService` owns state and is fed by persisted tool result messages rather than `ActivityEntryAdded` or stream events.
- Critic: ITERATE until ownership, DI, schema/session scoping, panel model, and live compaction hook points are explicit.
- Final plan status after amendments: implementation-ready for a desktop-only phase, with `SessionTaskProjectionService` as the mandatory authority boundary.

## ADR Draft

### Title
Adopt a session-scoped reference-faithful todo loop for ordinary desktop chat

### Status
Proposed

### Context
- The desktop app currently exposes `todo_write` via `src/Tools/TodoWriteTool.cs`, but the visible task UI in `Desktop/HermesDesktop/Views/Panels/TaskPanel.xaml.cs` reads from `src/tasks/TaskManager.cs`.
- This means ordinary chat task planning does not appear in the task panel, so the user cannot complete a real desktop task loop.
- The reference project `external/hermes-agent-main` implements tasks as a session-scoped todo store with full-list writes/reads, completion-time projection, transcript hydration, and active-task reinjection after compression.
- The desktop project intentionally removed gateway/messenger surfaces, so only those semantics and event-flow patterns are in scope.

### Decision
Implement the desktop ordinary-chat task loop with a dedicated session todo store and a reference-faithful `todo` contract. Add a `SessionTaskProjectionService` that consumes authoritative persisted `todo` tool result messages, owns the latest per-session snapshot, and exposes that live session state to `TaskPanel` as its primary source.

### Drivers
1. Close the existing ordinary chat -> tool -> panel gap with the smallest source-faithful change.
2. Preserve desktop-only scope and avoid reintroducing retired gateway/TUI surfaces.
3. Avoid overloading `TaskManager`/CoordinatorService durable project-task semantics onto ordinary session planning.

### Alternatives Considered

#### Alternative 1
Map chat todos directly into `TaskManager`.

Why not chosen:
- It couples ordinary chat planning to durable project-task semantics too early and diverges from the reference loop.

#### Alternative 2
Route ordinary chat task creation through `CoordinatorService`.

Why not chosen:
- Current evidence shows CoordinatorService is not called from the chat entry path, and expanding it first would not fix the broken ordinary task loop.

### Consequences

#### Positive
- Ordinary chat tasks become visible, recoverable, and reference-aligned.
- The plan stays within desktop scope.
- Future integration between session todos and durable tasks remains possible.
- The task UI no longer depends on truncated replay/activity summaries for correctness.

#### Negative
- The desktop app will temporarily have two task models with different responsibilities.
- Prompt/tool-name compatibility work is required during migration.
- A small projection service must be lifecycle-wired into transcript/session load/reset/compaction paths.

### Follow-Ups
1. After the ordinary chat loop is stable, decide whether to expose durable `TaskManager` tasks in the same panel with explicit filtering/grouping.
2. Evaluate whether coordinator-created long-lived tasks should optionally sync into archived session todo summaries.
