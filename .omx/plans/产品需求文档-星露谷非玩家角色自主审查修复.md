# 产品需求文档：星露谷非玩家角色自主审查修复

## 事实来源

- 访谈规格：`.omx/specs/deep-interview-autonomy-audit-resolution.md`
- 上下文快照：`.omx/context/autonomy-audit-validation-20260508T084753Z.md`
- 原始审查：`agent-autonomy-audit.zh-CN.md`

本计划绑定 A1-A5、B1-B6、C1-C2 全部台账项。任何一项都不能因为排期措辞而消失。执行可以按依赖顺序拆分，但每一项都必须有明确交付物、测试信号和边界归属。

## 问题

`agent-autonomy-audit.zh-CN.md` 混合了真实问题、过时源码定位、过重定性和合理的架构边界担忧。如果修复只处理最显眼的 prompt 问题，剩下的硬编码能力面、过时测试、timeout 恢复策略和误导配置会在上下文压缩后丢失。

## 目标

建立一条覆盖全部合理审查项的修复路线，同时保留当前星露谷非玩家角色架构：

- autonomy parent 继续输出 raw JSON intent；
- 保留 parent cloud decision 与 local executor/delegation 的分工；
- 工具权限由真实 tool surface 保证，而不是只靠 prompt 文案；
- runtime 事实安全指导在合适路径继续可用；
- 固定默认值要么移到可测试的策略/配置边界后面，要么明确标成预留未生效设置，并用测试防止误导。

## 非目标

- 不让 autonomy parent 直接调用 executor-only tools。
- 不移除 `NpcAutonomyLoop` 的 raw JSON intent contract。
- 不把 `RuntimeFactsGuidance` 当成星露谷 autonomy 行为 bug 删除。
- 不改变父云子本地的产品路由原则。
- 不新增依赖。
- 当原始审查与当前源码冲突时，不把 `agent-autonomy-audit.zh-CN.md` 当权威。

## RALPLAN-DR 摘要

### 原则

1. 每个台账项都有命名交付物和测试信号。
2. 除非审查证明是真实 autonomy 边界越界，否则保持既有行为。
3. runtime 能力边界必须由代码和测试保证，不能只靠 prompt 文案。
4. 配置声明必须诚实：要么真实生效，要么明确标成预留未生效。
5. prompt 与 skill 压缩必须来自仓库资产或中性结构，不能由宿主挑选语义重点。

### 决策驱动

1. 保护 parent/local executor 分工，同时移除宿主替 agent 选重点、选工具、选恢复策略的问题。
2. 让后续上下文丢失也不会丢修复依据。
3. 用依赖切片保持改动可审查、可验证。

### 可选方案

方案 A：只修明显 prompt 字符串和失败测试。
- 优点：首个 diff 更小。
- 缺点：A1、A2、A3、A5、B1-B6、C2 都会失去持久执行闭环。
- 结论：拒绝。

方案 B：以台账为完整边界，按依赖切片修复并写清测试。
- 优点：覆盖全部 A/B/C 项，保留架构边界，并给执行 lane 明确验证点。
- 缺点：计划和测试更细。
- 结论：采用。

方案 C：把 NPC autonomy 改成 parent agent 直接工具调用。
- 优点：某些动作的心智模型更直接。
- 缺点：违反 parent intent/local executor 分工，会撤销当前安全边界。
- 结论：拒绝。

## 架构决策记录

决策：采用完整台账修复路径。真实越界项直接修；过时测试和报告要校准；硬编码能力假设要移到明确策略边界，或标成预留未生效设置。

驱动：
- A1/A2/A3 是真实宿主越界。
- A4/C1/C2 说明旧审查文本和旧测试不是可靠权威。
- A5/B3/B4/B5/B6 是能力面和配置边界问题，不能丢。
- B1/B2 要保留并做资产/fallback 管理，不是删除。

被拒绝的替代方案：
- 只修 A1-A3：会丢掉配置和测试滞后问题。
- 本次直接实现完整 nested delegation depth enforcement：当前产品策略是 flat-only v1，`AgentTool` 深度语义目前也没有真实 enforcement。本计划通过把 `delegation.max_spawn_depth` 标成诚实的预留未生效设置来解决当前误导问题。
- 删除广义 prompt guidance：JSON intent 和 runtime facts guidance 是安全契约，不能按越界项误删。

后果：
- 执行时必须把每个台账编号关联到计划和测试。
- 部分项的交付形式是配置诚实化和 fallback 测试，而不是扩大行为面。
- 当前配置和日志不能再暗示 nested delegation depth 已经生效。

### S2：Skill contract 忠实披露

覆盖台账：A1、A2。

目标文件：
- `src/games/stardew/StardewNpcAutonomyPromptSupplementBuilder.cs`
- 需要时修改 `skills/gaming/**` 或 `src/skills/gaming/**`
- `Desktop/HermesDesktop.Tests/Stardew/StardewAutonomyTickDebugServiceTests.cs`

交付物：
- 移除用 `skillId switch -> needles` 决定 prompt 摘要重点的机制。
- 改用资产驱动或结构中性的机制：
  - 首选：skill 资产显式提供 compact contract section；
  - fallback：按 Markdown 结构确定性截取，不依赖 skill ID 的语义。
- 移除宿主生成的 fallback `skill_view(name=..., file_path=...)` 调用。
- 如果 `skill_view(...)` 原本由 skill 资产自己声明，则忠实保留。

退出条件：
- `BuildCompactSkillContract` 不再通过 `skillId switch` 和硬编码 needles 选行。
- 测试证明生成 prompt 的重点不是宿主语义关键词表决定的。
- 测试证明资产缺少 `skill_view(...)` 时 builder 不会自动补。
- 测试证明 skill 自己写的 reference guidance 会保留。

### S3：Timeout 事实交还给 agent

覆盖台账：A3。

目标文件：
- `src/games/stardew/StardewNpcAutonomyBackgroundService.cs`
- 需要时修改 `src/runtime/NpcRuntimeDriver.cs` 或相邻 runtime state/event 类型
- `Desktop/HermesDesktop.Tests/Stardew/StardewNpcAutonomyBackgroundServiceTests.cs`

交付物：
- 把 timeout 处理拆成两个职责：
  - 执行层清理：释放 claim、必要时取消/清理过期 bridge command、记录 command status；
  - agent 恢复决策：写入下一轮 autonomy tick 可见的 `action_slot_timeout` fact/status。
- 移除 action-slot timeout 的固定 restart cooldown 恢复策略。
- 保持资源释放和 bridge 安全清理的确定性。

退出条件：
- timeout 路径不再把 `PauseTrackerAsync` 加固定 restart cooldown 当成恢复决策。
- 结构化 timeout fact 能通过 runtime state、activity log 或 prompt-visible status 被下一轮 parent 看到。
- 测试仍证明 claim 被释放，过期 action state 不会永久挂住。
