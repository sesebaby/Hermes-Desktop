# 产品需求文档：桌面端开发者页与 NPC 观察台

## 1. 背景

用户对当前 Hermes Desktop UI 的主要不满不是缺少 NPC 后端能力，而是缺少一个能直观看清 NPC 决策全过程的开发者观察台。当前桌面壳已有 `Dashboard`、`Chat`、`Agent`、`Skills`、`Memory`、`Buddy`、`Settings` 导航项，导航映射集中在 `Desktop/HermesDesktop/MainWindow.xaml.cs:14`，`Dashboard` 可跳到 `Agent > Runtime`，但 `Agent > Runtime` 目前同时承载身份配置与 NPC runtime 调试，职责混杂。

本计划基于以下已完成的 `$deep-interview` 产物：

- `.omx/specs/deep-interview-desktop-developer-page.md`
- `.omx/interviews/desktop-developer-page-20260507T022045Z.md`
- `.omx/context/desktop-developer-page-20260507T020104Z.md`

约束是：先列方案文档，不直接动手实现；充分复用已有基础设施；不做双轨实现；UI 上不要中英文夹杂，除日志、JSON、工具名、路径、模型原文等原始内容外，普通界面文案使用中文。

## 2. RALPLAN-DR 摘要

### 原则

- 复用优先：读取和展示必须优先复用 `NpcRuntimeWorkspaceService`、`NpcRuntimeSupervisor`、`NpcNamespace`、`TranscriptStore`、`SessionTodoStore`、`runtime.jsonl` 与现有 Stardew bridge 诊断链路。
- 单一路径：不得创建第二套 NPC runtime、第二套 private chat、第二套本地执行器、第二套任务/待办、第二套记忆或第二套 trace 存储。
- 追踪优先：页面主目标不是控制 NPC，而是从“观察事实 -> 上下文 -> 模型回复 -> 工具/委托 -> 本地执行 -> bridge -> 结果”定位断点。
- 中文一致：页面标题、按钮、页签、状态解释和诊断说明使用中文；技术原文只出现在原始内容区域。
- 第一阶段只读：真实移动、说话、私聊、自然语言目标输入等有副作用动作不进入首版主范围。

### 决策驱动

- 可观测性：开发者要能快速回答“NPC 为什么没动、卡在哪一层、模型到底说了什么、工具到底调用了什么”。
- 架构一致性：现有桌面宿主已经是组合根，NPC agent 不是第二套 Hermes；UI 只能聚合现有事实，不能绕过宿主决策边界。
- 可审查增量：首版应足够小，可构建、可测试、可回退，不因大规模 UI 重写破坏现有调试入口。

### 可选方案

#### 方案 A：独立 `开发者` 页面，首屏为 `NPC 观察台`

优点：

- 与 `Dashboard` 的概览职责分开，与 `Agent` 的身份配置职责分开。
- 能承载高密度 trace、文件、日志、任务、工具调用和委托链路。
- 方便后续把 `Agent > Runtime` 的开发调试职责收敛到这里。

缺点：

- 需要新增导航项、页面、视图模型/聚合服务和中文资源。
- 如果第一阶段不处理旧入口，会短期存在两个入口。

#### 方案 B：扩展 `Agent > Runtime` 页签

优点：

- 改动入口最少，复用当前 `AgentPage.xaml` 中的 runtime 页签和 debug 按钮。
- 不增加左侧导航复杂度。

缺点：

- `AgentPage` 已承载 SOUL、USER、saved agents、NPC runtime 等多个职责，继续扩展会加剧混杂。
- 用户当前痛点正是“找不到能和 NPC agent 对话/观察的地方”，继续埋在 `Agent` 下不解决信息架构问题。
- 长时间线、原始模型回复、文件查看和日志关联不适合放进现有页签。

#### 方案 C：扩展 `Dashboard` NPC 卡片

优点：

- 入口曝光最高。
- 适合健康状态、最近错误、最后 trace 等摘要。

缺点：

- `Dashboard` 是运营总览，不适合承载原始响应、工具参数、文件内容、时间线和多页签详情。
- 容易把首页变成调试台，影响普通使用。

### 推荐

采用方案 A：新增独立中文 `开发者` 页面，首版聚焦 `NPC 观察台`。`Dashboard` 保留概览卡和跳转；`Agent > Runtime` 在首版保留最小摘要与跳转，后续再迁移固定 debug 按钮，避免一次性破坏既有调试路径。

## 3. 产品目标

- 开发者能从左侧导航进入 `开发者`。
- 开发者能选择 NPC，并看到状态、位置/上下文摘要、动作槽、私聊锁、下一次唤醒、最近 trace、最近错误、任务与待办摘要。
- 开发者能只读查看该 NPC 的 `SOUL.md`、`MEMORY.md`、`USER.md` 或等价上下文文件。
- 开发者能查看最近一次或指定 trace 的时间线，覆盖观察事实、上下文构建、模型回复、意图解析、本地执行器、工具调用、桥接命令和执行结果。
- 开发者能看到 AI API 回复和工具调用过程；如果现有持久化粒度不足，首版只补现有 transcript/runtime 日志链路上的最小字段。
- 首版“AI API 回复”定义为 Hermes 已持久化的 assistant 消息、reasoning 字段、tool call 请求和 runtime 诊断摘要，不承诺展示 provider HTTP 原始请求/响应包；如果需要 provider 级 raw payload，必须另开安全与隐私评审。
- 开发者能看到 `agent` 工具产生的子 agent 委托输出与工具调用结果，前提是这些内容已进入现有 transcript/tool result。
- 开发者能打开 Hermes 主日志、NPC `runtime.jsonl`、SMAPI 日志、bridge discovery 文件或 NPC runtime 目录。

## 4. 非目标

- 不实现新的 NPC 控制器。
- 不新增直接 NPC 自然语言目标输入。
- 不把移动、说话、私聊等有副作用动作作为首版默认操作。
- 不编辑 `SOUL.md`、`MEMORY.md`、`USER.md`。
- 不创建新数据库或新日志体系。
- 不重做整个桌面 UI。
- 不新增 NuGet 依赖。
- 不把 `todo` 伪装成 `TaskManager`，也不合并两套任务系统。
- 不保证首版能重放所有历史 trace；首版以“最新 trace + 最近活动”作为最小可交付。

## 7. 最小实现切片

### 切片 1：导航与页面骨架

- 在 `MainWindow.xaml` 新增中文资源驱动的 `开发者` 导航项。
- 在 `MainWindow.xaml.cs` 的 `PageMap` 新增 `developer` tag。
- 新增 `DeveloperPage.xaml` / `DeveloperPage.xaml.cs`，页面首屏只包含 `NPC 观察台`。
- 新增资源键到 `Strings/en-us/Resources.resw` 与 `Strings/zh-cn/Resources.resw`；中文资源必须完整。

### 切片 2：只读聚合服务

- 新增窄服务，例如 `DeveloperNpcInspectorService`，只做读取和投影。
- 复用 `NpcRuntimeWorkspaceService.GetSnapshot()`、`NpcRuntimeSupervisor.Snapshot()`、`NpcRuntimeInstance.TryGetTaskView()`。
- 通过 `NpcNamespace` 推导 NPC runtime 目录、`SOUL.md`、memory、transcripts、activity、trace 路径。
- 文件读取使用 async I/O，限制单次预览长度，避免 UI 线程阻塞。

### 切片 3：trace 时间线

- 解析选中 NPC 的 `activity/runtime.jsonl`。
- 以 `TraceId` 聚合 `NpcRuntimeLogRecord`。
- 将 `observation`、`diagnostic`、`local_executor`、`host_action`、`task_continuity` 等 action/stage 映射成中文事件类型。
- 对无法识别的日志项显示为 `原始事件`，并保留 JSON。

### 切片 4：transcript 详情

- 使用 `TranscriptStore.LoadSessionAsync(sessionId)` 读取 NPC 长期 session。
- 从 assistant 消息提取模型最终回复、reasoning 字段和 `ToolCalls`；不把 provider HTTP 原始包作为首版需求。
- 从 tool 消息提取 `ToolName`、`ToolCallId`、结果内容。
- 对 `agent` 工具结果提取子 agent 输出块，作为 `委托链路` 详情。
- 不新增 transcript 存储。

### 切片 5：最小观测补点

仅当实现前确认现有 transcript/runtime log 无法展示必要内容时，才允许补点：

- 在现有 `runtime.jsonl` 写入更多 `diagnostic` 记录，例如本地执行器工具名、参数摘要、模型是否返回工具调用、工具调用失败原因。
- 或在现有 transcript 消息中确保 `ToolCalls.Arguments` 与 tool result 已保存。
- 不写第二个日志文件，不建新数据库，不绕过 `TranscriptStore`。

### 切片 6：旧入口收敛

- `Dashboard` NPC 卡片继续显示摘要，跳转目标从 `Agent > Runtime` 改为 `开发者 / NPC 观察台`。
- `Agent > Runtime` 首版保留摘要与固定调试按钮，增加“前往开发者观察台”的入口。
- 第二阶段再把固定 Haley/Penny debug 按钮迁移为开发者页里的受控调试动作。

## 8. 验收标准

- 左侧导航出现 `开发者`，进入后标题为 `NPC 观察台`。
- 新增页面普通 UI 文案为中文；不出现 `task/TODO/diff/replay` 等混排控件标签。
- 原始日志、JSON、工具名、文件名、路径、异常堆栈和模型原文保持原样。
- 选中 NPC 后能看到 `NpcId`、显示名、runtime 状态、session id、最近 trace、最近错误、bridge 健康。
- 当前概览展示动作槽、pending work、私聊锁、下一次唤醒、事件游标和任务/待办摘要。
- 资料区能只读查看 `SOUL.md`、`MEMORY.md`、`USER.md`，并显示路径、更新时间和读取错误。
- 时间线能按 trace 展示至少这些中文事件：观察事实、模型请求、模型回复、意图解析、本地执行、工具调用、游戏桥接、执行结果、任务变化、诊断。
- `模型回复` 详情能展示 transcript 中保存的 assistant 内容；没有保存时显示中文缺失说明。
- `工具调用` 详情能展示工具名、参数、结果、工具调用 id；没有参数时显示中文缺失说明。
- `委托链路` 能展示 `agent` 工具的子 agent 输出；没有委托时显示“本次追踪未发现委托”。
- `任务与待办` 展示 `SessionTodoStore`/`NpcRuntimeTaskView` 的当前状态，不读取或写入 `TaskManager` 作为同一数据源。
- 相关日志入口能打开 Hermes 主日志、NPC runtime 目录、NPC `runtime.jsonl`、SMAPI 日志和 bridge discovery 文件。
- 文件读取、日志解析、transcript 读取不阻塞 UI 线程。
- 不新增依赖。
- 构建通过：`dotnet build .\Desktop\HermesDesktop\HermesDesktop.csproj -c Debug -p:Platform=x64`。

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 当前 `runtime.jsonl` 不保存完整模型原文或工具参数 | 页面无法满足“看到 API 回复和工具调用过程” | 优先从 transcript 读；缺失时只在现有日志/transcript 链路补最小字段 |
| 用户期望“AI API 回复”被理解为 provider HTTP 原包 | 需求承诺过大且有隐私风险 | 首版明确为 Hermes 持久化消息和诊断；provider raw payload 另开评审 |
| 展示 `SOUL.md`、`MEMORY.md`、模型回复可能暴露敏感信息 | 开发者页可能泄漏隐私 | 首版只本地显示；后续导出需单独评审；原始内容区域明确标注“本机调试内容” |
| `Agent > Runtime` 与 `开发者` 双入口短期并存 | 用户仍可能困惑 | 首版加跳转和说明；第二阶段迁移固定 debug 动作 |
| 长日志加载卡 UI | 桌面卡顿 | 后台读取、分页/尾部读取、预览长度限制、ListView 虚拟化 |
| 中文字符串变长导致布局挤压 | 可用性下降 | 使用自适应布局、`TextWrapping`、不固定窄宽度 |
| 过早加入控制动作 | 破坏只读定位目标 | 首版禁止副作用动作；动作进入第二阶段并明确标注 |

## 10. ADR

### 决策

新增独立中文 `开发者` 页面，首屏为 `NPC 观察台`，第一阶段只读观测，复用现有 NPC runtime、transcript、memory、todo、activity log 与 bridge 日志链路。

### 驱动

- 当前问题是开发诊断可见性不足，不是后端能力缺失。
- `Agent` 页已经职责过载，继续扩展会加重混乱。
- 用户明确要求复用后端基础设施、避免双轨实现、中文 UI 一致。

### 备选

- 扩展 `Agent > Runtime`：改动少，但职责继续混杂，不能形成直观观察台。
- 扩展 `Dashboard`：入口明显，但不适合高密度原始调试数据。
- 新建完整 trace 数据库：查询能力强，但违反首版轻量、复用和无双轨约束。

### 选择理由

独立 `开发者` 页面最符合信息架构目标；只读首版把风险控制在 UI 和投影层；最小观测补点保持在现有 transcript/runtime 日志链路内。

### 后果

- 首版需要新增页面、资源、只读投影服务和测试。
- 短期会保留 `Agent > Runtime`，但应明确其收敛方向。
- 若现有日志粒度不足，执行阶段需要补少量可观测字段。

### 后续

- 第二阶段把受控调试动作迁移到开发者页。
- 第三阶段增加 trace 筛选、诊断包导出、上下文注入差异视图。
- 若要暴露编辑人格/记忆或触发真实动作，必须另开评审。

## 11. 执行分工建议

### 可用 agent 类型
