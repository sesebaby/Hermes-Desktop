# 星露谷 NPC 可见移动修复上下文快照

## 任务陈述

修复手动测试中“海莉可以私聊、也能主动说话，但始终没有可见移动”的问题。修复必须遵守现有原则：NPC 自主行为仍由 `NpcAutonomyLoop` 把观察事实交给 agent 决策，不能新增非私聊事件驱动的移动逻辑；如果要提升移动意愿，必须通过提示词、可观察事实或工具语义让 agent 自己选择 `stardew_move`。

## 期望结果

- 海莉在合适的自主 tick 中能够基于事实和提示词主动选择 `stardew_move`，而不是只说“要走”。
- `stardew_move` 的目标不靠 agent 瞎猜坐标，而来自 bridge/status/world facts、玩家指令解析结果或明确安全测试目标。
- SMAPI bridge 的 move 不再只是瞬移完成；至少在 Phase 1 中提供可见的、可诊断的移动执行语义，手测能看到海莉位置变化或移动过程。
- 日志和 transcript 能清楚区分：agent 是否调用了 move、bridge 是否接受 move、move 是否被阻塞、是否完成。

## 约束

- 不允许 background service 以事件驱动方式替 NPC 决定移动；私聊入口除外。
- 不允许新增第二套 NPC 专用 agent/prompt/tool 组装链；保持桌面 agent 与 NPC agent 同源。
- 保持单 Desktop 进程托管多个 NPC runtime，不改成一 NPC 一进程。
- 不新增依赖，除非后续执行阶段证明现有 Stardew/SMAPI API 无法满足。
- 修改前应先写失败测试锁定：agent 可获得安全移动目标事实、move 工具描述/提示词可引导调用、bridge move 具备可诊断可见移动语义。

## 未知与开放问题

- Stardew 1.6 / SMAPI 中对 NPC 可见走路最合适的低风险 API 是 `controller`/pathfinding、临时 schedule，还是逐 tick tile stepping。
- HaleyHouse 附近哪些 tile 是稳定安全目标，是否应由 bridge 动态发现而非硬编码。
- Phase 1 是否只需要“短距离可见挪动/走到门口”来证明 move，还是要支持跨 location 到 Town。
- 是否需要新增 `stardew_world_snapshot` 工具给 agent 可读候选目标，还是扩展现有 `stardew_status` 的 facts。
