## 访谈记录

### Round 1

问题：这次逐项审查的裁剪基准到底是什么：用户想把 Hermes-Desktop 收缩成“只服务 Stardew MVP 的专用游戏宿主”，还是保留“Hermes 桌面 agent 宿主”这个产品边界，只删除休眠、误导、未接线或维护成本明显不值的部分？

回答：只服务 Stardew MVP 的专用游戏宿主。

影响：计划审查应以 Stardew MVP 专用宿主为边界；已实现但不服务 Stardew MVP 的通用平台能力可以进入删除/隐藏/归档讨论，但仍需逐项确认非目标、保留边界和验证标准。

### Round 2

问题：是否把第一版非目标写成“凡是不直接支撑 Stardew NPC 自主、私聊、配置、日志观测、运行时调试的能力都删除/隐藏/归档”？

回答：用户希望一项一项讨论。

影响：后续按计划条目逐项审查，每项单独给出事实、影响、初步判断，并单独确认删除/保留/隐藏/归档。

### Round 3

问题：第 1 项 Buddy 是直接删除，先隐藏 UI 保留代码，还是保留并改造成 Stardew 相关能力？

回答：用户提出条件问题：如果希望游戏中的宠物也介入 agent，Buddy 是否有用，还是应使用 NPC runtime 模式？

证据补充：

- `src/buddy/Buddy.cs` 的 `BuddyService` 是单个桌面 buddy 的生成、持久化、刷新 soul 与 ASCII 渲染服务。
- Buddy 生成的是 `Species`、`Rarity`、stats、`Name`、`Personality` 等桌面伙伴身份数据。
- `BuddyService` 只持久化一个配置文件路径下的 buddy，没有每个游戏实体独立 namespace、transcript、memory、todo、tool registry、Stardew tool surface、autonomy loop 或 host action execution。
- `NpcRuntimeContextFactory` 会为每个 NPC 创建独立 `SoulService`、`MemoryManager`、`TranscriptStore`、`SessionTodoStore`、`ContextManager`、`ToolRegistry` 等，才是游戏内 actor agent 的合适模式。

初步结论：游戏宠物 agent 应走 NPC runtime/actor runtime 模式；现有 Buddy 只能作为“宠物身份生成/外观选择”的小灵感来源，不应作为 pet agent 基础设施保留。

### Round 4

问题：是否确认第 1 项按“删除 Buddy，宠物 agent 未来走 NPC runtime 模式”定？

回答：删除。

决策：`Buddy` 系统删除。未来若做游戏宠物 agent，应复用/扩展 NPC runtime 或抽象 actor runtime，而不是基于 Buddy。

### Round 5

问题：第 2 项 Wiki 删除，还是保留作为未来游戏资料库？

回答：用户提出条件问题：未来想给游戏中的 NPC 添加游戏知识，例如矿洞情况、姜岛情况，Wiki 是否适合？

证据补充：

- `WikiManager` 是纯 Markdown 数据/搜索层，支持 schema、index、page save/delete/search，但无 LLM 调用、无 UI，并且当前未通过 `wiki_*` 工具接入 agent。
- 当前 Stardew 游戏知识的既有模式是 `skills/gaming/*` 和 reference 文件。例如 `stardew-navigation` 明确要求本地 executor 用 `skill_view` 读取地图资料和 POI reference。
- `NpcRuntimeContextFactory` 已把 `SkillManager` 接入 NPC runtime；`AgentCapabilityAssembler` 已注册 `skill_view` / `skill_invoke` 等工具。

初步结论：矿洞、姜岛、地点意义、路线、NPC 社会知识等应作为 Stardew skill/reference 或 NPC pack 资源进入 agent，而不是保留通用 Wiki。通用 Wiki 当前会形成第二知识面，并且没有工具接入。

### Round 6

问题：是否确认第 2 项按“删除 Wiki，游戏知识走 skills/reference”定？

回答：删除。

决策：`Wiki` 系统删除。游戏知识进入 Stardew skills/reference 或 NPC pack，不保留第二套通用 Wiki 知识库。

### Round 7 准备

条目：`CoordinatorService` / `AgentService`

证据补充：

- `src/agents/AgentService.cs` 提供通用子 agent spawn、worktree/remote isolation、`TeamManager`、`MailboxService`。
- `src/coordinator/CoordinatorService.cs` 通过 `AgentService` 和 `TaskManager` 做通用多 worker orchestration。
- `Desktop/HermesDesktop/App.xaml.cs` 注册了 `AgentService` 与 `CoordinatorService`。
- 搜索结果显示 Stardew 的 `npc_delegate_action`、local executor、NPC runtime 不引用 `AgentService` / `CoordinatorService`。
- `src/Tools/AgentTool.cs` 是独立的模型内工具式子 agent，由 `AgentCapabilityAssembler` 注册，不依赖 `AgentService`。
- Stardew 委托执行走 `NpcRuntimeSupervisor`、`NpcLocalExecutorRunner`、`DelegationChatClient`、`StardewNpcToolFactory`。

初步结论：如果只服务 Stardew MVP，可以删除 `AgentService` / `CoordinatorService` 这套通用多 worker/worktree/remote 编排能力，但必须明确保留 `AgentTool`、`npc_delegate_action`、Stardew local executor 和 delegation lane。
