# 草案：开发者页调试动作与诊断增强

## 1. 需求摘要

本阶段在已完成的开发者页只读观察台基础上继续推进三件事：

- 增加一个短手测命令，把海莉或潘妮明确放到 `Town`，便于验证她们从镇上主动跨地图去 `Beach`。
- 第二阶段把现有受控调试动作收敛到 `开发者 / NPC 观察台`。
- 第三阶段补齐追踪筛选、诊断包导出、上下文注入差异视图。

本计划只规划，不直接实施代码。

## 2. RALPLAN-DR 摘要

### 原则

- 单一路径：SMAPI 控制台命令、桌面按钮和日志审计必须复用同一个 bridge 调试动作实现，不各写一套放置 NPC 的逻辑。
- 明确副作用：把 NPC 放到 `Town` 是手测辅助和显式 teleport，不能被记录或解释为自然跨地图移动成功。
- 观察优先：开发者页继续以定位断点为主，动作只作为受控调试区的一部分，并带确认、执行结果和日志。
- 中文一致：普通 UI 文案、文档标题、页签和按钮使用中文；日志、路径、JSON、工具名、模型原文保持原样。
- 脱敏导出：诊断包默认本地生成，必须移除 bridge token、API key、Authorization、凭据类字段。

### 决策驱动

- 手测效率：当前 `teleport <npc>` 只是把玩家传到 NPC 附近，不能让 NPC 回到镇上，无法稳定复现 `Town -> Beach` 场景。
- 架构一致性：现有 NPC runtime、transcript、todo、runtime.jsonl、bridge HTTP 面都已接线，新增能力应挂在这些链路上。
- 可诊断性：用户需要看到模型回复、工具调用、委托、任务、待办、执行结果，以及为什么跨地图没有发生。

### 可选方案

#### 方案 A：bridge 调试动作核心 + SMAPI 短命令 + 开发者页复用

做法：在 bridge 侧抽一个 `BridgeNpcDebugRepositioner`，SMAPI 控制台命令和 `/debug/npc/reposition` HTTP 端点都调用它；桌面 `StardewNpcDebugActionService` 再调用该 HTTP 端点，开发者页展示按钮和结果。

优点：

- 一套放置逻辑、一套日志、一套失败码。
- 同时满足“短命令”和后续开发者页受控动作。
- 不触碰 NPC runtime 决策链，也不冒充自然移动。

缺点：

- 需要新增 bridge debug endpoint 和 DTO。
- 桌面端要补 `StardewNpcDebugActionService` 方法与 UI 状态。

#### 方案 B：只加 SMAPI 短命令，开发者页继续跳转和观察

做法：只在 bridge 控制台增加 `npc_town <npc>`，桌面不接这个动作。

优点：

- 最快，风险最低。
- 对桌面 UI 无影响。

缺点：

- 第二阶段“受控调试动作迁移到开发者页”仍未完成。
- 用户仍要在 SMAPI 控制台和桌面观察台之间切换。

#### 方案 C：直接实现完整跨地图分段执行

做法：跳过调试动作迁移，直接让 bridge 执行 `Town -> Beach` 全路径。

优点：

- 如果成功，用户无需手动放置 NPC。

缺点：

- 与当前 PRD 的“小步 route probe”节奏冲突。
- 风险集中在路线状态机，失败时更难判断是目标选择、执行层、bridge 还是游戏寻路问题。

### 推荐

推荐方案 A。它把“短手测命令”和“开发者页受控调试动作”合成同一个底层能力，同时保留跨地图自然移动验证的真实性。

## 3. 阶段边界

### 第一阶段：短手测命令与共用 bridge 调试动作

目标：能在 SMAPI 控制台输入短命令，把海莉或潘妮放到 `Town` 的可站立点。

推荐命令：

```text
npc_town haley
npc_town penny
```

中文别名仍由现有 `NpcTargetNameResolver` 支持：

```text
npc_town 海莉
npc_town 潘妮
```

实现边界：

- 不覆盖现有 `teleport` 命令；现有命令含义是“玩家到 NPC 附近”。
- 新命令含义是“调试放置 NPC 到镇上”。
- 目标位置优先复用 `BridgeDestinationRegistry` 的 `town.square` 或 `town.fountain`，并在附近找开放 tile。
- 执行使用 Stardew 的角色 warp/placement API，是明确调试 teleport。
- 写 SMAPI 日志，例如 `debug_npc_reposition_completed`，包含 `npcId`、原位置、目标位置、命令名。
- 如果世界未加载、NPC 不存在、目标地图不存在、目标 tile 不可站立，返回结构化失败日志。

主要触点：

- `Mods/StardewHermesBridge/Commands/TestTeleportCommand.cs:21` 现有 `teleport` 命令注册。
- `Mods/StardewHermesBridge/Commands/TestTeleportCommand.cs:55` 当前只调用 `Game1.warpFarmer(...)`。
- `Mods/StardewHermesBridge/Bridge/BridgeDestinationRegistry.cs:57` 已有 `town.square`。
- `Mods/StardewHermesBridge/Bridge/BridgeDestinationRegistry.cs:73` 已有 `beach.shore_photo_spot`，可用于手测目标对照。

### 第二阶段：受控调试动作迁移到开发者页

目标：`Agent > Runtime` 不再承担主要调试动作；`开发者 / NPC 观察台` 提供受控动作区。

动作范围：

- `放到镇上`：调用 bridge debug reposition endpoint，底层复用第一阶段的 `BridgeNpcDebugRepositioner`。
- `单步唤醒`：复用 `StardewAutonomyTickDebugService.RunOneTickAsync(...)`。
- `发送调试台词`：复用 `StardewNpcDebugActionService.SpeakAsync(...)`，首版只保留固定文案或明确标注调试文案。
- `打开运行目录 / 刷新观察台`：沿用现有开发者页能力。

UI 规则：

- 增加一个中文分区：`受控调试`。
- 有副作用动作必须有确认文案或二次点击保护，按钮旁显示最近执行结果。
- 按钮禁用时给出中文原因：bridge 未连接、未选 NPC、世界未加载、动作执行中。
- 保持开发者页主文案中文；原始错误码可保留原文。
- `Agent > Runtime` 保留摘要和“打开开发者 NPC 观察台”跳转，固定按钮迁移后不再重复展示。

### 第三阶段：筛选、导出、上下文差异

目标：开发者能从一次失败手测中快速导出证据，并比较模型实际收到的上下文。

能力范围：

- 追踪筛选：按 NPC、追踪 id、事件类型、commandId、工具名、错误码、文本关键词筛选。
- 诊断包导出：生成本地 zip，包含所选 NPC 的运行时文件、选中 trace、相关 transcript 摘要、Hermes 日志尾部、SMAPI 日志尾部、脱敏 bridge discovery 和诊断说明。
- 上下文注入差异视图：展示本次 NPC 调用中人格、记忆、技能、世界事实、待办、检索内容、最近消息等 block 的注入前后状态、字符数/估算 token、是否被预算策略裁剪。

实现边界：

- 不新增第二套 trace 数据库。
- 追踪筛选在现有 `runtime.jsonl` 和 transcript 投影上做。
- 诊断包使用 `System.IO.Compression`，不新增依赖。
- 上下文差异优先在现有 `runtime.jsonl` 写结构化诊断记录，例如 `context_injection_snapshot`，而不是另建存储。
- 上下文原文可能包含隐私；默认展示 block 名称、长度、hash、裁剪状态和短预览，导出时允许配置是否包含原文。

主要触点：

- `src/runtime/NpcDeveloperInspectorService.cs:48` 当前 trace/transcript 异步读取。
- `src/runtime/NpcDeveloperInspectorService.cs:125` 当前 trace 加载。

- `src/runtime/NpcDeveloperInspectorService.cs:445` 当前只读 SQLite transcript reader。

## 5. 验收标准

- 在 SMAPI 控制台输入 `npc_town haley` 或 `npc_town penny` 后，对应 NPC 位于 `Town` 的可站立 tile。
- SMAPI 日志明确写出这是 `debug_npc_reposition`，不能与自然移动 `task_move_enqueued/task_completed` 混淆。
- 从 `Town` 发起“去海边”的自然语言目标时，route probe 仍通过 mechanical target / bridge task status 链路观察，不能把放置命令当成跨地图成功证据。
- 开发者页出现中文 `受控调试` 分区，选中海莉或潘妮后可执行 `放到镇上`、`单步唤醒`、`发送调试台词`。
- 有副作用动作执行中会禁用按钮，执行后显示中文结果和原始错误码。
- `Agent > Runtime` 不再重复展示固定调试按钮，保留跳转到开发者页。
- 追踪列表能按事件类型、追踪 id、commandId、错误码和关键词筛选。
- 诊断包导出为本地 zip，并脱敏 bridge token、Authorization、API key 等字段。
- 上下文差异页签能展示注入块、裁剪状态、长度变化和短预览。
- 不新增第二套 NPC runtime、todo、memory、transcript、trace 数据库或工具 lane。

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 调试放置被误认为跨地图成功 | 验收结论失真 | 日志和 UI 明确标注为调试放置，不写入自然移动 task 成功 |
| 开放任意坐标 reposition | 可能破坏存档或卡 NPC | 首版只允许固定 `Town` 目标和已知 NPC |
| 诊断包泄露 token 或隐私 | 安全风险 | 默认脱敏，导出前列出包含文件，token 类字段强制替换 |
| 上下文差异记录过多原文 | 隐私和体积风险 | 默认记录元数据和短预览，原文导出另设显式选项 |
| 开发者页按钮和 SMAPI 命令逻辑分叉 | 后续维护困难 | 抽同一个 bridge helper，控制台和 HTTP 都调用它 |

## 8. ADR 草案

### 决策

采用“bridge 调试动作核心 + SMAPI 短命令 + 开发者页复用”的路线。短命令解决即时手测，开发者页动作通过同一个 bridge endpoint 复用该能力，诊断增强继续读取现有 runtime/transcript/log 链路。

### 驱动

- 用户需要稳定把 NPC 放回 `Town` 以触发 `Town -> Beach` 跨地图测试。
- 现有 `teleport` 命令只移动玩家，不满足需求。
- 开发者页一期已经是 NPC 观察中心，后续调试动作应收敛到这里。
- 不能创建第二套 runtime、记忆、任务、trace 或工具通道。

### 备选

- 只加 SMAPI 命令：最快，但桌面开发者页仍无法完成受控调试闭环。
- 只加桌面按钮：不满足短命令和 SMAPI 手测便利。
- 直接做完整跨地图分段执行：风险过大，不利于先定位 route probe。

### 后果

- bridge 会新增一个明确的 debug action surface。
- 开发者页从只读观察台进入“受控动作 + 观察”阶段。
- 诊断包导出和上下文差异需要严格脱敏和体积限制。

### 后续

- 继续验证 `skills/gaming/stardew-navigation/SKILL.md` 是否仍有“跨地点移动未来”之类滞后文案。
- 如果 route probe 通过，再规划真正的跨地图第一段移动执行。
