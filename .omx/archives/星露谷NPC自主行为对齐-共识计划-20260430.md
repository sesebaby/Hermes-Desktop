## RALPLAN-DR 摘要

### 原则

1. 先证明最小的真实 NPC 生命闭环，再扩展到整个村庄。
2. 通过 `gameId + saveId + npcId + profileId` 隔离 NPC 身份、记忆、转录、trace、活动和 prompt 上下文。
3. 所有游戏动作都只通过类型化的 Stardew bridge 契约和 `StardewCommandService` 发生。
4. 保持强自治：Agent 负责节奏、观察、决策和行动。
5. 将游戏/桥接事件仅视为事实。
6. prompt 组装保留在运行时本地的 `ContextManager` / `PromptBuilder`。
7. 在引入更丰富行为之前，先限制成本和失控行为。

### 决策驱动因素

1. 第一个真正有意义的里程碑是一个真实闭环：`self-driven tick -> observe -> context -> decide -> tool/action -> poll -> persist -> next tick`。
2. 实现必须复用现有 Hermes Desktop 原语：`NpcRuntime*`、`NpcNamespace`、`Agent`、`MemoryManager`、`ContextManager`、`PromptBuilder` 以及 Stardew bridge 契约。
3. 测试必须能在没有真实星露谷或 SMAPI 的情况下运行。

### 可行方案

#### 方案 A：直接把循环放进 `NpcRuntimeInstance`

优点：
- 新类最少。
- 生命周期状态已经在这里。

缺点：
- 把生命周期状态、后台编排、LLM 调用、桥接调用、trace 和记忆混在一个类里。
- 单独测试单个 tick 更困难。

#### 方案 B：添加由运行时生命周期拥有的专用 `NpcAutonomyLoop`

优点：
- 让 `NpcRuntimeInstance` 继续专注于生命周期和快照状态。
- 给执行提供一个直接、可测试的 `RunOneTickAsync`。
- 提供一个单一的、被允许进入 LLM/tool 决策的入口。
- 让事实型事件边界可以被强制执行。

缺点：
- 增加了一层小型编排和 DI 绑定。
- 需要注意不要再建一套 prompt 系统。

#### 方案 C：立即把通用 `Agent` 整个用作运行时

优点：
- 最接近 hermescraft 的长会话工具型 Agent 模型。
- 复用现有 Agent 工具循环。

缺点：
- 对 Phase 1 来说范围过大，除非 NPC 工具和上下文隔离已经解决。
- 有把桌面工具暴露给 NPC 的风险。

已选：方案 B，但仅在本地上下文和工具 allowlist 构建完成后，在循环内部使用一个 NPC 范围的 `Agent`。

## 不可协商的自治边界

实现必须不是事件驱动的。

SMAPI 事件、桥接事件、接近变化、对话点击、聊天/收件箱消息、调度 tick 和世界状态变化，都只能作为观察事实持久化。它们不能：

- 调用 LLM completion，
- 调用 `Agent.ChatAsync`，
- 调用 `StardewCommandService.SubmitAsync`，
- 调用 `move` 或 `speak`，
- 入队语义指令，例如“Haley 现在应该回应”。

只有 `NpcAutonomyLoop.RunOneTickAsync` 或未来显式的循环调度器才能进入 observe/decide/tool/action 路径。调度器可以决定 *何时* 跑 tick，但不能决定 NPC *做什么*。

## 需求摘要

- 先为 Haley 添加一个受限的单 NPC 自治循环。
- tick 在任何决策前都要先观察当前事实。
- tick 通过 `ContextManager` / `PromptBuilder` 构建 NPC 本地上下文。
- Agent 只能看到 NPC 安全的工具。
- move/speak/status 动作都通过 `StardewCommandService`。
- 长命令要轮询状态，直到进入终态或达到配置上限。
- trace/activity/memory 都写入 NPC 命名空间。
- 事件摄取只保留事实，并且必须有负向测试。

## 验收标准

1. 启动 Haley 会创建运行时命名空间和具备循环能力的句柄，同时保留 supervisor 快照。
2. `RunOneTickAsync` 在决策或行动之前先执行观察。
3. 事件/事实摄取在下一次显式 tick 之前不会产生 LLM/tool/command 副作用。
4. Haley 上下文通过运行时本地的 `ContextManager` / `PromptBuilder` 流动。
5. NPC Agent 只允许使用严格白名单内的 Stardew/NPC 本地工具。
6. move 决策通过 `StardewCommandService` 提交 `GameActionType.Move`。
7. speak 决策通过 `StardewCommandService` 提交 `GameActionType.Speak`。
8. 长命令轮询 `task/status`，直到终态或达到限制。
9. 桥接不可用/世界阻塞事实会生成 no-op/paused trace，而不是强制动作。
10. 一个完成的、有效的 tick 会写入 trace/activity，并尝试进行简洁的 NPC 记忆持久化。

## 实施步骤

1. 添加仅事实观察边界。
   - 添加 `IStardewWorldObserver` 和 `StardewWorldObserver`。
   - 为 `StardewNpcStatusData` 包装 `StardewBridgeRoutes.QueryStatus`。
   - 如果需要事件事实，则添加被动的 observation fact 类型/存储。

2. 添加 NPC 范围的 Stardew 工具。
   - 为 status、move、speak 和 task status 添加包装器。
   - 在运行时上下文内部绑定 `npcId`、`saveId`、`traceId` 和幂等性。
   - 复用 `StardewCommandService`；不要直接调用 SMAPI HTTP。

3. 使用现有基础设施连接运行时本地上下文。
   - 不要添加 `StardewPromptAssembler` 或 `NpcPromptContextBuilder`。
   - 根据需要把人格文件复制/种子化到 `NpcNamespace`。
   - 构建本地 `SoulService`、`MemoryManager`、`TranscriptStore`、`ContextManager` 和 `PromptBuilder`。
   - 添加一个窄技能绑定：从 `skills.json` 解析名称到 `skills/gaming/stardew-*.md`，并把文本输入到现有上下文输入中。

4. 添加 `NpcAutonomyLoop`。
   - 添加 `RunOneTickAsync(NpcRuntimeInstance instance, CancellationToken ct)`。
   - 流程：预算 -> 观察 -> 准备上下文 -> NPC 范围的 Agent/tool allowlist -> 决策回合 -> 命令轮询 -> trace/activity/memory -> 更新 `LastTraceId`。
   - 将持续后台调度保持为独立且显式的部分。

5. 安全连接生命周期。
   - 仅扩展 runtime/supervisor/host 到足以创建并启动一个可循环的 Haley runtime。
   - 不要在 Desktop 启动时自动启动所有发现的 NPC。
   - 保留 Dashboard 的发现行为。

6. 在每个切片之前或同时添加测试。
   - 优先在动作测试之前完成自治边界的负向测试。

## 风险与缓解

- 隐藏事件控制：用负向测试强制事实型事件摄取。
- 工具泄漏：用本地工具注册表构建 NPC Agent，并断言 allowlist。
- Prompt 漂移：在测试中要求走运行时本地的 `ContextManager` / `PromptBuilder` 路径。
- 失控成本：从单 tick、预算限制、取消和不开启应用启动循环开始。
- 真实游戏依赖：在测试中使用 fake observer、fake bridge client 和 fake command service。

## ADR

决策：将 Phase 1 实现为一个专用的、单 NPC、Agent 驱动的 `NpcAutonomyLoop`，使用运行时本地的 Hermes context/memory/soul 基础设施和 NPC 范围的 Stardew 工具，并通过 `RunOneTickAsync` 执行。

驱动因素：
- 最优先的缺口是 NPC 的自治运行时行为。
- 多 NPC 设计要求 NPC 自我观察/自我决策，以及仅事实型的 bridge 语义。
- 现有 Hermes context 和工具基础设施应该被复用，而不是绕开。

备选方案：
- 直接把循环放进 `NpcRuntimeInstance`。
- 立刻把通用 `Agent` 整个用作运行时。
- 事件驱动的 NPC 响应。
- 自定义 Stardew prompt 组装器。

选择原因：
- 专用循环提供了单一、可强制执行的自治边界，以及一个可测试的 tick 表面。
- 运行时本地的 `ContextManager` / `PromptBuilder` 满足设计文档要求，并避免 prompt 路径漂移。
- 事实型事件防止游戏变成隐藏的语义指挥器。

后果：
- 增加一个小型编排类和本地服务工厂/绑定。
- 需要围绕工具 allowlist 和事件不驱动行为添加显式测试。
- 持续后台模式将成为后续更安全的一步，需在单 tick 被证明后再推进。

后续事项：
- 仅在 Haley tick 通过测试后，再添加多 NPC 调度器。
- 添加 SocialRouter 时只作为消息投递，不承载语义。
- 只有在 status/move/speak/task-status 闭环稳定后，才加入更丰富的工具。

## 可用 Agent 类型清单

- `explore`：快速仓库定位和符号/文件映射。
- `architect`：架构审查和边界验证。
- `executor`：实现和重构。
- `test-engineer`：测试设计和覆盖。
- `debugger`：失败诊断。
- `build-fixer`：构建/工具链/类型失败修复。
- `verifier`：证据收集和验证。
- `code-reviewer`：最终风险审查。
- `security-reviewer`：如果 bridge auth 有变化，则审查 bridge/token/信任边界。
- `writer`：文档和迁移说明。
