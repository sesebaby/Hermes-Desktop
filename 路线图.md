# Hermes Desktop 路线图

> 最后更新：2026-04-25 | 当前版本：v2.4.0

本路线图旨在缩小与 Superset、Nimbalyst、Codex、Cursor、Claude Code 和 Capy 的竞争差距，同时保留 Hermes 的独特优势：原生 Windows UI、运行时模型切换、灵魂/身份系统和原生消息传递。

---

## 竞争地位

| 维度 | Hermes | 领先者 | 差距 |
|-----------|--------|---------|-----|
| 原生 Windows UI | WinUI 3 + Mica | Cursor (Electron), Codex (混合) | 领先 |
| 运行时模型切换 | Claude/GPT/Ollama/Qwen/DeepSeek 对话中切换 | 无此功能 | 领先 |
| 内存与身份 | 灵魂系统、wiki FTS5、12个模板、编译内存 | 无此深度 | 领先 |
| 工具广度 | 27+ 工具、并行只读(8个工作者)、权限控制 | 与 Claude Code 相当 | 持平 |
| 会话编排 | 扁平聊天列表 | Superset 看板、Nimbalyst 项目视图 | 落后 |
| 文件可追溯性 | 手动 `git diff` 在工作树中 | Nimbalyst 内联差异审查 | 落后 |
| IDE 集成 | 仅独立运行 | Cursor/Codex 在编辑器中运行 | 落后 |
| 云端执行 | 仅接口(Docker/SSH/Modal/Daytona 存根) | Devin 云端优先、Superset 工作树 | 落后 |
| MCP 生态系统 | 加载配置，从不注册工具 | OpenCode/Superset 一流 MCP | 落后 |
| 自动更新 | 无 | 所有竞争对手都有 | 落后 |
| 流式传输可靠性 | 在某些提供商上挂起(问题 #26) | Claude Code/Codex 稳健 | 落后 |
| 移动监控 | 仅 Telegram/Discord 机器人 | Nimbalyst iOS 应用 | 落后 |

## 架构原则

1. **始终本地优先**：所有功能离线工作。核心功能不依赖云端
2. **仅可添加**：每个新功能都可在设置中切换。喜欢 v2.4.0 的用户可以禁用新面板
3. **Git 状态管理**：会话历史、任务看板、文件变更 — 全部在 SQLite 中，带原子写入(现有模式)
4. **无 Electron**：VS Code 扩展通过 WebSocket 与本机 Hermes 通信。Hermes 保持 WinUI 3
5. **测试覆盖率**：每个新服务都有单元测试。每个 UI 页面都有冒烟测试"加载不崩溃"

## 风险登记

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|------------|--------|------------|
| VS Code 扩展维护负担 | 中等 | 中等 | 从只读会话列表 + 仅聊天发送开始 |
| Docker 后端需要 Docker Desktop | 高(对用户) | 低 | 可选。LocalBackend 保持默认 |
| 语义压缩需要 Ollama | 中等 | 中等 | 回退到关键词搜索(`SessionSearchIndex`) |
| 自动更新器破坏自定义构建 | 低 | 高 | 仅在 `AssemblyCompany == "RedWoodOG"` 时检查 GitHub Releases |
| 看板增加 UI 复杂性 | 中等 | 中等 | 单独页面，不替换聊天。用户选择加入 |
| SQLite-vec 扩展可用性 | 中等 | 中等 | 捆绑 `sqlite-vec` DLL 或使用 Ollama 嵌入 |
