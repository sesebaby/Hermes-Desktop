## Goal
将当前 Hermes Desktop 改造成一个**面向游戏接入的最小减法运行时**，优先服务星露谷等游戏场景。

核心目标不是抽象出一个过度瘦身的协议核，而是在**尽量保留 Hermes 的能力密度**前提下，只移除那些**明显不适合游戏运行时**的模块，最终便于后续封装为 DLL 或 EXE 供游戏接入。

未来目标约束：**一个游戏 NPC 对应一个 agent**。NPC 需要在游戏世界中像人一样持续生活，因此不能误删支撑“自主行动、记忆沉淀、日程/意图、反思、人格表达、感知与多 agent 协作”的底座能力。当前减法只删除外部桌面/开发/社交平台语境，不删除未来 NPC 生活模型需要的运行时原语。

关键修正：`GameRuntimeProfile` / 白名单只能作为**边界定义与验收工具**，不能成为长期双轨方案。实施时一旦某类能力被判定为“游戏运行时废弃”，必须删除或退役对应源码、注册入口、配置、skills 内容、启动链路与测试假设；不允许保留“完整桌面轨 + 游戏裁剪轨”两套并行运行时。

## Constraints
- **最小减法原则**：只删除“明显不适合游戏”的模块，不做大规模理念重写。
- **退役优先原则**：profile / allowlist 用来定义最终边界；进入实施后，废弃能力必须从运行时代码面删除或退役，不允许只靠配置隐藏。
- **MCP 不能删除，但其内容可裁剪**：`src/mcp/*.cs` 框架必须保留，但不适合游戏的 server 配置与目录内容需要删除。
- **skills 不能删除，但其内容可裁剪**：`src/Tools/SkillInvokeTool.cs` 与相关技能体系必须保留，但不适合游戏的 skill 条目、说明、触发内容需要删除。
- **AgentTool / subagent 编排绝对保留**：`AgentTool`、subagent 调度、agent service / coordinator 这条编排主链路不进入删除候选。后续删开发工具时，只能处理被判定为非游戏的具体执行器、配置或 profile 内容，不能通过删除 subagent 编排来“省事”。
- **Dreamer / todo / cron 绝对保留**：`DreamerService` 与 dreamer 相关自主循环、`TodoWriteTool`、`ScheduleCronTool`、`TaskManager` / cron scheduler 这一类日程和意图能力不进入删除候选。若当前实现绑定了 Discord、gateway、桌面工作流或开发语境，只能拆除或替换外部适配层，不能删除能力本体。
- **NPC 生活底座不误删**：自主后台循环、日程/意图队列、反思/梦境/记忆整理等概念能力与“一个 NPC 一个 agent”目标相关；若当前实现绑定了 Discord、gateway、桌面任务 UI 或开发语境，应拆掉这些外壳和依赖，而不是把底座概念整体删除。
- **记忆 / 人格 / 玩家交互不误删**：`MemoryTool`、`SessionSearchTool`、`MemoryReviewService`、`SoulService`、`AgentProfile`、`AskUserTool` 等支撑 NPC 连续性、身份、回忆和向玩家发问的能力不进入删除候选。
- **讨论方式约束**：后续方案讨论应聚焦“去掉什么、为什么去掉”，并始终先给推荐和理由。
- **Desktop UI 不做首轮重设计，但退役项 UI 入口必须删除**：`Desktop/HermesDesktop/**` 的 UI 宿主形态先不重构；但若被删除能力在 UI 上有入口，必须同步删除对应导航项、页面区块、按钮、设置项、状态卡、资源字符串与 code-behind 处理，不能留下空入口、灰掉入口或隐藏开关。
- **媒体与语音能力不删除**：后续要集成进游戏 mod，`ImageGeneration / TTS / Transcription / Vision` 不纳入删除计划。
- **保留核心运行时能力**：`src/Core/Agent.cs`、`src/memory/MemoryManager.cs`、`src/soul/SoulService.cs` 应保留。
- **游戏接入边界后置**：DLL / EXE、mod 接入、游戏事件等桥接细节不在当前减法阶段展开，另设后续阶段讨论。
- **状态分类必须明确**：每个候选项只能处于 `Retire Now`、`Retain`、`Defer` 三类之一；不得用“先排除但源码长期保留”作为完成态。

## Non-Goals
- 不删除 MCP 子系统框架，但允许删除其中不适合游戏的内容。
- 不删除 skills / skill invoke 能力框架，但允许删除其中不适合游戏的内容。
- 不删除 `AgentTool` / subagent 编排主链路。
- 不删除未来 NPC 自主生活所需的底座概念：长期记忆、人格、反思/梦境式后台思考、日程/意图队列、多 agent 协作。
- 不删除 `Dreamer`、`TodoWriteTool`、`ScheduleCronTool`、`TaskManager` / scheduler；它们是 NPC 自主生活、未来行动计划和日程系统的候选底座。
- 不删除 `MemoryTool`、`SessionSearchTool`、`MemoryReviewService`、`SoulService`、`AgentProfile`、`AskUserTool`；它们分别对应记忆写入、回忆、记忆整理、人格、NPC 身份和玩家交互。
- 不在第一轮重设计 Desktop UI 宿主；但不把 Desktop 当作保留废弃能力的备用轨。
- 不把被删除能力的 UI 入口保留为禁用按钮、隐藏配置项或“以后可能恢复”的占位页。
- 不删除媒体与语音能力，因后续要集成进游戏 mod。
- 不把项目重构成“只剩协议与模型”的空心内核。
- 不以“新增一套 game profile，但旧运行时继续暴露全量能力”作为最终方案。
- 不把“未注册 / 未打包 / 配置关闭”当作废弃完成；它只能是中间验收信号，最终必须清理源码、配置、skill 内容和引用链。

### NPC 目标下的误伤风险修正
从“一个 NPC 一个 agent，NPC 在游戏世界像人一样活着”的目标看，当前删除清单里必须保护的能力包括：
1. **Dreamer 绝对不能删除**：删除 Discord / gateway 投递适配是可以讨论的；但后台散步、反思、摘要、梦境式整理属于 NPC 自主生活底座，`DreamerService` 与相关 dreamer 源码列为 `Retain`。
2. **todo / cron 绝对不能删除**：桌面任务面板和工作流文案未来可以改造成 NPC 日程 UI；但 `TodoWriteTool`、`ScheduleCronTool`、调度器和任务管理底座是 NPC 生活模型需要的能力，列为 `Retain`。
3. **Memory / SessionSearch / MemoryReview 不能删除**：一个 NPC 要像人一样活着，必须能形成长期记忆、回忆过去互动、整理经验。
4. **Soul / AgentProfile 不能删除**：一个 NPC 一个 agent 意味着每个 NPC 要有人格、偏好、习惯和身份边界。
5. **AskUser / 玩家交互能力不能直接删除**：`AskUserTool` 当前未必注册，但它对应“NPC 向玩家询问、发起选择”的交互原语，应保留或后续改造成游戏对话桥。
6. **多 agent 协作能力不能直接删除**：`AgentTool`、`AgentService`、`CoordinatorService`、`MailboxService` 需要保留；它们支撑 NPC 之间的协作、分工和传话。
7. **媒体 / 语音 / 视觉能力不能删除**：已明确用于游戏 mod 集成，是 NPC 说话、听见、看见和生成内容的基础。

### 继续检查后的额外不可删除能力
结合当前 Desktop 注册表与源码清单，除了已确认的 MCP / skills / 媒体语音视觉 / AgentTool / Dreamer / todo / cron，还应保护：
1. **记忆链路**：`MemoryTool`、`MemoryManager`、`MemoryReviewService`、`HermesMemoryOrchestrator`、`SessionSearchTool`、`TranscriptRecallService`。理由：NPC 必须能记住、回忆和整理经历。
2. **人格链路**：`SoulService`、`SoulRegistry`、`SoulExtractor`、`AgentProfile`、`AgentProfileManager`。理由：一个 NPC 一个 agent 必须有稳定身份、人格、习惯和偏好。
3. **玩家交互原语**：`AskUserTool` / `IUserInteraction`。理由：NPC 将来需要主动问玩家问题、提供选择、等待玩家回应。
4. **多 NPC / 多 agent 协作候选**：`AgentTool`、`AgentService`、`CoordinatorService`、`MailboxService`。理由：多 NPC 之间可能需要协商、分工、传话或共同决策；即使当前实现有开发者工作流痕迹，也不能整体删除。
5. **Task / scheduler 支撑链路**：`TaskManager`、`ICronScheduler`、`InMemoryCronScheduler`、`TodoWriteTool`、`ScheduleCronTool`。理由：NPC 日程、目标、习惯和未来行动计划依赖这类能力。

暂不列为绝对保留、但也不建议首轮粗暴删除的是 `CheckpointTool`：当前实现是文件系统快照，更像开发安全工具；但“保存 / 回滚 NPC 状态”在游戏里可能有价值。建议列为 `Defer`，等 NPC 状态存档方案确定后再决定保留、改名或删除。

其他删除项暂不伤害 NPC 目标：聊天平台、浏览器/Web、shell/file/patch/LSP/OSV/sandbox、智能家居、生活办公外部集成，都属于桌面助手或外部平台语境，不是 NPC 在游戏世界中生活的必要底座。

### 事实修正：源码存在、当前注册、最终暴露必须分开判断
当前材料不能只按“文件存在”判断删除优先级。后续计划必须分清四层：
1. **Source Exists**：源码文件存在。
2. **Registered Now**：当前 Desktop / CLI 是否实际注册或启动。
3. **Game Runtime Exposed**：游戏运行时是否允许模型调用。
4. **Packaged**：最终 DLL / EXE 包是否包含。

删除优先级应优先处理“已注册或已启动、且明显非游戏”的能力；仅源码存在但未注册的能力，仍可列入退役，但优先级低于可达运行时入口。

### 后续阶段占位：游戏接入边界
当前阶段只做减法，不展开 DLL / EXE、mod 接入、游戏事件、权限、延迟等桥接设计。该主题在减法完成后另设阶段讨论。

## Recommended Deletion Order
1. **P0 Retire Now：聊天平台 / 外发消息链路**
   - 删除或退役 `gateway` 平台适配、`SendMessageTool`、相关后台启动链路、配置读取与 Dreamer 对 gateway / Discord digest 的外部投递适配；不得删除 Dreamer 本体。
2. **Retain：AgentTool / subagent 编排主链路**
   - `AgentTool`、AgentService、CoordinatorService、subagent 调度与相关编排入口绝对保留；后续删开发工具不得把它们列为候选。
3. **Retain：Dreamer、todo / cron、任务调度与 NPC 生活底座**
   - `DreamerService` / dreamer 源码、`TodoWriteTool`、`ScheduleCronTool`、TaskManager / scheduler、记忆回顾、人格 profile、玩家询问与多 agent 协作候选能力全部保留；允许后续改名、改 UI、改触发方式，不允许删除。
4. **P1 Retire Now：开发者 / 编程助手工具簇（不含 AgentTool / subagent 编排，不含 todo / cron）**
   - 删除或退役文件读写、shell、patch、LSP、OSV、sandbox 等非游戏执行器及其注册入口；若这些执行器被 subagent profile 引用，同步清理 profile 引用，不删除编排主链路。
5. **P1 Retire Now：浏览器 / Web 工具**
   - 删除或退役 `BrowserTool`、`WebFetchTool`、`WebSearchTool` 及相关配置暴露；若某项当前未注册，也要从源码和文档层面完成退役，而不是留下备用轨。
6. **P1 Retire Now：智能家居 / 外部设备工具**
   - 删除或退役 `HomeAssistantTool` 及相关环境变量/说明。
7. **P1 Retire Now：MCP / skills 内部的开发内容、生活办公内容**
   - 保留 MCP / skills 框架，删除不适合游戏运行时的 server 配置、skill 目录、说明、触发词和 bundled skill 暴露。
8. **Retain：MCP / skills 框架与游戏相关内容**
9. **Retain：媒体 / 语音 / 视觉能力（后续用于游戏 mod 集成）**
   - 注意：当前若这些工具未注册，不应误判为“已集成”；保留意味着保留源码与未来装配位，后续需要单独设计游戏侧暴露方式。
10. **Defer：Checkpoint / 存档回滚能力**
   - `CheckpointTool` 当前是文件系统快照，偏开发工具；但保存 / 回滚 NPC 状态可能是游戏运行时能力。先不纳入首轮删除，待 NPC 状态存档方案确定后再判断。
11. **Defer：Desktop UI 宿主重设计**
   - UI 形态不进入首轮；但被退役能力的 Desktop 注册、启动、后台链路和 UI 入口必须同步清理。
