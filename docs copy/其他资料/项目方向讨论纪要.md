# 项目方向讨论纪要

## 1. 这次讨论真正想解决的问题

当前最核心的矛盾不是“要不要服务器”，而是：

1. 想尽快把功能做成产品
2. 不想继续走重服务器、集中式编排的路线
3. 又不希望把产品核心能力完整裸露给玩家本机，导致太容易被抄走或二次分发

本次讨论后，已经接受一个现实前提：

- 绝对防破解做不到
- 真正可行的目标是提高破解成本，而不是追求无法破解

## 2. 为什么会考虑 Hermes

引入 `Hermes` 的核心动机不是追热点，而是希望：

1. 尽量少自研 agent orchestration
2. 尽量少自研 memory / tools / workflow 这一整套 runtime 能力
3. 能交给 Hermes 的尽量交给 Hermes
4. 以更快速度把产品先做出来

换句话说，`Hermes` 被看中的点是：

- 它有机会显著减少自研框架量
- 它更适合作为现成的本地 agent runtime 使用

## 3. 当前收敛出来的新方向

当前更倾向的新方向是：

`Hermes-first 的本地化 AI 游戏产品`

一句话解释：

- 智能能力主要在本地跑
- 云端只保留极薄控制面
- 不再依赖当前这种重服务器、集中式编排模式

## 4. 已经形成的总体架构倾向

当前讨论收敛后的架构倾向如下：

1. `Hermes` 跑在独立的本地 `Runtime` 中
2. `Rust` 只负责外壳和守护层
3. `Mod` 继续保持薄，只负责宿主桥接
4. 云端只保留极薄控制面

更具体地说：

### 4.1 Mod

职责：

1. 宿主取数
2. 宿主 UI / 交互入口
3. 最终宿主写回

原则：

- 不把 Hermes 直接塞进 Mod
- 不让 Mod 承担复杂安全逻辑
- 不让 Mod 承担复杂运行时逻辑

### 4.2 本地 Runtime

职责：

1. 跑 Hermes
2. 承担 agent / session / memory / tools / workflow
3. 承担角色运行时装配
4. 成为本地真正的智能中枢

原则：

- 这是主要运行层
- 这是减少服务器压力的关键落点

### 4.3 Rust 外壳

职责：

1. 拉起和守护 Hermes Runtime
2. 做版本控制
3. 做签名校验
4. 做配置缓存
5. 做 IPC
6. 做基础反篡改
7. 做远程片段获取与本地管理

原则：

- Rust 不是替代 Hermes 的智能层
- Rust 是产品外壳、守护层、加固层

### 4.4 极薄云端

职责：

1. 简单账号系统
2. 授权与权益判断
3. 远程能力开关
4. 远程配置分发
5. 少量高价值策略片段分发
6. 可选云同步

原则：

- 不再承接每个玩家的完整实时 agent 会话
- 不再成为主计算中心
- 只保留必要控制面和少量关键真相

## 5. 对“商业秘密”的新理解

本次讨论后，已经不再把“商业秘密”主要理解为：

1. 普通角色卡
2. 普通提示词文本
3. 玩家本来就能自定义的内容

因为这部分即使暴露，也不值得作为最高级保护对象。

当前更合理的理解是：

`真正该保护的不是单个文本文件，而是能让别人快速低成本复刻产品特色的规则层能力。`

## 6. 当前认定更值得保护的内容

优先值得保护的是：

1. `能力开关规则`
2. `tool 权限与可用性规则`
3. `workflow 路由规则`
4. `异常与降级策略`
5. `少量稀有模板片段`
6. `授权、版本、签名与远程控制真相`

不再优先死保的是：

1. 普通角色卡
2. 普通本地上下文拼装
3. 已准备对玩家开放自定义的内容

## 7. “少量高价值策略片段”目前的定义

所谓“少量高价值策略片段”，不是一整包完整秘密配置，而是：

`某个用户、某个版本、某个场景、某次运行所需的一小段高价值规则。`

这类片段当前主要包括：

1. `能力开关`
   - 某能力是否可用
2. `路由规则`
   - 当前场景该走哪条 workflow 分支
3. `权限规则`
   - 哪些 tool 当前可用、以什么权限可用
4. `模板片段`
   - 少量稀有玩法、活动、专属内容所需模板

这类片段的目标不是绝对防泄露，而是：

- 即使抓包，也尽量只暴露局部、临时、场景化片段
- 不让单次获取结果变成长期稳定、完整可复刻的“母本”

## 8. 对抓包与破解的认知已经达成的结论

目前已经接受以下判断：

1. 抓包不能彻底避免
2. 只要内容出现在玩家本机，就不可能绝对保密
3. Rust 外壳只能提高破解成本，不能提供绝对保护
4. 真正应该做的是把系统设计成“即使被抓包，也难以低成本长期复刻”

因此，当前认可的保护思路是：

1. 不让单次响应就是完整真相
2. 让远程片段具备版本性和时效性
3. 让本地缓存具备最小化、短期化、可失效特征
4. 让本地不同层各自只看到一部分信息，不轻易得到全貌

## 9. 当前已经明确不追求的东西

当前不再追求：

1. 绝对防破解
2. 继续维持重服务器、集中式 agent 编排
3. 为了保密把所有能力重新强拉回云端
4. 把普通角色卡当成最高价值机密

## 10. 当前已形成的总纲

一句话总纲：

`做一个 Hermes-first 的本地化 AI 游戏产品：本地跑智能，云端只保留轻控制面；目标是更快做出产品，同时通过 Rust 外壳、运行时分层、动态片段分发和轻云端授权，把破解和抄袭成本抬高。`

## 11. 后续继续讨论时的建议口径

后续继续讨论时，建议始终围绕下面这条主线，不再反复回到“要不要重服务器”的旧问题：

1. 本地 Runtime 如何承接 Hermes
2. Rust Shell 如何做守护、签名、缓存和加固
3. Mod 如何继续保持薄桥接
4. 云端最小控制面到底保留哪些能力
5. 高价值策略片段到底如何下发、缓存、失效和轮换

当前方向已经不是：

`是否继续维持当前服务器中心架构`

而是：

`如何把产品改造成以本地 Hermes Runtime 为核心、以轻云端控制面为补充的产品形态`
