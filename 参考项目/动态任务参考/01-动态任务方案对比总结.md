# 动态任务方案对比总结

这份文档不是仓库简介，而是从“你在 Godot 里要怎么做动态多步骤任务”这个角度做的实现对比。

---

## 一句话结论

如果你现在被“动态任务”卡住，最稳的做法不是先追求随机生成，而是先把任务拆成这几个层次:

- `QuestTemplate`
- `QuestRuntimeState`
- `Step`
- `Condition`
- `Effect`
- `BranchPolicy`

然后再决定任务是:

- 线性推进
- 分支推进
- 并行推进
- 世界状态驱动生成

这几个仓库基本都在做这件事，只是切入点不同。

---

## 1. `advanced-quest-system-example`

### 适合借鉴什么

这是最适合你直接参考的 Godot 样板。

关键点:

- 任务不是靠继承一堆 `KillQuest / FetchQuest / TalkQuest` 类型硬拆。
- 它引入了 `BaseQuestResource`，把任务定义成一个资源对象。
- 一个任务下面挂多个 `QuestStep`。
- step 按顺序完成，完成后推进到下一个。
- step 自己可序列化 / 反序列化，适合存档。

关键文件:

- `advanced-quest-system-example/quests/scripts/base_quest.gd`
- `advanced-quest-system-example/quests/scripts/quest_step.gd`
- `advanced-quest-system-example/quests/scripts/steps/gather_item.gd`
- `advanced-quest-system-example/quests/scripts/steps/deliver_item.gd`

### 它的强项

- 非常贴近 Godot 实战
- 结构简单
- 多步骤任务的最小实现清晰
- 对存档友好

### 它的短板

- 默认是线性 step 流
- 分支、并行、排他路线要你自己再扩
- “动态生成任务目标”这一层没有展开

### 你的用途

如果你现在连“任务如何拆 step”都还没稳定，这个应该先抄。

---

## 2. `quest-system`

### 适合借鉴什么

这是上面示例依赖的底层 Godot 任务框架。它提供的是资源化、模块化、可序列化的任务基础设施。

关键点:

- 任务是资源
- API 明确
- 支持自定义 quest
- 支持序列化 / 反序列化
- 单例管理器比较清晰

关键目录:

- `quest-system/addons`
- `quest-system/tests`

### 它的强项

- 基础设施稳
- 适合当底层框架
- 类型化和测试比很多 Godot 插件更规范

### 它的短板

- 它更像“任务框架底座”，不是“复杂动态任务解决方案”
- 你仍然要自己定义 step、分支、条件查询方式

### 你的用途

如果你打算自己做一套长期可维护的任务系统，可以学它的资源化和序列化边界，不一定要照搬 API。

---

## 3. `godot-questify`

### 适合借鉴什么

这个仓库最有价值的地方，是它把任务当成“图”而不是“数组”。

关键点:

- 有 `Start Node` 和 `End Node`
- objective 有 `active` / `completed` 两个状态
- 通过 `condition_query_requested` 把条件检查丢给外部系统
- 支持手动 `update_quests()`，不强绑轮询
- 支持 quest params，把运行时参数注入条件值
- 支持 `Any Previous` 和 `Exclusive Branch Connector`

关键文件:

- `godot-questify/addons/questify/quest_manager.gd`
- `godot-questify/addons/questify/scripts/model/quest_resource.gd`
- `godot-questify/addons/questify/scripts/model/quest_condition.gd`
- `godot-questify/addons/questify/scripts/utils/query_value_parser.gd`

### 它的强项

- 天然适合分支任务
- 能做并行目标
- 能做排他路线
- 条件系统和数据系统解耦
- 很适合接你自己的 `DataManager`

### 它的短板

- 图编辑器和运行时会比线性 step 系统更重
- 小项目如果只是做简单支线，可能有点过度设计
- 默认条件更新机制要考虑性能和时机

### 你的用途

如果你担心任务后面会越来越复杂，有“同一任务里多个可选路径”的需求，优先参考这个。

---

## 4. `scripted-quests`

### 适合借鉴什么

这个不是 Godot，但设计思想很强。

核心抽象:

- 一个 quest 文件包含多个 `quest_components`
- 每个 component 有:
  - `prerequisites`
  - `actions`
- 玩家每次交互时，组件按顺序重新评估
- 还支持 `rerun_components`

关键位置:

- `scripted-quests/README.md`
- `scripted-quests/tools/schema/prerequisites.json`
- `scripted-quests/tools/schema/action.json`
- `scripted-quests/plugin/src/main/java/com/playmonumenta/scriptedquests/quests/components`

### 它的强项

- 非常数据驱动
- 条件和动作都是组件
- 容易扩展
- 对“动态任务”很友好，因为不是写死状态流，而是持续重评估

### 它的短板

- 如果没有好的调试工具，配置会越来越难维护
- 纯重评估模型对复杂剧情流可能不如显式状态机直观
- 强依赖作者对前置条件设计的纪律

### 你的用途

如果你想做“NPC 对话 + 条件判断 + 动态推进”的任务系统，这个思路很值得借。

你完全可以在 Godot 里做一个简化版:

- `Condition[]`
- `Action[]`
- 每次事件发生时重新评估当前激活节点

## 5. `QuestFramework`

### 适合借鉴什么

这是 Stardew Valley MOD 框架，更偏“任务生命周期管理”。

关键点:

- 有 `Trigger`
- 有 `NextQuests`
- 有 `QuestOffer`
- 有 `IQuestObserver`
- 有自定义 quest type
- 有不同来源的 offer，例如 NPC、Mail、Board

关键位置:

- `QuestFramework/docs/content-pack-guide.md`
- `QuestFramework/docs/advanced-api-guide.md`
- `QuestFramework/src/Patches/QuestPatch.cs`
- `QuestFramework/src/Offers/QuestOffer.cs`

### 它的强项

- 任务链清晰
- 任务触发来源抽象得好
- 任务投放系统和任务本体分离
- 观察器机制适合复杂完成条件

### 它的短板

- 多步骤流程本身不是它最强的部分
- 更偏“框架层”和“投放层”
- 不像图任务那样天然支持复杂分支图

### 你的用途

它最值得你学的不是任务目标本身，而是:

- 任务如何被投放
- 任务如何被触发
- 完成后如何挂后续任务链

如果你的游戏以后会有:

- NPC 发放任务
- 物品触发任务
- 公告板任务
- 邮件任务

那这个设计很有参考价值。

---

## 6. `Cataclysm-DDA`

### 适合借鉴什么

这是“开放世界动态任务”最值得看的参考。

关键点:

- `MGOAL_CONDITION`: 动态条件目标
- `ORIGIN_SECONDARY`: 任务作为后续任务出现
- `followup`: 任务链
- `assign_mission_target`: 动态查找或创建任务地点
- `update_mapgen`: 在任务目标地点注入任务专属内容

关键位置:

- `Cataclysm-DDA/doc/JSON/MISSIONS_JSON.md`
- `Cataclysm-DDA/doc/JSON/MAPGEN.md`
- `Cataclysm-DDA/src/mission_util.cpp`
- `Cataclysm-DDA/src/mission_start.cpp`

### 它的强项

- 真正把“世界生成”和“任务生成”连起来了
- 不只是任务描述动态，连目标地点和内容都动态
- 很适合开放世界委托、悬赏、探索任务

### 它的短板

- 复杂度很高
- 不适合你一上来就照搬
- 它解决的是“大世界任务生成”，不是 Godot 小中型项目的最小落地方案

### 你的用途

如果你想做这类任务:

- 去随机营地救人
- 去世界里某个符合条件的地点调查
- 任务目标地点每次不同
- 任务开始后在地图上注入专属敌人 / 道具 / 机关

就重点看它。

---

## 7. `StardewMods-QuestEssentials/QuestEssentials`

### 适合借鉴什么

这个项目正好补上了 `QuestFramework` 的一个缺口: 多阶段 / 多目标任务本体。

关键点:

- 明确写了 `multi-stage (multiobjective) quests support`
- 有 `AdventureQuest`
- 有 `Tasks` 数组
- 每个 task 可以定义 `RequiredTasks`
- 未满足前置 task 时，该 task 不显示、也不可完成
- 任务完成检查通过消息和 checker 分发到各类 task

关键位置:

- `StardewMods-QuestEssentials/QuestEssentials/README.md`
- `StardewMods-QuestEssentials/QuestEssentials/docs/adventure-quests.md`
- `StardewMods-QuestEssentials/QuestEssentials/Quests/AdventureQuest.cs`
- `StardewMods-QuestEssentials/QuestEssentials/Tasks`
- `StardewMods-QuestEssentials/QuestEssentials/Framework/QuestTaskConverter.cs`

### 它的强项

- 多阶段任务抽象很直接
- task 依赖激活机制很实用
- 任务类型和 task 类型都可扩
- 比纯 step 数组更灵活，但又没图系统那么重

### 它的短板

- 更偏任务内容编排，不是底层引擎级框架
- 复杂分支表达力仍然不如显式任务图

### 你的用途

如果你想要的是:

- 一个任务有多个动态 objective
- objective 有依赖关系
- 前面的目标完成后才显示后面的目标

那这个实现非常值得借。

## 8. `CustomNPCs_1.12.2-Unofficial`

### 适合借鉴什么

这个仓库最值得记住的是“多目标任务的完成策略”。

README 里明确提到:

- quest objectives 数量从 3 提升到 9
- 完成策略支持:
  - `all`
  - `in turn`
  - `at least one`

还有 quest compass 和更完整的 quest log。

### 它的强项

- 直接回答“多目标任务怎么定义”
- UI / 任务日志 / 任务指南针一体化考虑
- 对任务内容编辑器友好

### 它的短板

- 整个项目体量很大
- 任务系统不是它唯一重点
- 代码噪音明显多于专门的任务框架

### 你的用途

如果你在设计任务数据结构时卡住了，可以直接借它的目标完成策略枚举。

比如在 Godot 里先定义:

- `ALL`
- `IN_ORDER`
- `AT_LEAST_ONE`

这会立刻让你的多步骤任务模型清晰很多。

## 横向对比

### 如果你的重点是“Godot 最快落地”

优先级:

1. `advanced-quest-system-example`
2. `quest-system`
3. `godot-questify`

### 如果你的重点是“复杂分支和并行目标”

优先级:

1. `godot-questify`
2. `StardewMods-QuestEssentials/QuestEssentials`
3. `CustomNPCs_1.12.2-Unofficial`

### 如果你的重点是“世界状态驱动的动态任务生成”

优先级:

1. `Cataclysm-DDA`
2. `scripted-quests`
3. `QuestFramework`

### 如果你的重点是“任务发放、任务链、任务来源”

优先级:

1. `QuestFramework`
2. `Cataclysm-DDA`
3. `scripted-quests`

## 我给你的推荐架构

如果这是一个 Godot 游戏，我建议你不要直接抄某一个仓库，而是做一个混合方案:

### 第一层: 任务模板

保存静态定义:

- 任务 id
- 标题 / 描述
- step 列表
- 分支规则
- 发放条件
- 奖励规则

### 第二层: 任务运行时状态

保存存档态:

- 当前激活 steps
- 已完成 steps
- 分支锁定情况
- 运行时参数
- 任务目标绑定对象

### 第三层: step

每个 step 只关心自己:

- 描述
- 完成条件
- 完成效果
- 是否可选
- 是否隐藏

### 第四层: 条件系统

建议做成可组合:

- `CheckFlag`
- `CheckItem`
- `CheckLocation`
- `CheckNpcTalk`
- `CheckKillCount`
- `CheckWorldState`
- `And / Or / Not`

### 第五层: 效果系统

- `GiveReward`
- `SetFlag`
- `StartNextQuest`
- `BindTarget`
- `SpawnOrInjectContent`
- `ShowDialog`
- `RevealMapMarker`

## 对你最重要的实现建议

### 建议 1

先做“线性多步骤任务”，不要先做完全随机任务。

也就是先做到:

- 接任务
- 当前 step 激活
- step 完成后推进下一步
- 可存档

### 建议 2

条件检查不要写死在 quest 类里，要做成独立条件对象。

这是从 `godot-questify` 和 `scripted-quests` 抽出来最有价值的一点。

### 建议 3

任务目标绑定要和模板分离。

比如:

- 模板只说“找一个村庄中的铁匠”
- 运行时才绑定为 `npc_id = smith_17`

这是从 `Cataclysm-DDA` 的 `assign_mission_target` 思路抽出来的。

### 建议 4

多目标任务从第一天就定义完成策略。

至少支持:

- 全部完成
- 按顺序完成
- 任意一个完成

### 建议 5

任务发放来源单独抽象，不要和任务主体耦合。

比如:

- NPC 发放
- 地图触发
- 道具触发
- 公告板发放

这是 `QuestFramework` 最值得借的地方。

## 最后给你的落地路线

### 第一步

照着 `advanced-quest-system-example` 做你自己的:

- `QuestTemplate`
- `QuestStep`
- `QuestRuntime`

### 第二步

从 `CustomNPCs` 引入目标完成策略:

- `ALL`
- `IN_ORDER`
- `AT_LEAST_ONE`

### 第三步

从 `StardewMods-QuestEssentials/QuestEssentials` 引入 task 依赖激活:

- objective 依赖 objective
- 前置未完成时不显示
- 任务列表按激活态动态变化

### 第四步

从 `godot-questify` 引入分支节点思想:

- 并行目标
- 任一路线完成
- 排他路线

### 第五步

从 `QuestFramework` 引入任务投放层:

- NPC
- 物品
- 公告板

### 第六步

最后才引入 `Cataclysm-DDA` 式动态目标生成:

- 动态地点
- 动态 NPC
- 动态场景注入

---

## 最简判断

如果你现在只能先选一个参考项目开始看:

- 想立刻在 Godot 里做出来: `advanced-quest-system-example`
- 想从一开始支持分支任务: `godot-questify`
- 想以后做开放世界动态委托: `Cataclysm-DDA`
