# Stardew 菜单窗口尺寸与布局参考

更新时间：2026-03-31

目的：

- 给当前项目的 Stardew 菜单实现一个可执行的窗口/布局参考，不再靠“再调几个数”反复试错。
- 把官方资料、主源码参考、成熟 Mod、当前项目代码位置收在一处。
- 明确哪些是硬规则，哪些只是实现风格。

---

## 先给结论

当前这类窗口反复修不住，根因不是单个 `width/height` 数值，而是一直没有把菜单当成下面这份合同来实现：

1. 固定外壳：窗口外框、顶部 speaker anchor、输入区、footer 都是固定区域。
2. 独立正文区：长文本只能在正文区内部滚动，不能顶掉输入区和 footer。
3. 统一锚点：所有坐标都基于 `Game1.uiViewport`、宿主菜单 bounds 或 safe area 计算，不能混着用世界坐标、旧窗口坐标和拍脑袋的偏移。
4. resize 重建：窗口尺寸变化后，要么重建菜单，要么完整重算内部布局；不能只修一两个控件坐标。
5. 生命周期分离：`MenuChanged` 只给“菜单切换快照”，真正当前菜单状态要看 `Game1.activeClickableMenu`。

一句话：以后不要再把“正文、状态、动作、输入”揉成一块一起挤。

## 1. 官方资料与主源码参考

说明：

- `SMAPI`/modding wiki 能说明事件、输入、绘制层级、缩放语义。
- `IClickableMenu`、`DialogueBox`、`Utility.getSafeArea()` 这类像素级布局规则，公开主文档写得不细，需要补读 Stardew 源码镜像。

### 1.1 SMAPI / Wiki 直接可用的规则

1. `MenuChanged` 是菜单“打开 / 关闭 / 替换后”的事件，但它给的是事件快照，不保证等于当前实时状态。
   来源：
   - https://www.stardewvalleywiki.com/Modding:Modder_Guide/APIs/Events
   - https://github.com/Pathoschild/SMAPI/blob/develop/src/SMAPI/Events/MenuChangedEventArgs.cs
   - https://github.com/Pathoschild/SMAPI/blob/develop/src/SMAPI/Framework/SCore.cs
   启发：
   - 识别对话框切换可以用 `MenuChanged`。
   - 真正要判断“此刻屏幕上还在不在这个菜单”，要读 `Game1.activeClickableMenu`，不能只信 `e.NewMenu`。

2. `RenderedActiveMenu` / `RenderingActiveMenu` 是最稳的菜单叠层钩子。
   来源：
   - https://www.stardewvalleywiki.com/Modding:Modder_Guide/APIs/Events
   - https://github.com/Pathoschild/SMAPI/blob/develop/src/SMAPI/Events/IDisplayEvents.cs
   启发：
   - 要在原菜单上面加 overlay，就走这个层，不要随便抢占 `activeClickableMenu`。
   - 要在菜单下层加背景或遮罩，也有明确钩子，不需要硬 patch 绘制顺序。

3. `WindowResized` 是正式事件，菜单和 overlay 必须有 resize 策略。
   来源：
   - https://www.stardewvalleywiki.com/Modding:Modder_Guide/APIs/Events
   启发：
   - 任何依赖 `uiViewport` 的菜单，一旦窗口变化，内部布局都要重算或重建。

4. 输入 API 里的像素坐标默认不是 UI 坐标；UI 缩放要显式处理。
   来源：
   - https://www.stardewvalleywiki.com/Modding:Modder_Guide/APIs/Input
   启发：
   - 鼠标命中框、文本框点击、滚轮滚动，不要混用未缩放坐标和 UI mode 坐标。
   - 如果自己在 overlay 里算命中区域，要么统一用 UI mode，要么统一做缩放转换。

5. `Game1.activeClickableMenu` 就是当前模态主菜单入口。
   来源：
   - https://wiki.stardewvalley.net/Modding:Modder_Guide/Game_Fundamentals
   - https://wiki.stardewvalley.net/Modding:Common_tasks
   启发：
   - 真正“打开一个菜单”的标准做法，仍然是赋值给 `Game1.activeClickableMenu`。
   - 如果只是加一层额外操作面，不一定要替换它。

### 1.2 Stardew 主源码参考给出的布局规则

1. `IClickableMenu` 的布局核心就是：
   - `xPositionOnScreen`
   - `yPositionOnScreen`
   - `width`
   - `height`
   来源：
   - https://github.com/WeDias/StardewValley/blob/main/Menus/IClickableMenu.cs
   启发：
   - 自定义菜单所有内部布局都应该从这四个量推导，不要到处散落自己的“屏幕中心 + 常量”公式。

2. `gameWindowSizeChanged(...)` 是菜单 resize 的标准承载点。
   来源：
   - https://github.com/WeDias/StardewValley/blob/main/Menus/IClickableMenu.cs
   启发：
   - 复杂自定义菜单建议直接 override。
   - 如果不 override，也至少要通过外层逻辑重建菜单。

3. `DialogueBox` 是“底部锚定对话框”，不是通用弹窗模板。
   来源：
   - https://github.com/WeDias/StardewValley/blob/main/Menus/DialogueBox.cs
   启发：
   - 要做原版风格私聊面，应该借它的视觉语言，不应该照搬它的尺寸逻辑。
   - 因为 `DialogueBox` 的设计目标是底部短文本对话，不是中等高度、带输入框、带内部滚动的聊天窗。

4. `Game1.uiViewport` 是 UI 布局真值，不要和 `Game1.viewport` 混用。
   来源：
   - https://github.com/WeDias/StardewValley/blob/main/Game1.cs
   启发：
   - 居中、safe area、菜单 bounds 一律按 `uiViewport` 走。

5. `Utility.getSafeArea()` 是真实存在的边界工具，不是概念建议。
   来源：
   - https://github.com/WeDias/StardewValley/blob/main/Utility.cs
   启发：
   - 靠边 tooltip、hover 卡片、右下角预览框都应该最后过一次 safe-area 裁切。

### ManagedQuestLog：固定窗口 + 可裁剪滚动正文

关键位置：

- `ManagedQuestLog` 类：`ManagedQuestLog.cs:19`
- `DrawWindow(...)`：`ManagedQuestLog.cs:157`
- `DrawManagedQuestDetails(...)`：`ManagedQuestLog.cs:179`
- `Utility.ConstrainScissorRectToScreen(...)`：`ManagedQuestLog.cs:228`

值得抄的点：

1. 不靠拉高整个窗口容纳长文本，而是给正文区单独开 `ScissorRectangle`。
2. 先测文本高度，再累计 `yPos` 排后续 objectives，最后反算 `_contentHeight`。
3. 文本区和奖励区是两套互斥布局，不混排。
4. `ConstrainScissorRectToScreen` 说明滚动裁剪区最后还要过屏幕约束。

结论：

- 这是最接近“固定 shell + 内部滚动内容区”的成熟做法。
- 你们的 AI 私聊正文区、群聊历史区、NPC 信息面板 tab 内容区，都应该先学这个，再谈视觉。

### CustomOrderBoard：固定大板 + 居中 + resize 直接重建

关键位置：

- `CustomOrderBoard` 类：`CustomOrderBoard.cs:11`
- 居中：`Utility.getTopLeftPositionForCenteringOnScreen(...)`：`CustomOrderBoard.cs:85`
- `gameWindowSizeChanged(...)`：`CustomOrderBoard.cs:148`
- `DrawQuestDetails(...)`：`CustomOrderBoard.cs:275`

值得抄的点：

1. 整窗尺寸先固定，再做双栏分区，不在渲染时临时拼凑。
2. CTA 是按栏中心定位，不跟正文末尾乱跑。
3. 文本太长时，优先压缩内容显示比例，不让布局骨架塌掉。
4. 窗口变化后直接 `new CustomOrderBoard(...)` 重建菜单，不做半残修正。

结论：

- 如果你们的窗口是“固定形式的剧情板 / 群聊板 / 信息板”，就该学这种方式。
- resize 以后重建整个菜单实例，比补偏移安全得多。

2. 文本框宽度和容器宽度有明确边距，不跟文字长度绑死。

结论：

- 你们当前 `AiDialogueClickableMenu` 没有 resize 重排逻辑，这就是风险点。

## 4. 以后写 Stardew 窗口的硬规则

下面这些不是建议，是应该固化成代码约束的规则。

### 4.1 菜单骨架规则

1. 先定 shell，再放内容。
   shell 固定包括：
   - portrait/speaker 区
   - 正文区
   - 输入区
   - footer

2. 正文区之外的区域不能被正文挤压。
   特别是：
   - 输入框高度固定
   - footer 永远可见

3. 所有内部坐标都从菜单 bounds 推导。
   不允许：
   - 某些控件用 `xPositionOnScreen + 常量`
   - 某些控件直接用 `Game1.uiViewport` 重新算
   - 某些控件再额外算自己的屏幕中心

### 4.2 resize 规则

1. 自定义菜单必须有 resize 策略：
   - 要么 override `gameWindowSizeChanged(...)` 重建菜单。
   - 要么由 presenter/overlay 在 viewport 变化时重建菜单实例。

2. viewport 变化后必须重算：
   - 外框位置
   - 正文区 bounds
   - 输入区 bounds
   - footer 位置
   - 滚动区可见行数或滚动容器可见高

3. 不允许只补一个控件坐标。
   这正是过去十几轮反复坏掉的来源。

### 4.3 长文本规则

1. 长文本只能滚正文区，不滚整窗。
2. 正文区要么：
   - 用 `ScissorRectangle`/裁剪区
   - 要么用独立内容容器
3. 换行宽度必须固定，不许随文本长度改变窗口宽度。

### 4.4 safe area 和 hover 规则

1. 靠边 hover 卡片、tooltip、右下角预览框，最后必须经过 safe-area 修正。
2. 不要假定“鼠标右下角一定放得下”。
3. hover 信息和正文信息不要共用同一区域。

### 4.5 生命周期规则

1. `MenuChanged` 只表示切换快照，不表示当前菜单已经稳定。
2. 与 `DialogueBox` 交接时，必须额外确认：
   - 当前 `activeClickableMenu` 是否还是目标菜单
   - transition 是否结束
   - `currentSpeaker` 是否对齐

---

## 5. 给当前项目的直接建议

如果只为当前 `AiDialogueClickableMenu` 落地，最小建议是：

1. 把这几个常量抽成统一的 shell token，不允许在别处复制一套：
   - `HorizontalInset`
   - `TopInset`
   - `PortraitSize`
   - `PortraitGap`
   - `NameGap`
   - `BodyBottomGap`
   - `FooterInset`

2. 给 `AiDialogueClickableMenu` 增加 resize 重建逻辑。
   参考：
   - `CustomOrderBoard.cs:148`
   - `ModConfigMenu.cs:314`
   - `BaseOverlay.cs:177-200`

3. 把正文区从“按行滚动”进一步升级成“正文容器滚动”或裁剪区滚动。
   参考：
   - `ManagedQuestLog.cs:228`
   - `GuidebookMenu.cs:89`
   - `GuidebookMenu.cs:200`

4. hover / portrait fallback / 将来的动作结果卡片，都要过 safe area。
   参考：
   - `GuidebookMenu.cs:626-643`
   - `Utility.getSafeArea()`

5. 对话切换期不要只靠 `MenuChanged` 抢菜单。
   参考：
   - `GiftDialogueHelper.cs:50-62`
   - `OpenDialogueBoxHandler.cs:19`

## 6. 不要再走的修法

1. 看到文本溢出，就直接增大整个窗口高度。
2. 输入框挡住正文，就单独把输入框往下挪。
3. footer 被挤掉，就继续调 `FooterInset` 常量。
4. 看到窗口居中不对，就直接改构造函数里的中心偏移。
5. 不处理 resize，只在当前分辨率下截图看着像对。
6. 在 `MenuChanged` 当帧里直接判定“原对话结束”。

这些修法共同的问题是：

- 只修表象，不修布局合同。
