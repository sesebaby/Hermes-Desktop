# GGBH OpenAIWorld 统一分析文档设计

## 1. 文档定位

本文定义 `GGBH_OpenAIWorld_20260326` 恢复资料的新一代统一分析文档体系。

它回答的是：

- 当前 `分析文档` 目录为什么会显得过多、重复、分散
- 新体系应该如何把“整体总览”和“分功能深挖”合并成一套稳定结构
- 哪些功能必须纳入首版，避免后续遗漏
- 每份文档应详细到什么程度，才能支撑后续复刻、迁移和源码回溯
- 旧文档应该如何在整理过程中暂存，以及何时删除

本文不是：

- `OpenAIWorld` 功能实现文档本体
- `AllGameInAI` 当前实现计划
- 对参考 mod 逐文件逐方法的最终分析正文

本文是后续整理 `recovered_mod/GGBH_OpenAIWorld_20260326/分析文档` 的设计真相源。

## 2. 背景与问题

当前目录 [`recovered_mod/GGBH_OpenAIWorld_20260326/分析文档`](/D:/Projects/AllGameInAI/recovered_mod/GGBH_OpenAIWorld_20260326/%E5%88%86%E6%9E%90%E6%96%87%E6%A1%A3) 已经积累出多篇分析稿。

现状问题不是“材料不足”，而是“拆分维度混杂”：

- 有些文档按功能链路拆
  - 如私聊、群聊、信息传播、主动世界
- 有些文档按底层能力拆
  - 如长期记忆、Prompt 存储、总架构
- 有些文档按迁移目的拆
  - 如跨游戏方案、MVP 模板、移植检查表

这导致 3 个后果：

1. 信息重复严重
2. 真正的功能全集不够稳定
3. 读者很难快速判断“某个玩法功能依赖哪些共用系统”

此外，已解密的 `31` 份 prompt 和版本更新日志还表明，现有文档对部分场景功能覆盖不足。

因此需要一套新的统一体系，满足以下前提：

- 一份主文档可以看到全貌
- 每个功能有单独落点
- 共用系统和上层玩法分层清楚
- 文档要足够细，能够回溯到源码文件、类、方法、数据结构和 prompt

## 3. 设计目标

### 3.1 主目标

新体系 MUST：

- 以“一份总览 + 多份分功能文档”的方式替代当前平铺式分析稿
- 采用“两层结构”组织
  - 先写底层共用系统
  - 再写上层具体玩法功能
- 首版纳入 `OpenAIWorld` 已确认的全部核心系统和场景功能
- 每篇文档尽量落实到源码路径、关键类、关键方法、关键数据结构、关键 prompt
- 同时服务于两个目标
  - 完整盘点参考 mod
  - 为后续迁移到 `AllGameInAI` 保留分析接口

### 3.2 次目标

新体系 SHOULD：

- 让后续读者不需要先读旧稿，也能建立完整心智模型
- 尽量减少底层机制在多篇玩法文档里的重复展开
- 把“直接证据”和“分析判断”区分开，降低误判风险
- 保持目录整洁，避免旧稿和新稿长期混放

## 4. 非目标与边界

### 4.1 非目标

本设计不负责：

- 立刻重写 `OpenAIWorld` 的全部分析正文
- 立刻删掉当前旧文档
- 直接进入代码实现或复刻
- 为每个源码文件都写一篇独立说明

### 4.2 删除边界

旧文档不会在设计阶段直接删除。

执行规则固定为：

- 先完成新体系文档
- 由用户确认新体系完整且可用
- 再删除旧文档

整理过程中允许把旧文档临时移动到归档目录，以保持目录整洁，但这不是长期保留承诺。

## 5. 总体信息架构

新体系固定采用以下目录结构：

- `分析文档/00_总览/`
- `分析文档/10_共用系统/`
- `分析文档/20_玩法功能/`
- `分析文档/90_附录/`
- `分析文档/archive_legacy_2026-04-03/`

各目录职责如下。

### 5.1 `00_总览`

总览区至少保留一份主入口和一份迁移提炼文档，负责：

- 一句话说明 `OpenAIWorld` 是什么
- 列出全功能矩阵
- 给出共用系统与玩法功能的依赖关系
- 给出推荐阅读顺序
- 给出后续复刻或迁移时最值得优先关注的能力
- 给出跨宿主迁移时的共享内核、宿主耦合点和优先复刻顺序

### 5.2 `10_共用系统`

只写被多个上层玩法共同依赖的能力层，不写单条玩法长流程。

这里的内容包括但不限于：

- 整体架构与主链路
- Prompt 与角色卡体系
- AI 请求层、服务配置与能力探测
- AI 路由、会话能力与多通道编排
- 消息、通信通道与消息持久化
- 关系、记忆、摘要机制
- 行为协议、解析、执行
- 世界事件数据模型与落地机制
- 宿主挂点、触发面与回写边界
- 容错、修复、持久化与版本演进

### 5.3 `20_玩法功能`

只写玩家可见、可感知、可触发的功能。

每篇文档重点说明：

- 这个功能对玩家而言是什么
- 它如何触发
- 它依赖哪些共用系统
- 模型输出如何最终落到游戏

### 5.4 `90_附录`

放高密度索引型内容，不承担长篇解释。

包括但不限于：

- 恢复来源、证据等级与边界
- 已解密 prompt 总表
- 源码锚点索引
- 版本更新功能演进表

### 5.5 `archive_legacy_2026-04-03`

该目录只用于整理过程中的临时承接。

用途是：

- 保持主目录整洁
- 让新旧文档不直接混在同一层
- 为整理过程提供回看缓冲

最终是否保留该目录，不由本文保证；默认在新体系获用户确认后删除旧稿。

## 7. 覆盖范围要求

首版统一分析文档必须覆盖两大类内容。

### 7.1 底层共用系统

至少包括：

- Prompt 资产解析、覆盖规则与角色卡体系
- AI 服务器、智能体、请求协议、格式支持与能力探测
- AI 路由、会话能力、多通道编排与 provider 选择
- 消息基类与私聊、群聊、联系人群消息模型
- 原始消息记录、JSON sidecar、镜像写回与消息持久化
- 关系、摘要、长期记忆、经历存储
- 行为协议、动作解析、动作执行、传播再入与容错修复
- 世界事件模型、地图事件模型、对象落地与回写
- 宿主挂点、UI/事件触发面与 authoritative writeback 边界

### 7.2 上层玩法功能

至少包括：

- 私聊
- 群聊
- 联系人群与固定群聊
- 传音与远程通信
- 信息裂变
- 主动对话与接触触发
- 主动世界演化
- 交易与给物
- 道具使用
- 身体交互
- 双修场景
- 竞技场相关 AI 功能
- 战败 AI 对话与战败后处理
- 自定义物品生成
- 自定义状态或气运生成
- 角色卡驱动的 NPC 创建
- 解签大师等特殊功能

不得因为“像外围玩法”而省略场景能力。

设计原则固定为：

- 首版先全量盘清楚
- 后续如需压缩阅读入口，再通过总览文档控制阅读顺序
- 不靠遗漏来换取结构整洁

以下内容即使不单列为独立玩法文档，也必须在对应文档中显式出现为子场景：

- 动态可选项对话
- 互留传音符 / 发送传音符
- 邀请洞府做客等邀约类场景
- 主动世界中的地标生成、可摧毁地标与自定义地标
- 加入宗门、退出宗门、成为家仆等强宿主动作场景

### 8.2 玩法功能文档模板

- `功能定义`

### 8.3 总览文档模板

主入口文档固定至少包含：

- `推荐阅读顺序`
- `迁移边界与优先复刻顺序`
- `最值得优先复刻的能力`

### 8.4 附录文档模板

附录以索引和证据为主，风格允许更紧凑，但仍需保证：

## 9. 详细度与证据要求

这是本设计的硬要求。

统一分析文档不得只停留在“功能名 + 大白话说明”层面。

### 9.1 最低证据粒度

每篇文档都 SHOULD 尽量落实到以下层级：

1. 源码文件路径
2. 关键类名
3. 关键方法或调用点
4. 关键数据结构
5. 对应 prompt 文件名
6. 相关配置项或版本演进来源

### 9.2 源码引用要求

优先引用以下类型的锚点：

- `decompiled/GGBH_OpenAIWorld_src/OpenAIWorld/*.cs`
- `decompiled/GGBH_OpenAIWorld_src/OpenAIWorld.mod/*.cs`
- `decompiled/GGBH_OpenAIWorld_src/OpenAIWorld.mod.data/*.cs`
- `decompiled/GGBH_OpenAIWorld_src/obfuscated_ns/*.cs`

对混淆文件的分析要求：

- 优先给出文件路径
- 能定位到类或方法时必须补充
- 若只能先给文件级锚点，也要标注“当前只确认到文件级”

### 9.3 Prompt 引用要求

涉及 AI 生成规则的文档，必须尽量关联到具体 prompt 文件，例如：

- `世界定义.md`
- `世界推演.md`
- `记忆压缩.md`
- `行为指令.md`
- `群聊.md`
- `传音群聊.md`
- `信息裂变.md`

### 9.4 事实与判断分离

每篇文档都必须区分：

- `已确认事实`
  - 直接来自源码、数据模型、prompt 明文、更新日志
- `分析判断`
  - 由多个证据交叉得出的解释

不能把推断直接写成既定事实。

### 9.5 证据等级要求

新体系统一采用显式证据等级。

最低要求如下：

- `A`
  - 可直接由源码、数据结构、prompt 明文、更新日志或恢复产物确认
- `B`
  - 由多处直接证据交叉支持，但仍存在少量解释空间
- `C`
  - 目前主要是分析推断，尚未拿到足够直接证据

若某篇文档不给证据等级，则视为不合格。

### 9.6 索引类附录的固定列

索引类附录至少应满足以下固定列约束：

- `已解密Prompt总表`
  - `Prompt名`
  - `解密文件名`
  - `原始Zip路径`
  - `功能归属`
  - `主要用途`
  - `证据等级`
- `源码锚点索引`
  - `功能域`
  - `文件路径`
  - `类/方法`
  - `作用说明`
  - `证据等级`
- `版本更新功能演进表`
  - `版本号`
  - `功能变化`
  - `功能归属`
  - `是否已在新体系落位`
  - `证据等级`

## 10. 执行顺序

新体系整理按以下顺序推进。

### 10.1 第一步：先固定恢复来源与证据边界

优先产出：

- `90_附录/00_恢复来源_证据等级与边界.md`

目的：

- 先固定当前分析所依据的真实 DLL、反编译目录、prompt 来源和更新日志来源
- 先明确哪些结论是直接证据，哪些只是恢复推断
- 避免总览文档建立在未固定的证据边界上

### 10.2 第二步：先补源码锚点与宿主挂点索引

优先补：

- `90_附录/02_源码锚点索引.md`
- `10_共用系统/09_宿主挂点_触发面与回写边界.md`

目的：

- 先把宿主接入面、Harmony 补丁面、UI 触发面和回写边界固定下来
- 避免后续总览和玩法分析失去统一锚点

### 10.3 第三步：归整旧稿

把当前旧文档整体移入：

- `分析文档/archive_legacy_2026-04-03/`

目的仅为：

- 清空新体系目录
- 保持当前目录整洁
- 让新旧结构不互相污染

### 10.4 第四步：先写总览与迁移提炼

优先产出：

- `00_总览/GGBH_OpenAIWorld_总览与功能矩阵.md`
- `00_总览/GGBH_OpenAIWorld_迁移复刻总提炼.md`

因为它负责先解决两个问题：

- 功能是否漏项
- 读者是否能快速导航

### 10.5 第五步：再补两份附录底座

优先补：

- `90_附录/01_已解密Prompt总表.md`
- `90_附录/03_版本更新功能演进表.md`

原因：

- prompt 总表从规则资产角度兜底
- 版本演进表从发布时间线角度兜底

这两份先做，最能防漏。

### 10.6 第六步：写其余共用系统

原因是玩法功能都会回链这些底层能力。

先写底层，可以减少重复说明。

### 10.7 第七步：写玩法功能

每篇玩法功能文档只讲自己的主链与玩家侧表现，并回链对应底层系统。

### 10.8 第八步：用户确认后删除旧稿

规则固定为：

- 新体系全部完成
- 用户确认完整可用
- 再删除旧文档

## 11. 完成标准

当以下条件都成立时，本设计视为被正确执行：

1. `分析文档` 新体系目录已建立
2. 恢复来源、证据边界、源码锚点和宿主挂点已先固定
3. 总览文档已能列出完整功能矩阵
4. 共用系统和玩法功能两层边界清晰
5. 首版纳入全部已确认功能，不遗漏场景玩法
6. 每篇文档都能尽量回溯到源码路径、类、方法、数据结构和 prompt
7. 每篇文档都明确标注证据等级与证据来源
8. 旧文档仍在，直到用户确认后才删除

## 12. 直接结论

`OpenAIWorld` 当前最需要的不是再新增若干平行分析稿，而是把现有成果重构成一套：

- 有唯一总览入口
- 有稳定分层
- 有完整功能覆盖
- 有源码级证据支撑
- 能直接服务后续迁移和复刻

的统一分析文档体系。

因此，本设计固定采用：

- `两层结构`
  - `共用系统`
  - `玩法功能`
- `全功能纳入`
- `源码可追溯级详细度`
- `先重构新体系，后删除旧稿`

这是后续整理 `GGBH_OpenAIWorld` 分析文档的唯一推荐路径。
