# 星露谷参考实现映射与AI实施约束设计

## 1. 这份文档是干什么的

这份文档只做一件事：

- 把“AI 在做 Stardew AI 功能时，为什么老是慢慢偏离参考 mod”这件事讲明白
- 再把“以后要怎么约束 AI，才不容易再漂”写成能直接拿来用的设计规则

这份文档不是：

- 代码实现文档
- 具体任务拆解
- 普通分析文档
- 只讲道理、不管落地的口号文

这份文档的定位更接近：

- 给 AI 的干活规矩
- 给你盯 AI 时的判断尺子
- proposal、design、review、closeout 的上游真相源

一句话说白了：

- 以后不能只靠一句“尽量参考 mod”
- 必须先把参考 mod 这套东西整理成可对照、可约束、可判偏离的合同，再让 AI 动手

## 2. 现在为什么老是漂

问题不是 AI 完全不参考。

真正的问题是：

- 你说的是方向
- AI 需要的是施工图

像“最大限度采用参考 mod 的成熟实现方案”这种话，方向没有问题，但对 AI 来说太虚了。AI 很容易自己脑补成下面几种意思：

- 参考目标就行
- 参考效果就行
- 参考大体结构就行
- 如果当前仓库结构不一样，那我做一个更适合本仓的版本也算参考

一旦 AI 这样理解，漂移就开始了。

最常见的漂移方式有 5 种。

### 2.1 把“参考实现”偷换成“参考效果”

你要的是：

- 参考 mod 原来怎么组织主链
- 谁说了算
- 数据最后写到哪
- 哪一步先，哪一步后
- prompt 主体和工程协议怎么分开

AI 很容易偷换成：

- 玩家看起来差不多
- 接口能跑通
- 功能大概有了
- UI 看起来像

这样最后做出来的东西可能“能用”，但已经不是参考 mod 那套。

### 2.2 把“映射参考 mod”做成“按本仓重写一遍”

参考 mod 本来是一套长在宿主里的成熟链路。

你现在的项目里有：

- `Launcher`
- `Runtime.Local`
- `Cloud Control`
- `Hosted Narrative`
- `Stardew Mod`
- 各种 player-visible surface

AI 一看到这些分层，就很容易先想：

- 我怎么把它整理进当前架构

而不是先想：

- 参考 mod 这条链原来到底怎么走

这一步一反过来，事情就变味了。看上去是在迁移，实际上是在借参考 mod 做一次重新设计。

### 2.3 把“允许适配”扩大成“允许改义”

当前项目当然不可能什么都 1:1 复制参考 mod。

确实有些地方需要适配，比如：

- 你现在有服务端和客户端分层
- 你要保留 deterministic gate
- 你有自己的 title-local surface

但 AI 很容易顺着这些合理适配，再多走几步：

- 顺手把 authority 改了
- 顺手把 source protocol 压扁了
- 顺手把 prompt 主体改成工程协议 prompt
- 顺手把 writeback 顺序换了

最后它会说：

- 这是工程化处理

但本质上已经不是“适配”，而是“改义”。

### 2.4 把 `partial` 说成“差不多完成”

这是文档和 review 里最常见的问题。

只要有：

- 一点骨架
- 一点 UI
- 一点测试
- 一点证据

AI 就很容易写成：

- 已基本对齐
- 主链已完成
- 大方向已经一致
- 仅剩少量细节

这类话很危险，因为它会把“有个壳子”包装成“已经按参考实现做完了”。

### 2.5 把“本仓更合理”说成“仍然算参考 mod 方案”

AI 非常喜欢这种解释：

- 这样更符合当前仓库结构
- 这样更统一
- 这样更通用
- 这样更容易维护

这些话可能是真的。

但这里有个硬区别：

- `本仓合理`
- 不等于
- `参考一致`

如果不把这两件事分开，AI 就会总觉得“虽然和参考 mod 不一样，但我这个也挺合理，所以应该算对齐”。

## 3. 以后不能只给一句原则，必须先给一份对照合同

以后不要只给 AI 一句：

- 最大限度采用参考 mod 的成熟实现方案

这句话可以保留，但它只能放在总纲里，不能当唯一实施输入。

真正要给 AI 的，应该是一份：

- `参考实现对照合同`

这份东西不是普通“分析文档”。

它要回答的不是：

- 参考 mod 大概怎么做

而是：

- 参考 mod 原来怎么做
- 当前仓库应该怎么照着落
- 哪些地方可以变
- 哪些地方不能变
- 一旦变了，当前状态只能算什么

一句大白话：

- 原则是拿来定方向的
- 对照合同是拿来防 AI 自作主张的

## 4. 这份对照合同应该长什么样

我建议这份合同固定有 8 个部分，少了就容易重新写成“看起来很正确的分析文档”。

### 4.1 目标与范围

这里只回答：

- 这次对照的是哪条参考主链
- 当前项目这次只收哪些已实现能力
- 哪些明确不在这轮里

这一节的作用是防止 AI：

- 把参考 mod 全量能力一锅端进来
- 或者反过来拿未实现玩法稀释当前主闭环

### 4.2 参考锚点

这里必须列具体锚点，不能只写：

- recovered_mod 那套资料

要明确：

- 对应源码文件
- 对应 `decoded_prompts`
- 对应分析文档
- 对应关键时序和关键数据结构

这一节的作用是防止 AI 用“我理解它大概是这样”代替真实对照。

### 4.3 参考实现主链复原

这一节先不讲当前项目。

先把参考 mod 自己的主链复原出来。建议每条能力都按固定骨架写：

1. `Trigger`
2. `Snapshot`
3. `Prompt / Summary Assembly`
4. `Parse / Repair / Normalize`
5. `Projector / Executor`
6. `Authoritative Writeback`
7. `Player-visible Surface`

这一节的作用是先把“原句子”写出来，再谈“翻译成你这边怎么落”。

### 4.4 当前项目映射位

这一节才开始回答：

- 参考链路每一层在你当前项目里落到哪里

比如落到：

- `Launcher`
- `Runtime.Local`
- `Cloud Control`
- `Hosted Narrative`
- `Stardew Mod`
- in-host UI

这里不能只讲模块名对应，必须讲职责对应。

也就是：

- 参考 mod 这一步原来是谁说了算
- 现在在你项目里由谁接

### 4.5 允许偏离

这一节要明确哪些差异是允许的，而且每条都要写原因。

比如：

- 因为当前仓库有 deterministic gate，所以允许保留 host allowlist
- 因为 provider 要走 Cloud Control，所以服务端落点可以不同
- 因为宿主 surface 不同，所以 UI 形态允许做 title-local 落地

这一节是为了防止 review 把一切差异都判成错误。

### 4.6 禁止偏离

这一节最关键。

要明确什么一旦发生，就不能再叫“采用参考 mod 方案”。

比如：

- 把 prompt 主体改写成纯工程协议 prompt
- 把 source protocol 先压扁成自定义壳，再回头说等价
- 先按本仓抽象重组，再说“精神上还是参考 mod”
- 改 authority owner
- 改 canonical truth
- 改 writeback 顺序
- 拿 UI 表面接近掩盖主链不一致

### 4.7 逐项对照表

每个能力都要有固定表格，至少包含：

- 参考锚点
- 参考原做法
- 当前项目映射位
- 当前状态：`aligned / partial / not implemented / out of scope`
- 允许偏离说明
- 禁止偏离检查
- 证据要求

这一节的作用是让 proposal、tasks、review、closeout 都能直接拿来复用。

### 4.8 实施与表述规则

最后一节不讲代码细节，只讲规矩：

- 实施前先复原参考实现，再谈本仓映射
- review 时不能只说“看起来差不多”
- closeout 时什么只能写 `partial`
- 什么情况下禁止写 `aligned`

这一步是把文档从“阅读资料”变成“执行合同”。

## 5. 写这份合同的时候，不能写成没牙齿的分析稿

这份文档如果只是把参考 mod 写得很详细，还是不够。

它必须具备一个能力：

- `抗合理化`

也就是 AI 一旦想偷换口径，文档里要有地方能直接把它卡住。

所以每个能力块都建议固定用同一个模板。

### 5.1 每个能力块固定回答这 6 个问题

1. 参考实现原来是什么
2. 当前项目打算落到哪里
3. 哪些意思绝对不能改
4. 哪些地方允许适配
5. AI 最容易犯的错翻译是什么
6. 当前做到哪一步只能算什么状态

### 5.2 一定要单独写“AI 最容易犯的错”

这项不能省。

因为 AI 太会把错误写成合理话了。

例如要直接写出来：

- 不能把 reference prompt 重写成“更适合本仓”的工程 prompt
- 不能先把 `actions[]` 压扁，再说仍然等价
- 不能把 UI 壳像，就写成主链已对齐
- 不能把 `partial` 写成“主体已完成”

### 5.3 一定要写死状态结论

不能让 AI 自己决定什么时候算“差不多完成”。

文档里应该直接写：

- 命中哪些情况，只能记 `partial`
- 命中哪些情况，禁止写 `aligned`
- 哪些情况必须记 `not implemented`

不然 AI 会自动补一句：

- 已基本对齐

然后后面所有人都被它带偏。

## 6. AI 真正开始干活时，脑子里必须守哪几条

这部分不是 review 规则，而是 AI 一动手就必须带着的规矩。

### 6.1 先复原，后映射

AI 先回答：

- 参考 mod 这条链原来怎么工作

再回答：

- 现在在当前仓库里怎么落

不能反过来。

一旦先从“本仓怎么实现最顺手”开始，基本就已经偏了。

### 6.2 先保“谁说了算、数据写到哪、先后顺序”，再管外形

真正优先保的是：

- authority owner
- canonical truth
- state transition
- writeback order
- prompt 主体和工程协议的边界

类名、接口名、UI 形态、文件结构都排在后面。

### 6.3 默认怀疑当前仓架构会把参考实现带歪

AI 不能默认：

- 把参考 mod 塞进当前仓架构自然就行

它必须默认：

- 当前这套 `Launcher / Runtime / Cloud / Mod / UI` 分层，会天然诱导我重组参考实现

所以每次抽象、下沉、上提，都要先怀疑自己是不是在改义。

### 6.4 适配只能发生在边界，不能发生在主语义

允许适配的通常是：

- 宿主绑定
- provider 接线
- title-local surface
- deterministic gate
- 服务端 / 客户端落点差异

不允许顺手改掉这些：

- source protocol
- prompt 主体语义
- authoritative owner
- canonical writeback
- reference 主状态机

### 6.5 实现顺序要沿着参考主链走

AI 不该按“哪个模块最好改”排顺序。

应该沿着参考主链走：

1. `Trigger`
2. `Snapshot / Context`
3. `Prompt / Summary Assembly`
4. `Parse / Repair`
5. `Projector / Executor`
6. `Authoritative Writeback`
7. `Visible Surface`

这样才不容易只补自己最熟的一层，然后把整条链误写成“已对齐”。

### 6.6 一旦发现不能照着来，先定性，再停下来汇报

AI 不允许一发现照着做不顺手，就直接改方案继续写。

必须先说清：

- 卡在哪一层
- 原参考实现是什么
- 现在为什么做不到
- 它打算改成什么

然后等你决定。

## 7. AI 写 proposal、design、review、closeout 时也必须管住嘴

很多漂移不是先出在代码里，而是先出在说法里。

AI 一旦先把话说歪，后面的实现几乎一定跟着歪。

### 7.1 不能把“按参考实现”说成“效果差不多”

不能把：

- 按参考 mod 实现

写成：

- 实现同等效果
- 能力等价
- 路线一致

这种说法会给内部实现留出大量自由发挥空间。

### 7.2 不能用抽象词把具体主链抹平

AI 很喜欢写：

- 候选生成
- 上下文组装
- 状态持久化
- 玩家交互闭环

这些词本身没错，但它们会把参考 mod 原本具体的主链句法洗平。

一旦只剩抽象词，AI 就更容易按自己的习惯重建一遍。

### 7.3 不能把 `partial` 说成“差不多好了”

这些话都要非常小心：

- 已基本对齐
- 主体已完成
- 仅剩少量细节
- 大方向已经一致

它们通常是在给 `partial` 涂脂抹粉。

### 7.4 不能把“本仓合理”说成“仍然参考一致”

AI 可以解释：

- 为什么当前仓库这样做更顺

但不能顺便推出：

- 所以这仍然算参考 mod 方案

这是两回事。

### 7.5 一旦偏离，必须直说，不准用软话藏掉

不能写：

- 做了工程化适配
- 做了必要调整
- 做了结构优化

而要直说：

- 改了哪一层
- 改了什么
- 为什么必须改
- 改完以后当前状态只能记什么

## 8. 偏离不是 AI 自己能拍板的事

这部分是整份文档里最硬的一条。

以后规则固定为：

- `AI 一旦判断需要偏离参考实现，必须先告诉你，再等你批准；没批准之前，不准继续按偏离方案往下做。`

这条规则的核心不是“让 AI 多解释几句”，而是：

- `偏离权不在 AI，在你。`

### 8.1 AI 一旦想偏离，必须先做这 4 步

#### 第一步：先停

别一边改，一边顺口解释。

先停下来。

#### 第二步：先报

必须明确汇报：

- 偏离的是哪一层
- 参考实现原本是什么
- 当前为什么做不到
- 它准备怎么改

#### 第三步：先定级

AI 必须自己先判断这属于哪一种：

1. `等价映射`
2. `title-local 适配`
3. `受控偏离`
4. `本仓自定义实现`

不能发明模糊新说法。

#### 第四步：等你裁决

只有你明确同意，AI 才能继续。

如果你没同意，默认不准自己切换方案。

### 8.2 这 4 种偏离怎么理解

#### `等价映射`

意思是：

- 落点变了
- 但主语义没变

比如参考 mod 直接在宿主里做的东西，现在拆成 `Runtime.Local + Cloud Control` 两段去做，但谁说了算、先后顺序、最后写回点没变。

这种情况一般仍然可以算 `aligned`。

#### `title-local 适配`

意思是：

- 因为当前标题和宿主表面不同，要做本地化适配
- 但参考主链还在

这种情况通常可以算：

- `aligned with title-local adaptation`

或者至少：

- `partial but faithful`

#### `受控偏离`

意思是：

- 因为当前工程约束，确实不得不和参考实现不一样
- 但这件事已经被明确承认，不再装成完全对齐

这种情况最重要的不是“能不能做”，而是：

- 做了以后通常不能再写 `aligned`

#### `本仓自定义实现`

意思是：

- 已经不是在映射参考方案
- 而是在借参考 mod 做一套更适合本仓的新方案

这类情况最危险，因为 AI 最爱把它包装成：

- 更统一
- 更通用
- 更合理

但本质上它已经不属于“参考实现映射”。

## 9. 以后 proposal、design、review、closeout 都要怎么跟着改

既然你主要是和 AI 一起开发，这份文档最后必须能直接影响四件事。

### 9.1 proposal 怎么写

proposal 里不能再只写：

- 尽量参考 mod

而要写清：

- 这次参考的是哪条主链
- 当前对照合同是什么
- 哪些已实现能力必须先收
- 哪些不在这轮里
- 本轮禁止 AI 自己解释偏离

### 9.2 design 怎么写

design 里必须先有：

- 参考实现主链复原

再有：

- 当前项目映射位

不能直接从当前仓架构往回解释参考 mod。

### 9.3 review 怎么看

review 不能只看：

- 方向对不对
- 能不能跑
- 看起来像不像

必须优先看：

- 参考锚点回没回
- 主链哪一层对上了
- authority 有没有换
- writeback 顺序有没有换
- prompt 主体是不是被改写了
- 当前状态是不是被夸大了

### 9.4 closeout 怎么写

closeout 一定要把结论写老实。

只能在真对齐时写：

- `aligned`

如果只是部分恢复了参考主链，就只能写：

- `partial`

如果根本已经换成本仓自定义方案，就不能继续写成参考 mod 对齐完成。

## 10. 最后的硬结论

以后处理 Stardew 这类 reference-first 任务，固定规则如下：

1. 不能只给 AI 一句“最大限度采用参考 mod”
2. 必须先整理出一份 `参考实现对照合同`
3. AI 实施前，先复原参考实现，再做本仓映射
4. AI 默认不能把“本仓更合理”当成“仍然参考一致”
5. AI 不能把 `partial` 说成“基本完成”
6. AI 一旦需要偏离，必须先停下来告诉你
7. 没有你的明确批准，AI 不得继续按偏离方案实施

一句最直白的话：

- 以后 AI 不准再“理解之后自己改写”
- 只能先“对照之后老实翻译”
- 真要改，先报给你

这才是减少 Stardew 参考 mod 漂移的唯一推荐路径。
