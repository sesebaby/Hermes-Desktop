# Superpowers 多游戏复用重构总设计

## 1. 文档定位

本文档用于把 `Superpowers` 从“当前围绕 Stardew 先长出来的一套实现”重收成：

- `可服务多游戏 Mod 的统一主线设计`

本文档回答的是：

- 哪些能力必须纳入完整方案，不能因为当前阶段没先做就丢掉
- 哪些东西应该做成共享层，哪些东西必须留在每个游戏自己那一层
- `Launcher / Launcher.Supervisor / Runtime.Local / Runtime.Adapter / Mod / Cloud` 到底怎么拆
- 为什么当前实现会长歪
- 以后第二个游戏接入时，到底只应该新写哪些东西

本文档不是：

- 当前阶段完成判断文档
- 某一个游戏的实现任务清单
- 旧实现兼容方案

当前明确采用：

- `允许对现有实现做大改`
- `不以向前兼容为优先`
- `一切以以后多游戏 Mod 接入和长期可复用为优先`

## 3. 当前问题判断

当前项目已经初步实现了：

- Cloud 端
- `Runtime.Local`
- `Runtime.Stardew`
- Stardew Mod

但当前实现不符合长期目标，核心问题不是“少几个功能”，而是“边界错了”。

当前主要问题判断如下：

1. 共享层、游戏层、桌面前台层、本地执行层的职责还不够干净。
2. 一部分本该属于“共享流程骨架”的内容，长在了 Stardew 具体实现里。
3. 一部分本该属于“游戏自己处理”的内容，又被吸进了 Runtime 或桌面侧。
4. 提示词资产、游戏语义、宿主写回边界如果继续混写，第二个游戏接入时大概率会重复当前问题。
5. 当前设计如果只围绕 `Stardew` 收口，而不把全量能力放进完整方案，后续群聊、传播、世界事件、对象生成等能力会再次断层。

因此本次设计的目标不是做一个 MVP 方案，而是：

- `先把完整方案定清，再按阶段实施`

## 4. 治理声明

### 4.1 当前阶段

- 当前 phase：`post-M1-reference-grade-hardening`
- 当前 workflow：`solo operator + manual verification only`

### 4.2 本次命中面

本次设计同时命中：

- `Cloud Control`
- `Launcher`
- `Launcher.Supervisor`
- `Runtime.Local`
- `Runtime.<game>` Adapter
- `Game Mod`
- `player-visible UI`
- `AI orchestration / prompt asset / memory / structured intent`

### 4.3 玩家首先看到的宿主

当前玩家首先看到结果或失败的宿主固定为：

1. `Launcher`
2. 某个游戏自己的宿主 UI

其中：

- 账号、下载、更新、启动、兑换、通知、求助，先在 `Launcher` 看见
- 对话、群聊、面板、物品文本、世界事件等游戏内结果，先在游戏宿主里看见

### 4.4 authority owner 与 truth source

本设计固定以下 owner：

1. `提示词资产 authority owner = Cloud`
2. `提示词编排 authority owner = Cloud`
3. `聊天正本 truth source = Cloud`
4. `记忆正本 truth source = Cloud`
5. `launchReadinessVerdict authority owner = Launcher.Supervisor`
6. `runtimeHealthFact authority owner = Runtime.Local`
7. `最终宿主执行 authority owner = Game Mod`
8. `游戏语义翻译 authority owner = Runtime.<game> Adapter`

### 4.5 单路径声明

本设计固定采用：

- `单一路径`

不允许：

- 云端一套、本地备用一套
- 旧私聊链和新私聊链长期并行
- Adapter 里一套、Mod 里再偷偷一套
- 某个能力先长一条临时旁路，后面再说并回主线

## 5. 设计目标

本设计要同时满足以下目标：

1. 方案必须完整覆盖参考 mod 的全量能力，不因为当前阶段先不做就漏位。
2. 实施可以分阶段，但分阶段不能改变完整方案的边界。
3. 提示词资产必须在云端，且按游戏隔离，不进入共享层。
4. 云端必须承担提示词编排、provider 通信、聊天正本、记忆正本、审计明文。
5. 本地必须保留 deterministic gate、宿主前置校验、执行证据、恢复入口。
6. 每个游戏未来接入时，只应该新写少量游戏专属块，而不是重做整条链。
7. 开发阶段允许大改，优先清掉错误边界，而不是保老接口。
8. 完整方案覆盖范围不自动改写当前 `phase boundary`、`claim matrix`、`listing state` 和 `ship gate`。

## 6. 非目标

本文档明确不追求以下目标：

1. 不追求对当前错误边界做温和过渡。
2. 不追求保留现有 Stardew 形状的合同当长期共享合同。
3. 不追求把所有能力都做成一个大而全共享内核。
4. 不追求让桌面前台承接后台运营术语和内部编排视图。
5. 不追求让 `Runtime.Local` 变成本地 AI 大脑或本地运营平台。

## 7. 总体分层

本设计推荐采用以下 7 层结构：

1. `Cloud`
2. `Launcher`
3. `Launcher.Supervisor`
4. `Runtime.Local`
5. `Runtime.<game> Adapter`
6. `Game Mod`
7. `Host Game`

### 7.1 Cloud

负责：

- 每个游戏自己的提示词资产
- 提示词编排
- 连玩家 provider 或服务器 provider
- 聊天正本
- 记忆正本
- 审计明文
- AI 结构化建议结果
- 成本归因正本
- entitlement / claim / listing / access 真相

不负责：

- 直接改宿主游戏
- 直接替代本地 deterministic gate
- 直接宣布宿主已经成功落地

### 7.2 Launcher

`Launcher` 是玩家真正看到和操作的桌面前台，不只是启动按钮壳子。

固定承担：

1. `账号面`
   - 注册
   - 登录
   - 账号状态
   - 退出登录
2. `游戏面`
   - 首页
   - 游戏列表
   - 游戏工作区
   - 当前状态
   - 启动 / 继续运行 / 更新 / 修复入口
3. `产品与兑换面`
   - 产品介绍
   - 外部购买跳转
   - Key 兑换
   - 我的权益
   - `BYOK` / `托管` 分开显示
4. `通知面`
   - 重要通知
   - 我的消息
   - 更新记录
   - 回执
5. `支持与帮助面`
   - 一键修复
   - 重新检查
   - 提交问题
   - 问题记录
   - 常见问题
6. `设置面`
   - 账号
   - 启动器
   - 游戏管理
   - 隐私与数据

固定不承担：

- 提示词真源
- 聊天正本
- 记忆正本
- provider / prompt / orchestration 控制面
- sellability / waiver / claim 真源

### 7.3 Launcher.Supervisor

`Launcher.Supervisor` 是本地桌面管家。

负责：

- 拉起 / 停止 `Runtime.Local`
- 启动前检查
- 汇总本地健康状态
- 生成唯一玩家可见的 `launchReadinessVerdict`
- 协调 Launcher、Runtime.Local、游戏运行状态
- 执行受控修复、更新、重启动作

### 7.4 Runtime.Local

`Runtime.Local` 是本地执行核，不是完整桌面产品前台。

负责：

- 收结构化事实包
- 发给 Cloud
- 收结构化结果
- 做统一校验、修复、归一
- 生成本地执行清单
- 管 trace / health / recovery / execution evidence
- 维护 `runtimeHealthFact`
- 维护 deterministic gate

不负责：

- 提示词真源
- 记忆正本
- 产品包前台
- 通知前台
- 支持页前台
- 直接改游戏

### 7.5 Runtime.<game> Adapter

每个游戏一个专属适配层。

负责：

- 把这个游戏的真实情况翻成统一事实包
- 把归一后结果翻成这个游戏自己的执行清单
- 处理游戏字段映射
- 处理频道语义差异
- 处理该游戏的动作落点对齐

不负责：

- 拼最终提示词
- 存正本
- 直接改游戏

### 7.6 Game Mod

负责：

- 从宿主游戏里取真实情况
- 显示玩家可见结果
- 最后真正改宿主
- 回传宿主执行结果和宿主证据

### 7.7 Host Game

宿主游戏本体是最终被读取、被展示、被写回的真实对象。

## 8. 桌面前台完整职责

### 8.1 Launcher 一级导航固定为

1. `首页`
2. `游戏`
3. `产品与兑换`
4. `通知`
5. `支持与帮助`
6. `设置`

### 8.2 Launcher 必须完成的 4 个闭环

1. `启动闭环`
   - 登录 -> 选游戏 -> 看状态 -> 启动 / 修复 / 更新
2. `兑换闭环`
   - 产品介绍 -> 外部购买 -> 输入 Key -> 权益生效可见
3. `通知闭环`
   - 平台发出 -> Launcher 接收 -> 玩家看见 -> 继续操作
4. `支持闭环`
   - 玩家提交问题 -> 平台处理 -> 回执返回 Launcher

### 8.3 功能包必须进入桌面和商业方案

桌面侧与产品侧必须明确承接以下产品包：

1. `试用包`
2. `基础包-BYOK`
3. `基础包-托管`
4. `高级包-绘画`
5. `高级包-视频`
6. `高级包-语音`

并且固定规则如下：

1. `基础包-BYOK` 和 `基础包-托管` 在玩家前台必须分开展示
2. 产品包卖什么、显示什么，不由 Launcher 自己推断
3. 产品包可见能力、listing 状态、entitlement 状态必须回链：
   - `narrative-base-pack-contract`
   - `capability-claim-matrix`
   - `sku-entitlement-claim-matrix`

## 9. 全量能力分层

本设计固定采用：

- `全量功能纳入方案`
- `分层放位`
- `分阶段实施`

但明确：

- `全量功能纳入设计，不等于全量功能纳入共用核心`

### 9.1 第一层：共用核心层

这层是跨游戏最值得先做稳的核心主线。

包括：

- 私聊
- 记忆压缩 / 长期记忆
- 结构化行为协议
- 交易 / 给物 / 承诺
- 基础道具使用反馈
- 容错 / 修复 / 归一
- 统一审计链路

### 9.2 第二层：共用扩展层

这层也是正式能力，但比核心层复杂。

包括：

- 群聊
- 联系人群 / 固定群聊
- 远程一对一通信
- 远程多人群聊
- 信息裂变 / 社会传播
- 主动对话 / 主动接触
- 群聊历史回流个人历史与记忆
- 更完整的动作白名单

### 9.3 第三层：先定正式接口层

这层现在就必须在方案里给正式位置，但不要求立刻长成成熟共享内核。

包括：

- 世界事件 / 主动世界演化
- 自定义物品生成
- 自定义状态生成
- 角色卡与 NPC 创建
- 工具入口
- 更重的对象生成和世界写回链

### 9.4 第四层：游戏专题层

这层也必须进入方案，但默认不进入第一版共享主线。

包括：

- 竞技场
- 战败对话与战败后处理
- 身体交互
- 双修场景
- 解签大师
- 特殊场景玩法
- 强依赖某个游戏规则的小玩法

## 11. 统一主线

所有能力，不管是私聊、群聊、传播、还是世界事件，最终都必须经过同一种大主线。只是不同能力会走不同分支。

### 11.1 主线步骤

1. `宿主触发`
2. `Mod 取真实情况`
3. `Runtime.<game> Adapter 整理结构化事实包`
4. `Runtime.Local 做统一入口检查`
5. `Runtime.Local 发给 Cloud`
6. `Cloud 做提示词编排并连 provider`
7. `Cloud 返回结构化结果`
8. `Runtime.Local 做统一校验、修复、归一`
9. `Runtime.<game> Adapter 翻成游戏执行清单`
10. `Mod 最后真正改游戏`
11. `执行结果回写`

### 11.2 统一 owner

1. `宿主触发 owner = Mod`
2. `事实整理 owner = Runtime.<game> Adapter`
3. `提示词编排 owner = Cloud`
4. `本地归一 owner = Runtime.Local`
5. `执行清单 owner = Runtime.<game> Adapter`
6. `最终宿主执行 owner = Mod`

### 11.3 单路径硬规则

1. 同一种能力任何时刻只能有一条正式实现主线。
2. Cloud 只负责“建议怎么做”，不负责直接改宿主。
3. 本地只有一个统一校验口。
4. 任何结果都必须能说清：还没做、已建议、已归一、已执行、执行失败。
5. 所有能力都必须挂到同一套 trace 和审计链上。

## 12. 共用对象与统一合同

共享层共用的是：

- `对象外形`
- `状态顺序`
- `校验规则`

不是共用每个游戏的提示词内容、宿主数据结构和 UI 细节。

### 12.1 7 个共用对象

1. `触发单`
   - 说明为什么这次开始处理
2. `事实包`
   - 说明当前游戏里真实发生了什么
3. `云端编排包`
   - Cloud 内部对象，只在 Cloud 里保留
4. `AI 结构化结果`
   - 模型返回的原始建议结果
5. `归一后结果`
   - Runtime.Local 校验后仍可继续的结果
6. `执行清单`
   - 这个游戏准备实际做什么
7. `执行回执`
   - 最后到底有没有真的做成

### 12.2 统一合同

必须统一至少 3 组合同：

1. `能力合同`
2. `动作白名单合同`
3. `失败分类合同`

### 12.3 固定状态变化

所有能力都必须经过这 4 次状态变化：

1. `AI 原始结果`
2. `本地归一结果`
3. `游戏执行清单`
4. `执行回执`

## 13. 生命周期、时序与并发规则

### 13.1 固定生命周期

每次能力处理固定经历：

1. `已触发`
2. `事实已冻结`
3. `云端已返回建议`
4. `本地已归一`
5. `执行清单已生成`
6. `宿主已执行`
7. `已提交`

### 13.2 状态红线

1. `云端返回` 不等于成功
2. `本地归一通过` 不等于成功
3. `执行了` 不等于玩家看见了
4. `有日志` 不等于有正本

### 13.3 并发与时序规则

1. 一轮只对应一份冻结后的事实包
2. 旧结果不能覆盖新状态
3. 同一个可见面一次只能认领当前有效请求
4. 传播和世界事件必须携带 hop / 轮次 / 来源编号
5. 执行失败必须有显式回执

## 14. 正本、明文与审计

本设计固定以下规则：

1. `聊天正本 = Cloud`
2. `记忆正本 = Cloud`
3. `提示词资产正本 = Cloud`
4. `提示词编排正本 = Cloud`
5. `审计明文 = Cloud`

并明确：

1. Cloud 保留提示词明文和记忆明文，用于审计和复盘。
2. 本地不保存完整提示词明文正本。
3. 本地不保存完整记忆明文正本。
4. 本地只保留：
   - 玩家可见文本
   - 执行证据
   - 回放投影
   - trace 证据
   - recovery 证据

## 15. 失败、退役与验证

### 15.1 固定失败分类

至少固定以下失败类别：

1. `云端没通`
2. `云端回来了但结果不合格`
3. `Adapter 翻执行清单失败`
4. `宿主执行失败`
5. `玩家可见提交失败`

### 15.2 退役规则

新主线一旦接管某个能力，旧入口必须同步退役。

每次接管必须写清：

1. 替代谁
2. 什么时候算接管完成
3. 什么时候删旧代码
4. 退役后还剩什么

### 15.3 验证分类

必须同时具备：

1. `结构验证`
2. `主线验证`
3. `玩家可见验证`

### 15.4 完成标准

一个能力只有同时满足以下 4 件事，才算真正接管完成：

1. 新主线跑通
2. 玩家真的看见或宿主真的落地
3. 证据已留
4. 旧入口已退役

## 16. 分阶段实施

### 16.1 第一阶段：新骨架立起来

必须完成：

1. 新 7 层结构落地
2. 新共用对象与统一合同落地
3. 第一层核心能力接到新主线
   - 私聊
   - 记忆
   - 交易 / 给物 / 承诺
4. 旧实现直接退役，不搞双轨

### 16.2 第二阶段：共用扩展层接进来

必须完成：

1. 群聊
2. 联系人群 / 固定群聊
3. 远程一对一
4. 远程多人群聊
5. 信息裂变 / 社会传播
6. 主动接触 / 主动对话

重点是统一：

- 线程和会话模型
- 多参与者顺序
- 历史镜像和回流
- 失败分类
- committed 条件

### 16.3 第三阶段：正式接口层接实

必须完成：

1. 世界事件 / 主动世界演化
2. 自定义物品生成
3. 自定义状态生成
4. 角色卡 / NPC 创建
5. 工具入口

重点是：

1. 输入输出合同先于实现稳定
2. 对象生成和世界写回保留强宿主边界
3. 云端建议与宿主真实落地彻底分开

### 16.4 长期扩展层：游戏专题能力

长期按游戏单独推进：

1. 竞技场
2. 战败链
3. 身体交互 / 双修
4. 解签大师
5. 特殊场景玩法
6. 明显强依赖某个游戏规则的专题能力

### 16.5 阶段红线

1. 每进一阶段，都要同步写清退役谁。
2. 第二阶段和第三阶段虽然分后做，但第一阶段就必须在合同和目录上留正式位置。
3. 阶段顺序只服务于实施，不改变完整方案覆盖范围。

## 17. 未来新游戏接入的最小新增面

本设计固定未来新游戏接入时，尽量只新增以下 4 类内容：

1. 这个游戏自己的提示词资产
2. 这个游戏怎么取真实情况
3. 这个游戏怎么显示给玩家
4. 这个游戏怎么真正改宿主

尽量不重写：

1. Cloud 编排主线
2. 请求 / 响应外形
3. 本地校验和修复
4. 记忆正本
5. 审计链路
6. 桌面前台主结构

## 18. 代码结构重切建议

本设计建议后续实现按以下原则重切代码结构：

1. `Superpowers.Runtime.Contracts`
   - 放真正共享的共用对象与统一合同
   - 不继续放强 `Stardew` 痕迹的长期共享对象
2. `Superpowers.CloudControl`
   - 放提示词资产、编排、provider、审计、商业与控制面
3. `Superpowers.Launcher`
   - 放玩家桌面前台
4. `Superpowers.Launcher.Supervisor`
   - 放 readiness、runtime 监督、修复、更新、启动编排
5. `Superpowers.Runtime.Local`
   - 放本地执行核、统一校验、trace、health、recovery
6. `Superpowers.Runtime.<game>`
   - 放游戏专属 Adapter
7. `games/<game>/...Mod`
   - 放宿主取事实、宿主显示、最终写回

并明确：

- 当前开发阶段允许大改
- 不要求保留现有错误边界的兼容层
- 不要求现有 Stardew 风格合同继续存在

## 19. 最终结论

本设计的最终结论如下：

1. `Superpowers` 必须采用完整方案设计，不允许因为当前阶段先不实现某些能力就把它们排除在方案外。
2. 提示词资产必须留在 Cloud，且按游戏隔离；提示词编排与 provider 通信都在 Cloud。
3. 聊天正本、记忆正本、审计明文必须留在 Cloud，本地不保留完整正本。
4. 桌面侧必须拆成 `Launcher + Launcher.Supervisor + Runtime.Local`，不能再把桌面前台和本地执行核混成一层。
5. 本地最终执行权固定留给 `Game Mod`，`Runtime.Local` 和 `Runtime.<game> Adapter` 都不直接改游戏。
6. 所有能力都必须经过统一主线：`触发 -> 事实包 -> 云端建议 -> 本地归一 -> 执行清单 -> 执行回执`。
7. 参考 mod 的全量能力都必须进入方案，但按 4 层放位，而不是全塞进共享核心。
8. 开发阶段允许对现有实现大改；目标不是保护旧实现，而是为以后多游戏 Mod 真正铺路。
