# Superpowers 参考 Mod 能力映射附件

## 1. 文档定位

本文把 `GGBH_OpenAIWorld` 已恢复的能力，逐项映射到 `Superpowers` 新总设计。

本文回答的是：

1. 参考 mod 的每项能力在 `Superpowers` 里归哪一层
2. 每项能力到底落到哪几个层去承担
3. 哪个宿主先看到结果
4. 失败先暴露到哪
5. 当前 phase 先落哪批，以后阶段按什么顺序接
6. 第一阶段究竟按参考 mod 的哪个代码单元来做
7. 第一阶段到底是 `A / B / C / D` 哪一种实现结论
8. 后面是否允许迁到 AFW

固定回链：

- `docs/superpowers/specs/attachments/2026-04-07-superpowers-reference-code-unit-reconstruction-and-afw-migration-appendix.md`

## 2. 映射总原则

固定原则：

1. 参考 mod 的全量能力都要入表，不漏项。
2. 入表不等于全部进入共用核心。
3. `Cloud / Launcher / Launcher.Supervisor / Runtime.Local / Runtime.<game> Adapter / Game Mod` 都必须写清职责。
4. 第一阶段如果没有写清参考文档、参考 prompt、绑定代码单元、实现结论，就不算设计完成。

## 4. 社交与世界两大主链补充规则

### 4.1 社交主链

固定结论：

1. 私聊和远程一对一共用同一条人物关系聊天/记忆主骨架。
2. 群聊有独立群历史，但关键内容要回流个人后续判断，且必须带来源标记。
3. 动态选项属于正式结构化输出。
4. 交易 / 给物 / 承诺 / 接受 / 拒绝 属于社交主链里的正式动作协议。
5. 信息传播走 `ConveyMessage` 这类独立传播协议，不是普通文本效果。
6. 传播必须先经过通信载体，不允许直接改第三方脑子。

### 4.2 世界主链

固定结论：

1. 世界事件是主骨架。
2. 造物 / 造状态 / 造 NPC / 地标变化默认挂在世界事件上下文里。
3. 不同游戏支持力度不同，因此必须走：
   - 共享世界语义
   - title support matrix
   - 本地硬拦截
4. 社交/行为链可以提议创建 NPC，但正式创建权统一收口到世界主链。

## 5. 对未来新游戏的直接要求

以后新游戏接入时，默认只应该新增：

1. 该游戏的提示词资产
2. 该游戏的事实采集
3. 该游戏的宿主 UI 承载
4. 该游戏的最终宿主写回
5. 该游戏的 title support matrix
6. 该游戏对 `U10` 的实现

如果某个能力接入时需要重写：

- Cloud 编排主线
- Launcher 主结构
- Runtime.Local 统一 gate

默认视为边界又漂了。
