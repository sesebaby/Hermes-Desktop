# Superpowers 世界事件、对象生成与工具入口附件

## 1. 文档定位

本文细化第三阶段需要接实的正式接口层：

1. 世界事件
2. 自定义物品生成
3. 自定义状态生成
4. 角色卡 / NPC 创建
5. 工具入口

固定回链：

- `docs/superpowers/specs/attachments/2026-04-07-superpowers-reference-code-unit-reconstruction-and-afw-migration-appendix.md`

## 2. 总原则

这些能力必须现在就定接口，不然后面一定断层。  
但它们不等于今天就和私聊一样成熟。

固定原则：

1. 先把门、路、输入输出、owner 定死。
2. 再按阶段接入实现。
3. 对象生成和世界写回始终保留强宿主边界。
4. 默认先按参考 mod 的世界主链做。
5. 不同游戏支持力度不同，必须走 title support matrix。

## 2.1 本附件绑定代码单元

本附件固定绑定：

- `U2 AI 服务方案 / 会话编排 / 输入组装`
- `U8 世界推演 / 世界事件创建语义`
- `U9 自定义物品 / 状态 / NPC 生成语义`
- `U10 宿主 hook / UI 注入 / 最终写回`

第一阶段固定实现结论：

- `U2 = B`
- `U8 = B`
- `U9 = B`
- `U10 = C`

固定规则：

1. `U8 / U9` 第一阶段不允许 AFW 进入生产主链。
2. 世界事件是主骨架。
3. 造物 / 造状态 / 造 NPC / 地标变化默认挂在世界事件上下文里。
4. 不同游戏是否支持这些能力，由 title support matrix 决定。
5. `Adapter + Runtime.Local` 必须按支持矩阵做硬拦截并留明文审计。

## 3. 世界事件

固定规则：

1. 世界推演触发权在宿主，不在 Cloud。
2. Cloud 负责事件建议、事件候选/正式正本、审计明文。
3. `Runtime.Local` 负责事件 schema 校验、fail-closed、trace。
4. `Runtime.<game> Adapter` 负责：
   - 把事件建议翻成该游戏事件对象
   - 标出该游戏哪些字段必填
   - 按支持矩阵决定哪些字段和对象允许落
5. `Mod` 负责：
   - 最终在宿主创建或推进事件
   - 回传事件执行回执

首次可见宿主：

- 游戏内事件宿主

## 4. 自定义物品 / 状态生成

固定规则：

1. 自定义物品 / 状态默认属于世界主链里长出来的结构化结果。
2. Cloud 给结构化生成建议。
3. `Runtime.Local` 拦格式不完整、超白名单、超支持矩阵的建议。
4. `Runtime.<game> Adapter` 把建议翻成该游戏对象模型。
5. `Mod` 最终创建实例并回传证据。

固定补充：

1. 不支持的 title 不能硬落。
2. 被挡掉、被裁剪、被降级的部分必须进入 Cloud 审计明文。

## 5. 角色卡 / NPC 创建

固定规则：

1. 角色卡资产和创建建议在 Cloud。
2. 社交/行为链可以提出创建角色候选。
3. 正式创建权统一收口到世界主链。
4. `Runtime.Local` 必须对创建动作走审批门和 fail-closed。
5. `Runtime.<game> Adapter` 把创建建议翻成该游戏的角色创建命令。
6. `Mod` 负责最终创建 NPC 和可见面接入。

这是当前方案批准的 `controlled deviation`：

- 参考 mod 有双入口味道
- 当前方案统一收口，避免 authority 分裂

## 6. 工具入口

固定规则：

1. 工具入口不是第三条平行主链。
2. 工具是社交主链和世界主链都可能调用的受控工具层。
3. 工具注册边界在 Cloud。
4. 第一阶段不允许 AFW 进入工具生产主链。
5. 结构化输出必须走 schema 或 typed 校验。
6. 高风险工具必须有审批闭环。
7. 最终宿主动作仍要落到 `Runtime.Local gate -> Adapter -> Mod` 这条链上。

## 7. review 必查点

1. 是否把世界事件 / 生成能力提前写成成熟共享内核
2. 是否让 Cloud 越权直接改宿主
3. 是否让 Runtime.Local 直接懂具体宿主对象模型
4. 是否把角色创建和工具入口绕过了统一 gate
5. 是否把世界推演触发权错误上收到了 Cloud
6. 是否为不支持的 title 硬落了对象生成
