# All Game In AI 人物关系主状态专项设计

> Historical design note as of 2026-03-27.
>
> This document remains useful for capability semantics, but not for current orchestration placement or service/local ownership.
> If anything here appears to conflict with the active superpowers framework or governance docs, treat this file as historical context only.

## 1. 文档定位

本文档用于补齐当前 `superpowers` 设计中对“人物关系”定位与承载方式不够明确的问题。

它不替代以下现有设计：

- `M1 / M2 / M3` 分阶段重构设计
- 主技术栈与授权架构设计
- 跨游戏类型扩展性专项设计

本文档要明确两件事：

1. 对任何想做出“真实人物世界”的游戏，`人物关系` 都是必要基础能力，而不是可选 flavor
2. 平台最终不能只依赖各游戏原生关系信息做高层消费，而应维护自己的 `social relationship table`，作为后续关系动态发展的主状态基础

这里的“人物关系”同时包括：

- `玩家-NPC`
- `NPC-NPC`

本文档的目标不是要求 `M1` 立刻做成完整社会模拟，而是把：

- `M1` 最少要建立什么
- `M1` 暂时不统一什么
- `M2` 才统一什么
- `M3` 才增强什么

写成清晰的阶段边界。

## 2. 问题定义

当前文档已经出现了以下信号：

- `relationshipFacet`
- 好感度 / 关系值 / 亲密度映射
- 基于关系触发的 NPC 送礼
- 多主体 identity 与 typed refs

这些内容说明现有设计已经意识到“关系”很重要。

但当前设计仍存在四个不足：

1. 关系更多被写成一个 `facet`、一个触发条件或一种文案语境，而没有被明确写成“真实性的必要基础能力”
2. 文档更偏向 `玩家-NPC` 关系，对 `NPC-NPC` 关系的长期价值写得不够清楚
3. 文档还没有明确平台是否要维护自己的持续关系主状态
4. 文档还没有明确区分：
   - `M1` 先建立主关系表，但允许大量 `game-local`
   - `M2` 才开始统一关系 schema
   - `M3` 才进入更强关系驱动逻辑

如果这层判断不写清，后续设计容易在三个错误之间来回摆动：

- 把关系降成 flavor，只影响几句台词
- 永久依赖各游戏原生关系表示，导致后续无法推动动态发展
- 过早要求 `M1` 完整统一 schema 和完整社会模拟

## 3. 核心结论

### 3.1 总原则

对任何想做出“真实人物世界”的游戏，`人物关系` 都应被视为必要基础能力。

原因很简单：

- 真实的人物世界，不是若干孤立角色分别对玩家说话
- 真实感来自“谁认识谁、谁更亲近谁、谁偏向谁、谁疏远谁、谁替谁说话、谁与谁有旧事”
- 如果系统只能生成单次对话，却不能持续表达人物之间的关系语义，真实性会很快见顶

因此平台不应只把关系理解为：

- 玩家好感度
- 单次触发条件
- 一段可有可无的 flavor 文案

而应把关系理解为：人物互动、记忆、语境、事件触发与后续世界行为的重要基础状态之一。

### 3.2 平台必须拥有自己的关系主状态

如果后续目标是让关系能动态发展，并持续影响互动与世界变化，那么平台最终不能只停留在“读取各游戏原生关系信息”。

平台必须维护自己的 `social relationship table`。

这张表不是单纯缓存原生数据，而是：

- 后续关系动态发展的主状态基础
- 高层 prompt / event / dialogue / memory / gift 等能力的持续关系来源
- 跨回合、跨事件、跨上下文累积关系语义的主要承载层

这也意味着：

- 原生关系信息仍然重要
- 但它们更像初始化输入、世界事实输入、作者设定输入、同步参考输入
- 平台不能把“关系真相”永远外包给各游戏原生系统

### 3.3 范围定义

本设计中的“人物关系”至少覆盖：

- `玩家-NPC`
- `NPC-NPC`

并且不应只限于“亲密度数值”，还应允许承载更广的关系语义，例如：

- 熟悉 / 陌生
- 亲近 / 疏远
- 偏爱 / 反感
- 帮助 / 亏欠
- 依附 / 跟随
- 站队 / 倾向
- 旧事 / 历史背景 / 静态社会设定

本文档当前不要求一次性把这些全部统一 schema 化，但要求从平台方向上承认它们属于同一能力领域。

## 4. 分阶段设计

### 4.1 M1：先建立主关系表，不先强统一 schema

`M1` 必须引入平台自有的 `social relationship table`，作为人物关系持续状态的主承载层。

但 `M1` 的正确目标不是：

- 完整社会模拟
- 完整统一关系 schema
- 全游戏统一关系词典
- 全游戏统一关系强度刻度

`M1` 的正确目标是：

- 先把关系主状态建起来
- 先让这张表能承载 `玩家-NPC` 与 `NPC-NPC`
- 先让高层能力能稳定消费关系上下文
- 先为后续动态发展留出主状态基础

因此 `M1` 的关键判断应是：

- `M1` 要有主关系表
- `M1` 不要求强统一关系 schema
- `M1` 允许大量 `game-local`

### 4.2 M1 的数据来源

`M1` 的关系主状态至少来自四类输入：

1. `原生显式关系`
   例如游戏内已有的好感度、婚姻、亲属、仇恨、立场、显式社会关系字段

2. `原生静态设定`
   例如作者写死的人物背景、家庭关系、固定村庄关系、门派或社会背景

3. `游戏内事件`
   例如送礼、争执、帮助、共同经历、任务结果、剧情事件

4. `受控 AI 推断`
   AI 可以补充关系解释和趋势判断，但在 `M1` 不应无约束地产生硬状态

这四类输入共同进入平台的 `social relationship table`，形成后续关系演化的主状态基础。

### 4.3 M1 的三款首发承载方式

三款首发游戏在 `M1` 不应被要求以同一种原生方式提供关系数据。

更准确的承载方式应为：

- `环世界`
  优先导入游戏已有的动态社会关系信息，再叠加事件更新

- `鬼谷八荒`
  优先导入游戏已有的人际/关系相关信息，再叠加事件更新

- `星露谷物语`
  优先导入原生友谊、婚恋、家庭以及作者写死的人物关系背景，再叠加事件更新

也就是说，三款首发游戏在 `M1` 的共同点不是“原生关系结构完全相同”，而是：

- 最终都汇入平台自有的 `social relationship table`
- 都成为后续动态关系发展的起点

### 4.4 M1 的变化边界

`M1` 的关系表不是静态缓存，但也不应变成重型社会模拟系统。

`M1` 允许的变化来源应收敛为：

- 显式事件驱动变化
- 对 `玩家-NPC` 关系读取和吸收原生好感度等现成数值
- 对 `NPC-NPC` 关系做少量、可解释的事件更新

`M1` 暂不作为普遍硬要求的内容包括：

- 普遍性的时间衰减 / 维持模型
- 大规模自动 NPC 社会传播
- 复杂三角关系连锁演化
- 阵营政治自动重组
- 高密度全图 NPC 自主演化

因此 `M1` 的关系演化原则应写成：

- 以显式事件驱动为主
- `玩家-NPC` 先重点接原生数值
- `NPC-NPC` 先做轻量状态和轻量更新
- 不以全局社会模拟为目标

### 4.5 M1 的 schema 策略

`M1` 必须统一的是“有一张表”和“有持续状态”，而不是立刻统一所有关系语义。

因此 `M1` 明确允许：

- `relationshipKind` 先保持大量 `game-local`
- `relationshipStrength` 先保留各游戏原本表示
- 每个游戏通过自己的 adapter / translator 写入关系表
- 高层能力先消费这些 `game-local` 关系上下文

`M1` 明确不要求：

- 统一关系类型词典
- 统一关系强度数值区间
- 强制三款首发先对齐同一关系 schema

### 4.6 M2：开始统一关系 schema

`M2` 才是关系 schema 开始统一的阶段。

只有当多游戏中反复出现稳定复用的关系语义后，才应逐步推进：

- 共享 `relationshipKind` 分类
- 共享关系强度映射
- 更明确的 shared relationship semantics
- 更稳定的跨游戏关系查询与行为消费方式

因此 `M2` 的重点是：

- 不是从零开始建关系表
- 而是在已有主关系表基础上，把反复复用的关系语义收敛成统一 schema

### 4.7 M3：关系成为更强世界逻辑

只有在多个游戏都证明需要更强关系能力后，`M3` 才应进一步把关系沉淀为更重的平台能力。

在这一阶段，关系才有资格进一步参与：

- 更强的世界行为驱动
- 更复杂的阵营 / 组织 / 社会网络表达
- 更持续的行动倾向、合作冲突、资源与事件流向

因此可以把三阶段总结为：

- `M1`：先建立主关系表，允许 `game-local`
- `M2`：开始统一关系 schema
- `M3`：关系成为更强的世界逻辑之一

## 5. M1 关系表最小结构建议

`M1` 的 `social relationship table` 应至少支持“谁对谁有什么关系，以及它从哪里来、最近为何变化”。

建议最小字段包括：

- `sourceActorId`
- `targetActorId`
- `relationshipKind`
- `relationshipStrength`
- `relationshipValence`
- `confidence`
- `sourceOfTruth`
- `lastUpdatedAt`
- `lastUpdateReason`
- `gameLocalPayload`

其中：

- `relationshipKind` 在 `M1` 允许保持大量 `game-local`
- `relationshipStrength` 在 `M1` 允许保留各游戏原本表示
- `gameLocalPayload` 用于承载暂未统一的关系细节

`M1` 先不要求：

- 全平台统一关系词典
- 全平台统一强度区间
- 复杂网络传播参数
- 高级社会模拟参数

## 6. 对现有文档的 Redline 结论

### 6.1 对主技术栈设计的补强

主技术栈设计中已存在 `relationshipFacet`，但建议额外明确：

- `relationshipFacet` 的存在并不只是为了容纳“关系值字段”
- 它对应的是“真实人物世界”的基础语义面之一
- 平台最终需要自己的持续关系主状态，而不只是读取游戏字段

还应明确：

- 关系相关语义不应永久停留在各游戏私有上下文中
- 但 `M1` 也不要求马上完全统一
- `M1` 先建主关系表，`M2` 再统一 schema

### 6.2 对 M1 / M2 / M3 分阶段设计的补强

当前分阶段文档已经明确了：

- 好感度等触发源
- 关系触发送礼
- `relationshipFacet` 作为少数保留 facet

但仍建议补写以下判断：

- `M1` 的关系目标不只是“关系可被玩家感知”，还包括建立平台自有的持续关系主状态
- `M1` 的关系主状态允许大量 `game-local` 表达，不要求立即统一 schema
- `M1` 中 `玩家-NPC` 先重点接原生好感度等现成数据，`NPC-NPC` 先重点接原生关系和静态设定，再叠加少量事件更新
- `M2` 才逐步把人物关系推进成统一的结构化 schema
- `M3` 才进一步把关系推进成更强的世界驱动逻辑

### 6.3 对跨类型扩展设计的补强

跨类型扩展设计已经补了：

- `Identity Graph`
- typed `subjectRefs[]`
- `ownerId` / `controllerId`
- `relations`

但这些仍更偏向“多主体建模能力”，还不是对“平台必须拥有自己的关系主状态”的正面声明。

因此建议补写：

- 多主体字段不只服务复杂事件表达
- 它们还应服务于平台关系主状态的建立与后续演进
- `NPC-NPC` 关系不应被视为次要附属场景，而应被视为真实人物世界的重要组成部分

## 7. M1 验收口径建议

为了避免“写了原则但无法验收”，建议为 `M1` 增加以下口径：

1. 平台已存在可持久化的 `social relationship table`
2. 三款首发游戏都能把自己的原生关系数据或原生关系设定导入这张表
3. 高层 prompt / event / dialogue / gift / memory 至少已有一部分能消费这张表中的关系上下文
4. `玩家-NPC` 关系至少已能稳定承载原生好感度或同类原生关系信息
5. `NPC-NPC` 关系至少已能承载原生关系、作者设定或轻量事件更新
6. `M1` 不要求三款首发先统一关系类型词典和统一强度刻度

## 8. 推荐回写语句

以下句子适合回写到现有 spec：

### 8.1 总原则表述

- 对任何想做出“真实人物世界”的游戏，`人物关系` 都是必要基础能力，而不是可选 flavor。
- 这里的人物关系同时包括 `玩家-NPC` 与 `NPC-NPC`，不能只将其理解为玩家好感度。

### 8.2 M1 表述

- `M1` 必须建立平台自有的 `social relationship table`，作为 `玩家-NPC` 与 `NPC-NPC` 关系持续状态的主承载层。
- `M1` 允许关系类型、关系强度和部分关系语义保持大量 `game-local` 表达，不要求立即统一 schema。
- `M1` 中 `玩家-NPC` 先重点接原生好感度等现成关系信息，`NPC-NPC` 先重点接原生关系和静态设定，再叠加少量事件更新。

### 8.3 M2 表述

- `M2` 才开始把多游戏中稳定复用的关系语义收敛为统一 schema。

### 8.4 M3 表述

- `M3` 才进一步把关系推进成更强的世界逻辑与持续行为驱动能力。

## 9. 最终结论

当前 `superpowers` 文档并不是完全没有写到“关系”。

真正的问题在于：

- 已经写到了“关系存在”
- 但还没有足够明确地写出“关系为什么是一等能力”
- 也还没有足够明确地写出平台是否要维护自己的关系主状态
- 还没有足够清楚地写出 `M1`、`M2`、`M3` 在关系承载方式上的阶段边界

因此本专项设计给出的收敛结论是：

- 必须把“人物关系是必要基础能力”写成总原则
- 必须把平台自有的 `social relationship table` 写成长期方向
- 必须把 `M1` 定义为“先建立主关系表，但允许大量 `game-local`”
- 必须把 `M2` 定义为“开始统一关系 schema”
- 必须把 `M3` 定义为“关系成为更强的世界逻辑之一”

可压缩成一句最核心的话：

- `M1` 先把关系主状态建起来
- `M2` 再把反复复用的关系语义统一起来
- `M3` 再让关系更强地塑造人物互动与世界行为
