# Superpowers 可复用程序设计

## 1. 文档定位

本文定义 `Superpowers` 的共享叙事中间层应该怎么设计。

它回答的是：

- 参考 `GGBH_OpenAIWorld` 已经证明了哪些可复用事实
- `Superpowers` 第一版应该如何把这些事实整理成共享设计
- 首版能力应该如何分层
- 共享对象、宿主边界、实施约束和审核要求应该如何落到文档里

本文不是：

- `GGBH_OpenAIWorld` 恢复事实本体
- 当前 phase 的完成判断文档
- 未来平台化蓝图总文

当前读取顺序固定为：

1. `docs/superpowers/governance/current-phase-boundary.md`
2. `docs/superpowers/specs/2026-03-27-superpowers-master-design.md`
3. 本文
4. 对应 contract / profile / evidence / threat-model

本文采用两类口径，必须明确区分：

- `参考事实`
  - 指 `recovered_mod/GGBH_OpenAIWorld_20260326/分析文档` 已经证实的内容
- `设计归纳`
  - 指 `Superpowers` 为了统一实现、验证和审核而做的第一版共享设计

硬规则：

- 不允许把 `设计归纳` 写成 `参考事实`
- 不允许拿未来平台化设想改写当前 source-faithful 主链

## 2. 参考事实与设计归纳边界

### 2.1 已由参考 mod 和分析文档证实的事实

当前可以稳定承接的事实主要有：

- 存在统一的 prompt 资产层，而不是若干散落文案
- prompt 资产支持本地覆盖、导出、变量注入
- 角色卡存在 `公开段 / 私密段 / 角色定义段` 这样的分段结构
- 存在服务配置、方案路由、JSON 能力探测、thinking 参数注入
- 会话层真实支持 `messages / tools / customData`
- 工具调用机制存在，但具体业务使用范围还没有全部拆清
- 消息层不是一次性字符串，而是有统一消息主记录、JSON sidecar、群相关消息结构
- 记忆层存在“原始消息 + 摘要压缩 + 时间桶”这类可复用思路
- 行为层存在“AI 先给结构化结果，再由宿主白名单执行”的边界
- `JsonRepair`、解析、修复、归一这类容错能力真实存在
- 群聊、固定联系人群、远距离通信、信息传播、世界事件、角色卡与 NPC 创建等功能面真实存在

但同样必须承认：

- 不同功能面的证据强度并不一样
- 有些链路已经有强源码支持
- 有些链路仍然主要依赖交叉材料或旧稿补强
- 因此不能把所有功能面都写成“已经恢复到同样强度的运行时事实”

### 2.2 本文作为 `Superpowers` 第一版的设计归纳

本文允许做以下设计归纳：

- 用统一的“触发 -> 原始事实 -> 上下文 -> AI 结构化结果 -> 检查修复 -> 执行计划 -> 宿主执行结果”来描述共享主线
- 用统一对象名和接口名帮助实现、测试和审核
- 定义共享层的最小动作族、最小关系边界、最小对象流
- 把首版能力分成不同优先级和不同里程碑

但这些都是：

- `Superpowers` 的设计选择
- 不是说参考 mod 原本就已经以这些对象名和这些层级存在

### 2.3 本文不做的事

本文不做以下断言：

- 不宣称参考 mod 已经原样长成 `Kernel*` 这一套对象
- 不宣称参考 mod 已经严格证明“只有这 6 层、且只能这 6 层”
- 不宣称所有世界事件、传播链、角色创建链都已经达到同样强度的源码确定性
- 不宣称当前 phase 已经允许把所有功能面拉成同级实现任务

## 3. 首版完整能力清单与分层

本节先列出首版完整功能面，再明确哪些现在先做稳，哪些必须后续做出来，哪些现在先把边界定清。

### 3.1 当前主线必须先做稳的功能

这一层决定当前主线能不能站住。

- 玩家和 NPC 的一对一私聊
- 对话时给玩家生成动态可选回复
- 把旧聊天和旧经历压缩成记忆摘要，并在后续对话里继续使用
- 给东西
- 要东西
- 发起交易
- 接受交易
- 拒绝交易
- 做出承诺
- 撤回承诺
- 关系的最小变化，例如好感升降、敌意升降

一句话总结：

`先把“能聊、能记住、能交易、能承诺、关系会有最小变化”这条主线做稳。`

### 3.2 首版必须做出来，但按后续里程碑并入同一主线的功能

这一层不是未来 backlog，而是首版总体方案里必须真正做出来的能力。

- 多人群聊
- 群聊里由系统决定谁先说、谁后说
- 固定联系人群和固定群后端
- 远距离的一对一通信
- 远距离的多人群聊
- 信息传播，也就是一个人知道的事继续传给别人
- 群聊内容回流到个人聊天和个人记忆里
- 角色主动接触玩家、主动触发交流
- 角色卡在运行时真正进入上下文
- 受限制的自定义物品扩展位
- 更完整的一层核心动作白名单

这一层的意思是：

- 这些能力首版必须实现
- 但不等于当前 hardening 阶段的同级完成门槛
- 它们必须进入明确里程碑
- 只能逐步并入同一条正式主线

### 3.3 现在先定义正式接口与边界，后续再并入主线的功能

这一层现在先把位置和边界定清，后续再接入同一主线。

- 世界事件
- 创建新 NPC 的正式入口
- 自定义状态生成的正式入口
- 工具调用的正式入口

这一层的意思是：

- 先把门、路、边界、接法定清楚
- 但当前不把它们抬成首版主线的大块实现任务
- 后续进入明确里程碑后，再并入同一条正式主线

### 3.4 不进入首版共享主线，只保留为各游戏自己处理的专题能力

这些功能面在参考 mod 中真实存在，但当前不应该上拉成共享首版主线。

- 竞技场相关玩法
- 身体互动、双修这类强题材玩法
- 战败对话、战败按钮这类特殊分支
- 黑市、特殊奇遇、特殊场景
- 解签、算命这类特殊玩法
- 任何强依赖某个具体游戏规则的小玩法

### 3.5 能力分层硬规则

- 第二层和第三层都不是可无限延期的 future backlog
- 第二层必须进入明确里程碑
- 第三层只有在进入明确里程碑后，才允许从接口定义转为实现任务
- 第二层和第三层都只能并入同一条正式主线
- 不允许因为后续能力的接入，反过来打乱第一层当前主线的收口顺序

## 4. 共享对象与接口设计

### 4.1 主线信息流，用白话讲清楚

共享设计里的信息流按下面这条主线向前走：

1. `触发`
   - 为什么这次 AI 流程会开始
   - 例如玩家发起私聊、系统触发群聊、社交行为需要继续判断
2. `宿主原始事实`
   - 从游戏里读取当前真实情况
   - 包括人物、地点、背包、关系、最近消息、已有事件
3. `可送给 AI 的上下文`
   - 把原始事实、历史消息、记忆摘要、角色卡信息整理成一次可发送输入
4. `AI 返回的结构化结果`
   - AI 只能返回结构化建议，不能直接改游戏
5. `检查、修复、归一`
   - 先检查格式
   - 再修复
   - 再统一字段和枚举
   - 只有合法内容才能继续往后走
6. `执行计划`
   - 把合法结果整理成一份“准备怎么落地”的计划
   - 这里还不是直接改游戏
7. `宿主执行结果`
   - 最终能不能改、改了什么、哪些被拒绝，由宿主说了算

### 4.2 共享对象命名规则

本文允许定义统一的共享对象名，例如：

- 触发记录
- 宿主快照
- 上下文封套
- AI 意图结果
- 归一后的合法结果
- 执行计划
- 宿主执行结果

但必须明确：

- 这些名字是 `Superpowers` 的共享设计对象
- 它们的作用是统一语义、统一验证、统一审核
- 不是说参考 mod 原本就已经按这些对象名存在

### 4.3 共享动作边界

第一版共享动作层可以冻结一组核心动作族，用于统一语义和统一审核：

- 给东西
- 要东西
- 发起交易
- 接受交易
- 拒绝交易
- 发出邀约
- 做出承诺
- 撤回承诺
- 传播消息
- 关系的最小变化

但以下能力目前不进入第一版共享核心动作层，只保留正式位置：

- 创建新 NPC
- 新增状态
- 删除状态

原因很简单：

- 这些能力确实存在
- 但当前还不适合写成“第一版共享核心动作已经稳定冻结”

### 4.4 共享关系边界

第一版允许定义最小共享关系视图，但必须把它写成 `Superpowers` 的设计选择，而不是恢复事实本体。

第一版关系边界建议只保留：

- 谁对谁
- 当前关系标签
- 好感数值
- 敌意数值
- 最近更新时间

第一版 AI 的关系改动边界建议写成：

- 只允许调整关系数值
- 不允许直接改正式关系标签

这样做的原因是：

- 可以先统一最小共享语义
- 又不把参考 mod 没有强证到位的细节硬写成既定事实

### 4.5 宿主需要承担的 4 类职责

共享层不能代替宿主。宿主至少要承担下面 4 类职责：

1. `读取游戏真实状态的入口`
   - 负责把人物、地点、背包、关系、消息、事件读出来
2. `声明当前游戏允许做哪些动作的入口`
   - 负责告诉共享层这个游戏允许哪些动作、参数怎么校验
3. `把执行计划落到当前游戏里的入口`
   - 负责最终验证并执行合法动作
4. `记忆落点入口`
   - 负责把消息、摘要、长期记忆写回当前游戏的存储里

### 4.6 统一资产、路由、角色卡、工具的设计边界

本文允许把以下内容写成共享设计的一部分：

- 统一资产层
- 本地覆盖规则
- 变量注入规则
- 角色卡进入上下文的方式
- 服务方案与能力探测

同时也允许定义下面这些共享设计对象：

- 路由配置对象
- 资产绑定对象
- 角色卡绑定对象
- 工具能力配置对象

但必须明确：

- 它们是 `Superpowers` 为统一实现和审核而定义的对象
- 不是说参考 mod 已经天然按这套对象定型

### 4.7 迁移与退役原则

共享对象的引入只能服务于统一主线，不能制造历史债务。

因此必须同时满足：

- 新对象不能和旧实现长期并行运行
- 新对象一旦开始正式接管某段职责，就必须定义旧实现的退役方案
- 不允许“先都留着，以后再清”
- 不允许“为了兼容先长期双保留”

## 5. 单主线实施约束与文档证据要求

### 5.1 任何时候只允许一条正式实现主线

这条是硬规则：

- 不允许旧做法和新做法长期一起跑
- 不允许先做一条旁路，后面再说并回主线
- 不允许同一个能力同时维护两套真正可运行的实现

一句白话总结：

`只能有一条正式做法。`

### 5.2 开发阶段不允许降级凑合跑

开发阶段只允许两种结果：

- 成功
- 失败并直接阻断

开发阶段不允许：

- 失败后先退成一个“看起来还能用”的版本
- 用“先能跑起来再说”掩盖主线问题
- 把降级策略当成开发期默认设计

一句白话总结：

`开发时不准糊弄，错了就停。`

### 5.3 不允许留下历史债务

只要新功能已经开始接管旧职责，就必须同步推动旧实现退役。

每次接管都必须写清：

- 被替代的是谁
- 什么时候退
- 什么条件算退干净
- 最后删哪些代码、文档、配置

不允许：

- 先留着旧实现“以防万一”
- 先兼容着以后再清
- 新旧长期并存

一句白话总结：

`新功能进来，旧功能就要准备出去。`

### 5.4 每项实现都必须留下仓库内可审核的文档证据

没有文档记录，就不算完成。

每份文档证据至少要回答 4 件事：

1. 做了什么
2. 替代了什么
3. 怎么验证的
4. 还剩什么限制

日志、截图、测试结果可以作为辅证，但主证据必须是 repo-local 正式文档。

一句白话总结：

`做了不写清楚，等于没做。`

### 5.5 后续能力只能按里程碑并入同一主线

对于第二层和第三层能力：

- 必须做
- 但必须按里程碑逐步并入同一条正式主线
- 不允许先在别处长一套临时实现

### 5.6 世界事件现在先定接口，不抢主线

世界事件当前的正确定位是：

- 先把位置、输入输出、边界、接法定清楚
- 后续到了明确里程碑，再接入主线
- 接入时也不能另外长出一套系统

### 5.7 当前 phase 的最终优先级

当前 `post-M1-reference-grade-hardening` 阶段下：

- 第一层能力仍然是当前主线收口优先级最高的内容
- 第二层能力必须进入首版里程碑，不允许长期停留在占位符
- 第三层能力现在先把接口定清，后续再接入
- 第四层能力继续保留在 title-local 专题，不进入首版共享主线

## 6. 最终结论

`Superpowers` 的可复用程序设计，第一版应被理解为：

- 一个承接参考 mod 已证事实的共享叙事中间层设计
- 一套用来统一语义、统一验证、统一审核的共享对象与接口设计
- 一份把首版完整能力清单按优先级和里程碑拆清的实施约束文档

同时必须长期坚持以下 4 条硬规则：

1. 只能有一条正式实现主线
2. 开发阶段不允许降级凑合跑
3. 不允许留下历史债务
4. 没有 repo-local 文档证据就不算完成
